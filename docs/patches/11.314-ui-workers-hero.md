# 11.314 -- Workers page hero (v1.11.332)

New `web/src/pages/Workers.tsx` feature page with a
glance-able hero header. Component-scope only, no
daemon-side change and no `c4` CLI surface change.

## Why

The c4 web UI currently surfaces worker state in the
sidebar `WorkerList` -- a vertical scroll of every
worker the daemon knows about. The list is the
operator's primary navigation tool, but it has two
gaps:

1. No "at a glance" pane. An operator who just opened
   the dashboard needs to count rows manually to know
   how many workers are busy vs idle. The numbers
   matter for capacity decisions and incident
   triage, and the sidebar doesn't surface them.

2. No primary spawn affordance. Spawning a worker is
   one of the most common operator actions, but it's
   currently driven from the `+` button on the AppHeader
   menu -- two clicks away from the dashboard's
   primary view.

Patch 11.314 ships a feature page that fixes both: a
hero card with live counts + last-hour trend + a
prominent spawn CTA.

## What changed

### `web/src/pages/Workers.tsx` (new file)

Hero card surfaces:

- Three count blocks (Busy / Idle / Lost) rendered with
  `BadgeCounter` (v1.11.296). Tones:
  - Busy: `accent` (primary palette).
  - Idle: `muted` (muted-foreground palette).
  - Lost: `danger` (destructive palette).
- Sparkline (`Sparkline` primitive, v1.11.235) showing
  the trend of total worker count over the last 12
  polling buckets. The buffer is persisted to
  `localStorage['c4:workers:hero-sparkline']` so the
  trend survives page reloads inside the same browser
  session.
- Primary "Spawn worker" Button. The Button primitive's
  `loading` prop (v1.11.326) handles the pending state
  -- spinner replaces the icon, button auto-disables,
  `aria-busy="true"` is emitted. Spinner size adapts
  to the Button's `md` size.
- Toast feedback via `ToastProvider` (v1.11.298) +
  `useToast`. Success: "Spawn request sent. Track
  progress in the worker list." Failure: "Spawn
  failed: <reason>" where `<reason>` is the error
  message bubbled up from the rejected promise.

### State

- Counts derived from `GET /api/list` polled every
  5 seconds (matches the sidebar's polling cadence).
  Busy/idle classification:
  - Busy when `state` (or `status` fallback) is
    `busy` / `running` / `dispatching`.
  - Idle otherwise.
  - Lost comes from the response's top-level
    `lost` array (the `LOST (daemon restart)`
    section emitted by `c4 list`).
- Sparkline buffer: each successful counts update
  pushes the `total` onto a rolling buffer capped at
  `SPARKLINE_BUCKETS = 12`. The buffer is persisted
  to localStorage so a page reload does not blank
  the trend. Malformed storage (non-JSON,
  non-array, non-number entries) is filtered to an
  empty list silently.

### Spawn handler

- Default: `POST /api/workers` with body
  `{ "name": "auto-<seconds>-<base36>" }`. Reads
  the response body as the error message on
  non-2xx. The default name shape is intentionally
  conservative (no tier, no target, no profile)
  -- a richer spawn modal is a follow-up.
- Override: `<Workers onSpawnRequest={fn} />` lets
  tests / specialised mounts intercept the
  request without mocking `fetch` at the page
  boundary. The vitest suite uses this override
  for every spawn-related case.

### Registration

- `web/src/pages/registry.ts` gains a new
  `workers-hero` entry under the `operations`
  category with the `LayoutDashboard` icon.
- i18n entries added under
  `feature.workersHero.label = "Workers"` and
  `feature.workersHero.description = "At-a-glance
  worker dashboard ..."` in both `en.json` and
  `ko.json` (lockstep preserved; ko mirrors en
  text pending a translation).

### Data attributes

- `data-section="workers-hero"` on the body
  container.
- `data-section="workers-hero-trend"` on the
  sparkline container.
- `data-section="workers-hero-count-block"` +
  `data-tone` on each count block.
- `data-testid` hooks: `workers-hero-card`,
  `workers-hero-count-busy`, `workers-hero-count-idle`,
  `workers-hero-count-lost`, `workers-hero-spawn-cta`,
  `workers-hero-sparkline`.

## Tests

`web/src/pages/Workers.test.tsx` -- 11 vitest cases:

- Hero shell + title + description.
- `data-section="workers-hero"` selector.
- Three count blocks render.
- Counts derived from `/api/list` (busy/idle/lost
  split).
- Spawn CTA renders.
- CTA click fires the spawn callback.
- Success toast on a successful spawn.
- Error toast (with the rejection message) on a
  failed spawn.
- Sparkline renders with an accessible label.
- Count blocks carry `data-tone` for theming.
- Malformed `/api/list` response keeps counts at 0
  (no crash).

`fetch` is stubbed via `vi.stubGlobal` so the
network flow is deterministic. The `onSpawnRequest`
prop is the seam used by the spawn-related cases.

All 11 green via

## Notes

- The hero page does NOT duplicate the sidebar's
  worker list. The two surfaces are intentional:
  the sidebar is the primary navigation tool, and
  the hero is the "operator dashboard" feature
  surface. An operator looking for a specific
  worker still uses the sidebar; the hero is for
  capacity / health overview.
- The Sparkline trend starts blank on first mount
  (the buffer fills as polling cycles complete).
  The component pads the data to >= 2 samples so
  the sparkline line renders smoothly during the
  first few seconds.
- The `ToastProvider` is scoped to the hero card
  rather than the page root. This keeps the
  toasts visually anchored to the surface the
  operator is interacting with and prevents
  collision with other pages that mount their own
  ToastProvider.
- The default spawn handler is intentionally
  minimal -- a follow-up patch can replace the
  Button with a "Spawn worker" trigger that opens
  a modal with the full spawn-options form
  (tier, target, profile, branch, --auto-mode,
  etc).
- localStorage is the right persistence layer for
  the sparkline buffer because the trend is
  per-browser-session and doesn't need to survive
  a daemon restart. A future patch could move the
  buffer to a daemon-side timeseries endpoint if
  the cross-browser view becomes valuable.
