# patch 11.194 -- UI code-split audit (v1.11.212)

## Goal

Audit the React.lazy code-split boundaries in the web bundle, confirm
the heavy feature pages each emit their own chunk, and pin the
registry contract with regression tests.

Scope is strict: component-level audit only. No bundle analyzer
runs, no Vite config changes, no eager-import conversions when the
registry already uses dynamic imports.

## Audit findings

### `web/src/pages/registry.ts`

Already uses the `load: () => import('./<Name>')` zero-arg dynamic
import thunk for every entry in `FEATURES`. 21 pages total, grouped
across five categories (operations / automation / cost / config /
diagnostics). The heavy candidate set from the task spec maps onto
the registry as:

| heavy candidate | in registry | pattern |
|-----------------|:-----------:|---------|
| Health          | yes         | `load: () => import('./Health')` |
| Plan            | yes         | `load: () => import('./Plan')` |
| Templates       | yes         | `load: () => import('./Templates')` |
| Profiles        | yes         | `load: () => import('./Profiles')` |
| Risk            | yes         | `load: () => import('./Risk')` |
| TokenUsage      | yes         | `load: () => import('./TokenUsage')` |
| Workers         | n/a         | rendered as `WorkerDetail` from `App.tsx` (already `lazy()`) |
| Sessions        | n/a         | rendered as `SessionsView` from `App.tsx` (already `lazy()`) |

Workers and Sessions are not registry features -- they sit on the
top-level view switcher in `App.tsx`, which already wraps them in
`React.lazy()` (see below).

### `web/src/components/layout/FeatureView.tsx`

Invokes the registry loader via `lazy(feat.load)` and renders the
result inside a local `<Suspense>` boundary that falls back to a
`<Skeleton>` panel matching the page frame. The lazy wrapper is
recomputed inside `useMemo` keyed on `feat.id` so navigating between
features does not re-trigger a chunk reload for the active feature.

### `web/src/App.tsx`

Top-level views are all lazy-wrapped at module scope:

```ts
const WorkerDetail   = lazy(() => import('./components/WorkerDetail'));
const ChatView       = lazy(() => import('./components/ChatView'));
const ControlPanel   = lazy(() => import('./components/ControlPanel'));
const HistoryView    = lazy(() => import('./components/HistoryView'));
const SessionsView   = lazy(() => import('./components/SessionsView'));
const MeetingsView   = lazy(() => import('./components/MeetingsView'));
const SpecialistsView= lazy(() => import('./components/SpecialistsView'));
const WikiView       = lazy(() => import('./components/WikiView'));
const AutonomousView = lazy(() => import('./components/AutonomousView'));
const Chat           = lazy(() => import('./components/Chat'));
const WorkflowEditor = lazy(() => import('./components/WorkflowEditor'));
const SettingsView   = lazy(() => import('./components/SettingsView'));
```

The `topView` switcher is wrapped in a single `<Suspense>` with a
`LoadingSkeleton` fallback (`rows={6}`), so any lazy chunk that is
not yet loaded suspends to the skeleton instead of unmounting the
shell. `Login`, `AppHeader`, `Sidebar`, `MetricsBar`, and
`AnnounceRegion` remain eager because they are part of the initial
paint (`v1.10.509` commit message in the file documents the same
rationale).

## Before / after

Before: registry contract was already correct, but only `auto`,
`risk`, `queue`, `token-usage` had a dedicated lazy-shape test
(`page-lazy.test.tsx` from v1.11.102). A regression that replaces
`load: () => import('./X')` with an eager `import X from './X'` +
`load: async () => ({ default: X })` would still pass the previous
test (still a function, still resolves to a default-exporting
module) but would silently fold the heavy chunk back into the main
bundle.

After: new `web/src/pages/registry.test.ts` adds five cases:

1. Each heavy candidate (`health`, `plan`, `templates`, `profiles`,
   `risk`, `token-usage`) exposes a `load` function with arity 0.
2. Full-registry sweep -- every entry in `FEATURES` passes the same
   shape check, catching the case where a new page is added with an
   eager loader.
3. `load()` returns a `Promise` resolving to a module that has a
   function `default` -- end-to-end smoke against `Health`.
4. `React.lazy(feat.load)` returns an exotic component tagged with
   the `Symbol.for('react.lazy')` `$$typeof` sentinel for every
   heavy candidate.
5. `featuresByCategory()` continues to surface every heavy id
   through a category bucket, so the audit assumptions remain
   reachable from the sidebar.

## Test note

The autonomous worktree at `/root/c4-worktree-auto-w178` does not
have its own `node_modules` installed. The task spec explicitly
allows skipping the vitest run in that case. Running

```
env -C /root/c4-worktree-auto-w178/web /root/c4/web/node_modules/.bin/vitest run src/pages/registry.test.ts
```

against the shared `/root/c4/web/node_modules` fails at the vite
config load step because `vitest/config` and `@vitejs/plugin-react`
resolve relative to the worktree, not the install root. The tests
will pick up automatically once the manager merges the branch and
the install root sees the new file.
