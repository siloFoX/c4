# 11.221 -- Card primitive surface refresh (tone variants)

Component-scope-only addition. No daemon-side change and no `c4` CLI
surface change. Card primitive (`web/src/components/ui/card.tsx`)
gains a `tone` prop with subdued tint + matching border per ARPS
token. Default tone is byte-identical to the prior surface so every
existing consumer keeps its current rendering.

## Why

The Card primitive only carried one surface look (`bg-card` +
`border-border`). Status-keyed containers built on Card had to layer
their own ad-hoc tint classes on top via `className`, which fragmented
the surface language across consumers (each page picked its own
`bg-destructive/8` vs `bg-destructive/12` mix). Patch 11.221
centralises the three status surfaces inside the primitive so all
future status-keyed containers share the same tint depth and border
weight.

## What changed

### `web/src/components/ui/card.tsx`

- New exported type `CardTone = 'default' | 'success' | 'warning' | 'danger'`.
- New `tone` prop on `CardProps`. When omitted or `'default'`, the
  Card keeps its prior `bg-card` + `border-border` classes byte-for-
  byte (preserving snapshot stability).
- New tone branches:
  - `tone='success'` -> `bg-success/5 border-success/30`
  - `tone='warning'` -> `bg-warning/5 border-warning/30`
  - `tone='danger'`  -> `bg-destructive/5 border-destructive/30`
- When a non-default tone is set the surface drops the implicit
  `bg-card` + `border-border` (otherwise Tailwind would still emit
  the default `bg-card` and the tint would lose). The `border` class
  itself is kept so the `border-<tone>/30` colour modifier still
  paints a visible edge.
- Tone composes cleanly with the v1.11.143 `interactive` prop:
  hover-lift, active-press, and focus-visible ring classes are still
  appended in the same order, so a `tone='warning' interactive` Card
  retains its keyboard activation contract.
- Children render path (`CardHeader`, `CardTitle`, `CardDescription`,
  `CardContent`, `CardFooter`) is untouched.

### `web/src/components/ui/card.test.tsx`

Seven new vitest cases under the `<Card tone>` block:

1. Omitted tone keeps `bg-card` + `border-border`.
2. `tone='default'` renders the same as omitted.
3. `tone='success'` renders `bg-success/5` + `border-success/30`.
4. `tone='warning'` renders `bg-warning/5` + `border-warning/30`.
5. `tone='danger'` renders `bg-destructive/5` + `border-destructive/30`.
6. Caller `className` still passes through alongside a tone.
7. `tone` composes with `interactive` (cursor + focus-ring intact).

All pre-existing cases in `card.test.tsx` and
`card.snapshot.test.tsx` remain green; default-tone Cards render
byte-identically to the v1.11.238 output.

### `web/src/pages/DesignSystem.tsx`

`CardPanelDemo` now renders one example per tone (success / warning
/ danger) directly below the default Card so the design system page
documents the new surface language alongside the existing primitive.

## In-tree adoption note

The task brief asked for three status-aware adoption sites in
addition to the primitive itself. After grepping every `<Card>` site
in `web/src/`, the three obvious candidates were:

- `Login.tsx` (auth error)
- `WorkerDetail.tsx` (terminal error banner)
- `AutonomousView.tsx` (dispatch / escalation errors)

In each case the error is rendered as an inline `<div role="alert">`
inside an otherwise persistent Card shell, not as a status-keyed
container. Swapping the parent Card's tone would tint the whole
shell red even when no error is present, which is the wrong
semantic. The `<Alert>` primitive remains the right surface for
those transient inline banners.

The new Card tones are therefore registered for future consumers
that own a genuinely status-keyed Card -- e.g. a future "halted
worker summary" card or "release blockers" card -- and the
DesignSystem showcase documents the surface so those consumers have
a single reference to copy from.
