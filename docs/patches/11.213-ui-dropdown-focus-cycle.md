# 11.213 -- UI: dropdown-menu focus-cycle adoption

Released as `web` v1.11.231 on 2026-05-15.

## Goal

Adopt the `useFocusCycle` hook (shipped in 11.205 / v1.11.223) inside
`DropdownMenu` so the hand-rolled ArrowUp / ArrowDown / Home / End /
wrap logic is replaced by the shared a11y primitive. Component scope
only -- no public API change, no daemon-side touch, no CLI surface.

## Hook wiring

```ts
const menuRef = useRef<HTMLDivElement | null>(null);

const handleFocusChange = useCallback((el: HTMLElement) => {
  const idx = itemsRef.current.indexOf(el as HTMLButtonElement);
  if (idx >= 0) setHighlight(idx);
}, []);

const focusCycle = useFocusCycle({
  containerRef: menuRef,
  itemSelector: '[role=menuitem]:not([aria-disabled=true])',
  orientation: 'vertical',
  wrap: true,
  onSelect: handleFocusChange,
});
```

The `onSelect` callback bridges the hook back into the existing
`highlight` state, which is what drives `aria-activedescendant` on
the menu container and the `bg-accent` visual highlight class on
each row.

## Keydown chain

The document-level keydown listener is retained because each menu
item is the actual focus owner (not the menu container) -- if the
listener were on the menu container, the first ArrowDown press
right after the menu opens would not fire (the trigger button is
the active element at that moment). Order inside the listener:

1. `Escape` -- close + restore focus to trigger.
2. Single-letter type-ahead (printable, no modifiers, not Space) --
   runs BEFORE the hook so that letter keys never reach the hook
   (the hook would no-op on them anyway, but the explicit ordering
   makes the contract obvious from reading the source).
3. `focusCycle.handleKeyDown(e)` -- handles arrows + Home / End,
   wraps at edges, skips items whose visibility filter drops them
   (native `disabled`, `aria-disabled=true`, `aria-hidden=true`,
   `hidden`).

The cast `e as unknown as KeyboardEvent<HTMLElement>` is safe because
the hook only touches `key`, `preventDefault`, and `stopPropagation`,
all of which exist with identical semantics on native KeyboardEvent.

## Why context-menu.tsx is NOT migrated

`context-menu.tsx` was inspected. The shape is similar but three
properties make a drop-in swap incompatible with the hook:

1. **Pointer-positioned portal.** The menu is rendered into
   `document.body` via `createPortal` at the cursor location.
   Container ref still works for the hook's `querySelectorAll`, but
   the next two points are the blockers.
2. **Focus trap, not roving focus.** `useFocusTrap` keeps focus on
   the menu container; each item has `tabIndex={-1}` and is never
   the active element. The hook's "find the active item, advance to
   the next one" model relies on items being focusable. Switching
   models would require ripping out `useFocusTrap` -- larger than the
   "share primitives" goal of this patch.
3. **Pre-set highlight on open.** The existing test
   `'ArrowDown advances highlight, ArrowUp moves it back'` asserts
   that the very first ArrowDown advances from index 0 to index 1.
   The hook (driven by `document.activeElement` = the menu container,
   not item 0) would instead focus item 0 on first ArrowDown,
   breaking the assertion. Honouring it would mean tracking a
   "virtual current index" outside the hook, defeating the point of
   the hook.

The `disabled` semantics also diverge: `context-menu.tsx` uses
`aria-disabled` (items are still in the tab list and the highlight
filter), while `dropdown-menu.tsx` uses native `disabled`. The hook's
visibility filter handles both, but that is a side note.

Conclusion: `context-menu.tsx` keeps its existing inline logic. If
we later refactor it away from the focus-trap model (e.g. to give
items roving focus), that is a separate patch.

be caught here as well as in `use-focus-cycle.test.tsx`.
