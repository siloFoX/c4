# 11.1081 -- UX FIX: chart-line showcase gallery route (v1.11.1099)

## Scope

1154 `chart-line-*` primitives live under
`web/src/components/ui/` but App.tsx imported none of
them, so they were dead code with no way for an operator
to see them. This adds a `gallery` top-view route backed
by a new `ChartLineGallery` component that lazy-loads a
curated, representative slice of those primitives and
renders them in a responsive grid with a search filter.

## Files

- `web/src/components/ChartLineGallery.tsx` -- new
  gallery component (lazy registry + universal sample +
  responsive grid + search + per-tile error boundary).
- `web/src/components/layout/TopTabs.tsx` -- added
  `'gallery'` to the `TopView` union and a TABS entry
  (LineChart icon, `tab.gallery` label).
- `web/src/lib/preferences.ts` -- added `'gallery'` to
  `TOP_VIEW_VALUES` so the route persists across reloads.
- `web/src/App.tsx` -- lazy-imported `ChartLineGallery`,
  added the `topView === 'gallery'` branch, and a
  `go.gallery` command-palette entry.
- `web/src/i18n/en.json` / `ko.json` -- `tab.gallery`
  label ("Gallery" / "갤러리").
- `web/e2e/showcase-gallery.spec.ts` -- new Playwright
  spec (visual verification).
- `package.json` / `web/package.json` /
  `package-lock.json` / `web/package-lock.json` ->
  1.11.1099.

## Design

- **Lazy loading.** Each registry entry is a
  `React.lazy(() => import('./ui/chart-line-...').then(
  m => ({ default: m.ChartLineX })))`. The chart chunks
  stay out of the main bundle and only fetch when the
  gallery is visited, so wiring the gallery in does not
  regrow the initial paint. Named exports are mapped to
  `default` to satisfy the `lazy` contract.
- **Universal sample.** Every chart-line primitive
  validates and ignores fields it does not read, so a
  SINGLE sample point carrying every field
  (`high`/`low`/`close`/`value`/`volume`) feeds all of
  them regardless of input shape -- no per-chart shape
  metadata is needed. The series is a sine + drift wave
  of 96 bars, long enough to clear the longest warmup
  (Ichimoku senkouB = 52) so crosses / divergences
  actually draw.
- **Per-tile error boundary.** Each tile is wrapped in a
  `UIErrorBoundary` so one misbehaving primitive degrades
  to an inline "Chart failed" card instead of blanking
  the grid.
- **Responsive grid + search.** `grid grid-cols-1
  sm:grid-cols-2 xl:grid-cols-3 gap-4`; a case-insensitive
  search filter (`data-section="chart-gallery-search"`)
  narrows by id or label via a deferred value.
- **Registry size.** 41 verified primitives are wired in
  (export names confirmed against components/ui). The
  registry is one-line-per-chart, so extending coverage
  toward the full 1154 is trivial follow-up work.

## Navigation wiring

- New `gallery` tab in the top tab bar (LineChart icon,
  between Features and Settings).
- `Cmd/Ctrl+K` -> "Go to Gallery" command (keywords:
  charts, chart-line, showcase, primitives).
- `gallery` is a persisted top-view, so a reload restores
  it. `getTopViewLoader('gallery')` returns undefined
  (the gallery ships its own lazy chunk via App.tsx's
  `React.lazy`), so the TopTabs hover-prefetch is a
  harmless no-op for this route.

## Visual verification (mandatory)

`web/e2e/showcase-gallery.spec.ts` drives the running dev
server (localhost:5173) headless:

1. **renders >= 12 chart tiles with drawn SVGs** -- boots
   directly into the gallery (init script seeds
   `c4.topView='gallery'` and `c4.onboardingTour.v1='seen'`
   so no modal occludes the capture), waits for >= 12
   `chart-gallery-tile` containers AND >= 12 drawn
   `<svg>` elements inside the grid, then screenshots
   into `test-results/showcase-gallery.png`.
2. **search filter narrows the grid** -- types "ichimoku"
   and asserts the visible tile count drops below the
   full count (and >= 1 survives).

Auth: a single `/api/**` route handler fulfills
`/api/auth/status` with `{ enabled: false }` (so
`useAuthState` resolves to `disabled` and the dashboard
renders without a login round-trip), serves empty
`/api/list` + `/api/events`, and aborts every other
`/api` call (a network error, not an HTTP 401, so the
AUTH_EVENT that would flip back to Login never fires).

Both tests pass headless (chromium):

The runs take ~50s because the first dev visit transforms
the gallery chunk plus ~40 chart chunks on demand; the
per-test timeout is raised to 90s to accommodate. A
production build pre-bundles these, so this latency is a
dev-only artifact. The capture confirms a 3-column grid
of live charts (DI cross, SAR flip, momentum/ROC/awesome
divergence, CMO, CCI overbought/oversold, ...) drawing
the shared sample wave with cross/divergence markers.

## Risk

- Additive feature: a new route + component + nav entry.
  No existing view's behaviour changes. The only edits to
  shared files are a union member, a TABS row, a
  TOP_VIEW_VALUES entry, a lazy import, a topView branch,
  a palette command, and two i18n keys.
- The gallery is curated (41 of 1154). It is not an
  exhaustive index -- a comprehensive auto-discovery
  registry would need per-chart sample-shape metadata or
  a build-time manifest. The universal-sample + one-line-
  per-entry design makes incremental expansion cheap.
- Dev-mode first-visit latency (~50s to transform ~40
  lazy chunks) is a dev-server artifact, not a production
  concern. The e2e timeout accommodates it.
- Verified against a byte-identical staging copy in
  `/root/c4/web` because the worker worktree's
  `node_modules` is a symlink that blocks Node's ESM
  config resolution for vite / playwright when run from
  the worktree path; the spec runs cleanly from any
  checkout whose `node_modules` is a real directory.
