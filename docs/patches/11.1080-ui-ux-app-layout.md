# 11.1080 -- UX FIX: App.tsx centered layout container (v1.11.1098)

## Scope

Fix the "whole app crowds to the top-left on wide
viewports" bug. `web/src/App.tsx` rendered the page
content flush against the left edge of `<main>` with no
max-width cap or horizontal centering, so on a 1440px+
display the content hugged the left and left a large dead
zone on the right.

The fix wraps the page-content region (the lazy-loaded
view switcher) in a centered responsive container:

```
mx-auto flex min-h-0 w-full max-w-6xl flex-1 flex-col px-4
```

- `max-w-6xl` (1152px) caps the column on wide viewports.
- `mx-auto` centers the capped column -> symmetric side
  margins (~144px each at 1440px).
- `w-full` + no min-width means narrow / mobile viewports
  use the full width (the cap never engages below
  1152px), so mobile is unaffected.
- `px-4` gives edge breathing room on narrow viewports.
- `flex min-h-0 flex-1 flex-col` preserves the existing
  per-view overflow / scroll discipline (each topView
  branch still manages its own `overflow-hidden` +
  `min-h-0` chain).

The full-width chrome strips above the content
(MetricsBar, AutonomousStatusBanner, RouteProgressBar,
GridDebugOverlay) stay outside the container by design --
only the page content is centered, so status bars still
span the viewport.

## Visual verification (mandatory)

`web/e2e/layout-app-shell.spec.ts` drives the running dev
server (localhost:5173) headless and proves the centering:

1. **1440px viewport** -- navigates to `/`, screenshots
   into `test-results/layout-app-shell-1440.png`, and
   asserts the `app-main-container` bounding box:
   - `box.x > 120` (left edge sits ~144px inside the
     viewport, proving the column is centered, not
     flush-left).
   - `box.width <= 1200` (the column is actually capped
     at max-w-6xl = 1152px, so the centering is real and
     not just full-width).
2. **375px mobile viewport** -- asserts the column hugs
   the viewport (`box.x < 40`, only the px-4 inset) and
   fills it (`box.width > 330`), proving mobile uses full
   width.

Auth handling: the dev daemon has auth ENABLED, so the
spec installs a single `/api/**` route handler that
fulfills `/api/auth/status` with `{ enabled: false }`
(so `useAuthState` resolves to `disabled` and the
dashboard renders without a login round-trip), serves
empty `/api/list` + `/api/events` payloads, and ABORTS
every other `/api` call. Aborting yields a network error
(not an HTTP 401), so the AUTH_EVENT that would flip the
UI back to the Login surface never fires. A single
conditional handler avoids route-precedence ambiguity --
`/api/auth/status` is always fulfilled (an earlier
catch-all + separate-override approach aborted
auth/status and left the app stuck on the Loading
screen).

Both tests pass headless (chromium):

```
2 passed (22.2s)
```

The 1440px capture confirms the worker sidebar + content
column start ~144px from the left edge (header chrome
remains full-width), with symmetric right margin.

## Risk

- Pure layout / Tailwind change plus a new e2e spec. No
  runtime logic touched; the flex chain that drives
  per-view scrolling is preserved.
- `max-w-6xl` (1152px) caps content width on very wide
  displays. For data-dense views (worker terminal,
  multi-pane editors) this trades some horizontal real
  estate for centered readability -- the deliberate UX
  decision behind this dispatch. If a future view needs
  full bleed it can opt out by rendering above the
  container, the same way the chrome strips do.
- The e2e spec depends on a running dev server on
  localhost:5173 (the project's e2e convention -- no
  `webServer` block in playwright.config.ts; the operator
  runs vite separately). It stubs all network so it does
  not depend on a live daemon.
- Verified against a byte-identical staging copy because
  the worker worktree's `node_modules` is a symlink that
  blocks Node's ESM config resolution for vite /
  playwright when run from the worktree path; the spec
  runs cleanly from any checkout whose `node_modules` is
  a real directory (CI, `/root/c4/web`, a fresh clone).
