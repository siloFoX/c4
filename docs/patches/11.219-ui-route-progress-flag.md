# 11.219 -- UI: route-progress feature flag

Status: shipped in v1.11.237 (2026-05-15)
Scope: `web/` only. No daemon-side change, no `c4` CLI surface change.

## Why

Patch 11.198 introduced a component-scoped feature flag registry
(`web/src/lib/feature-flags.ts`) with a `routeProgress` key that
defaulted to `true` but had no consumer wired up yet. The
`RouteProgressBar` component (top-of-page trickle bar driven by
route key changes) was the obvious first consumer: it is always
mounted at the layout root, is purely cosmetic, and is the kind of
thing that QA / accessibility reviewers want a hard kill switch
for. Wiring this flag closes the loop opened by 11.198 and
establishes the canonical "consume a flag inside a component"
pattern for the rest of the registry keys to follow (`motion`,
`pageTransitions`, etc.).

## What

`web/src/components/RouteProgressBar.tsx` now calls
`useFeatureFlag('routeProgress')` from inside the component body.
The flag value is checked at the JSX boundary, not before the
other hooks, so every hook in the component runs unconditionally
on every render -- that keeps the React rules-of-hooks happy when
the flag flips. When the flag is `false` the component returns
`null` and renders nothing; when the flag is `true` (the default)
the bar renders exactly as before.

```tsx
const [enabled] = useFeatureFlag('routeProgress');
// ... all other hooks (useReducedMotion, useState, useRef,
// useCallback, useImperativeHandle, useEffect) run unconditionally
if (!enabled) return null;
return <div data-testid="route-progress-bar" ... />;
```

Two non-obvious decisions:

1. The hook lives inside the component, never at module scope.
   `RouteProgressBar` is imported once at the layout root, so a
   module-scope flag read would freeze the value to the first
   render and bypass the `feature-flag-changed` event bus.
2. The early `return null` is placed AFTER every other hook
   call, not before them. React's hook order must be stable
   across renders even when the visible output is `null`, so
   short-circuiting before `useEffect` / `useImperativeHandle`
   would crash with "Rendered fewer hooks than expected" on the
   render where the flag flips back to `true`.

## Behaviour parity

- Default install: `routeProgress` resolves to `true` via
  `FLAGS[].defaultValue`, so the bar renders unchanged. All eight
  pre-existing test cases (trickle / done / reduced-motion /
  colour / className / forwardRef / routeKey-cycle / hidden-by-
  default) still pass.
- Flag flipped to `false`: the root `<div data-testid=
  "route-progress-bar">` is not in the DOM. The trickle interval
  is never armed (no `start()` runs), so no timers leak.
- Flag flipped back to `true`: a fresh mount is performed by
  React; the bar renders in its initial hidden state, ready for
  the next route change to call `start()`.

## Tests

`web/src/components/RouteProgressBar.test.tsx` -- three new
cases on top of the existing eight:

1. Renders the progress bar when the `routeProgress` flag
   defaults to `true` (asserts the `data-testid=
   "route-progress-bar"` node is in the DOM).
2. Renders nothing when the flag is set to `false` before mount
   (`setFlag('routeProgress', false)` then `render(...)`;
   asserts `container.firstChild === null`).
3. Toggle cycle: mount with the default, flip to `false` via
   `setFlag` inside `act(...)`, assert the node disappears,
   flip back to `true`, assert the node reappears.

`beforeEach` / `afterEach` clear `c4:feature-flags` from
`localStorage` and call `resetFlags()` so the eight pre-existing
cases keep running against the default-on state and stay green.
