# 8.17 - External Claude session import (`c4 attach`)

## Summary

Adds a read-only import path for Claude Code JSONL transcripts that live
outside a c4-managed worker. An operator points `c4 attach` at either
an absolute `.jsonl` file or a bare session UUID under
`~/.claude/projects/` and the session shows up in the Web UI's Sessions
tab next to normal spawned workers. The original transcript is never
copied or rewritten - the attach store only persists a pointer and some
metadata, so detaching does not touch the underlying file.

Built on top of the 8.18 session-parser: attach does not re-parse the
JSONL, it re-uses `parseJsonl` / `listSessions` / `defaultProjectsRoot`.

## Module layout

```
src/session-attach.js                     # pure module (store + resolver)

```

## Persistence

File: `~/.c4/attached.json` (override via `AttachStore({ storePath })`).

```json
{
  "sessions": [
    {
      "name": "attached-1234abcd",
      "jsonlPath": "/abs/path/to/session.jsonl",
      "sessionId": "fixture-1",
      "projectPath": "/home/test/demo",
      "createdAt": "2026-04-18T10:00:00.000Z",
      "lastOffset": 2118
    }
  ]
}
```

Persisted attachments survive daemon restart. Malformed files are
treated as empty (the parser never throws back into the daemon).

## `src/session-attach.js`

Zero runtime dependencies. Public surface:

| Export | Shape | Purpose |
|---|---|---|
| `AttachStore({ storePath? })` | class | load / list / add / remove attachments |
| `defaultAttachedStorePath()` | `string` | resolves `~/.c4/attached.json` |
| `resolveSessionPath(input, { projectsRoot? })` | `{ path, sessionId, projectDir, projectPath } \| { error, code }` | path or UUID lookup |
| `attach(input, { store?, projectsRoot?, name? })` | `{ ok, record, summary } \| { error, code }` | register + summarize |
| `detach(name, { store? })` | `{ ok } \| { error, code }` | remove a registration |
| `summarize(path)` | `{ sessionId, projectPath, createdAt, updatedAt, model, turns, tokens, warnings }` | compact parse output |
| `listAttached({ store? })` | `AttachedSession[]` | convenience list |
| `getShared()` | `AttachStore` | daemon-wide singleton (tests build their own) |

`resolveSessionPath` treats the input as a path when it contains a path
separator or ends with `.jsonl`; anything else is looked up as a
session id by walking `projectsRoot`. Ambiguous ids (two files with the
same session UUID under different project directories) return code
`AMBIGUOUS` with a `matches` array so the operator can re-run against
the full path. Zero-match returns `NOT_FOUND`, non-`.jsonl` returns
`BAD_EXT`, and missing files return `ENOENT`. All filesystem errors
become structured `{ error, code }` return values rather than thrown
exceptions so CLI, daemon, and Web UI formatters stay simple.

Attempting to attach the same path twice returns
`{ code: 'ALREADY_ATTACHED', existing }` instead of silently duplicating
the row. Duplicate *names* append a `-2`, `-3`, ... suffix up to 99
before giving up with `NAME_COLLISION`.

## Daemon endpoints

All four live under the existing auth middleware from 8.14 and are
gated by `rbac.ACTIONS.WORKER_CREATE`. Attaching a session is
conceptually a zero-pty spawn, so re-using the existing role keeps the
permission matrix from growing a one-off action.

```
POST   /api/attach              { path?, sessionId?, name? }
GET    /api/attach/list
DELETE /api/attach/:name
GET    /api/attach/:name
GET    /api/attach/:name/conversation
```

### `POST /api/attach`

Request body accepts either `path` or `sessionId` (exactly one; the
CLI picks the flag based on input shape). Optional `name` sets a custom
alias instead of the auto-derived `attached-<session-short>` form.

Response:

```json
{
  "name": "attached-fixture1",
  "sessionId": "fixture-1",
  "projectPath": "/home/test/demo",
  "jsonlPath": "/abs/path/session.jsonl",
  "createdAt": "2026-04-18T10:00:00.000Z",
  "turns": 8,
  "tokens": { "input": 150, "output": 60 },
  "model": "claude-opus-4-7",
  "warnings": 1
}
```

Status codes: `400` on missing input / bad extension; `404` on
`ENOENT` or `NOT_FOUND`; `409` on `AMBIGUOUS` or `ALREADY_ATTACHED`.

### `GET /api/attach/list`

Returns `{ sessions: AttachedSession[], total }`. Records are the
persisted shape above.

### `DELETE /api/attach/:name`

Removes the pointer. Does **not** delete the underlying `.jsonl` file.
Returns `{ ok: true, name }` on success, `404` when the name is
unknown.

### `GET /api/attach/:name`

Returns `{ record, summary }` so the Web UI can show totals in the
list row without fetching the full Conversation. The summary re-runs
`parseJsonl` so stale counts are never surfaced.

### `GET /api/attach/:name/conversation`

Returns the full parsed `Conversation` payload - exactly the shape the
8.18 `/api/sessions/:id` endpoint emits. Returns `410 Gone` when the
underlying `.jsonl` no longer exists on disk (the caller should call
DELETE to prune the registration).

## CLI

```
c4 attach <session-id-or-path> [--name alias]
c4 attach list
c4 attach detach <name>
```

The first form POSTs to `/api/attach` and pretty-prints the summary:

```
Attached as attached-fixture1
  sessionId: fixture-1
  project:   /home/test/demo
  turns:     8
  tokens:    150 in / 60 out
  model:     claude-opus-4-7
  warnings:  1
```

`list` renders a compact table and `detach <name>` just prints
`Detached <name>` after the DELETE call returns `ok: true`.

## Web UI

The Sessions tab grows an **Attached** sub-section at the top of the
sidebar, distinct from the per-project groups below it. Each row
shows the alias, short session id, project path (or the JSONL path as
fallback), creation timestamp, and a trash icon that calls
`DELETE /api/attach/:name`. A collapse chevron hides the list when it
gets long.

A new `+ Attach new...` button opens a modal with two inputs (JSONL
path or session UUID, plus an optional alias). On submit it POSTs to
`/api/attach`, refreshes the list, and auto-selects the new row so the
viewer pane renders immediately.

The detail pane re-uses `ConversationView` via its new `snapshotUrl`
prop - attached rows point the component at
`/api/attach/<name>/conversation` instead of `/api/sessions/:id`. This
keeps all markdown / tool / thinking rendering code in one place.

## Manager worker record `kind` field

`pty-manager.list()` now stamps `kind: 'spawned'` on every row. Web UI
consumers that want a unified list of spawned + attached workers can
concatenate `/list`'s `workers[]` with `/attach/list`'s `sessions[]`
(the Web UI in 8.17 keeps them visually separate). The field is
forward-compatible: a future `kind: 'remote'` for 9.6 fleet peers would
land alongside `'attached'` without schema churn.

## Deferred scope (P2 / P3)

- **P2 Bidirectional resume** (`POST /api/attach/:name/resume` spawning
  `claude --resume <sessionId>`): not implemented. The persisted record
  already carries the session id, so wiring this up later is a pure
  daemon-side change - no parser or Web UI shape changes.
- **P3 Auth ACL** (only the importing operator sees their attached
  sessions): deferred. The current RBAC gate is `WORKER_CREATE`, which
  any admin / manager already holds; multi-user visibility isolation
  would need a new `attach.read` action plus per-record owner metadata.
  Tracked here so the gap is not lost in review.

## Interface stability

Two contracts are load-bearing for downstream work:

1. The persisted shape of `~/.c4/attached.json` (`sessions[].name`,
   `jsonlPath`, `sessionId`, `projectPath`, `createdAt`, `lastOffset`).
2. The HTTP response shapes above.

Breaking changes to either need a major-version bump on `c4` plus
updates to any fleet peer that proxies these endpoints.
