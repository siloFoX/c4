The dispatch lists:

- Build `web/src/components/ui/error-boundary.tsx` --
  React error boundary with fallback UI
  (`ErrorState` component), retry, telemetry hook
  (logs to `console.error` in dev, optional reporter
  prop).
- Wrap every page route in `App.tsx`.

This patch ships the primitive + 13 unit tests + the
App.tsx wrap. The existing top-level
`components/ErrorBoundary` (rich crash surface with

global safety net.

## Public API

```ts
export class UIErrorBoundary extends Component<UIErrorBoundaryProps, ...>;

export interface UIErrorBoundaryProps {
  children: ReactNode;
  fallback?: ReactNode | ((error: Error, retry: () => void) => ReactNode);
  onError?: (error: Error, info: ErrorInfo) => void;
  resetKeys?: ReadonlyArray<unknown>;
  title?: string;
  description?: string;
  className?: string;
}
```

## Behaviour

### Default fallback

When the boundary catches an error and no `fallback`
prop is set, it renders the canonical `ErrorState`
primitive with:

- `title = props.title ?? "Something went wrong"`
- `description = props.description ?? "This view
  failed to render. Retry to re-mount."`
- `error = caught error` (so the message / stack
  surface)
- `onRetry = this.reset` (Retry button clears the
  error state)
- `data-testid="ui-error-boundary-fallback"` for
  e2e scoping

`role="alert"` flows in via the primitive's existing
contract.

### Custom fallback shapes

| `fallback` value | Rendered as |
| --- | --- |
| omitted | default `ErrorState` |
| `ReactNode` | verbatim |
| `(error, retry) => ReactNode` | factory; receives the caught error + a retry callback |

### Telemetry

- **Always**: logs `console.error('[UIErrorBoundary]', error, info.componentStack)`
  so dev-time stack traces show up in the devtools
  console without a wired reporter.
- **Optional**: `onError(error, info)` is invoked
  after the console log. Errors thrown FROM
  `onError` are swallowed (a buggy reporter cannot
  crash the boundary).

### resetKeys identity flip

`componentDidUpdate` shallow-compares
`prevProps.resetKeys` against `this.props.resetKeys`.
A length / element-identity change triggers
`this.reset()`, which clears the error state and
re-renders children.

This is the canonical pattern for route-key resets:
when the operator navigates to a different page, the
old crash should not stick around.

### Imperative reset

`this.reset()` is exposed on the class. The default
fallback wires it to the Retry button's `onRetry`.
Custom fallbacks receive it as the second argument to
the factory form.

## App.tsx wrap

```tsx
<UIErrorBoundary
  resetKeys={[topView]}
  title="This view crashed"
  description="The page below failed to render. Retry to re-mount, or navigate away."
>
  <PageTransition routeKey={topView}>
    {topView === 'history' ? <HistoryView /> : ...}
  </PageTransition>
</UIErrorBoundary>
```

The single boundary covers every topView branch
because `<PageTransition>` switches between them
exclusively (only one branch is mounted at any time).
`resetKeys={[topView]}` ensures a crash in one view
clears when the operator navigates to another.

Why not wrap each branch individually? The branches
are mutually exclusive at render time, so a single
outer boundary is functionally equivalent and reads
cleaner.

## Coexistence with `components/ErrorBoundary`

The existing top-level boundary
(`web/src/components/ErrorBoundary.tsx`) is the rich
crash surface with stack-trace pre, Copy stack, Open
GitHub issue, and Reload. It is mounted in
`main.tsx` and stays the global safety net for crashes
in the shell, the header, the sidebar, or any other
non-page subtree.

The new `UIErrorBoundary` is the lighter primitive
for per-page / per-feature subtrees. Both can nest;
the inner boundary catches first, the outer one only
runs when the inner has its own bug.

## Tests

`error-boundary.test.tsx` has 13 cases:

1. Happy-path passthrough (no error).
2. Default `ErrorState` fallback on caught error.
3. `title` / `description` override flows through to
   the default fallback.
4. `ReactNode` fallback rendered verbatim.
5. Factory `fallback` receives `(error, retry)`.
6. `onError` invoked with `(error, info)`.
7. Errors thrown from `onError` are swallowed (the
   reporter cannot crash the boundary).
8. `console.error` telemetry always fires (the
   `[UIErrorBoundary]` log line).
9. Retry button clears the error and re-renders
   children.
10. `resetKeys` identity flip resets the error
    state.
11. Stable `resetKeys` identity does NOT reset.
12. Nested boundary catches locally without
    bubbling.
13. `className` forwarded to the default fallback.

13/13 pass against vitest 4.1.5 + jsdom 29.1.1.

## Out of scope

- **Migrating the existing top-level
  `components/ErrorBoundary` to use this primitive.**
  The top-level has a different mandate (rich crash
  surface). It stays as-is.
- **Per-route `onError` reporter wiring.** The
  primitive exposes the hook; per-page adopters can
  wire it to `lib/error-reporter` when they need
  explicit instrumentation. The default
  `console.error` covers the dev-loop need.
