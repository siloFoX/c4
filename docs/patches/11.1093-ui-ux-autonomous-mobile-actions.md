# 11.1093 -- UX FIX: Autonomous status-card actions clipped on mobile (v1.11.1111)

## Scope

At 375 the AutonomousView card headers overflowed: the digest card's
Refresh + Pause buttons (the red Pause especially) and the trailing
status message clipped at the right edge, and the escalation card's
show-resolved control clipped too.

## Root cause

`CardHeader` defaults to `flex flex-col`, but both AutonomousView
headers overrode it to a single horizontal row
(`flex-row ... justify-between`). At 375 the title plus the action
group exceeded the width with no wrap, so the trailing controls ran off
the right edge.

## Change (src/components/AutonomousView.tsx)

- Digest card header: `flex-row items-center justify-between` ->
  `flex-col gap-2 sm:flex-row sm:items-center sm:justify-between`, so
  the Refresh/Pause actions stack below the title below sm and return
  to the single row at >= sm. The actions group gains `flex-wrap` as a
  final safety.
- Escalation card header: the inner row likewise becomes
  `flex flex-col gap-2 sm:flex-row sm:items-center sm:justify-between`,
  and the show-resolved control group gains `flex-wrap`.
- Added `data-testid` to the Refresh button
  (`autonomous-refresh-btn`), the Pause button (`autonomous-pause-btn`),
  and the show-resolved label (`autonomous-show-resolved`) so the e2e
  can assert their positions.

Layout-only; reuses existing token-backed classes (no new colours /
spacing). The six other views the prior audit flagged as adequate are
untouched.

## Verification (mandatory)

`web/e2e/autonomous-mobile-actions.spec.ts` (new) boots into the
Autonomous view and, for each of 375 / 768 / 1440, asserts the three
header controls each render with a bounding box fully inside the
viewport (left >= 0, right <= viewport width) -- i.e. no clip -- and
screenshots the 375 layout. Result: `3 passed`. The 375 screenshot
confirms Refresh + the red Pause stacked below the title and the
show-resolved control below the escalations title, all fully visible.

Note: the spec must stub `/api/metrics` with a full body
(`totals.liveWorkers` etc.) because `MetricsBar` (global chrome) reads
`m.totals.liveWorkers` without guarding a partial response; an
unstubbed metrics call returns `{}` and crashes the view in the test
harness. That MetricsBar hardening is a separate concern (the live
daemon returns the full shape) and is out of this todo's scope.

## Risk

- Pure layout change to two headers + three test-only `data-testid`s.
  At >= sm the headers render identically to before (verified at 768 /
  1440). No behaviour change.
