# 11.306 -- theme tokens validator (v1.11.324)

New theme-token policy enforcer at
`web/src/lib/theme-tokens-validator.ts` plus a companion
test that walks every `web/src/components/ui/*.tsx` file
and asserts zero raw hex / rgb / hsl color literals.
Component-scope only, no daemon-side change and no `c4`
CLI surface change.

## Why

The c4 web app's design system is built on Tailwind
utility classes that resolve to CSS variables (the
"tokens" -- `--color-foreground`, `--color-background`,
`--color-primary`, etc) plus a fixed spacing scale. Any
component that hardcodes a raw hex color (`#fff`,
`#1a1a1a`), an `rgb(...)` literal, or an `hsl(...)`
literal silently bypasses the token layer and breaks the
canonical theming contract:

- The dark-mode / light-mode flip stops working for that
  surface (the hex stays the same regardless of theme).
- The palette migration story (ARPS theme refresh,
  future brand re-skin) has to grep every component
  instead of editing one CSS variable.
- Token-vs-hex drift hides inside a 4000-line repository
  diff and is found only when a designer opens the
  dark-mode build and sees an off-brand splash of
  colour.

Patch 11.306 ships a synchronous scanner + a regression
test that fails CI immediately if a future commit
re-introduces a raw color literal in the UI tree.

## What changed

### `web/src/lib/theme-tokens-validator.ts` (new file)

A pure-text scanner -- no AST parsing, no Tailwind /
PostCSS dependency. Takes file contents and returns a
list of `ThemeViolation` objects:

```ts
interface ThemeViolation {
  file: string;
  line: number;
  column: number;
  match: string;
  rule: 'hex-color' | 'rgb-fn' | 'hsl-fn';
  excerpt: string;
}
```

Three rules enforced:

- `hex-color`: 3-, 4-, 6-, or 8-digit hex literals like
  `#fff`, `#1a1a1aff`. 5- and 7-digit hex sequences are
  skipped because they are not valid CSS colors and
  almost always indicate a build hash, content id, or
  similar non-color usage.
- `rgb-fn`: `rgb(...)` and `rgba(...)` function calls.
- `hsl-fn`: `hsl(...)` and `hsla(...)` function calls.

Skip rules tuned to avoid false positives:

- `// line comments` are stripped before scanning, so
  documentation like `// canonical color was #ff0000`
  is not flagged.
- `/* block comments */` (including multi-line JSDoc)
  are stripped before scanning, so legacy palette
  references in module-level documentation pass.
- URL fragments (`href="#anchor"`), id selectors
  (`'#main-content'`), and other non-color hex-like
  substrings are filtered via a non-word look-behind
  on the `#` and a "plausible hex length" guard.
- `currentColor`, `transparent`, `none` SVG attribute
  values are not banned (they are intentionally
  non-tokenised).
- `var(--token-x)` CSS variable references are not
  banned (they ARE the canonical token reference).

Exported helpers:

- `scanSourceForViolations(filePath, contents)` ->
  `ThemeViolation[]` (test-friendly, no I/O).
- `scanFilesForViolations(files)` ->
  `ThemeViolation[]` (reads each file from disk).
- `formatViolations(violations)` -> string. Yields a
  human-readable `file:line:col [rule] match` report
  used by the integration test failure message so a CI
  log points at the exact line that needs fixing.

Spacing (`px`, `rem`, `em`) values are NOT banned. A
caller-supplied default like `width = '320px'` is a
tunable knob, not a token violation; the design system
has no opinion on the default value as long as the
consumer can override it. (The UI tree has a small
number of these prop defaults; they remain in-scope as
caller-overridable knobs.)

### `web/src/lib/theme-tokens-validator.test.ts` (new file)

Unit cases (15): hex literals of varying lengths,
href/`#anchor` exemption, 5/7-digit hex exemption,
rgb/rgba, hsl/hsla, line comment exemption, block
comment exemption, `var(--token)` allowance,
`currentColor`/`transparent`/`none` allowance,
file/line/column reporting, `formatViolations`
behaviour.

Integration case (1): walks every
`web/src/components/ui/*.tsx` non-test file (>10 files
on the current main + branch) and asserts the scanner
returns zero violations. The integration test currently
passes -- the UI tree is byte-clean. The test now
guards against future regressions.

All 16 green via
`vitest run --project unit src/lib/theme-tokens-validator.test.ts`.

## Notes

- The scanner is intentionally text-based (regex + line
  walk) rather than AST-based. Pulling in `@babel/parser`
  or `typescript`'s compiler API for this check would
  cost a couple of seconds at test time for marginal
  precision gain. The text rules are conservative enough
  to avoid false positives on the current UI tree, and
  the false-negative risk (a hex literal hidden inside
  a template literal or computed string) is low because
  the design-system convention is to use Tailwind class
  strings, not inline color computations.
- The integration test is intentionally placed at the
  `web/src/components/ui/` directory boundary (not at
  the whole-app root). Adopting tokens everywhere is a
  long-tail effort; the dispatch scope is just the UI
  primitives, which are the most visible surface area
  and the right place to set the policy floor.
- Future extension: a `lint:theme-tokens` npm script
  that runs the same scanner against the entire
  `web/src/` tree as a soft-warning would make sense
  once the surrounding pages adopt the same policy.
  Out of scope for this patch.
