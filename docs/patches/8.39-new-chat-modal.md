# 8.39 — New Chat modal (Sessions tab)

Branch: `dgx-spark/new-chat-modal`

## What ships

A claude.ai-style "start a new conversation" entry point inside the
Sessions tab. Operators no longer have to drop to a terminal + run
`c4 task` to spawn a fresh worker — they can pick a model, an agent
template, type an initial prompt, and submit from the Web UI.

## Surface area

- **`web/src/components/SessionsView.tsx`**
  - New `NewChatModal` component (model + agent + prompt selectors,
    stop-propagation backdrop, autofocus on textarea, reset on open).
  - `MODEL_CHOICES`: `default` / `claude-opus-4-7` / `claude-sonnet-4-6`
    / `claude-haiku-4-5`. Each carries a one-line `hint` rendered under
    the select.
  - `AGENT_CHOICES` mirrors the builtin templates from
    `src/pty-manager.js _getBuiltinTemplates()`: `generic` (no template),
    `planner` (Opus + design prompt prefix), `executor` (Sonnet + impl
    prefix), `reviewer` (Haiku + review prefix). The `/api/task` route
    aliases `profile` → `template` so passing `profile: 'planner'`
    applies the planner template's model + prompt prefix.
  - `New Chat` button added to the Sessions header, sitting next to the
    existing `Attach new...` button.
  - `handleNewChatSubmit` POSTs to `/api/task` with no `name` (the
    daemon auto-generates one from the prompt's first words via
    `_generateTaskName`), and refreshes both the sessions list and the
    attached list on success so the new worker's JSONL shows up as soon
    as Claude Code starts writing it.

- **`src/daemon.js`** — minor fix to the `POST /task` route. When the
  caller omits `name`, the audit / Slack-emit / history-record calls
  used to log `worker: undefined`. Now we resolve the name through
  `result.name` (set by `pty-manager` after auto-generating) so every
  downstream record references the real worker.

## How to test manually

1. `c4 daemon start`
2. `cd web && npm run dev` and open http://192.168.10.222:5173
3. Sessions tab → click `New Chat` (top right of the session list).
4. Enter "build a hello-world Express server", pick model = Sonnet 4.6,
   agent = Executor, click `Start chat`.
5. Modal closes; the daemon spawns a worker named `w-build-a-hello-world-express-server` (truncated to 30 chars), the executor template applies the model + prompt prefix, and a JSONL session appears in the sessions list within a few seconds.

(added in review round 1):

   `'default'`, `profile` only attached when agent isn't
   `'generic'`, `autoMode: false`, refresh both
   `refreshSessions()` and `refreshAttached()` on success.
3. **daemon /task resolvedName fallback**: source-grep on the
   `resolvedName` variable + the audit / Slack-emit / history
   call sites (`target: resolvedName`, `worker: resolvedName`,
   `planner.setPlanDocPath(resolvedName, ...)`); behavioural
   shim verifies the exact precedence (explicit name >
   result.name > empty string).

## Review round 1 fixes (2026-05-01)

- **a11y bug**: NewChatModal had no Escape key handler — standard
  expectation for `role="dialog" aria-modal="true"`. Added a
  window-level keydown listener that fires `onClose()` only when
  not busy so an accidental Esc during submit doesn't drop the
  in-flight POST result.
- **UX bug**: backdrop click (`onClick={onClose}`) unconditionally
  closed the modal, including while a submit was in flight. The
  POST would still resolve, but the modal's error region would be
  unmounted and any failure response would be dropped silently.
  Fix: split the handler into `handleBackdropClick` that no-ops
  while `busy`.
- **Test gap**: PR #35 originally shipped without any dedicated
  tests. Added `tests/new-chat-modal.test.js` with 21 assertions
  across 3 suites covering NewChatModal contract,
  handleNewChatSubmit body shape, and daemon-side resolvedName
  fallback behaviour.
