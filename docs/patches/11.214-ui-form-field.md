# 11.214 -- UI: FormField primitive

Released as `web` v1.11.232 on 2026-05-15.

## Goal

Introduce a shared `FormField` primitive that owns the
label + control + helper/error layout for form rows, and adopt
it at three existing call sites. Component scope only -- no
public API change to existing primitives, no daemon-side touch,
no CLI surface.

## API

```ts
interface FormFieldProps {
  id?: string;
  label: string;
  required?: boolean;
  helperText?: string;
  error?: string;
  children: ReactElement;
  className?: string;
  layout?: 'vertical' | 'horizontal';
}
```

- `id` -- the control id. If omitted, `useId()` supplies one.
- `label` -- required string; rendered into a `<label htmlFor>`
  pointing at the control id. `required` toggles a small red
  `*` indicator next to the text.
- `helperText` / `error` -- the helper renders in
  `text-muted-foreground`; the error renders in `text-destructive`
  with `role="alert"`. When both are provided the error wins:
  only the error renders, and `aria-describedby` points at the
  error id. `aria-invalid="true"` lands on the control whenever
  `error` is truthy.
- `children` -- a single `ReactElement`. The element is
  `cloneElement`-augmented with `id` (when it is a form control)
  and `aria-describedby` (always). Pre-existing values on the
  child take precedence over the FormField-supplied ones for
  `id` and `aria-describedby`.
- `layout` -- defaults to `'vertical'`. `'horizontal'` switches
  the wrapper to `flex-row`, gives the label a ~30% column
  (`w-[30%] shrink-0 pt-2`), and lets the control side expand
  (`flex-1`). The wrapper also carries a `data-layout` attribute
  so the test suite (and future styling overrides) can target
  the chosen layout directly.

## Wrapper-aware cloneElement

The Login adoption forced a small design wrinkle. Login's two
form rows look like:

```tsx
<FormField id="c4-user" label={t('login.user')}>
  <div className="relative">
    <User aria-hidden="true" className="... absolute left-3 ..." />
    <Input id="c4-user" ... className="pl-10" />
  </div>
</FormField>
```

The `children` element is a wrapping `<div>`, not the `<input>`.
If FormField blindly cloned `id="c4-user"` onto the wrapper, two
elements would carry the same id and `getByLabelText('User')`
(via the existing `Login.test.tsx`) would resolve `htmlFor` to
the wrapper `<div>` instead of the `<input>`.

FormField guards against that: when `typeof children.type ===
'string'` and the tag is NOT in `{input, select, textarea}`, the
clone only sets `aria-describedby` and the caller's own
`<input>` keeps the real id. The label's `htmlFor` still uses the
FormField-resolved id, which the caller is responsible for also
placing on the inner control. This keeps the icon-overlay sites
working without forcing every adopter to flatten its DOM.

## Adoption picks

Three sites were swapped:

1. `web/src/pages/Plan.tsx` -- the plan-worker `<select>` row.
   Cleanest case: a literal `<Label htmlFor="plan-worker">` +
   `<select id="plan-worker">` pair. FormField cleans it up to
   a single declarative wrapper, and the cloneElement path
   reuses the existing `plan-worker` id.
2. `web/src/components/Login.tsx` -- the c4-user row.
3. `web/src/components/Login.tsx` -- the c4-password row.
   Both Login rows go through the wrapper-aware path described
   above so the icon overlay keeps working.

Sites intentionally left alone:

- `SettingsView.tsx`'s `ChoiceGroup` is a radiogroup `<div>`,
  not a form control row.
- `Swarm.tsx`'s swarm-worker `<select>` is wrapped in a
  `<Tooltip>`; the tooltip's own cloneElement / event wiring
  would conflict with FormField's id / aria injection. Out of
  scope for this patch.
- `NewChatModal.tsx` uses a deliberately custom
  `text-xs uppercase tracking-wide` label style that does not
  fit the shared `text-sm font-medium` look. Restyling the
  modal is a separate visual decision.
- `Input.tsx` / `Textarea.tsx` / `Select.tsx` already embed
  their own label / hint / error slots; the new primitive
  composes alongside them rather than replacing them.

## Tests

`form-field.test.tsx` covers 12 cases:

1. Renders label connected to control via htmlFor / id.
2. Auto-generated id is unique across two FormField siblings.
3. Externally provided id is honoured.
4. `required` renders the indicator.
5. Default (no `required`) omits the indicator.
6. `helperText` renders in muted tone and is wired via
   `aria-describedby`.
7. `error` renders in destructive tone, carries `role="alert"`,
   sets `aria-invalid="true"` on the control, and links via
   `aria-describedby`.
8. Error overrides helperText when both are provided.
9. `layout="horizontal"` flips the wrapper to flex-row and
   gives the label a ~30% column.
10. Default layout is vertical (`flex-col`).
11. A child-supplied id wins over a FormField-supplied id.
12. Caller `className` merges onto the wrapper.

The existing `Login.test.tsx` (including locale-flip assertions
that go through `getByLabelText`) and `Plan.test.tsx` both stay
green. Local run: 86 / 86 across the three files.

## Out of scope

- The `arps-design-system-v1` tokens were checked only for class
  names (`text-muted-foreground`, `text-destructive`, `text-sm`,
  `gap-1.5` etc.). No new tokens are added by this patch.
- Visual polish for `required` -- the indicator is the bare
  asterisk in the destructive tone described in the task.
  Tooltip / screen-reader-only "required" copy can be a later
  follow-up.
- `Fieldset` integration / radiogroup support is out of scope --
  FormField targets the single-control row case only.
