# 11.291 -- Textarea variant + char count + data selectors (v1.11.309)

Component-scope-only enhancement of the existing `Textarea`
primitive. No daemon-side change and no `c4` CLI surface change.
The default surface is preserved byte-for-byte; new affordances
opt in via the new `variant`, `showCharCount`, and `maxLength`
props.

## Why

Two pain points across the recent editor surfaces:

1. Templates body editor + Snapshot note + diff comments are
   all code-adjacent surfaces -- the operator is editing text
   that includes shell commands, file paths, or
   minimally-structured markdown where character alignment
   matters. The default sans-serif font fights the content;
   callers were inline-overriding the className to add
   `font-mono` (fragmenting the type vocabulary).
2. Description-style textareas wanted a "23/280" character
   counter under the field so the operator can pace their
   note-writing. Callers either built the counter inline (one
   `<p>` per editor) or skipped it entirely.

Patch 11.291 puts both affordances inside the primitive so
every consumer reaches for the same surface vocabulary.

## What changed

### `web/src/components/ui/textarea.tsx`

- New exported type `TextareaVariant = 'default' | 'mono'`.
- New `variant?: TextareaVariant` prop (default
  `'default'`). `default` keeps the sans-serif look
  byte-for-byte. `mono` swaps in `font-mono` (the
  Tailwind design token, not a hard-coded font stack).
- New `showCharCount?: boolean` prop. When true, renders a
  `<p data-section="textarea-char-count">` slot under the
  textarea with the current length over the `maxLength`
  denominator (`"5/20"`). When `maxLength` is unset, just
  the current length is shown (`"5"`).
- `maxLength` (standard HTML attr) is now forwarded twice:
  once to the underlying `<textarea>` (native input guard)
  AND used to drive the char-count math + `data-overflow`
  flag. Callers passing a controlled value longer than
  `maxLength` see the destructive overflow state even
  though the browser would have capped further typing.
- Data-attribute selectors:
  - `data-section="textarea"` on the `<textarea>` element +
    `data-variant` + `data-error`.
  - `data-section="textarea-row"` on the slot-wrapper div
    when `label` / `hint` / `error` / `showCharCount` is
    present.
  - `data-section="textarea-hint"` on the helper text.
  - `data-section="textarea-error"` on the error message.
  - `data-section="textarea-char-count"` on the char-count
    paragraph + `data-overflow="true|false"`.

### Adopted

- Primitive enhancement only this commit. Existing Textarea
  call sites (Plan, Queue editor, ChatComposer, Templates
  editor, etc.) inherit the new defaults + data attributes
  without any caller change.

## Deferred (dispatch follow-ups)

- Templates body editor monospace adoption: thread
  `variant="mono"` into the Templates editor surface so the
  body text reads as code.
- Snapshot note monospace adoption: same story -- the
  primitive supports `variant="mono"`; the per-page
  adoption is a follow-up.
- Profiles description editor: depends on the planned
  Profiles editor surface that does not exist today.

## Tests

- `web/src/components/ui/textarea.test.tsx` -- 13 new vitest
  cases:
  - `variant="default"` does NOT apply `font-mono`.
  - `variant="mono"` applies `font-mono`.
  - `data-variant` attribute present.
  - Char-count slot hidden by default.
  - `showCharCount + maxLength` renders `"<len>/<max>"`.
  - `showCharCount` without `maxLength` renders just
    `"<len>"`.
  - Overflow class + `data-overflow="true"` flip when length
    exceeds maxLength.
  - No overflow flag when at or below max.
  - Char count falls back to `defaultValue` when
    uncontrolled.
  - `data-section="textarea"` on the element.
  - `data-section="textarea-row"` on the slot wrapper.
  - `data-section="textarea-hint"` + `"textarea-error"` per
    state.
  - `data-error="true|false"` on the textarea per error
    prop.
  All 36 (23 original + 13 new) green.

## Versions

- 1.11.308 -> 1.11.309 across root + web package.json + both
  lockfiles. CHANGELOG.md entry under `## [1.11.309]`.

## Notes

- `showCharCount` is intentionally opt-in. Most existing
  Textarea adoptions (chat composer, plan body, queue
  detail editor) do not benefit from a character counter,
  so the default remains "no counter slot".
- The mono variant uses Tailwind's `font-mono` token so a
  future design-system hue migration follows automatically.
