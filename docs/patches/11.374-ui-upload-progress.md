## Goal

Dispatch:

- Build `web/src/components/ui/file-upload-progress.tsx`
  -- per-file progress bar with cancel
  button, retry on failure, total upload
  summary. Integrate with file-drop
  component.

`<FileDrop>` (11.270 / 11.352) covers file
intake + a SINGLE inline progress bar. The
new primitive owns the *list* of per-file
rows + aggregate summary, while the
dropzone keeps its existing single-bar
slot for the simple "uploading one
snapshot" case.

`lib/file-upload` (11.352) is the XHR
helper that surfaces `onProgress` events.
Hosts forward those into per-row state.

## Architecture

```
+----------------+        onAdd
| <FileDrop>     | ---------------> host (snapshot page, templates import, ...)
+----------------+
                                 |
                                 v
                         uploadFiles(...) with per-row signal
                                 |
                                 v
                         host updates items[]
                                 |
                                 v
+--------------------------+
| <FileUploadProgress>     |
|  - items                 |
|  - onCancel(id)          |
|  - onRetry(id)           |
|  - onRemove(id)          |
+--------------------------+
```

The component is purely render-driven.
State lives in the host so a host can
share it across surfaces (e.g., a global
upload tray driven by Zustand).

## API

```tsx
import {
  FileUploadProgress,
  formatBytes,
  summarizeUploads,
  type UploadItem,
} from './components/ui/file-upload-progress';

<FileUploadProgress
  items={uploads}
  onCancel={(id) => abortMap.get(id)?.abort()}
  onRetry={(id) => restartUpload(id)}
  onRemove={(id) => setItems((u) => u.filter((x) => x.id !== id))}
  formatSize={formatBytes}
  showSummary
  ariaLabel="Snapshot uploads"
/>
```

### `UploadItem`

```ts
interface UploadItem {
  id: string;
  name: string;
  size?: number;       // bytes
  progress: number;    // 0..1
  status: 'pending' | 'uploading' | 'success' | 'error' | 'cancelled';
  error?: string;      // surfaced under the row when status='error'
}
```

### Per-status affordances

| status | progress variant | badge label | action button |
| --- | --- | --- | --- |
| `pending` | `default` (primary) | "Queued" | Cancel |
| `uploading` | `info` | "Uploading" | Cancel |
| `success` | `success` | "Done" | Remove |
| `error` | `destructive` | "Failed" | Retry |
| `cancelled` | `warning` | "Cancelled" | Remove |

Buttons only render when the matching
callback is supplied. The button
`aria-label` includes the filename
("Cancel snapshot.zip", "Retry snapshot.zip",
"Remove snapshot.zip") so screen-reader
users always know which row a button
targets.

### Summary helper

```ts
export function summarizeUploads(items: UploadItem[]): UploadSummary;

interface UploadSummary {
  total: number;
  done: number;
  failed: number;
  cancelled: number;
  uploading: number;
  pending: number;
  overallProgress: number; // 0..1 average across rows
  bytesTotal: number;      // sum of size for rows that supplied one
  bytesDone: number;       // sum of size * effective-progress
}
```

Success rows are treated as 1.0 in the
aggregate even if the host did not flush
the `progress` field. Cancelled / error
rows surface the LAST `progress` value
(not zero) so the operator sees how far
the upload got before the abort.

### Bytes formatter

```ts
export function formatBytes(bytes: number): string;
```

Default IEC kibibyte ladder: `B / KiB /
MiB / GiB`. Callers can override via
`formatSize` -- the override threads
through both the per-row size cell and
the summary bytes row.

### Data attributes

Wrapper:
- `data-section="file-upload-progress"`
- `data-count`, `data-done`, `data-failed`,
  `data-uploading` mirror the
  `summarizeUploads` aggregate so
  downstream tests / storyshots can
  inspect without parsing children.

Per row:
- `data-section="file-upload-progress-item"`
- `data-status`, `data-id`.

Inner sections:
- `-summary`, `-summary-text`, `-summary-bytes`
- `-list`, `-empty`
- `-item`, `-name`, `-size`, `-status`,
  `-bar`, `-error`, `-actions`,
  `-cancel`, `-retry`, `-remove`,
  `-icon-success`, `-icon-error`.

## Pairs with existing primitives

- `<FileDrop>` (11.270 / 11.352) -- file
  intake. Forward `onAdd` -> upload
  pipeline -> per-row state ->
  `<FileUploadProgress>`. Drop the
  single-bar `progress` prop on the
  dropzone when this primitive is in
  use (the per-row bars take over).
- `lib/file-upload` (11.352) -- XHR
  helper. Forward each row's
  `onProgress` to per-row progress
  state; pair `signal` with the
  cancel button via an
  `AbortController` map keyed by
  `UploadItem.id`.

## Out of scope

- **Inline file thumbnails** in the row.
  `<FileDrop>`'s preview slot (11.352)
  already covers thumbnails before
  upload; duplicating inside the row is
  design duplication.
- **Auto-removal** of completed rows.
  The host owns the lifecycle.
- **Drag-to-reorder** the upload queue.
- **Network-speed estimate** (KiB/s,
  ETA). `lib/file-upload`'s
  `UploadProgressEvent` has the data
  but the smoothing + display story
  belongs in a separate patch.
- **`<FileDrop>` adoption swap.** This
  primitive is additive; existing
  `<FileDrop>` callers can keep their
  single-bar surface or wire the new
  list when ready.
