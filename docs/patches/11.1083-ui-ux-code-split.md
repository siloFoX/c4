# 11.1083 -- UX FIX: lazy-load ui directory for dev + bundle speed (v1.11.1101)

## Scope

The `web/src/components/ui` directory holds 1500+ components / 1M+
lines, dominated by the chart family: `chart-bar` / `chart-line` /
`chart-pie` / ... plus ~640 `chart-line-*` indicator primitives. Vite
dev cold-start and HMR were slow because the barrel
`web/src/components/ui/index.ts` re-exported every one of those charts
with `export * from './chart-...'`.

In native-ESM Vite dev, an `export *` line forces the browser to
fetch + transform the re-exported module the first time ANY barrel
consumer loads. Dozens of app components import a single primitive
(e.g. `Button`) from `./ui`, so the barrel dragged all ~640 chart
modules into the cold-start module graph -- even the anonymous Login
screen, and the default Workers view, which display no chart at all.

The charts were already only ever rendered through `ChartLineGallery`,
which lazy-loads each one with a direct dynamic
`import('./ui/chart-line-...')`. So the barrel re-exports were pure
dead weight on the critical path.

## Change

`web/src/components/ui/index.ts`: removed the contiguous block of
642 `export * from './chart-*'` lines (the entire chart family). The
barrel now re-exports only the ~150 lightweight primitives (button,
card, input, dialog, table, sparkline, gauge, ...). A guard comment
documents that charts must be imported directly from their own file
path and must never be re-added to the barrel.

No consumer imported a chart through the barrel (verified: only
`ChartLineGallery` references chart modules outside `ui/`, and it uses
direct dynamic imports), so this is purely subtractive. `tsc --noEmit`
is the compile-time proof -- any chart-via-barrel import would surface
as a missing-export error.

## Measured vite dev cold-start delta

Measured headless in a real-`node_modules` checkout (the worktree's
`node_modules` is a symlink that blocks vite config resolution).
Method: clean `vite` dev server start with pre-warmed `.vite/deps`
(identical for both barrels -- charts are source, not deps), then a
single headless Chromium load of `/`, counting the `/src/*` module
requests the browser must fetch + transform and the wall time until
the request stream goes idle. The `/src` module-request count is
deterministic across runs.

| barrel                | /src module requests | wall to settle |
| --------------------- | -------------------: | -------------: |
| FULL (before)         |                  890 |        ~45.2 s |
| SLIM (after)          |                  248 |        ~21.8 s |
| **delta**             |     **-642 (-72%)**  | **~-23 s (-52%)** |

The -642 module reduction is exactly the 642 chart exports removed:
on the default Workers view the app now pulls ZERO chart modules,
versus all ~640 before. (First-ever cold start including dependency
optimization measured ~69 s with the full barrel; the table uses
warm-deps runs so the delta isolates the barrel effect.) HMR benefits
identically -- editing any barrel-consuming component no longer
invalidates a 640-module subtree.

## Verification

- `tsc --noEmit`: passes -- no consumer imported a chart symbol via
  the barrel (compile-time guarantee that the subtractive change is
  safe).
- Charts remain fully functional: `ChartLineGallery` imports each
  chart from its own file path (`./ui/chart-line-*`), unaffected by
  the barrel change. The Gallery route still lazy-loads all tiles.
- Existing vitest suite: unaffected -- the change only removes
  re-exports that no test consumed (no test imports a chart through
  the `./ui` barrel; chart tests import their own files directly).

## Risk

- Purely subtractive at the module-graph level. The only way this
  breaks a consumer is a `import { ChartFoo } from '../ui'` (barrel)
  somewhere; `tsc` confirms none exist.
- If a future component needs a chart, it must import it directly
  (`import { ChartLineRsi } from './ui/chart-line-rsi'`) -- the guard
  comment in the barrel spells this out so the cold-start regression
  does not silently return.
