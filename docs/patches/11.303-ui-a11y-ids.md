# 11.303 -- useA11yIds helper bundle (v1.11.321)

New `useA11yIds` hook at `web/src/hooks/use-a11y-ids.ts`.
Component-scope only, no daemon-side change and no `c4`
CLI surface change.

## Why

Several form-control primitives in the c4 web app each
implement the same WAI-ARIA labelling boilerplate:

```tsx
// form-field.tsx (today)
const autoId = useId();
const fieldId = providedId ?? autoId;
const helperId = `${fieldId}-helper`;
const errorId = `${fieldId}-error`;
const warningId = `${fieldId}-warning`;
const messageIds = [
  childExistingDescribedBy,
  state === 'error' ? errorId : undefined,
  state === 'warning' ? warningId : undefined,
  state === 'ok' && helperText ? helperId : undefined,
].filter(Boolean);
const ariaDescribedBy = messageIds.length > 0
  ? messageIds.join(' ')
  : undefined;
const ariaInvalid = state === 'error' ? true : undefined;
```

The same shape appears in `radio-group.tsx` (helper +
error), `checkbox.tsx` (helper + error), `select.tsx`
NativeSelect, `tag-input.tsx` (helper + error), and
`number-input.tsx`. Each call site re-derives the four
ids, the `aria-describedby` join, and the `aria-invalid`
flip. Drift between sites is easy:

- One site uses `${id}-help`, another uses
  `${id}-helper`. Shared CSS / e2e selectors silently
  break.
- One site joins with `,` (broken), another with `'\n'`
  (broken), one with `' '` (correct).
- One site emits an empty `aria-describedby` when no
  message is active -- axe-core flags that as a
  warning.
- One site flips `aria-invalid` on warning (wrong --
  warning is a soft hint, the field is still
  submittable).

Patch 11.303 ships `useA11yIds` so every form control
reaches for the same vocabulary.

## What changed

### `web/src/hooks/use-a11y-ids.ts` (new file)

- `useA11yIds(opts?)` returns
  `{ controlId, labelId, helperId, warningId, errorId,
    ariaDescribedBy, ariaInvalid, state }`.
- Options:
  - `id?: string` -- caller-provided id overrides the
    React `useId` fallback.
  - `helper?: string | boolean | null | undefined` --
    truthy means the helper message is rendered.
  - `warning?: string | boolean | null | undefined` --
    truthy means the warning message is rendered.
  - `error?: string | boolean | null | undefined` --
    truthy means the error message is rendered.
  - `describedBy?: string` -- upstream
    `aria-describedby` value that should be merged
    with the helper/warning/error ids.
- Behaviour contract:
  - Empty string / `false` / `null` are all treated as
    "no message" so callers can pass raw prop values
    without coercing.
  - `aria-describedby` is `undefined` (never an empty
    string) when no slot is active, so axe-core does
    not warn.
  - Precedence: `error > warning > ok` for `state`.
    `ariaInvalid` is `true` only when error is
    active.
  - Memoised on the input set so adopters that pass
    inputs verbatim across renders get a stable
    result object.
- The id suffix shape is fixed: `-label`, `-helper`,
  `-warning`, `-error`. Documented in tests so future
  rewrites cannot drift the contract.

### Adopted

- Hook only this commit. The migration of FormField,
  RadioGroup, Checkbox, NativeSelect, TagInput, and
  NumberInput is deferred to per-primitive follow-up
  patches so each migration lands with its own test
  refresh.

## Deferred (dispatch follow-ups)

A representative swap looks like:

```tsx
// Before -- inline 4-useId + ad-hoc describedBy in
// form-field.tsx
const autoId = useId();
const fieldId = providedId ?? autoId;
const helperId = `${fieldId}-helper`;
const errorId = `${fieldId}-error`;
const warningId = `${fieldId}-warning`;
const messageIds = [/* ... */];
const ariaDescribedBy = messageIds.length > 0
  ? messageIds.join(' ')
  : undefined;
const ariaInvalid = state === 'error' ? true : undefined;

// After -- using useA11yIds
const a11y = useA11yIds({
  id: providedId,
  helper: helperText,
  warning,
  error,
  describedBy: childExistingDescribedBy,
});
const ariaDescribedBy = a11y.ariaDescribedBy;
const ariaInvalid = a11y.ariaInvalid;
const helperId = a11y.helperId;
const errorId = a11y.errorId;
const warningId = a11y.warningId;
```

Per-primitive follow-ups:
`web/src/components/ui/form-field.tsx`,
`web/src/components/ui/radio-group.tsx`,
`web/src/components/ui/checkbox.tsx`,
`web/src/components/ui/select.tsx` (NativeSelect),
`web/src/components/ui/tag-input.tsx`,
`web/src/components/ui/number-input.tsx`.

## Tests

`web/src/hooks/use-a11y-ids.test.tsx` -- 17 vitest
cases:

- Stable `controlId` derived from `useId` when no id
  prop.
- Explicit `id` prop is honoured.
- All four ids share the documented prefix shape.
- `aria-describedby` is `undefined` when no slot is
  active.
- Helper-only: `ariaDescribedBy=x-helper`, `state=ok`.
- Warning-only: `ariaDescribedBy=x-warning`,
  `state=warning`, `ariaInvalid=undefined`.
- Error-only: `ariaDescribedBy=x-error`,
  `state=error`, `ariaInvalid=true`.
- Helper + error: both ids in
  `aria-describedby`, state=error, ariaInvalid=true.
- Precedence: error>warning>ok regardless of which
  slots are populated.
- Empty string / false / null all count as
  no-message.
- Upstream `describedBy` merges before the derived
  ids.
- Upstream `describedBy` alone is preserved when no
  slots are active.
- Memoisation stability across no-op renders (same
  object reference).
- New object on input change.
- Truthy boolean `true` counts as an active slot.
- `useId` fallback yields unique ids across two
  separate hook instances.
- Documented suffix shape: `-label`, `-helper`,
  `-warning`, `-error`.

## Notes

- The hook lives in `web/src/hooks/` (not
  `web/src/components/ui/`) because it is a
  behavioural hook, not a renderable primitive. The
  `index.ts` barrel under `components/ui/` is
  intentionally NOT touched.
- The truthy semantics (`helper: ''` counts as
  "no helper") let adopters pass their raw prop
  values without coercing. The most common adopter
  shape is
  `helper={hint} warning={warningText} error={errorMessage}`
  where each value is a string-or-undefined; empty
  strings are routinely passed when a state is
  pending and should not affect the rendered ids.
- Precedence order in the joined
  `aria-describedby` is helper -> warning -> error
  so screen readers announce the field's purpose
  (helper) before the validity state (warning /
  error). The describedBy upstream value, when
  present, leads the join so it is announced first.
- `state` is included separately from
  `ariaInvalid` because some adopters (FormField)
  use `data-state` for theming while still relying
  on `aria-invalid` for screen readers; both are
  derived from the same precedence rule.
