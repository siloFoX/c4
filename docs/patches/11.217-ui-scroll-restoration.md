# 11.217 -- UI: scroll-restoration hook (v1.11.235)

Component-scope-only addition. No daemon-side change and no `c4` CLI
surface change.

## Why

Long scroll surfaces in the web UI (History view sidebar, worker
list) reset to the top every time the operator navigates away and
back. The History sidebar in particular -- virtualized through the
11.209 `useListVirtualizer` hook -- can easily span 1k+ rows, and
losing the cursor every time a detail pane is opened is friction the
operator feels on every back/forward.

`window.history.scrollRestoration` only covers the document scroll
position, not nested overflow containers, so the fix lives in a
small reusable hook rather than a router-level shim. Persisting to
`sessionStorage` (per-tab) is the right granularity: scroll position
should not survive a tab close, but should survive component
unmount / remount within the same session.

## Surface

```ts
useScrollRestoration({
  containerRef,        // RefObject<HTMLElement>
  storageKey,          // string; bare keys get namespaced as c4:scroll:<key>
  debounceMs?: number, // default 100ms (trailing-edge)
}): { reset(): void }
```

- On mount: reads `sessionStorage[<resolvedKey>]` and assigns to
  `containerRef.current.scrollTop`. Missing / invalid values are
  no-ops (initial position stays 0).
- On scroll: writes `scrollTop` back to storage after the debounce
  window. Rapid scrolls coalesce to a single write of the latest
  value.
- On unmount: removes the listener, cancels any pending timer, and
  flushes the last observed value so detaching during a debounce
  window does not lose state.
- `reset()`: clears the stored position (useful when a list is
  filtered or refreshed and the previous offset no longer makes
  sense).

SSR-safe: all `window` / `sessionStorage` access is gated by
`typeof` checks and runs inside `useEffect`, never during render.
Catches `try/catch` around storage access so quota / private-mode
errors stay silent (best-effort).

## Storage key conventions

Two namespaces in this patch:

- `history:sidebar` -- the History view sidebar virtualizer scroll
  container.
- `workers:list` -- the WorkerList root scroll surface.

Bare keys are auto-prefixed with `c4:scroll:` so the final keys
written to `sessionStorage` are `c4:scroll:history:sidebar` and
`c4:scroll:workers:list`. Callers can pass a fully qualified key
(containing `:`) to opt out of the prefix.

## Tests

`web/src/hooks/use-scroll-restoration.test.tsx`, 9 cases:

1. Restores `scrollTop` from `sessionStorage` on mount.
2. Writes `scrollTop` after the debounce window elapses.
3. Multiple rapid scrolls debounce to a single write of the latest
   value (spy on `Storage.prototype.setItem`).
4. `reset()` clears the stored position.
5. Unmount removes the scroll listener.
6. Empty `sessionStorage` leaves initial `scrollTop` at 0 (no
   crash).
7. `null` `containerRef` is a safe no-op.
8. Two hook instances with different storage keys persist
   independently.
9. Unmount during a pending debounce flushes the value rather than
   dropping it.

Run:

```
env -C web /root/c4/web/node_modules/.bin/vitest run \
  src/hooks/use-scroll-restoration.test.tsx
# Tests  9 passed (9)
```

## Adoption

- `web/src/components/HistoryView.tsx`: `SidebarVirtualizedList`
  acquires a `useRef<HTMLDivElement>` on the existing scroll
  container (`overflow-auto` + `height: 60vh`) and calls
  `useScrollRestoration({ containerRef, storageKey:
  'history:sidebar' })`. The virtualizer's own `onScroll` handler
  stays wired -- the hook only listens; it does not replace the
  virtualizer offset computation.
- `web/src/components/WorkerList.tsx`: the root `<div>` gains a
  ref and explicit `overflow-y-auto`, then the hook is wired with
  `'workers:list'`. The parent `<ScrollArea>` in
  `layout/Sidebar.tsx` still owns scroll when the WorkerList fits
  its content; the inner overflow only kicks in when the worker
  count exceeds the available column height.

## Non-goals

- Cross-tab persistence: out of scope -- intentionally
  `sessionStorage`, not `localStorage`.
- Programmatic "scroll to ID" deep linking: separate concern; a
  future patch could layer `scrollIntoView` on top of the hook.
- Resize observers: scroll restoration runs once on mount.
  Containers whose layout shifts after restoration (e.g., async
  data loads taller than the saved offset) will clamp naturally on
  the next user-initiated scroll; we did not add a re-restore on
  content-grow because it would fight the operator's manual
  scrolling.
