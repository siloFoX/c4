# TODO 8.21b -- `/api/auth/status` 401 on trailing-slash variants

**Status:** done
**Branch:** `c4/auth-fix`
**Scope:** Harden the daemon's route resolution so an unauthenticated
probe of `/api/auth/status/` (trailing slash) no longer bounces the Web
UI out of its login flow.

## Summary

`resolveApiRoute()` in `src/static-server.js` now normalizes trailing
slashes on both API and non-API paths before the route string is
compared against `auth.OPEN_API_ROUTES` and the daemon's exact-match
route table. Any proxy, browser URL canonicalizer, or client library
that appends `/` to a GET (e.g. `GET /api/auth/status/`) is now
indistinguishable from the canonical `GET /api/auth/status` as far as
the auth middleware and handler dispatch are concerned.

The fix is a pure string normalization at the single entry point every
inbound request already passes through. It does not touch
`auth.checkRequest`, `OPEN_API_ROUTES`, or any individual handler, so
the authenticated surface stays exactly as wide as before.

## Reproduction (before the fix)

With the daemon running at `127.0.0.1:3456` and `auth.enabled = true`:

```
$ curl -is http://127.0.0.1:3456/api/auth/status  | head -1
HTTP/1.1 200 OK

$ curl -is http://127.0.0.1:3456/api/auth/status/ | head -1
HTTP/1.1 401 Unauthorized
```

The React app ships `fetch('/api/auth/status')` with no trailing slash,
but any of the following can silently rewrite that to
`/api/auth/status/`:

- an `nginx` or Cloud Run front end with a default `merge_slashes` /
  canonical redirect,
- an operator testing through a browser address bar that auto-appends
  `/`,
- a downstream client library using `URL` normalization before the
  request leaves the process.

Once the status probe 401s, `fetchAuthStatus()`'s fail-safe returns
`{ enabled: true }` (from TODO 8.21 / 4aa808b). That *is* the correct
UX -- but the UI renders the login card even though the backend would
have accepted the user. Any follow-on login attempt hits the same
trailing-slash trap for the subsequent `/api/auth/login` call the user
reported anecdotally as "Web UI login silently broken in production."

## Root cause

```js
// src/static-server.js -- before
function resolveApiRoute(rawPath) {
  const p = rawPath || '/';
  const isApi = p === '/api' || p.startsWith('/api/');
  const route = isApi ? (p.slice(4) || '/') : p;
  return { isApi, route };
}
```

`/api/auth/status/` therefore resolved to `{ isApi: true, route:
'/auth/status/' }`. `OPEN_API_ROUTES` contains the canonical form
without a trailing slash, so `auth.checkRequest` fell through to the
"no bearer -> 401" branch. The same mismatch affected every exact-match
handler in `daemon.js` (e.g. `route === '/auth/login'`) -- the
trailing-slash variant would have silently 404'd instead of reaching
the login handler.

## Fix

`resolveApiRoute` now strips one or more trailing slashes from the
resolved route, unless the route is `/` itself:

```js
// src/static-server.js -- after
function resolveApiRoute(rawPath) {
  const p = rawPath || '/';
  const isApi = p === '/api' || p.startsWith('/api/');
  let route = isApi ? (p.slice(4) || '/') : p;
  if (route.length > 1 && route.endsWith('/')) {
    route = route.replace(/\/+$/, '') || '/';
  }
  return { isApi, route };
}
```

This is the minimum-surface change:

- `OPEN_API_ROUTES` semantics are untouched; only the canonical key
  shape (no trailing slash) needs to match.
- Every protected route stays protected -- `/api/list/` still
  normalizes to `/list` and `checkRequest` still 401s anonymous
  callers.
- Exact-match handler dispatch in `src/daemon.js` also benefits:
  `/api/auth/login/`, `/api/health/`, `/api/dashboard/` etc. now reach
  the intended handler instead of falling through to the 404 tail.

## Verification

After the fix, repeating the reproduction yields:

```
$ curl -is http://127.0.0.1:3456/api/auth/status  | head -1
HTTP/1.1 200 OK

$ curl -is http://127.0.0.1:3456/api/auth/status/ | head -1
HTTP/1.1 200 OK
```

Protected routes stay protected:

```
$ curl -is http://127.0.0.1:3456/api/list  | head -1
HTTP/1.1 401 Unauthorized

$ curl -is http://127.0.0.1:3456/api/list/ | head -1
HTTP/1.1 401 Unauthorized
```
