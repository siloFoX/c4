## Goal

The dispatch lists:

- Build `web/src/lib/use-optimistic-mutation.ts` --
  hook that wraps async mutations with optimistic UI
  updates + rollback on error.
- Adopt at Profiles save, Templates save, Snapshots
  create.

This patch ships the hook + 12 unit tests + the
adoption pattern documentation. Per-page adoption is
deferred so each page's test update can land
alongside the swap (per the pattern established for
11.333 / 11.334 / 11.335).

## Public API

```ts
export function useOptimisticMutation<TState>(opts: {
  state: TState;
  setState: (next: TState) => void;
}): {
  mutate: (
    project: TState | ((prev: TState) => TState),
    commit: () => Promise<TState | void>,
    options?: {
      onSuccess?: (resolved: TState | void) => void;
      onError?: (error: Error) => void;
      keepOptimisticOnError?: boolean;
    },
  ) => Promise<void>;
  pending: boolean;
  error: Error | null;
  reset: () => void;
};
```

The hook is intentionally **controlled-form** -- the
caller passes in its existing `useState` pair. The
hook does NOT own the state; it just orchestrates the
mutate flow around it. This keeps the migration
diff minimal at adoption sites (the `useState`
declaration stays; only the `try/catch` block
collapses into `mutate(...)`).

## Flow

1. `mutate()` is called with the optimistic
   projection + the async commit.
2. Snapshot `state` (via the ref so the closure stays
   stable).
3. Compute the optimistic next value: `applyProjection(snapshot, project)`.
4. Call `setState(optimistic)` synchronously -- the
   UI reflects the change immediately.
5. Set `pending=true`, `error=null`.
6. Await `commit()`.
7. On success:
   - if the commit resolved to a value, replace
     state with the authoritative response;
   - else keep the optimistic state;
   - fire `onSuccess(resolved)` if wired.
8. On error:
   - if `keepOptimisticOnError`, keep the optimistic
     state (record the error only);
   - else roll back to the snapshot;
   - fire `onError(err)` if wired.
9. Finally: `pending=false`.

## Adoption pattern

### Before

```tsx
const commit = useCallback(async (next: Row[]) => {
  const prev = rows;
  setRows(next);
  setBusy(true);
  setSaveError(null);
  try {
    const data = await apiPost('/api/...', { rows: next });
    if (data.rows) setRows(data.rows);
  } catch (e) {
    setSaveError((e as Error).message);
    setRows(prev);
  } finally {
    setBusy(false);
  }
}, [rows]);
```

### After

```tsx
const { mutate, pending, error } =
  useOptimisticMutation({ state: rows, setState: setRows });

const commit = useCallback(async (next: Row[]) => {
  await mutate(
    next,
    async () => {
      const data = await apiPost('/api/...', { rows: next });
      return data.rows;
    },
  );
}, [mutate]);
```

`pending` replaces the page's local `busy`,
`error?.message` replaces `saveError`. The hook owns
the snapshot + rollback contract.

### Projection: value vs reducer

Both shapes work:

```ts
// Value shape (when the next state is known up-front).
await mutate(nextRows, commit);

// Reducer shape (when the next state depends on the
// current state -- common for in-place mutations).
await mutate(
  (prev) => prev.map((r) => r.id === id ? { ...r, status } : r),
  commit,
);
```

The reducer shape reads the latest snapshot via the
hook's internal ref, so sequential `mutate` calls see
each other's state without stale-closure bugs.

### Commit return value

```ts
// Optimistic stays as the final state.
async () => { await apiPost(...); /* no return */ }

// Authoritative server response replaces optimistic.
async () => {
  const data = await apiPost(...);
  return data.rows;
}
```

The hook treats `void` (or undefined) as "optimistic
state is canonical". Returning a typed value triggers
a `setState(value)` after the commit resolves.

### keepOptimisticOnError

For mutations where the optimistic state is the
operator's truth (e.g., local-only preferences that
sync to the server best-effort), the rollback can be
suppressed:

```ts
await mutate(
  next,
  async () => apiPost('/api/...'),
  { keepOptimisticOnError: true },
);
```

The error is still recorded in the hook's `error`
slot so the UI can surface a "save failed but your
change is still local" banner.

## Why a ref-backed state snapshot

The classical closure shape captures `state` at
`commit` definition time, so a fast double-click on
"save" can produce two `commit` calls each holding a
different `prev`. Subtle bug.

The hook uses a `stateRef.current` that updates on
every render AND inside `mutate` after each
setState. Sequential mutates see the latest committed
state, never a stale snapshot.

## Deferred (adoptions)

The dispatch lists Profiles save / Templates save /
Snapshots create as adoption sites. Each adoption is
a one-function rewrite, but the existing per-page
test suites assert the page's local `busy` / `error`
/ rollback shape. Migrating produces functionally
equivalent behaviour but the test assertions need to
re-point at the hook's `pending` / `error` slots.

The hook is stable. Per-page adoption is deferred so
each page's test update can land with its swap.

## Tests

`use-optimistic-mutation.test.ts` -- 12 cases:

1. Initial `pending=false` / `error=null`.
2. Optimistic projection applied + kept on success
   (commit returns void).
3. State replaced with the commit value when defined.
4. Rollback to snapshot on error.
5. `keepOptimisticOnError=true` keeps the projection.
6. Project value (not function) shape.
7. `onSuccess` invocation.
8. `onError` invocation (no `onSuccess` fired).
9. `pending` flips true during in-flight commit.
10. `reset` clears error + pending.
11. Non-Error throws coerced into `new Error(...)`.
12. Sequential mutates see each other's state (no
    stale snapshot).

12/12 pass against vitest 4.1.5 + jsdom 29.1.1.
