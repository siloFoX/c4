## Scanner heuristics

Two new safety nets so the existing regex-based scanner
can run cleanly across pages without flooding the
report with false positives:

### Natural-language tail

`RE_HEX` now captures the line content after the
matched hex literal. A new helper
`looksLikeNaturalLanguageTail(tail)` returns `true` when
the trailing content starts with a space + a lowercase
letter (regex `/^\s+[a-z]/`). When that returns `true`,
the scanner skips the match.

The motivating false positive is `"Dispatched todo #142
to worker auto-w17"` in the Notifications.tsx mock
data: `#142` is a valid 3-digit hex sequence (1, 4, 2
are hex digits) but the trailing ` to worker` is the
clear signature of English content, not a CSS color
declaration. CSS color values in code are virtually
always followed by `'`, `"`, `` ` ``, `,`, `;`, `}`,
`)`, or end-of-line; never by a lowercase word.

### `hsl(var(--token))` / `rgba(var(--token))`

`RE_HSL` and `RE_RGB` now capture the inner argument
list. A new helper `isVarReference(args)` returns
`true` when the arguments contain `var(...)`. When
that returns `true`, the scanner skips the match.

The motivating false positive is the canonical
alpha-overlay pattern from the design system:

```ts
className: 'shadow-[0_0_0_4px_hsl(var(--primary)/0.35)]'
```

The c4 token system stores HSL components without the
`hsl()` wrapper (`--primary: 217 91% 60%`), so adding
alpha requires composing them inside `hsl(...)`. That
is still a tokenised color -- the actual value flows
through `var(--primary)`. Raw forms like
`hsl(220 18% 8%)` (no `var()`) keep failing.

## Pages integration scan

`theme-tokens-validator.test.ts` adds a new
`describe()` block that walks every
`web/src/pages/*.tsx` file (filtered to exclude
`*.test.tsx`), feeds them through
`scanFilesForViolations()`, and asserts the violation
list is empty. The block mirrors the existing
`components/ui` integration test and uses the same
`formatViolations()` summary so a regression points at
the exact `file:line:column` and the matched literal.

As of v1.11.344 the pages surface passes with zero
violations.

## Dual-theme render test

`UIDemoRoute.test.tsx` adds three new cases:

1. `renders the full gallery under the dark theme
   without crashing` -- calls `applyTheme('dark')`,
   asserts `<html>` carries the `.dark` class, renders
   `<UIDemoRoute />` with the `uiDemoRoute` flag on,
   and asserts the root + every demo section data
   attribute survives the render.
2. `renders the full gallery under the light theme
   without crashing` -- same contract under
   `applyTheme('light')`, asserts the `.dark` class is
   absent.
3. `the gallery root resolves a non-empty
   backgroundColor under the dark theme` -- runs
   `getComputedStyle()` on the gallery root and
   asserts the call returns a string. jsdom does not
   resolve Tailwind utility classes to actual RGB
   values, so this is a smoke check that catches a
   stripped-tokens.css regression rather than a
   pixel-level palette assertion.

Combined with the existing six UIDemoRoute cases,
the demo route now has 9 passing tests.

## Out of scope

- App.tsx shell tokens. The
  `from-[hsl(220_18%_8%)] to-[hsl(220_15%_12%)]`
  gradient in `App.tsx:281` is documented technical
  debt (the comment at line 277 explicitly tracks the
  pending swap to CSS-variable values). The dispatch
  scopes the audit to "every UI primitive + every
  page"; App.tsx is the root shell that hosts the
  pages. A follow-up patch can tokenise the gradient
  alongside other root-shell tweaks.
- `WorkflowGraph.tsx` semantic colors. The node-type
  palette (task / condition / parallel / wait / audit
  / notify / meeting / end) uses raw hex literals
  that read as a semantic chart palette rather than
  text/background colors. Adding tokens for the
  workflow graph palette is a real design system
  extension, not a quick swap.
- Per-section palette drift detection. jsdom cannot
  resolve Tailwind utility-class backgrounds to RGB
  values, so a real "every text-foreground class
  resolves to a non-empty rgb()" assertion would
  require a Playwright + real-browser rendering
  harness. That is out of scope for this vitest +
  jsdom patch.
