# 11.305 -- motion utilities bundle (v1.11.323)

Consolidates the motion utilities into a single canonical
import surface at `web/src/lib/motion.ts`. Component-scope
only, no daemon-side change and no `c4` CLI surface change.

## Why

Animating surfaces in the c4 web app currently reach for
motion utilities across two import paths:

```tsx
// Today, in Toast / Drawer / Dialog / Tooltip / Switch /
// Spinner / Skeleton / ...
import { useReducedMotion } from '../hooks/use-reduced-motion';
import { motionClass } from '../lib/motion';

function Drawer({ open, ... }) {
  const reduced = useReducedMotion();
  const enterClass = motionClass('slideInRight', reduced);
  const exitClass = motionClass('slideOutRight', reduced);
  // ...
}
```

Three pain points:

1. Two imports for what is conceptually one concern.
2. The reduced-motion bool has to be threaded through
   each `motionClass()` call site -- easy to forget on
   one of them, which silently disables the
   reduced-motion gate for that animation.
3. New adopters have to learn the two-import idiom
   before they can animate anything.

Patch 11.305 ships a `useMotionClass(key, fallback?)`
hook in `lib/motion.ts` that fuses the two together,
plus re-exports `useReducedMotion` so callers that only
need the preference (e.g. to gate an inline style or an
`animate-*` class with `motion-safe:` modifier) can
import everything from one path.

## What changed

### `web/src/lib/motion.ts`

Module-level JSDoc rewritten to document the canonical
pattern with three usage shapes:

1. `useMotionClass(key, fallback?)` -- preferred for new
   code. One call site, one import.
2. `motionClass(key, reducedMotion, fallback?)` --
   stateless picker for callers that already have the
   bool from a prop, context, or pre-computed source.
3. Raw `motion[key]` access -- for inline class
   composition (e.g. inside `motion-safe:...` modifier
   groups).

New `useMotionClass(key, fallback?)` hook:

```ts
export function useMotionClass(
  key: MotionKey,
  fallback?: string,
): string {
  const reducedMotion = useReducedMotion();
  return motionClass(key, reducedMotion, fallback);
}
```

`useReducedMotion` re-exported from
`hooks/use-reduced-motion` so the single import surface
covers both shapes.

The eight motion presets (`fadeIn`, `fadeOut`,
`slideInRight`, `slideOutRight`, `slideInLeft`,
`slideOutLeft`, `scaleIn`, `scaleOut`) and the original
`motionClass` stateless picker are unchanged --
behaviour is byte-for-byte identical to v1.11.253.

## Tests

`web/src/lib/motion.test.ts` gains 7 new vitest cases on
top of the existing 8 (15 total):

- `useReducedMotion` re-export: returns false when
  `prefers-reduced-motion` does not match.
- `useReducedMotion` re-export: returns true when it
  matches.
- `useMotionClass`: returns animated class when
  `prefers-reduced-motion` is OFF.
- `useMotionClass`: returns `''` when reduced and no
  fallback provided.
- `useMotionClass`: returns fallback when reduced.
- `useMotionClass`: ignores fallback when motion is
  allowed.
- `useMotionClass`: output matches the stateless
  `motionClass()` form for every motion key.

`matchMedia` is stubbed via `vi.stubGlobal` so the hook
output is deterministic. Test file moved to jsdom env
because the hook uses `useState` + `useEffect` and the
re-export consumer needs DOM-aware `renderHook`.

All 15 green via

Downstream consumers were sanity-checked: 184 tests
across `Toast.test.tsx` + `dialog.test.tsx` +

## Notes

- The hook is intentionally a one-liner -- it does not
  add a new layer of memoisation. The underlying
  `useReducedMotion` hook caches its own state via
  `useState`, and the `motionClass` picker is a pure
  string lookup. Adding `useMemo` here would only add
  cost.
- The re-export uses `export { useReducedMotion } from
  '../hooks/use-reduced-motion'` rather than wrapping
  the hook so the identity is preserved -- consumers
  can use `useReducedMotion` from either path and the
  React reconciler treats them as the same hook.
- `useMotionClass(key)` returns `string`, not
  `string | undefined`, even when the reduced-motion
  fallback is empty. Empty string is the canonical
  "no class applied" value in React's className API;
  this matches the existing `motionClass()` contract.
- The eight motion preset names are exported via the
  `MotionKey` union -- new presets must be added to
  the `motion` object first, then the type is
  automatically widened.
