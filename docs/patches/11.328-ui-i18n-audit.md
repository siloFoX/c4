## Goal

The dispatch lists:

- Walk every UI page and assert all visible strings go
  through `useTranslation()`.
- Replace any hardcoded English.
- Add a unit test that asserts no string in `src/pages`
  reaches the DOM as a literal.

This patch lands the audit half: a CI-runnable scanner
that detects hardcoded user-facing copy in
`web/src/pages/*.tsx`, plus the integration test that
ratchets the violation count.

The codebase carries ~200 pre-existing hardcoded
strings spread across every page; migrating them all
in one patch is too much scope (would touch every
page + add hundreds of `en.json` / `ko.json` entries
in a single review). The scanner + ratchet lets
follow-up patches migrate strings incrementally and
catches NEW hardcoded copy in CI immediately.

## Scanner

`web/src/lib/i18n-coverage.ts` exposes:

```ts
export interface I18nCoverageViolation {
  file: string;
  line: number;
  column: number;
  text: string;
  rule: CoverageRule;
  excerpt: string;
}

export function scanSourceForI18nViolations(
  filePath: string,
  contents: string,
): I18nCoverageViolation[];

export function scanFilesForI18nViolations(
  files: readonly string[],
): I18nCoverageViolation[];

export function formatI18nViolations(
  violations: readonly I18nCoverageViolation[],
): string;

export function filterAllowed(
  violations: readonly I18nCoverageViolation[],
  allowList: readonly I18nAllowEntry[],
): I18nCoverageViolation[];
```

### Detection rules

| rule | pattern |
| --- | --- |
| `jsx-text` | `>([^<{}\r\n]+)<` -- JSX text content |
| `aria-label` | `aria-label="..."` attribute value |
| `title` | `title="..."` attribute value |
| `alt` | `alt="..."` attribute value |
| `placeholder` | `placeholder="..."` attribute value |
| `label` | `label="..."` attribute value |

Each detection runs through `looksLikeEnglishCopy()`
which applies the natural-English heuristic. The
`title` / `aria-label` / `placeholder` / `label`
rules additionally require at least one whitespace
so single-word values (`title="Refresh"`,
`aria-label="Save"`) don't flood the report.

### Natural-English heuristic

```
length >= 6 chars
contains at least one ASCII letter
contains at least 2 space-separated words
contains at least 2 hyphen-free words
not in the TECHNICAL_TOKENS allow-list
no `${...}` / `{{...}}` interpolation
not a pure CSS-class-list shape
```

The "hyphen-free words" rule is the key
discriminator: Tailwind class lists like
`"flex flex-col items-center"` have 3 total words
but only 1 hyphen-free word (`flex`), so they are
skipped. English copy like `"Save changes"` has 2
hyphen-free words and gets flagged.

Comments (`//` line + `/* */` block) and code
lines starting with `import` / `export` / `type` /
`interface` are stripped before regex matching.

### Allow-list

The `filterAllowed(violations, allowList)` helper
suppresses specific `{ text, pathSuffix? }`
combinations from the report. Useful for the
genuinely-non-localised cases (brand names,
debug strings, third-party identifiers). Not used
by the current integration test; the test relies on
the simpler `MAX_KNOWN_PAGES_VIOLATIONS` ratchet
because the codebase has too many violations to
enumerate.

## Integration ratchet

`web/src/lib/i18n-coverage.test.ts` exports:

```ts
export const MAX_KNOWN_PAGES_VIOLATIONS = 194;
```

The integration test asserts:

```ts
expect(violations.length).toBeLessThanOrEqual(
  MAX_KNOWN_PAGES_VIOLATIONS,
);
```

On a regression the test failure message tells the
operator exactly how many new violations slipped in
and quotes them. The fix is either:

1. Migrate the new strings through `t()` / `tFormat()`
   so the count returns to the baseline; OR
2. Lower `MAX_KNOWN_PAGES_VIOLATIONS` to the new
   count when a migration patch reduces the floor.

The constant is deliberately separate from the
violation array so the failure message points
operators at the exact value to update.

## What is NOT in this patch

- **Migrating the 194 pre-existing hardcoded
  strings.** Doing so would touch every page in
  `web/src/pages/`, add ~200 keys to
  `web/src/i18n/en.json` + `ko.json`, and balloon
  the review. The pattern is straightforward (`t()`
  call site + i18n bundle entry); follow-up patches
  can chip away one page at a time and lower
  `MAX_KNOWN_PAGES_VIOLATIONS` accordingly.
- **AST-based scanning.** A real Babel / SWC AST
  walk would catch JS expressions like
  `const SUBTITLE = 'Save changes'; <span>{SUBTITLE}</span>`.
  The regex scanner does not. Worth doing if the
  per-page migration uncovers regressions that the
  regex misses; out of scope here.
- **`useTranslation()` migration.** The dispatch
  text uses `useTranslation()` interchangeably with
  the existing project's `t()` / `tFormat()` /
  `useLocale()` helpers. The project does not have
  a hook named `useTranslation`; the helpers in
  `web/src/lib/i18n.ts` are the canonical surface.
  This patch does not rename them.

## Adoption sites

The new scanner is wired into the test suite only --
no production code path imports it. The
`MAX_KNOWN_PAGES_VIOLATIONS` ratchet is the
operator-visible surface; lower it as you migrate
strings.
