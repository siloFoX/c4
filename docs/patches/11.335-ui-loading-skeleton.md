## Goal

The dispatch lists:

- Build `web/src/lib/use-loading-skeleton.ts` -- shared
  hook that returns skeleton placeholders based on
  loading state with min-display-time (avoid flash).
- Adopt at HistoryView, Workers, Sessions, Templates.

This patch ships the hook + 12 unit tests + the
adoption pattern documentation. Per-page adoption is
deferred to follow-up patches so each page's test
update can land alongside the swap.

## Why two gates

The naive pattern has two failure modes:

1. **Flash-of-skeleton.** A fast response (sub-100ms)
   flashes the skeleton for one or two frames then
   replaces it with content. The user perceives a
   glitch.
2. **Flash-of-content.** A slow response that almost
   finishes during the skeleton's first paint replaces
   it instantly. The user sees the skeleton blink out
   the moment it appeared.

The hook orchestrates two gates that defeat both:

- `showAfterMs` (default **120**) -- only START
  showing the skeleton if loading is STILL true after
  this delay. Sub-120ms responses never paint a
  skeleton.
- `minDisplayMs` (default **300**) -- once shown,
  KEEP showing for at least this many ms even if
  loading clears. The transition is always
  skeleton -> content, never skeleton-flash ->
  content.

## API

```ts
export function useLoadingSkeleton(
  loading: boolean,
  options?: { showAfterMs?: number; minDisplayMs?: number },
): boolean;

export const LOADING_SKELETON_DEFAULT_SHOW_AFTER_MS:  120;
export const LOADING_SKELETON_DEFAULT_MIN_DISPLAY_MS: 300;
```

## Adoption pattern

```tsx
// Before:
return loading ? <Skeleton /> : <Content />;

// After:
import { useLoadingSkeleton } from '../lib/use-loading-skeleton';

const showSkeleton = useLoadingSkeleton(loading);
return showSkeleton ? <Skeleton /> : <Content />;
```

For compound predicates (the "skeleton only when we
have nothing else to show" pattern):

```tsx
const showSkeleton = useLoadingSkeleton(
  loading && items.length === 0,
);
return showSkeleton ? <Skeleton /> : <Content />;
```

Per-page test files that drive loading state
explicitly need to advance fake timers past the gate,
OR pass `showAfterMs: 0` (legacy synchronous shape):

```tsx
const showSkeleton = useLoadingSkeleton(loading, {
  showAfterMs: 0,
  minDisplayMs: 0,
});
```

## Behaviour table

| loading transition | timing | result |
| --- | --- | --- |
| `false -> true` then `true -> false` within `showAfterMs` | 50ms total | skeleton never shows |
| `false -> true` held past `showAfterMs` | t=120ms | skeleton shows |
| skeleton visible, `loading -> false` immediately | t=130ms (10ms after show) | skeleton stays until t=420ms (300ms min-display) |
| skeleton visible, `loading -> false` after `minDisplayMs` | t=500ms | skeleton hides on next tick |
| `loading` flips back to true during the min-display window | -- | pending hide cancelled; skeleton stays |
| `loading` flips back to false during the show-after window | -- | pending show cancelled; skeleton never appears |
| unmount mid-window | -- | both timers cleared (no setState-after-unmount) |

## Deferred (adoptions)

The dispatch lists four adoption sites:

- **HistoryView** -- per-row loading skeleton.
- **Workers** (page) -- per-worker tile skeleton.
- **Sessions** -- per-session card skeleton.
- **Templates** -- per-template row skeleton.

Each adoption is a one-line swap (the `loading ? <Skeleton /> : <Content />`
expression is replaced with the hook's return value),
BUT each page's existing test suite asserts the
skeleton appears synchronously after `loading=true`.
The hook's default `showAfterMs=120` breaks that
assumption -- the test must either:

1. Wrap the assertion in `vi.useFakeTimers()` +
   `vi.advanceTimersByTime(120)`; or
2. Pass `{ showAfterMs: 0 }` to keep the legacy
   synchronous shape (but lose the flash-of-skeleton
   protection).

The hook is shipped and stable. Per-page adoption is
a routine swap that should land with its corresponding
test update; rolling all four into this patch would
balloon the diff to four page files + four test
files. Each adoption is a one-PR follow-up.

## Tests

`use-loading-skeleton.test.ts`:

- 12 unit cases covering:
  - initial render returns `false`;
  - default delay constants exposed;
  - sub-`showAfterMs` responses never show;
  - past `showAfterMs` flips the skeleton on;
  - `minDisplayMs` hold;
  - post-`minDisplayMs` immediate hide;
  - `showAfterMs` override;
  - `minDisplayMs` override;
  - mid-window flip up cancels hide;
  - mid-window flip down cancels show;
  - consecutive cycles each gate independently;
  - unmount clears pending timers.

12/12 pass against vitest 4.1.5 + jsdom 29.1.1 with
`vi.useFakeTimers()`.
