trend versus range from price travel.

## Indicator math

The Vertical Horizontal Filter (Adam White) decides whether a
market is trending or ranging from the geometry of the price
path.

- `computeLineVhfWindow(window)` -- score a single window. The
  `range` is the net travel of the close (its highest value
  minus its lowest); the `travel` is the total distance walked
  (the sum of the absolute bar-to-bar moves); the `vhf` is
  `range / travel`. The VHF sits in `[0, 1]` because the net
  displacement can never exceed the total path.

- `computeLineVhf(values, period)` -- the rolling estimate. For
  each bar the trailing window of `period` closes is scored:

```
VHF[i] = (highest close - lowest close) / sum(abs bar-to-bar move)
```

  A VHF near 1 marks a trending market -- the price moved in one
  direction so the range nearly equals the path; near 0 a
  ranging market -- the price travelled far but went nowhere. A
  flat window has a zero travel and an undefined VHF and is
  null, as are bars before the window is full.

### Worked fixture

The VHF is a ratio of a max-minus-min and a sum of absolute
moves -- no transcendentals -- so the whole pipeline is exact.
The fixture (`period 4`) is hand-tuned so every window's travel
is exactly 16, hence every VHF divides cleanly by 16:

```
vhf  [., ., ., 1, 1, 0.625, 0.375, 0.375, 0.5, 0.375, 0.375, 0.5]
```

A monotone window has `range = travel`, so it reads exactly 1; a
zigzag window travels far but spans little, so it reads low; a
flat window reads null.

## API

- `getLineVhfFinitePoints(points)` -- drop points with a
  non-finite `x` or `value`.
- `normalizeLineVhfPeriod(period, fallback)` -- coerce to an
  integer of at least 2.
- `computeLineVhfWindow(window)` -- the `{ range, travel, vhf }`
  score of a single window.
- `computeLineVhf(values, period)` -- the rolling VHF series,
  null through the warm-up and for flat windows.
- `runLineVhf(points, options?)` -- sort finite points by `x`,
  run the pipeline, return `{ series, period, vhf, samples,
  vhfFinal, vhfMin, vhfMax, trendingCount, rangingCount, ok }`.
  Each sample is classified `trending` / `ranging` / `neutral`.
  `ok` is false for fewer than two points.
- `computeLineVhfLayout(options)` -- stack the price panel above
  a VHF panel with a fixed `[0, 1]` y-domain; project the price
  and VHF paths plus markers and the 0.5 reference line.
- `describeLineVhfChart(data, options?)` -- ARIA description.

## Component

`ChartLineVhf` is a `forwardRef` region (`role="region"`) with
an ARIA-linked description, a `role="img"` SVG, two stacked
panels (price on top, VHF below) with their own axis ticks and
grid, panel labels, a config badge showing the period, a
two-series legend (Price / VHF) with toggle, a hover/focus
tooltip exposing the value, VHF and regime, and a
`motion-safe:animate-fade-in` entrance. It consumes
`{ x, value }` points. The VHF panel has a fixed `[0, 1]`
y-domain with a dashed 0.5 reference line and regime-coloured
markers (green trending, red ranging, grey neutral). Every node
carries a `data-section` attribute.

## Distinctness

- **vs the deviation oscillators (cog / pgo / tii)** -- those
  measure the price relative to a reference (a moving average or
  a centroid). The VHF measures the *geometry of the price path*
  itself: net displacement over total distance walked. It needs
  no moving average.
- **vs chart-line-hurst** -- both are fixed `[0, 1]` gauges, and
  both judge trend versus chop, but the Hurst exponent uses
  rescaled range analysis (a statistical scaling law) while the
  VHF is a direct path-geometry ratio.
- **vs chart-line-tii** -- the TII reports the directional bias
  of deviations; the VHF reports how efficiently the price
  travelled, regardless of direction.
