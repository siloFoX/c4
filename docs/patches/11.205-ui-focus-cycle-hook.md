# 11.205 -- UI: reusable focus-cycle hook

`web/v1.11.223`. Component scope only.

## Summary

A new `useFocusCycle` hook centralises ARIA roving-tabindex arrow-key
navigation so UI primitives stop reinventing it. The hook is wired up
in `tabs.tsx` and left available for the other primitives that
currently use a different (and deliberately preserved) pattern.

## Public API

```ts
useFocusCycle({
  containerRef: RefObject<HTMLElement>,
  itemSelector?: string,
    // default: '[role=menuitem], [role=tab], [role=option], button:not([disabled])'
  orientation?: 'horizontal' | 'vertical' | 'both',  // default 'vertical'
  wrap?: boolean,                                    // default true
  onSelect?: (el: HTMLElement) => void,
}): {
  handleKeyDown: (e: KeyboardEvent<HTMLElement>) => void,
  focusFirst: () => void,
  focusLast: () => void,
  focusIndex: (idx: number) => void,
}
```

- `handleKeyDown` consumes `ArrowDown`/`ArrowUp` (vertical),
  `ArrowLeft`/`ArrowRight` (horizontal), or all four (`'both'`), plus
  `Home`/`End`. `Tab` is passthrough.
- Items with the `disabled` attribute, `aria-disabled="true"`,
  `aria-hidden="true"`, or `[hidden]` are filtered out at query time.
- `wrap=true` cycles last->first / first->last; `wrap=false` clamps
  at the boundary.
- `onSelect` fires after focus moves so adapters can mirror focus
  into selection state.
- When focus is outside the matched item set, `ArrowDown` lands on
  the first item and `ArrowUp` on the last.

## Refactor scope

The task asks to wire the hook into four primitives **if** their
inline pattern already matches the hook's contract. Per the
"keep behaviour + tests identical" rule and the "if a primitive
currently has NO inline focus cycling, skip it and document"
escape hatch, only `tabs.tsx` was refactored. The other three are
deferred for a follow-up that expands the hook's contract -- the
rationale is in [Deferred primitives](#deferred-primitives) below.

### tabs.tsx (refactored)

The `Tabs` component used a hand-rolled `handleKeyDown(idx)` factory
that walked a memoised `enabledIndices` array and called
`focusAndSelect(nextIdx)` to focus the next tab **and** fire the
`onChange` prop. The refactor:

- Replaces the per-button factory with a single `handleKeyDown`
  returned by `useFocusCycle({ orientation: 'horizontal', wrap: true,
  itemSelector: '[role=tab]:not([disabled])' })`.
- Adds a `ref` to the tablist `<div>` (used as `containerRef`).
- Tags each tab `<button>` with `data-tab-value={item.value}` so the
  hook's `onSelect(el)` callback can resolve the focused element
  back to the original `TabsItem` and call the `onChange` prop --
  preserving the focus+select coupling the existing tests assert.
- Drops the per-button index `refs` array. The `tabIndex={active ? 0
  : -1}` roving-tabindex setup is unchanged.

All 21 existing tabs tests pass unmodified, including:
- ArrowRight / ArrowLeft moving selection + focus.
- Wrap-around on first and last tabs.
- `Home` / `End` jumps.
- Disabled-tab skipping during arrow navigation.
- The "arrow keys are a no-op when focus is outside the strip" case
  (still passes because the hook is attached to each tab's
  `onKeyDown`, not a document-level listener).

### Deferred primitives

`dropdown-menu.tsx`, `context-menu.tsx`, and `popover.tsx` are
**intentionally left in place**. Their cycling logic is structurally
different from the hook's contract -- a refactor would change
behaviour, not just consolidate it:

- **Document-level `keydown` listener.** All three attach their
  arrow-key handler to `document` (or the global `window`) while the
  menu is open, so a key pressed anywhere on the page navigates the
  menu. `useFocusCycle` returns a `handleKeyDown` meant to be wired
  to the container `onKeyDown` (or a child item's `onKeyDown`); it
  has no opinion about the document scope and would silently break
  the document-level dismiss/navigation behaviour the existing tests
  rely on.
- **Type-ahead.** `dropdown-menu.tsx` runs a 500 ms single-letter
  type-ahead buffer driven by a `setTimeout` ref. The hook has no
  type-ahead surface today; folding it in would expand the hook's
  contract beyond what the task scopes.
- **`aria-activedescendant` highlight.** The menus track a
  `highlight: number` in component state and render it via
  `aria-activedescendant` on the menu container. The hook focuses
  the DOM element directly (true roving tabindex). Both are valid
  ARIA patterns, but they are not interchangeable -- swapping one
  for the other is an a11y behaviour change and would need a new
  test pass.
- **popover.tsx has no inline focus cycling.** The task explicitly
  says "If a primitive currently has NO inline focus cycling, skip
  it and document". `popover.tsx` matches that case: no `ArrowDown`/
  `ArrowUp`/`Home`/`End` handler at all.

Adapting any of these three would either widen the hook's contract
(typeahead + activedescendant) or change observable behaviour
(document-level vs container-level keydown). Both are out of scope
for "build a reusable hook and wire up where the inline pattern
already matches". A follow-up patch can extend the hook with a
`mode: 'roving' | 'activedescendant'` switch and an optional
`typeahead: { resetMs }` block, at which point dropdown / context
menus can adopt it without behaviour drift.

## Accessibility

Follows the ARPS design system a11y patterns documented in
`arps-design-system-v1/USAGE.md`: focus is moved with
`HTMLElement.focus()`; disabled / hidden items are filtered before
selection so screen-readers never announce a phantom focus jump; the
hook does not touch the active element when no items match (so it
composes safely with a `useFocusTrap` wrapper). The roving-tabindex
contract (`tabIndex={active ? 0 : -1}`) is left to the consumer --
the hook only handles arrow-key dispatch.

## Tests

Hook (`use-focus-cycle.test.tsx`, 11 cases):

1. ArrowDown moves focus 0 -> 1 -> 2 (vertical).
2. ArrowUp moves focus 2 -> 1 -> 0.
3. `wrap=true`: ArrowDown from last goes to first.
4. `wrap=false`: ArrowDown from last stays on last.
5. `wrap=false`: ArrowUp from first stays on first.
6. `Home` / `End` jump to first / last.
7. Skips disabled items (`disabled` attribute is filtered out by the
   default selector).
8. `orientation='horizontal'` responds to ArrowLeft / ArrowRight,
   ignores ArrowDown / ArrowUp.
9. `orientation='both'` responds to all four arrow keys.
10. `onSelect` fires with the focused element on each move.
11. `Tab` is passthrough (no focus change, no preventDefault path).
