# 11.186 -- UI: Notifications feed page

## Why

The autonomous loop fires lifecycle webhooks (dispatch / complete / halt /
escalation) and the daemon emits system events, but operators had no in-app
surface to scroll through that stream. Slack/Discord remained the only feed
and there was no way to triage from the web UI.

## What

New `<Notifications>` page at `web/src/pages/Notifications.tsx` rendering a
full notifications feed view:

- Header: `Notifications` title + `Mark all read` button (header action slot
  on `PageFrame`). Mark-all-read is a local handler placeholder that flips
  the unread `New` badges off until the next fetch.
- Filter row: six `<Chip>` toggles (`All` / `Dispatch` / `Complete` / `Halt`
  / `Escalation` / `System`). Selected chip uses `tone='primary'` +
  `variant='solid'`; chips are wrapped in a `<button>` so the press state is
  exposed via `aria-pressed`. A `role='group'` wraps the row.
- Feed: existing `<Timeline>` primitive with `groupByDay`. Each entry shows
  the type icon (lucide), a title, optional description, and the timestamp.
  The tone-to-type map is `dispatch -> primary`, `complete -> success`,
  `halt -> warning`, `escalation -> danger`, `system -> neutral`.
- Empty state: `<EmptyState>` with a `BellOff` icon + tailored copy when no
  notifications match the active filter.
- Pagination: `Load more` button (Bell icon) under the feed when the
  filtered list exceeds the 50-item default.

## Data source

The page tries `GET /api/notifications` once on mount via the global
`fetch`. On non-2xx or thrown errors the catch arm seeds 20 inline mock
items spanning the last 7 days (mixed types, alternating read/unread) and
surfaces a `sample data` badge above the filter row so it is obvious the
content is not real. The endpoint shape expected is
`{ notifications: NotificationItem[] }` where each item has
`{ id, type, title, description?, timestamp, read? }`.

## Nav integration

Registry entry added to `web/src/pages/registry.ts` between Health and
DesignSystem in the `diagnostics` category, using the `Bell` lucide icon.
The label/description go through `t()` so en/ko bundles need a matching
pair:

- `feature.notifications.label` = `Notifications`
- `feature.notifications.description` = (en) `Unified feed of lifecycle
  events ...` / (ko) Korean translation.

## Tests

`web/src/pages/Notifications.test.tsx` (vitest + RTL) covers:

- Renders the page title.
- Renders the six filter chips.
- Mark-all-read button is present in the header.
- Timeline rows render when items are present.
- Clicking the `Halt` chip narrows the visible entries to halt-only.
- Empty state surfaces when the active filter matches no items.
- Falls back to the `sample data` badge when fetch throws.

`globalThis.fetch` is stubbed per test to avoid touching the network.

## Out of scope

- No UI primitive changes -- this is a new page only.
- Backend `/api/notifications` route lives elsewhere; this patch only wires
  the front-end consumer + sample data fallback.
- `Mark all read` is a placeholder (no POST yet).
- `Load more` simply expands the local slice; cursor pagination lands when
  the API surface is finalized.
