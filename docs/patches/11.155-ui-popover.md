# 11.155 -- UI: Popover anchored-panel primitive (v1.11.173)

## Summary

Add a `<Popover>` UI primitive at `web/src/components/ui/popover.tsx` for
lightweight anchored panels (info trigger, small overflow menu, hint
panel). Renders the panel via `createPortal` to `document.body` so it
escapes `overflow:hidden` ancestors. Smart positioning flips to the
opposite side when the chosen side overflows the viewport; jsdom's
layout-less environment cannot exercise the flip, so the tests only
assert that one of the chosen-or-opposite sides is recorded on
`data-popover-placement`.

## API

```tsx
<Popover
  trigger={<button>Open</button>}
  content={<p>Hi</p>}
  placement="bottom"
  align="center"
  offset={6}
  closeOnClickOutside
  closeOnEsc
/>
```

- `trigger`: anchor `ReactElement` -- cloned with an `onClick` toggle
  plus `aria-haspopup="dialog"`, `aria-expanded={open}`, and
  `aria-controls={panelId}` (`useId`-derived).
- `content`: `ReactNode` rendered inside the floating dialog panel.
- `open` / `defaultOpen` / `onOpenChange`: controlled + uncontrolled
  modes coexist; controlled mode is detected when `open !== undefined`.
- `placement`: `'top' | 'bottom' | 'left' | 'right'` (default
  `'bottom'`).
- `align`: `'start' | 'center' | 'end'` (default `'center'`).
- `offset`: pixel gap between trigger and panel (default `6`).
- `closeOnClickOutside`: default `true`. Listener registered in the
  `mousedown` capture phase so a click inside the panel does NOT close.
- `closeOnEsc`: default `true`.
- `className`: applied to the floating panel.

The panel is `role="dialog"`, `tabIndex={-1}`, and auto-focuses the
first focusable on open. Tab / Shift-Tab cycle inside the panel; focus
returns to the trigger on close.

## Adoption sites

1. `web/src/pages/Plan.tsx` -- info `HelpCircle` button next to the
   planning-flow `<Stepper>` opens a Popover summarising the four
   stages (select worker -> describe task -> dispatch -> view plan).
2. `web/src/pages/Config.tsx` -- info `HelpCircle` button on the
   `DispatchLifecycleDemo` panel header opens a Popover listing the
   five dispatch lifecycle stages with their descriptions.
3. `web/src/pages/Health.tsx` -- info `HelpCircle` button inline with
   the health-status badge opens a Popover noting the 10s poll cadence
   and where durable accounting lives (audit log, not the in-memory
   counters reset on daemon restart).

The originally listed targets (`WorkerActions` overflow menu, StatCard
`?` trigger, KeyboardShortcutsModal close) were not found in their
described shapes -- `WorkerActions` is a flat Button row, `StatCard`
has no info trigger, and `KeyboardShortcutsModal` is Dialog-based
(task spec keeps Modal/Dialog widgets intact). Substituted with three
info-trigger sites on pages that already use other UI primitives
(`Stepper`, `DashboardGrid`).

## Tests

`web/src/components/ui/popover.test.tsx` covers 17 cases: closed by
default; opens on trigger click; toggle on second click; closes on
Escape (default); does not close on Escape when `closeOnEsc=false`;
closes on outside `mousedown`; does not close on inside click; does
not close on outside click when `closeOnClickOutside=false`;
controlled `open` prop drives visibility; `onOpenChange` fires for
each transition; `aria-expanded` + `aria-haspopup` on trigger;
`role=dialog` + `aria-controls` link; focus moves into panel on
open; focus returns to trigger on close; className merges;
`data-popover-placement` records the chosen side (jsdom cannot
exercise the flip); stable `displayName`.

## Out of scope

- Smart-positioning flip cannot be unit-tested under jsdom (no
  layout). A browser-mode test could exercise it later.
- The primitive does not animate enter/exit. Tailwind motion-safe
  classes can be applied via `className` if a callsite wants them.
- Existing Modal/Dialog-based widgets (e.g. `KeyboardShortcutsModal`,
  `Dialog`, `ConfirmDialog`) are intentionally untouched -- Popover
  is for lightweight anchored panels only.
