# 11.319 -- Snapshots page polish (v1.11.337)

Polishes `web/src/pages/Snapshots.tsx` with an age-bucket
SegmentedControl filter and a success-path Toast on the
take-snapshot action. Component-scope only, no daemon-side
change and no `c4` CLI surface change.

## Why

Most of the dispatch's "polish" items already shipped:

- v1.11.288: FileDrop upload (the "Import from JSON"
  slot).
- v1.11.289: TimeAgo for the Created column.
- v1.11.262: Delete confirm Dialog + UndoToast.
- v1.11.275: AlertBanner for action errors.
- v1.11.282: Pagination footer.
- v1.11.284: Toolbar per-row actions.

The two remaining dispatch items were:

1. **SegmentedControl for filter.** The snapshots list can
   grow into the hundreds once an operator takes daily
   snapshots. Pagination handles the long tail, but an
   operator looking for "last week's recovery point"
   should not have to scrub through pages to find it.

2. **Toast for save/delete feedback.** Delete already
   surfaces an UndoToast (v1.11.262); the take-snapshot
   action returned silently on success -- the operator
   had to read the new row appearing in the table to
   know the save worked.

Patch 11.319 ships both.

## What changed

### `web/src/pages/Snapshots.tsx`

- **Age-bucket SegmentedControl filter.** New control
  above the snapshots table with four options:
  - `All (<count>)` -- default, no filter.
  - `<=24h (<count>)` -- snapshots created in the last
    24 hours.
  - `<=7d (<count>)` -- snapshots created in the last
    7 days (excluding `<=24h`).
  - `Older (<count>)` -- everything else.
  Each option carries a live count showing how many
  snapshots fall in the bucket before clicking.
- **Success toast on take.** The `handleTakeSubmit`
  callback now calls `showToast(...)` with the
  appropriate message after a successful save:
  - When a label is supplied: `Snapshot "X" saved.`
  - When no label is supplied: `Snapshot saved.`
  The toast appears in the standard top-right slot
  rendered at the bottom of the page. Failure paths
  still flow through the AlertBanner (`actionError`)
  for higher visibility -- the toast is the success
  channel only.
- **Age-filtered pagination.** Pagination now reads
  the age-filtered list length so a filter switch
  correctly reflows the pagination state. Both the
  row slice and the pagination total compute against
  the filtered set.
- **Helper export.** New `classifyAge(iso)` helper
  alongside the page returns `'recent' | 'week' |
  'older'` for any ISO timestamp. The helper is
  page-scoped (not exported from the module) so the
  surface stays minimal.

## Deferred (dispatch follow-ups)

- The dispatch listed **DataList for snapshot rows.**
  The current `<table>` layout conveys all five
  columns (Label / ID / Created / Size / Actions)
  with full keyboard + screen-reader support, and
  swapping to DataList would lose the columnar
  comparison. Deferred pending a follow-up
  evaluation -- the dispatch's intent (richer row
  metadata) is already served by the existing
  table.
- **ConfirmDialog for delete** and **FileDrop
  upload** were already shipped in earlier patches
  (v1.10.564 + v1.11.262 for delete + UndoToast,
  v1.11.288 for FileDrop). Those surfaces are
  preserved.

## Notes

- The age-bucket cutoffs (24h, 7d) were picked to
  match the operator's mental "freshness" categories
  -- snapshots taken today (recovery from a recent
  mistake), this week (rollback target after a
  branch merge), and older (long-term archives).
  The cutoffs are constants inside `classifyAge`;
  a follow-up patch could expose them as props if a
  use case emerges.
- The success toast is a standard top-right slot
  (matches the Profiles / Templates / Workers
  pattern). Delete continues to use its own
  bottom-center UndoToast so the two channels do
  not visually collide.
- The pagination wiring fix is the same shape as
  the v1.11.336 Templates patch: pagination reads
  the filtered length, not the raw length, so a
  filter switch doesn't strand the operator on an
  empty page.
- `classifyAge` returns `'older'` for missing /
  invalid timestamps so the filter never throws on
  malformed daemon responses. The bucket count for
  "Older" includes these edge cases, which is the
  right default -- an operator filtering for
  recent items should not surprise-hit a row with
  no timestamp.
