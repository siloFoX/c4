## Storage contract

```
localStorage.c4:bindings = { commandPalette: 'ctrl+shift+p', ... }
```

Reads merge defaults with the stored map. Malformed JSON, non-object payloads,
and non-string values fall back to defaults. `setBinding(id, defaultValue)`
or `setBinding(id, '')` removes the override.

## Event contract

```
window.dispatchEvent(new CustomEvent('keyboard-bindings-changed', {
  detail: { bindings: { commandPalette: 'mod+k', ... } }
}))
```

`useBindings()` also subscribes to the `storage` event so cross-tab edits
propagate without manual reloads.

## Best-effort wiring notes

- `CommandPalette.tsx` reads `getBinding()` for the two action commands
  whose displayed shortcut maps cleanly onto a `BindingId`
  (`action:toggle-theme` -> `toggleTheme`, `action:show-shortcuts` -> `help`).
  The remaining action shortcuts (`q`, `r`, `s`) are not in the canonical
  binding set and stay hard-coded.
- `KeyboardShortcutsModal.tsx` is intentionally untouched: its
  `SHORTCUT_ROWS` constant is locked by
  `KeyboardShortcutsModal.test.tsx` (the test asserts each `row.keys`
  string renders in the DOM via `screen.getAllByText(row.keys)`). Wiring
  the modal to consume `useBindings()` requires a coordinated update of
  the test snapshot and is the next consumer pass; the remap surface is
  the new authoritative override store regardless.
- The actual key-handler call sites (the global `?` listener,
  `mod+k` palette opener) live in their respective component files and
  still match against the default combos. Those handlers should switch
  to `parseCombo()`-based matching against `getBinding(...)` in a
  follow-up pass; the data contract + storage layer + edit surface are
  in place so the rewire is purely consumer-side.
