## Technique

A route-change crossfade implemented with **Tailwind utilities only**
-- no `framer-motion`, no new runtime dependency.

The component stores the previous `routeKey` + previous `children`
in state. On a `routeKey` change it transitions through three
phases:

1. **idle** -- single layer, the current tree, no animation.
2. **prepare** -- both layers mounted. The outgoing layer renders
   at its initial state (`opacity-100`) and the incoming layer at
   its initial state (`opacity-0 translate-x-2` for horizontal,
   `translate-y-2` for vertical). The browser commits these
   styles to the DOM.
3. **animate** -- a `requestAnimationFrame` callback flips both
   layers to their final state: outgoing `opacity-0 -translate-x-2`,
   incoming `opacity-100 translate-x-0`. Because the initial state
   was already painted in phase 2, the browser fires the CSS
   transition normally.

After `duration` ms a `setTimeout` drops the outgoing layer and
returns to **idle** with the incoming tree as the new current.

The outgoing layer is positioned `absolute inset-0` inside a
`relative` container so it cannot push the incoming layer's
layout around. The incoming layer drives the container's
intrinsic height as soon as it is committed in phase 2, so the
container does not collapse during the transition.

## prefers-reduced-motion

`window.matchMedia('(prefers-reduced-motion: reduce)')` is consulted
synchronously when a `routeKey` change is observed. When the user
has reduced motion enabled, the component **skips all three phases**:
the new tree replaces the old one in a single state update, the
outgoing layer is never mounted, and no timers fire.

## Accessibility

- `aria-live="polite"` on the outer container so screen readers
  announce the new route content without interrupting current
  speech.
- `aria-hidden="true"` on the outgoing layer so the screen reader
  does not read both the old and the new tree during the 200ms
  overlap.

## App.tsx integration line

```tsx
import PageTransition from './components/PageTransition';

// inside <Suspense>, wrapping the topView ternary switcher:
<PageTransition routeKey={topView} className="flex min-h-0 flex-1">
  {/* topView === 'history' ? <HistoryView/> : ... existing branches ... */}
</PageTransition>
```

The wrapper keys off `topView` (the daemon web shell's top-level
view selector -- this codebase does not use react-router). Per-branch
`key={topView}` divs with `motion-safe:animate-in motion-safe:fade-in`
classes continue to drive their own enter animations for nested
state transitions (e.g., `detailMode` switching inside the worker
view), so the crossfade composes with the existing micro-motion
instead of replacing it.

## Tests

Eight cases in `PageTransition.test.tsx`:

- renders children
- both layers present transiently during routeKey change
- after `duration` ms only the incoming tree remains
- prefers-reduced-motion path swaps content without mounting both
- forwardRef exposes the outer container
- aria-live attribute is present
- className merges onto the container
- `direction="vertical"` applies the vertical transform class

### jsdom caveat

jsdom does not implement `window.matchMedia`. Each test that cares
about it stubs `window.matchMedia` via `Object.defineProperty` so
the query returns the desired `matches` value. The fake-timer
tests rely on `vi.useFakeTimers()` to drive both `setTimeout` and
the `requestAnimationFrame` polyfill that vitest installs alongside
fake timers; `vi.advanceTimersByTime(50)` is enough to flush the
rAF callback into the `animate` phase while still leaving the
200ms swap timer pending so both layers remain mounted.
