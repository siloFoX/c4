# TODO 8.22 — UX visual regression + terminal auto-fit fix

**Status:** done
**Branch:** `c4/ux-visual`
**Scope:** Two-in-one task — repair the terminal auto-fit flow that
stopped producing correct cols after the 8.19 `withApiPrefix` refactor
(P1), and stand up a screenshot-driven visual regression pass on top of
the 8.20-era `tools/ux/explore.mjs` so layout breakage at common
desktop / tablet widths surfaces in CI-style reports without a human
eyeballing the UI (P2). P3 (mobile device emulation) is deferred to
TODO 8.23 — this patch deliberately does not touch it so the two
workers can land independently.

## Summary

Terminal auto-fit now catches parent flex reflows + sidebar toggles
via a `ResizeObserver` on the terminal `<pre>` that shares a 120 ms
debounce with the existing `window.resize` listener, clamps measured
cols to the daemon's 20..400 bounds, and never issues a `POST 0`.
Historical scrollback lines re-wrap at the new cols when the user
shrinks the terminal so the client no longer renders old wide lines
against a narrower `<pre>`. A `VITE_AUTOFIT_DEBUG` env toggle stays
wired so future regressions surface without code changes.

A new visual-regression pass in `tools/ux/explore.mjs` sweeps three
desktop / tablet viewports across every top-tab route, writes stable-
filename PNGs into `patches/ui-audit-<date>/screens/`, detects
off-viewport elements + ellipsis / overflow clipping in-page, compares
each frame against baselines at `patches/ui-audit-baseline/` via
`pixelmatch` + `pngjs` (flagged at > 0.5 % pixel delta), and captures
a terminal auto-fit anchor (2000 → 600 px) so the 8.22 P1 regression
has a browser-side check alongside the unit tests.

## P1 — Terminal auto-fit regression

### Root cause

Before this patch `WorkerDetail.tsx` only scheduled `recomputeFit`
from `window.addEventListener('resize')`. That fires on viewport
changes but not on:
- sidebar open / close (the main flex child shrinks without a window
  resize),
- font-size bumps that reflow the `<pre>` without resizing the window,
- parent flex reflows triggered by data loads that swap other
  columns' widths.

Result: the last cached `cols` stayed in localStorage, the
`/api/resize` POST never fired for the new width, and the server kept
emitting PTY output at 160 cols — rendered wrapped against the new
narrower `<pre>` on the client.

Separately, `ScreenBuffer.resize(cols, rows)` only trimmed the active
grid on `cols` shrinks. Existing scrollback lines kept their
pre-shrink width and the client wrapped them at the new `<pre>` width
on render — the history looked wrong even though fresh output was
correct.

### Fix (web/src/components/WorkerDetail.tsx)

- **Shared 120 ms debounce.** `scheduleRecompute()` wraps
  `window.clearTimeout` + `setTimeout(recomputeFit, 120)` and is used
  by both the existing `window.resize` listener and a new
  `ResizeObserver` on the `<pre>`. The observer falls back silently
  when the browser lacks `ResizeObserver` (some mobile Safari builds
  on flex children); the window listener stays wired as a fallback so
  we still recover in that case.
- **Functional `setCols`.** `recomputeFit` now calls
  `setCols(prev => prev === nextCols ? prev : nextCols)` so `cols`
  stays out of the callback's dependency list. Before this,
  `recomputeFit` was recreated on every cols change and tore down /
  re-attached the `ResizeObserver` on every fit cycle.
- **POST URL sanity.** The `apiFetch` call remains at `/api/resize`
  (the 8.19 `withApiPrefix` path). Regression-locked by the new
  `tests/ux-visual.test.js` assertion that rejects any `apiFetch('/resize')`
  shape.
- **20..400 clamp + never-POST-0 guards.** `if (inner <= 0) return;`
  and `if (!Number.isFinite(raw) || raw <= 0) return;` fire before
  `clamp(raw, MIN_COLS, MAX_COLS)`, so a layout pre-measurement (zero
  width) or a degenerate ruler never leads to a `POST { cols: 0 }`.
- **`VITE_AUTOFIT_DEBUG` toggle.** Read once at module load; when on,
  every measurement logs `[autofit] measured cols=%d (inner=%d,
  charW=%f, font=%d)` and every resize POST logs `[autofit] cols=%d
  rows=%d -> POST /api/resize`. Off by default; flip by writing
  `VITE_AUTOFIT_DEBUG=1` into `web/.env.local` before `npm --prefix
  web run dev`.

### Fix (src/screen-buffer.js)

Extended the existing `resize(cols, rows)` so a `cols < this.cols`
shrink also re-flows the `scrollback` buffer: each stored line longer
than the new width is split into consecutive chunks of length `c`, the
re-wrapped list replaces `this.scrollback`, and the result is capped
at `maxScrollback` (keeping the newest rows). Cols-grow is an
intentional no-op — the stored shorter lines render fine at wider
cols. One-line doc comment on the `resize` method calls out the new
behaviour.

### Manual verification checklist

- 2000 px viewport → cols ≈ 240 at 13 px monospace.
- 600 px viewport → cols ≈ 80.
- Font-size `-` / `+` triggers exactly one `POST /api/resize` per
  click.
- Network panel shows at most one `POST /api/resize` per debounce
  window under rapid window resizing.
- Shrink the viewport; old scrollback lines re-wrap in place instead
  of running past the `<pre>` right edge.

## P2 — Puppeteer visual regression

### Run

```
node tools/ux/explore.mjs --iteration <n>
```

Credentials stay at `/tmp/c4-silofox-cred` (username `silofox`). The
existing functional click-through (`desktop` + `mobile` viewports,
login / logout / 401-recovery / etc.) runs first and writes
`ux-reports/<n>/report.json`. Then the new visual pass runs and writes
an expanded copy to `patches/ui-audit-<date>/ui-audit-report.json`.

### Where artifacts land

```
patches/ui-audit-<YYYY-MM-DD>/
  ui-audit-report.json            # full report with the new `visual` key
  screens/
    desktop-xl-root.png
    desktop-xl-workers.png
    desktop-xl-features.png
    ...
    tablet-settings.png
  diffs/
    desktop-xl-root.png           # pixelmatch overlay for each flagged page
    ...

patches/ui-audit-baseline/
  desktop-xl-root.png             # first-run baselines; persistent
  ...
```

Both paths sit under `patches/` (alongside the existing
`1.11.x-*.md` / `1.10.x-*.md` patch notes at repo root). The
per-run `ui-audit-*` folders and the `ui-audit-baseline/` directory
are added to `.gitignore` — the PNGs are large and non-deterministic
across chrome versions, so re-generate them locally instead of
committing. Operators who want cross-machine baselines can copy the
`baseline/` directory out-of-band.

### Viewports

```js
const VIEWPORTS_VISUAL = [
  { name: 'desktop-xl', width: 1920, height: 1080 },
  { name: 'desktop-md', width: 1366, height: 768 },
  { name: 'tablet',     width: 1024, height: 768 },
];
```

Mobile emulations (iPhone 13 / iPhone SE / Galaxy S20 / iPad mini in
portrait + landscape) are intentionally out of scope — TODO 8.23
ships them as a separate worker and extends the same report
structure. `VIEWPORTS_VISUAL` is the single extension point.

### Page sweep

Seven top-tab routes × three viewports = 21 PNGs per run:

```
/, /workers, /chat, /history, /workflows, /features, /sessions,
/settings
```

Navigation clicks `button[data-testid="top-tab-<id>"]` to match the
existing flow; `/` falls back to `page.goto(BASE + '/')`. Missing
top-tab selectors skip the page and surface a `warn` issue on the
main report.

### Overflow + clipping detection

Two in-page detectors run after each `page.goto` + layout-settle wait:

- **Overflow:** walks `document.querySelectorAll('*')` and records any
  element whose `getBoundingClientRect().right` exceeds
  `window.innerWidth + 1`. Capped at 20 samples per (viewport, page).
- **Clipping:** walks the same tree but filters to elements that use
  `text-overflow: ellipsis` *or* `overflow: hidden` / `overflow-x:
  hidden`, then flags any where `scrollWidth > clientWidth`. Captures
  the element text truncated to 80 chars so operators can eyeball
  which label was cut.

Both counts + first 20 offender samples land in the report's `visual`
key.

### Baseline diff

Per-page pixel diff runs via `pixelmatch` + `pngjs` (both dynamic-
imported from `tools/ux/package.json` — zero runtime impact on the
main app). If a baseline exists, the candidate is compared at
threshold 0.1; when > 0.5 % of pixels change the page is flagged and
a diff overlay is written to `patches/ui-audit-<date>/diffs/`. When
no baseline exists, the candidate is copied in as the new baseline
and the report records `baseline: 'captured'` for that page. Size
mismatches are treated as 100 % diff (always flagged) so a layout
that reshapes the full page never silently diffs clean.

### Terminal auto-fit anchor

After the viewport sweep a single standalone `captureAutofitAnchor`
pass launches one more browser, logs in, clicks the first worker in
the sidebar, toggles `Auto-fit` on, then:

1. resizes the viewport to 2000 × 1200, reads `dims` from the
   `Terminal session - dims {cols} x {rows}` label, stores as
   `visual.autofit.before2000`,
2. resizes the viewport to 600 × 900, reads `dims`, stores as
   `visual.autofit.after600`.

Both values land in the report so 8.22 P1 has a browser-side
regression anchor on top of the unit tests. When the sidebar is empty
the report records `autofit.note: 'no workers in sidebar'` instead.

### Regenerating baselines

Delete `patches/ui-audit-baseline/` and re-run `node
tools/ux/explore.mjs`. The first run after deletion seeds every frame
and records `baseline: 'captured'` for every page. A follow-up run is
then a sanity check that the diff against the freshly-seeded
baselines stays flagged-false (it should be ~0 %).
