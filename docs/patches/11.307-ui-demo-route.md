# 11.307 -- storybook-style demo route (v1.11.325)

New `/ui-demo` feature page at
`web/src/pages/UIDemoRoute.tsx` that renders every UI
primitive with its main variants for visual QA. Behind
a new dev-only `uiDemoRoute` feature flag.
Component-scope only, no daemon-side change and no `c4`
CLI surface change.

## Why

The c4 web app already ships a comprehensive interactive
`DesignSystem.tsx` demo page that wires every primitive
into a stateful playground with variant matrices, prop
tables, and code snippets. That page is excellent for
exploring the API surface but poor for visual QA passes:

- Async state machines mean the page does not render
  the same DOM on every screenshot run.
- Variant matrices are spread across collapsible groups
  -- a designer comparing dark-mode vs light-mode has
  to expand each group to see anything.
- The bundle weight is non-trivial because every demo
  imports its own sample data.

Patch 11.307 ships a separate `UIDemoRoute.tsx` that is
intentionally compact and non-interactive:

- One section per primitive family, with the main
  variants laid out flat in a single scrollable column.
- No async state, no fetch calls, no toast / dialog /
  drawer mounts. Just static primitives with default
  props.
- Gated behind a dev-only feature flag so the route
  exists but renders an explicit "disabled" message
  for production users who hit it by URL.

## What changed

### `web/src/pages/UIDemoRoute.tsx` (new file)

Compact gallery rendering nine primitive sections:

1. Buttons (primary / secondary / ghost / destructive
   / disabled).
2. Badges and Chips (5 badge variants + Chip +
   BadgeCounter at 3 different magnitudes).
3. Status (4 StatusDot tones + 3 StatusPill tones).
4. Avatars (xs / sm / md / lg sizes).
5. Form controls (Input + Textarea + Switch + Checkbox
   x 2 + RadioGroup).
6. Feedback (4 Alert variants + 2 Progress values + 3
   Spinner sizes).
7. States (Card + Panel + EmptyState + ErrorState).
8. Layout helpers (AspectRatio 16:9 + AspectRatio 1:1).
9. Navigation and misc (Breadcrumbs + Kbd pair +
   Tooltip-wrapped Button + Separator).

Each section is wrapped in a `<section>` with
`data-section="ui-demo-section"` and
`data-demo-section="<title>"` for e2e targeting.

The page reads the `uiDemoRoute` feature flag via
`useFeatureFlag('uiDemoRoute')`. When false (default),
the page renders a short "enable the flag" panel with
`data-section="ui-demo-disabled"`. When true, the page
renders the full gallery wrapper with
`data-section="ui-demo-root"` and
`data-demo-enabled="true"`.

The page title ("UI Demo") and description render in
the `PageFrame` header regardless of flag state -- the
flag only gates the body content.

### `web/src/lib/feature-flags.ts`

Added a new `uiDemoRoute` key to the `FeatureFlagKey`
union and registered it in the `FLAGS` array with
`defaultValue: false`. The Feature Flags page picks it
up automatically since `FLAGS` is the source of truth
for the rendered list.

### `web/src/pages/registry.ts`

Added a new feature entry under the `diagnostics`
category:

```ts
{
  id: 'ui-demo',
  labelKey: 'feature.uiDemo.label',
  descriptionKey: 'feature.uiDemo.description',
  category: 'diagnostics',
  Icon: LayoutGrid,
  load: () => import('./UIDemoRoute'),
}
```

i18n entries added under `feature.uiDemo.label` and
`feature.uiDemo.description` in both `en.json` and
`ko.json` so the i18n-lockstep validator is satisfied.
The ko entry currently mirrors the English text -- a
proper Korean translation can land in a follow-up.

## Tests

`web/src/pages/UIDemoRoute.test.tsx` -- 5 vitest cases:

- `data-section="ui-demo-disabled"` is present and
  `data-section="ui-demo-root"` is absent when the
  feature flag is off (the default).
- `data-section="ui-demo-root"` with
  `data-demo-enabled="true"` is present and
  `data-section="ui-demo-disabled"` is absent when the
  feature flag is on.
- All 9 documented sections are present in the gallery
  via `data-demo-section="..."` selectors.
- The page title ("UI Demo") renders in both flag
  states.
- The gallery contains >=9 elements with
  `data-section="ui-demo-section"`.

Existing 9 feature-flags tests also pass against the
new `uiDemoRoute` key.

All 5 green via
`vitest run --project unit src/pages/UIDemoRoute.test.tsx`.
All 9 feature-flags tests green via
`vitest run --project unit src/lib/feature-flags.test.ts`.

## Notes

- The gallery is intentionally non-interactive. The
  Switch / Checkbox / RadioGroup do have local state
  (so the controlled-input contract is respected) but
  the demo deliberately does not wire any side-effects
  -- toggling them does not trigger fetches, navigate
  routes, or alter app state. This keeps screenshot
  diffs stable.
- Tooltips are present but not auto-open in the
  default render. A future enhancement could expose
  a "preview tooltip" hover style so screenshot runs
  can capture the open state.
- The `LayoutGrid` icon was chosen for the registry
  entry because the page is a literal grid of variant
  cells, distinct from the `Palette` icon used by the
  existing `DesignSystem` interactive demo. Both
  pages live in the sidebar; operators can pick the
  one that fits their task.
- The feature flag is registered as a normal
  `FeatureFlagDef` rather than a separate dev-only
  registry so the operator can toggle it from the
  Feature Flags page without environment-variable
  gymnastics. The "dev-only" intent is enforced by the
  default value (false) plus the documented warning in
  the flag description, not by a build-time gate.
