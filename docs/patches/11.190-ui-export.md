## What

primitive that composes it with the existing `<Button>` + `<DropdownMenu>`
primitives. Three first-party adoptions ship in the same patch so the
helper has real surface area on day one.

### API: `web/src/lib/export.ts`

```ts
export interface ColumnDef<T> {
  key: keyof T | string;
  header?: string;
  format?: (val: unknown, row: T) => string;
}

export function toCSV<T>(rows: T[], columns: ColumnDef<T>[]): string;
export function toJSON<T>(rows: T[]): string;
export function downloadFile(content: string, filename: string, mimeType: string): void;
export function exportData<T>(opts: {
  rows: T[];
  columns?: ColumnDef<T>[];
  format: 'csv' | 'json';
  filename: string;
}): void;
```

- `toCSV` builds a header row from `columns` (header override defaults
  to `String(key)`), then renders each row by indexing the row with the
  column `key`. Any cell containing `,`, `\r`, `\n`, or `"` is wrapped in
  double quotes; inner quotes are escaped by doubling per RFC 4180.
- `toJSON` is `JSON.stringify(rows, null, 2)` -- 2-space pretty print.
- `downloadFile` early-returns when `typeof window === 'undefined'`, so
  the helper is safe to import from SSR / test code that pre-renders the
  tree.
- `exportData` composes the above and, when the caller's filename omits
  the `.csv` / `.json` extension, appends `-YYYYMMDD.<ext>` so multiple
  exports on the same day are still distinguishable from one another.

### API: `web/src/components/ui/export-button.tsx`

```tsx
<ExportButton
  rows={visibleRows}
  columns={[{ key: 'name', header: 'Worker' }, { key: 'total' }]}
  filename="token-usage"
  formats={['csv', 'json']}      // default both
  disabled={visibleRows.length === 0}
/>
```

- Single-format `formats={[...]}` makes the trigger fire `exportData`
  immediately on click.
- Multi-format renders a `<DropdownMenu>` with CSV / JSON items below the
  trigger.
- `forwardRef` is plumbed through to the trigger `<button>` so callers
  can grab focus or measure layout.

disables it when its row set is empty.

## Out of scope

- No daemon endpoint. The current helper is purely client-side; very
  large datasets (>1k rows) would benefit from a streaming export but
  the existing pages all paginate or window first, so the in-memory
  path is acceptable today.
- No XLSX support. Adding it would pull in a multi-hundred-KB dep for
  one optional output -- left as a follow-up if the request shows up.
