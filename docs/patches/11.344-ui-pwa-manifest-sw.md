### Reference: ARPS color

ARPS design tokens (`/root/c4/arps-design-system-v1/tokens.css`) define
`--surface-canvas: 220 18% 8%` (HSL). The hex
equivalent is `#0D1B2A` -- the same navy used by the
logo file. Using this as both `theme_color` AND
`background_color` keeps the launcher splash, the
title bar, and the first paint of the React app
visually continuous.

## Service worker (`web/src/sw.ts`)

Pure-TS source. The pure handlers are exported
independently so vitest drives them under jsdom
without a real `ServiceWorkerGlobalScope`. The
bottom of the file is a runtime-only block that
wires the `install` / `activate` / `fetch` /
`message` listeners when `self` quacks like a SW
scope.

### Caches (versioned)

| name             | purpose                                                   |
| ---------------- | --------------------------------------------------------- |
| `c4-shell-v1`    | app shell -- `/`, `/index.html`, `/manifest.webmanifest`, `/favicon.svg`, `/logo.svg`. Populated at `install`. |
| `c4-runtime-v1`  | cache-first store for same-origin JS/CSS/SVG/font assets. |
| `c4-api-v1`      | network-first store for `/api/*` GETs (fallback when offline). |

`handleActivate` prunes every `c4-*` cache that is
not in the keep list, so old hashes from prior
releases free up the quota the next time the user
visits. Foreign caches (non-`c4-*`) are never
deleted.

### Route classifier

```ts
classifyRequest(request, { origin, apiPrefix }):
  | 'api'         // same-origin + path starts with apiPrefix (default '/api/')
  | 'navigation'  // request.mode === 'navigate' OR destination === 'document'
  | 'static'      // same-origin AND pathname matches RE_STATIC_EXT
  | 'opaque'      // everything else (cross-origin, unknown same-origin)
```

`RE_STATIC_EXT` matches `js / mjs / css / svg / png
/ jpg / jpeg / gif / webp / avif / ico / woff / woff2
/ ttf / otf` with an optional query string. Compound
selectors like `.foo *` are not relevant here -- this
is the URL-path classifier, not the CSS audit.

### Strategies

- **api -- `networkFirst`**: try the network; on
  success, cache the response (GET only); on
  failure, fall back to the cached copy. POST /
  PUT / DELETE pass straight through (write paths
  must never be served from cache). A 5xx is not
  cached.
- **navigation -- `navigationHandler`**: same as
  `networkFirst` for the runtime cache, then on
  failure look up `/index.html` then `/` in the
  shell cache so the React app boots even offline.
  The cached shell carries the manifest link tag,
  so the install banner survives.
- **static -- `cacheFirst`**: serve the cached copy
  immediately; fire-and-forget revalidate in the
  background. Cold-cache lookups go to the network
  and cache the response.
- **opaque**: passthrough. Cross-origin requests
  (CDN fonts, telemetry endpoints) are never
  cached because the SW cannot inspect them.

### Test harness

`web/src/sw.test.ts` ships `FakeCache` +
`FakeCaches` that implement the `CacheLike` /
`CacheStorageLike` contracts the handlers consume.
The constructor for `Request` in jsdom rejects
`mode: 'navigate'` (browser-engine reserved value),
so the test helper attaches `mode` and `destination`
via `Object.defineProperty` after construction --
the classifier reads them via property access.

## SW registration (`web/src/lib/sw-register.ts`)

```ts
registerServiceWorker({
  scriptUrl,         // default '/sw.js'
  scope,             // default '/'
  disabled,          // default false
  onUpdateAvailable, // optional ServiceWorkerRegistration => void
  onError,           // optional unknown => void; never rethrown
}): Promise<{ registered: boolean; reason?: string; registration?: ServiceWorkerRegistration }>
```

Idempotent: a second call returns the same in-flight
promise so multiple App mounts (HMR, tests, multiple
bootstraps) do not double-register the worker.
`resetServiceWorkerRegistrationForTests` clears the
gate so the test suite can re-run.

`attachUpdateListener` fires `onUpdateAvailable`
only when an installed worker arrives AND a
controller already exists -- this is the spec
pattern for "update available", not "first install".
`activateWaitingServiceWorker` posts the
`SKIP_WAITING` message so the host can wire an
"Update now" button.

## Out of scope

- **Vite build step** that emits `sw.js` from
  `src/sw.ts`. Source-of-truth is in place; a
  follow-up patch wires the build (`vite-plugin-pwa`
  or a hand-rolled `?worker` import) and the
  matching `main.tsx` registration.
- **Pre-cache of hashed Vite chunks.** Hashed
  filenames change every deploy; we rely on
  `cacheFirst` to populate them on first use rather
  than enumerating the chunk list at install.
- **Background sync** for queued `/api/` POSTs
  while offline. Out of scope for the first install
  -- the SW only buffers reads, never writes.
- **Workbox / runtime cache TTL eviction.** The
  cache-first revalidate handles freshness for now;
  a future patch can add LRU pruning if the cache
  ever bloats.
- **Web Push.** The manifest does not declare the
  `gcm_sender_id` / VAPID hook; we have no
  push-service contract yet.
