# 11.211 -- UI: toast queue prioritization

Component-scope-only extension to the existing Toast layer. The
visual `<Toast>` component, the `useToast` hook, the portal target,
and the legacy `{toast, showToast, dismissToast}` triple all stay
exactly where they are. The new surface area is:

- A `TOAST_PRIORITY` map -- `error: 3, warning: 2, success: 1,
  info: 0`. Same-priority ties fall back to insertion order (ascending
  id), which keeps FIFO semantics within a single tier.
- A `TOAST_VISIBLE_LIMIT = 3` constant -- the maximum number of toasts
  the user sees at once.
- A `partitionToasts()` helper -- pure, deterministic split of the
  raw queue into `{ visible, pending }` with the same sort key.
- A `ToastStack` component -- renders the top-N visible toasts by
  priority and, if there is any overflow, mounts a small `+N more`
  Chip into the `#toast-root` portal so the user can see that more
  notifications are queued behind the visible row. The chip is
  hidden entirely when nothing is pending.
- A fourth `warning` tone on the existing Toast component (icon =
  `AlertTriangle`, classes = `border-warning/40 bg-warning/15
  text-warning`) so the priority rubric has a corresponding visual.

## Files

- `web/src/components/Toast.tsx` -- adds the `warning` tone, the
  `TOAST_PRIORITY` / `TOAST_VISIBLE_LIMIT` constants, the
  `partitionToasts()` helper, the `ToastEntry` / `ToastStackProps`
  types, and the new `ToastStack` component. The single-toast
  `Toast` default export and its props are untouched.
- `web/src/components/Toast.test.tsx` -- adds the warning-tone case
  plus three new `describe(...)` blocks covering the priority map,
  the `partitionToasts()` split, and the `<ToastStack>` rendering
  contract (15+ new assertions across 11 cases).

## Behaviour

The dispatch contract is intentionally simple: callers keep using
`useToast().showToast(message, type)` and pass `useToast().toasts`
into `<ToastStack>`. The stack then derives its own view -- it does
not mutate or re-order the source array. Dismissals bubble back via
`onDismiss(id)` so the parent can drop the entry from `useToast`,
which on the next render lets the stack promote the next-highest-
priority pending toast into the freed visible slot. Each `<Toast>`
inside the stack continues to own its own auto-dismiss timer and
swipe-to-dismiss gesture; the stack adds nothing that interferes
with those.

The `+N more` Chip uses the existing `Chip` primitive from
`components/ui/chip.tsx` (`tone="neutral"`, `variant="subtle"`,
`size="sm"`). It is portaled into `#toast-root` so it sits visually
with the toasts even when the consumer renders `<ToastStack>` from
deep inside the React tree.

## Out of scope

- No daemon endpoint, no new API surface, no audit-log line.
- `useToast` is unchanged. Existing FIFO + `TOAST_QUEUE_LIMIT = 5`
  semantics are preserved; the priority split happens at render
  time inside `ToastStack`, not at insert time inside `useToast`.
- The single-toast `Toast` component public props (`message`,
  `type`, `onDismiss`, `duration`) are unchanged. The only addition
  is the fourth `'warning'` member of the `ToastType` union.
