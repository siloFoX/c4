## Summary

New `ChartLineJma` UI primitive at
`web/src/components/ui/chart-line-jma.tsx` -- a pure-SVG line chart
with a Jurik-style low-lag low-noise moving average overlay.

## Indicator math

A Jurik-style Moving Average -- the price run through a three-stage
adaptive filter that delivers low lag with low noise:

```
beta       = 0.45*(length-1) / (0.45*(length-1) + 2)
alpha      = beta^2
phaseRatio = clamp(phase, -100, 100) / 100 + 1.5
e0 = (1 - alpha)*price + alpha*e0'
e1 = (price - e0)*(1 - beta) + beta*e1'
e2 = (e0 + phaseRatio*e1 - jma')*(1 - alpha)^2 + alpha^2*e2'
jma = e2 + jma'
```

`beta` sets the base smoothing; the `phase` parameter trades a touch
of overshoot for even less lag. The filter is seeded at the first
bar with that bar's price (`e0 = price[0]`, `e1 = e2 = 0`,
`jma = price[0]`), so a flat series stays flat and there is **no
warm-up period** -- the JMA produces a value from the very first
bar. The filter is recursive from the second bar onward.

### Worked verification

The recursive filter with `beta^2` is not exactly representable, so
the tests anchor on cases that are exact and one that is
fraction-exact:

- A **flat series** holds exactly at its constant (the seed keeps
  `e0 = c`, `e1 = e2 = 0`).
- A **length of 1** makes `beta = 0` and `alpha = 0`, so the filter
  degenerates to the identity -- the JMA equals the price exactly.
- For input `[0, 1]` with `length 2`, `phase 0` the recursion gives
  `jma[1] = 638029696000 / 678223072849 = 0.9407372...`, asserted
  with `toBeCloseTo`.

For the non-degenerate adaptive behaviour the tests assert robust
properties: the seed (`jma[0] = price[0]`), that the JMA lags just
below a rising ramp, that a different phase produces a different
result, and that a value is produced for every bar.

## API

- `getLineJmaFinitePoints(points)` -- drop points with a non-finite
  `x` or `value`.
- `normalizeLineJmaLength(length, fallback)` -- coerce to a positive
  integer; non-finite or sub-1 falls back, fractional floors.
- `normalizeLineJmaPhase(phase, fallback)` -- coerce the phase to
  [-100, 100]; non-finite falls back, out-of-range clamps.
- `computeLineJma(values, length, phase)` -- the JMA series, one
  value per input bar (no warm-up nulls).
- `runLineJma(points, options?)` -- sort finite points by `x`,
  compute the series, return `{ series, length, phase, jma,
  samples, jmaFinal, jmaMin, jmaMax, aboveCount, belowCount, ok }`.
  Each sample carries the price's `above` / `below` / `on` position
  versus the JMA. `ok` is false for fewer than two points.
- `computeLineJmaLayout(options)` -- project the price and JMA onto
  one shared panel. Returns the panel, ticks, the price path, the
  JMA path, and a JMA marker for every bar.
- `describeLineJmaChart(data, options?)` -- ARIA description.

## Component

`ChartLineJma` is a `forwardRef` region (`role="region"`) with an
ARIA-linked description, a `role="img"` SVG, axis ticks and grid, a
config badge calling out the length and phase, a two-series legend
(Price / JMA) with toggle, a hover/focus tooltip exposing the value,
JMA and position, and a `motion-safe:animate-fade-in` entrance.
Every node carries a `data-section` attribute.

## Distinctness

- **vs chart-line-dema / chart-line-tema** -- those are *fixed
  linear combinations* of nested EMAs (`2*EMA - EMA(EMA)` and
  `3*EMA - 3*EMA(EMA) + EMA^3`). The JMA is a *three-stage adaptive
  filter* with a `phase` parameter that explicitly tunes the
  lag-versus-overshoot tradeoff, and -- unlike the EMA cascades,
  which have a multi-period warm-up -- it produces a value from the
  very first bar.
- **vs chart-line-kama / chart-line-vidya / chart-line-frama** --
  those derive their smoothing from a single market-state index
  (efficiency ratio, the CMO, the fractal dimension). The JMA's
  low-lag, low-noise behaviour is built into its multi-stage filter
  structure and the phase parameter rather than an external gauge.
