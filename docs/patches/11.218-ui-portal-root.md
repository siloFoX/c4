# 11.218 -- UI: portal-root manager

Status: shipped in v1.11.236 (2026-05-15)
Scope: `web/` only. No daemon-side change, no `c4` CLI surface change.

## Why

Three components -- `Toast`, `Dialog`, `Popover` -- each carried their
own logic for choosing where `createPortal` should mount: the toast
layer lazily created `<div id="toast-root">`, while the dialog and
popover mounted directly into `document.body`. That meant four
slightly different policies for the same job (lazy create / reuse,
tag the node, survive route changes, stay SSR-safe), and every new
portal-using component had to re-invent the same pattern. Future
work that wants to drop a sibling layer (escalation banner,
dropdown menu, command palette, tooltip) needs a single audit point.

## What

`web/src/lib/portal-root.ts` exposes two functions:

```ts
function getPortalRoot(
  id: 'toast-root' | 'dialog-root' | 'popover-root' | 'dropdown-root' | string,
): HTMLElement | null;

function cleanupPortalRoot(id: string): void;
```

- `getPortalRoot(id)` is lazy + idempotent: the first call appends a
  `<div id={id} data-portal-root="true">` under `document.body`;
  subsequent calls return the same node. The `data-portal-root`
  attribute lets callers query "all portal targets" generically (for
  e.g. axe-core scoping or visual debugging).
- `cleanupPortalRoot(id)` removes the node iff it has no children.
  Test helpers can call it safely after each case; it is a no-op
  when the node is still in use, so leaking a stale `#dialog-root`
  is not possible.
- SSR-safe: returns `null` when `typeof document === 'undefined'`.
  Callers must null-check (Dialog and Popover fall back to
  `document.body` for the pathological "document present, body
  missing" case).
- `document.body` missing: one-shot `console.warn` (de-duplicated
  via a module-level flag) then returns `null`.

## Consolidation

| Component | Before                                  | After                                                                  |
| --------- | --------------------------------------- | ---------------------------------------------------------------------- |
| Toast     | Inline `getToastRoot` (createElement)   | `getPortalRoot('toast-root')` + toast-specific class on first create   |
| Dialog    | `createPortal(node, document.body)`     | `createPortal(node, getPortalRoot('dialog-root') ?? document.body)`    |
| Popover   | `createPortal(panel, document.body)`    | `createPortal(panel, getPortalRoot('popover-root') ?? document.body)`  |

`DropdownMenu` does not yet use a portal; the `'dropdown-root'` id is
reserved for the next iteration that lifts the menu surface out of
its trigger's stacking context.

## Behaviour parity

The dialog test asserts `document.body.contains(dialog) === true`.
`Node.contains` is recursive, so nesting the dialog inside
`<div id="dialog-root">` (itself a child of `body`) keeps the
assertion green. All 80 pre-existing Toast / Dialog / Popover tests
pass unchanged after the swap.

## Tests

`web/src/lib/portal-root.test.ts` -- 10 cases:

1. Creates a node with the requested id under `document.body`.
2. Tags the node with `data-portal-root="true"`.
3. Returns the same element on repeated calls (idempotent).
4. Different ids create separate elements.
5. Reuses a pre-existing element that already carries the id.
6. `cleanupPortalRoot` removes the element when it has no children.
7. `cleanupPortalRoot` is a no-op when the element has children.
8. `cleanupPortalRoot` is a no-op when the element does not exist.
9. SSR branch (`delete globalThis.document`) returns `null`.
10. Missing `document.body` warns once and returns `null`.

Run:

```bash
env -C web ./node_modules/.bin/vitest run src/lib/portal-root.test.ts
```
