# 11.316 -- Sessions page polish (v1.11.334)

New `web/src/pages/Sessions.tsx` feature page with the
dispatch's UI requirements: Tabs for active/archived/all,
Avatar with status overlay per row, SearchBar with
debounce, and right-anchored Drawer for the detail
view. Component-scope only, no daemon-side change and
no `c4` CLI surface change.

## Why

The dispatch asked to "polish web/src/pages/Sessions.tsx"
-- but that file does not exist in the current tree.
Session-related components (`SessionsAttachedSection`,
`SessionsHeader`, `SessionsAttachedRowActions`, etc)
live under `web/src/components/`, but there is no
feature page at `web/src/pages/Sessions.tsx`. The
daemon similarly has no list endpoint -- only the
per-id `/api/sessions/<id>` route used by
`lib/use-conversation`.

Rather than skip the dispatch, patch 11.316 builds the
feature page from scratch with the requested shape and
a deterministic mock dataset that can be swapped for a
live `/api/sessions` endpoint once the daemon route
lands.

## What changed

### `web/src/pages/Sessions.tsx` (new file)

A feature page that surfaces:

- **Three Tabs** (`active` / `archived` / `all`) via
  the Tabs primitive. `active` is default-selected so
  the operator's first glance lands on live sessions.
- **Per-row Avatar** with `status="online"|"offline"`
  driven by the session's status field. Avatar
  primitive exposes `data-section="avatar-root"` and a
  sibling `data-section="avatar-status"` so e2e tests
  can assert per-row presence.
- **SearchBar with debounce**. `debounceMs=200` (the
  primitive's default). The filter is applied across
  `name`, `worker`, `taskPreview`, and `id` so the
  operator can find a session by any visible attribute.
  The list re-renders inside the active tab so the
  search and tab filters compose.
- **Right-anchored Drawer** (360px wide) for the
  detail view. Click any row to open. The Drawer
  carries `data-testid="sessions-drawer"` and the
  detail body inside it carries
  `data-section="sessions-detail"` +
  `data-session-id=<id>` so e2e can pin the open
  drawer to a specific session.
- **Detail body** uses TimeAgo (long variant) for the
  started + last-active timestamps and renders the
  optional `notes` slot in a Panel.

### Data shape

```ts
interface SessionEntry {
  id: string;
  name: string;
  status: 'active' | 'archived';
  worker: string;
  startedAt: string;     // ISO timestamp
  lastActiveAt: string;  // ISO timestamp
  taskPreview: string;
  notes?: string;
}
```

The page accepts an optional `sessions?: SessionEntry[]`
prop. Tests pass a fixed fixture; production mounts pass
nothing and get a 4-row demo dataset (auto-mgr,
auto-w91, plan-w42, auto-w74) that exercises both
status values + the notes slot.

### Registration

- `web/src/pages/registry.ts` gains a new `sessions`
  entry under the `operations` category with the
  `ListChecks` icon.
- i18n entries added under `feature.sessions.label`
  and `feature.sessions.description` in both
  `en.json` and `ko.json` (lockstep preserved; ko
  mirrors en text pending a translation).

### Data attributes

- `data-section="sessions-page"` on the page wrapper.
- `data-section="sessions-list"` + `data-count="<n>"`
  on the row list.
- `data-section="sessions-row"` on each `<li>`, with
  `data-session-id` and `data-session-status`.
- `data-section="sessions-row-name"` on the name span.
- `data-section="sessions-detail"` on the drawer body
  with `data-session-id`.
- `data-testid="sessions-search"` on the SearchBar.
- `data-testid="sessions-row-<id>"` on each row
  button.
- `data-testid="sessions-drawer"` on the Drawer.

## Deferred (dispatch follow-ups)

- **Real daemon endpoint.** The `sessions` prop
  override is the test seam. A follow-up patch will
  replace the demo dataset with a
  `useFetchSlot('/api/sessions')` call once the
  daemon route lands. The page shape stays the same.
- **Richer detail surface.** The Drawer currently
  shows a static detail view (task preview + timing
  + notes). A live transcript preview, an
  "Attach to this session" button, and an archive
  toggle would be the natural next additions.

## Notes

- The demo dataset uses `new Date(Date.now() - ...).toISOString()`
  inside the module body so TimeAgo always renders
  sensible relative strings ("just now", "27m ago",
  "9h ago", "22h ago"). The timestamps update on
  page reload but are deterministic across test
  runs because the test fixture pins them to fixed
  ISO strings.
- The Avatar primitive's `online` / `offline`
  status overlay is the canonical signal for
  active vs archived. Both states render the same
  initials tile; the dot in the corner is the
  visual differentiator.
- The Drawer mounts only when a row is clicked
  (`open={drawerId !== null}`). When no session is
  selected, the Drawer is not rendered, so the
  page DOM stays minimal until the operator
  interacts.
- The `matchesFilter` helper does a case-insensitive
  substring match. An empty / whitespace-only filter
  matches every session.
- The 360px drawer width matches the existing
  drawer convention (HelpDrawer ships with the same
  width) so the panel slides in consistently with
  other detail surfaces.
- The dispatch's "Tabs for active/archived/all"
  requirement is satisfied by the three Tabs. The
  internal tab keys are kept short (`active` /
  `archived` / `all`) so future routing-by-tab
  patches can read the value from a URL fragment
  without extra mapping.
