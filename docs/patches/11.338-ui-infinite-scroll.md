## Goal

The dispatch lists:

- Build `web/src/components/ui/infinite-scroll.tsx` --
  IntersectionObserver-based infinite scroll with
  loading sentinel, error retry, end-of-list marker.
- Adopt at HistoryView task rows (replace pagination),
  Sessions list, Notifications.

This patch ships the primitive + 22 unit tests + the
adoption pattern documentation. Per-page adoption is
deferred per the 11.33x rollout pattern (each adopter
needs paired test updates).

## Public API

```ts
export function InfiniteScroll(props: InfiniteScrollProps): JSX.Element;

export interface InfiniteScrollProps {
  children: ReactNode;
  hasMore: boolean;
  loading: boolean;
  error: Error | null;
  onLoadMore: () => void;
  onRetry?: () => void;
  loadingContent?: ReactNode;
  errorContent?: ReactNode | ((retry: () => void) => ReactNode);
  endContent?: ReactNode;
  rootMargin?: string;          // default '200px 0px'
  threshold?: number | number[]; // default 0
  root?: Element | null;
  className?: string;
  ariaLabel?: string;
  scrollRef?: React.Ref<InfiniteScrollHandle>;
}

export interface InfiniteScrollHandle {
  triggerLoadMore(): void;
  getSentinel(): HTMLDivElement | null;
}
```

## Behaviour

### Sentinel + IntersectionObserver

The primitive renders a 1px-tall `<div>` after the
children and observes it via IntersectionObserver.
When the sentinel intersects the root (default:
viewport), the primitive fires `onLoadMore()`.

The default `rootMargin='200px 0px'` fires the
callback 200px BEFORE the bottom edge is visible so
the next page is pre-fetched while the operator is
still scrolling. Adopters can override
(`rootMargin='0px'` for "wait until visible",
`'500px 0px'` for aggressive pre-load).

### Guards

`onLoadMore` is gated by three flags. The primitive
SKIPS the call when any of them are truthy:

| flag | reason |
| --- | --- |
| `loading` | a fetch is already in flight |
| `error` | the previous fetch failed; wait for retry |
| `!hasMore` | no more data to load |

The guards apply to BOTH the observer callback and
the imperative `triggerLoadMore()` handle.

### Terminal-state content

The sentinel's inner content surfaces ONE of four
states, in priority order:

| state | content | data-state |
| --- | --- | --- |
| error truthy | inline retry card | `error` |
| loading truthy | "Loading more..." line | `loading` |
| !hasMore | "End of list" marker | `end` |
| idle | empty (observer fires onLoadMore when visible) | `idle` |

The wrapper carries `data-state="idle|loading|error|end"`
so e2e + styling can branch on the active state.

### Default error card

Renders:

```
<div role="alert">
  <span>{error.message || 'Failed to load more.'}</span>
  {onRetry ? <Button onClick={onRetry}>Retry</Button> : null}
</div>
```

The button is `data-testid="infinite-scroll-retry"`.

### Factory error content

```tsx
errorContent={(retry) => (
  <div className="custom-error">
    <p>Something broke.</p>
    <button onClick={retry}>Try again</button>
  </div>
)}
```

The factory receives the same `onRetry` callback the
default uses. When `onRetry` is unset the callback
is a no-op (so the rendered button is still wired
without throwing).

### Imperative handle

```tsx
const ref = useRef<InfiniteScrollHandle>(null);

<InfiniteScroll scrollRef={ref} ...>
  ...
</InfiniteScroll>

<Button onClick={() => ref.current?.triggerLoadMore()}>
  Load more
</Button>
```

Useful for adopters who want both an observer-driven
auto-load AND an explicit "Load more" button without
re-implementing the guard logic.

## jsdom mocking

jsdom does not implement IntersectionObserver. The
test file installs an inline mock in `beforeEach`:

```ts
class IntersectionObserverMock implements IntersectionObserver {
  constructor(callback, options) {
    captured.push({ callback, options, observed: [], disconnected: false });
  }
  observe(el) { ... }
  disconnect() { ... }
}
globalThis.IntersectionObserver = IntersectionObserverMock;
```

Tests then call `fireIntersection(observerIndex, isIntersecting)`
to simulate the observer firing. Adopters can use the
same mock or call `triggerLoadMore()` via the ref to
bypass the observer entirely.

## Deferred (adoptions)

Three sites: HistoryView, Sessions list,
Notifications. Each adoption needs:

1. Swap the page's pagination state (page counter,
   manual "Next page" button) for a cursor + buffer
   that exposes `{ items, hasMore, loading, error,
   loadMore, retry }`.
2. Wrap the row list in `<InfiniteScroll>` and pass
   the cursor flags through.
3. Update per-page tests to use the
   IntersectionObserver mock OR call
   `triggerLoadMore()` via the ref handle.

The primitive is stable. Per-page adoption is a
routine swap that should land with its test update.

## Tests

`infinite-scroll.test.tsx` -- 22 cases:

1. Children render verbatim.
2. Sentinel rendered at the bottom.
3. IntersectionObserver attached to the sentinel.
4. `onLoadMore` fires when the sentinel intersects.
5. NOT fired when the sentinel is not intersecting.
6. NOT fired while `loading=true`.
7. NOT fired when `error` is set.
8. NOT fired when `hasMore=false`.
9. Default loading content renders + role=status.
10. Default end-of-list marker renders.
11. Default error card renders + retry fires
    `onRetry`.
12. Falls back to default error message when
    `error.message` is empty.
13. Retry button omitted when `onRetry` is unset.
14. Custom `loadingContent` override.
15. Custom `endContent` override.
16. ReactNode `errorContent` override.
17. Factory `errorContent` override.
18. `data-state` reflects all four states.
19. `rootMargin` forwarded to IntersectionObserver.
20. Imperative `triggerLoadMore` + `getSentinel`
    handle.
21. `triggerLoadMore` no-ops while loading=true.
22. Observer disconnected on unmount.

22/22 pass against vitest 4.1.5 + jsdom 29.1.1 (with
an inline IntersectionObserver mock).

## Out of scope

- **Variable-row-height virtualization.** This is
  the `VirtualizedList` (v1.11.351) primitive's
  territory. The two compose: a virtualized list can
  use `<InfiniteScroll>` as its outer wrapper to
  drive `onLoadMore` when the virtual scroll position
  approaches the bottom.
- **Cursor-state management.** The primitive does NOT
  own `{ cursor, loading, error, items }` -- the
  adopter does. A future patch could ship a companion
  `useInfiniteCursor` hook that pairs with this
  primitive to encapsulate the common
  `apiGet('/api/...?cursor=' + cursor)` pattern, but
  the cursor shape varies enough across endpoints
  that a generic helper is out of scope here.
