## API

```tsx
<ThemeCustomizer
  initialConfig={{ radius: 8, hue: 264, contrast: 50 }}
  storageKey="c4:theme-customizer"
  target="scope"
  exportFormat="css"
  onChange={(config) => persist(config)}
  onExport={(text, format) => recordSnapshot(text, format)}
  ariaLabel="Theme customizer"
  copyFeedbackMs={1500}
/>
```

```ts
interface ThemeConfig {
  radius: number;   // 0..24
  hue: number;      // 0..360
  contrast: number; // 0..100
}

interface ThemeCustomizerProps {
  initialConfig?: Partial<ThemeConfig>;
  storageKey?: string;
  target?: 'root' | 'scope';
  onChange?: (config: ThemeConfig) => void;
  onExport?: (text: string, format: 'css' | 'json') => void;
  className?: string;
  ariaLabel?: string;
  exportFormat?: 'css' | 'json';
  copyFeedbackMs?: number;
}
```

## Behaviour

### Three knobs

Each slider drives a slice of the
canonical token surface:

| slider | range | drives |
| --- | --- | --- |
| radius | 0..24 px | `--radius-{sm,md,lg,xl,2xl}` |
| hue | 0..360 | `--brand`, `--brand-hover`, `--brand-pressed`, `--brand-subtle` |
| contrast | 0..100 | `--text-primary`, `--text-secondary`, `--text-tertiary`, `--border-default` |

Radius ladder:
- `sm = base * 0.25`
- `md = base * 0.5`
- `lg = base * 0.75`
- `xl = base`
- `2xl = base * 1.5`

Hue family (canonical S+L preserved):
- `--brand:         H 80% 65%`
- `--brand-hover:   H 80% 72%`
- `--brand-pressed: H 75% 58%`
- `--brand-subtle:  H 50% 22%`

Contrast lightness ramp (linear in
contrast/100):
- `text-primary:   75..100%`
- `text-secondary: 55..85%`
- `text-tertiary:  40..65%`
- `border-default: 18..38%`

### Two host modes

| target | mechanism | scope |
| --- | --- | --- |
| `scope` (default) | inline `style` on root div | only descendants of the customizer |
| `root` | `documentElement.style.setProperty` | the whole document |

`target='root'` is reversible via the
exported `clearThemeCustomizerRoot()`
helper -- removes the 13 root-scoped
vars in a single call. Used by tests
+ adopters that need an "exit theming
mode" affordance.

### Initial state precedence

1. `localStorage[storageKey]` parsed
   JSON (if `storageKey` is set + the
   value parses + the resulting object
   is non-null).
2. `initialConfig` normalized.
3. `DEFAULT_THEME_CONFIG`.

Malformed JSON, `null`, missing key
all degrade to step 2.

### Persistence

When `storageKey` is set, the latest
config writes to localStorage on every
state change via JSON.stringify. Write
failures (quota / privacy mode) are
swallowed -- the in-memory state is
authoritative.

### Export

The "Copy" button:
1. Serializes the current config as
   CSS (`exportFormat='css'`, default)
   or JSON (`'json'`).
2. Fires `onExport(text, format)`
   regardless of clipboard
   availability.
3. If
   `navigator.clipboard.writeText` is
   available, writes there too; flips
   the button label to "Copied!" for
   `copyFeedbackMs` (default 1500ms).
4. Clipboard rejection is swallowed
   (the `onExport` path is the
   reliable side-channel).

CSS shape:
```css
:root {
  --radius-sm: 2px;
  --radius-md: 4px;
  --radius-lg: 6px;
  --radius-xl: 8px;
  --radius-2xl: 12px;
  --brand: 264 80% 65%;
  --brand-hover: 264 80% 72%;
  --brand-pressed: 264 75% 58%;
  --brand-subtle: 264 50% 22%;
  --text-primary: 220 15% 87.5%;
  --text-secondary: 220 10% 70%;
  --text-tertiary: 220 8% 52.5%;
  --border-default: 220 10% 28%;
}
```

JSON shape (normalized):
```json
{
  "radius": 8,
  "hue": 264,
  "contrast": 50
}
```

### Reset

The "Reset" button restores
`DEFAULT_THEME_CONFIG` (8 / 264 / 50)
and triggers the persistence +
onChange + style-application chain
just like a slider change.

### onChange contract

- Does NOT fire on initial mount (the
  caller already knows
  `initialConfig`).
- Fires after every state change with
  the new normalized config.
- Latest-handler-wins via a ref so
  closures from prior renders never
  leak.

## ARIA + data attributes

| location | attrs |
| --- | --- |
| root | `role=region`, `aria-label`, `data-section="theme-customizer"`, `data-target`, `data-radius`, `data-hue`, `data-contrast` |
| control row | `data-section="theme-customizer-control"`, `data-control-key={radius|hue|contrast}` |
| slider | native `<input type="range">`, `aria-label`, `data-control-input` |
| value badge | `data-control-value` |
| preview block | `data-section="theme-customizer-preview"` + `-preview-{brand,label,label-secondary,button}` |
| export btn | `data-section="theme-customizer-export"`, `data-copied="true|false"`, dynamic `aria-label` ("Copy CSS to clipboard" / "Copy JSON to clipboard") |
| reset btn | `data-section="theme-customizer-reset"`, `aria-label="Reset to defaults"` |

## Pairs with existing primitives

- The token consumers (`Button`,
  `Card`, `Input`, ...) already pull
  `hsl(var(--brand))` etc., so a
  `target='root'` mount instantly
  re-skins the entire app without
  any per-component changes.
- `<Slider>` (11.273) -- the
  customizer uses native
  `<input type=range>` to stay
  self-contained; an alternate
  variant on top of `<Slider>`
  belongs in a follow-on.

## Out of scope (deferred)

- Saturation control. Brand family
  canonical saturation values are
  preserved; saturation is a
  separate token-set knob.
- Light / dark theme toggle. The
  tokens.css is dark-themed; a
  light-mode swap belongs to a
  follow-on.
- Per-component overrides.
- Import-from-clipboard parser.
  Inverse of export; deferred.
- Cross-tab sync via `storage`
  event. The singleton is per-tab.
- Adoption sweep through existing
  components. The token-using
  components already consume the
  vars; opting in per page is a
  separate dispatch.
- Animated transitions between
  config changes. Sliders update
  instantly; CSS transitions on
  the affected properties belong
  in a separate motion layer.
