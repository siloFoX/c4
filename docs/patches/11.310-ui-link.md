# 11.310 -- Link primitive (v1.11.328)

New `Link` primitive at `web/src/components/ui/link.tsx`.
Component-scope only, no daemon-side change and no `c4`
CLI surface change.

## Why

Every anchor in the c4 web app needs the same set of
affordances:

- A consistent text colour + hover underline based on
  the design tokens (so the dark-mode / light-mode flip
  works without editing every link).
- External links automatically open in a new tab
  (`target="_blank"`) with `rel="noopener noreferrer"`
  so a malicious destination cannot reach back via
  `window.opener` and the referrer is not leaked.
- External links carry a trailing icon so operators
  see they will leave the app before clicking.

Today these affordances are inlined at every call site
(raw `<a>` tags with bespoke className strings) or
skipped entirely. A handful of docs-surface external
links omit `rel="noopener"` and rely on the "new tab"
intent alone, which is a known phishing vector.

Patch 11.310 ships the canonical Link primitive so
every anchor reaches for the same shape.

## What changed

### `web/src/components/ui/link.tsx` (new file)

A `forwardRef` `<a>` wrapper with:

- Three variants:
  - `default` -- primary-coloured link with hover
    underline. Main "action" links (CTAs, "View
    details", etc).
  - `muted` -- muted-foreground link with hover
    underline. Secondary / metadata links (timestamps
    that link to a detail page, breadcrumb segments).
  - `inline` -- inherits text colour, always-underlined.
    For "click here" affordances inside running prose.
- Internal/external auto-detection. Any href that
  matches `^(?:[a-zA-Z][a-zA-Z0-9+\-.]*:|\/\/)` is
  treated as external (covers `https:`, `http:`,
  `mailto:`, `tel:`, `ftp:`, etc, plus
  protocol-relative `//cdn.example.com/x.png`).
  Everything else (relative paths, absolute paths,
  anchor fragments) is internal.
- Auto-applied security tokens. External + `_blank`
  target adds `rel="noopener noreferrer"`. The
  caller's own `rel` is layered (no duplicates) so
  callers can append `nofollow` etc.
- Trailing external icon (lucide `ExternalLink`,
  3.5x3.5 with `aria-hidden="true"`) on external
  links by default.
- Override props:
  - `external?: boolean` -- force the classification
    when auto-detect guesses wrong.
  - `hideExternalIcon?: boolean` -- suppress the
    trailing icon (compact link rows).
  - `target` -- caller-supplied target wins
    (`_self` for an external link that should
    navigate in-place).
- Stable focus-visible ring across all variants.

### Exported helpers

- `isExternalHref(href: string): boolean` -- the same
  classification logic exposed as a function so call
  sites can branch without rendering a Link
  (e.g. choose whether to prefetch).

### Adopted

- Primitive only this commit. The dispatch asked for
  raw `<a>` tags across the app to be replaced --
  that sweep is a follow-up per-page patch so each
  migration lands with its own test refresh.

## Notes

- The external-icon-by-default is a slight UX
  trade-off: operators who already know a destination
  may find the icon noisy. The primitive supports
  `hideExternalIcon` for compact link rows, and the
  ARPS UX research bias is toward making the "you
  will leave the app" intent explicit by default.
- The security `rel="noopener noreferrer"` tokens are
  added ONLY when the target is `_blank`. Callers
  that override the target to `_self` (e.g. an
  external login flow that needs the referrer) get
  no rel tokens unless they pass their own. This
  matches the WHATWG default.
- `isExternalHref` is permissive about the scheme
  pattern (`[a-zA-Z][a-zA-Z0-9+\-.]*:`) so it
  catches both common schemes (`https:`, `mailto:`)
  and long-tail schemes (`obsidian:`, `vscode:`)
  without an explicit allowlist. Internal anchor
  fragments (`#x`) and absolute paths (`/x`) are
  intentionally excluded.
- The Link primitive does NOT integrate with React
  Router or Next.js -- it's a pure `<a>`-render so
  the consumer is free to wrap it with their router's
  navigation interceptor (e.g.
  `<Link href={to} {...}>...</Link>` inside a
  `<RouterLink>` component-binding). A router-aware
  variant can land in a follow-up if a clear use
  case emerges.
- The `external` prop is the escape hatch for the
  "auto-detect guessed wrong" case. Two scenarios
  motivate it:
  - An internal route exposed through a reverse
    proxy that includes the full `https://...` URL
    in the canonical href.
  - An external endpoint that the operator wants
    rendered as if it were internal (no new tab, no
    icon) because the navigation is part of a flow
    the operator already understands.
