## Goal

The dispatch lists:

- Audit every focusable surface and verify
  focus-visible ring renders correctly.
- Apply consistent focus-ring class via tailwind
  plugin or shared utility.

This patch ships the shared utility (4 canonical class
string constants) + a CI-runnable scanner that asserts
every focusable element in `web/src/components/ui` and
`web/src/pages` carries a `focus-visible:` ring. The
codebase is already well-disciplined; the audit
surfaces just 1 baseline violation. The scanner is
the regression gate going forward.

## Public API

```ts
// Constants:

export const FOCUS_RING_INSET:   string; // 2px primary, no offset
export const FOCUS_RING_SUBTLE:  string; // 1px primary/50 + 1px offset

export interface FocusRingViolation {
  file: string;
  line: number;
  column: number;
  tag: string;
  excerpt: string;

```

`APP_SHELL_FOCUS_RING` (v1.11.343) is a back-compat

## Audit scanner

`focus-ring-audit.ts` walks supplied source files and
flags focusable elements with no `focus-visible:`
prefixed class in their open-tag content.

### Focusable tag patterns

- Native: `<button>`, `<a>`, `<input>`, `<select>`,
  `<textarea>`.
- Role-based: any element with
  `role="button"` / `link` / `menuitem` / `tab` /
  `switch` / `option` / `checkbox` / `radio` /
  `searchbox` / `textbox`.

### Skip rules

The scanner skips elements with:

- `type="hidden"`
- `hidden` (bare attribute)
- `aria-hidden="true"`
- `tabIndex={-1}` or `tabIndex="-1"`

Plus block + line comments are stripped before regex
matching; lines starting with
`import` / `export` / `type` / `interface` are
ignored.

### Double-count guard

`<button role="button">` is reported once (via the
native-tag pass). The role pass skips when the tag
name is already in the focusable-tag set.

### Integration ratchet

```ts
export const MAX_KNOWN_FOCUS_RING_VIOLATIONS = 1;
```

`scanFilesForFocusRingViolations(uiFiles +
pageFiles)` must return `<=` this many violations.
The current canonical baseline is 1 (a single
existing inline `<a>` or similar that does not carry
a `focus-visible:` class). Follow-up cleanup patches
should lower this constant when they fix existing
violations.

## Adoption pattern

```tsx
import { FOCUS_RING_DEFAULT } from '../../lib/focus-ring';
import { cn } from '../../lib/cn';

<button className={cn('rounded-md bg-primary px-3 py-1', FOCUS_RING_DEFAULT)}>
  Save
</button>
```

The constant + `cn()` composition keeps the call site
short and the focus-ring contract centralised. New
primitives should default to `FOCUS_RING_DEFAULT`;
reach for `FOCUS_RING_INSET` for compact rows where
the outset offset would overflow a parent, and
`FOCUS_RING_SUBTLE` for surfaces with already-heavy
chrome where the full ring would compete.

Existing primitives that inline the class string are
NOT required to migrate -- both forms produce
identical CSS and both satisfy the scanner.

## Why no Tailwind plugin?

The dispatch mentions "tailwind plugin or shared
utility". This patch ships the utility. A plugin that
intercepts every focusable selector in Tailwind
output would either:

- Use `@apply focus-visible:ring-*` inside the base
  layer, which competes with author-supplied class
  strings (the cascade is order-sensitive); or
- Generate a new `focus-ring` utility class that
  shorthand-expands to the canonical string, which is
  semantically identical to the constant.

The constant + scanner combination produces the same
gate behaviour without the plugin's surface area.
Adopters get a one-line import; the audit catches
regressions.

## Out of scope

- **Cosmetic migration of inline class strings to
  the constants.** Both forms produce identical CSS.
  A follow-up patch can swap them if the design
  system gets a new ring color, but neither the
  audit nor the runtime cares about the source-text
  shape.
- **Per-page audit beyond `web/src/pages` and
  `web/src/components/ui`.** Components in
  `web/src/components/` outside `ui/` are not
  scanned; the integration ratchet can be widened
  in a follow-up when the focus-ring discipline
  there is verified.
