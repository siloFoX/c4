# 11.201 -- UI: locale switcher

Drop the inline EN/KO toggle button in the web UI header in favour of a
dropdown that surfaces both locales side-by-side, matches the
AccountMenu / theme picker conventions, and reuses the existing
`useLocale()` / `setLocale()` plumbing in `web/src/lib/i18n.ts`.

## Files

- `web/src/components/LocaleSwitcher.tsx` (new) -- the component.
- `web/src/components/LocaleSwitcher.test.tsx` (new) -- vitest + RTL
  coverage (6 cases).
- `web/src/components/layout/AppHeader.tsx` -- swap the `Languages`
  `IconButton` for `<LocaleSwitcher />` in the actions slot.
- `web/src/components/layout/AppHeader.test.tsx` -- update the locale

## Design notes

### Why a dropdown

The header previously cycled the locale on every click (`en` -> `ko`
-> `en` -> ...). With only two locales today that is still a usability
papercut: the trigger does not advertise the other choice, screen
reader users have to flip blindly to discover the second locale, and
the surface stops scaling the moment a third locale lands. A dropdown
fixes all three without adding new primitives: the existing
hand-rolled `DropdownMenu` already drives the AccountMenu and is
keyboard- and a11y-tested.

### Trigger

Ghost `<Button>` with `Globe` (lucide) + uppercase locale code (`EN`
/ `KO`) + `ChevronDown`. The trigger width is stable across locales
because the code stays ASCII uppercase. `aria-label` resolves through
`t('common.language')` -- the i18n key already existed for the old
button, so no new keys land. `data-locale` is exposed on the trigger
so tests + future styling can target the active locale without
re-rendering against the label string.

### Menu items + active state

Items list both `LOCALES` entries by native name: `English` and
`Hangul`. Native names so a user landing in the wrong locale still
recognises their language (`Korean` in English copy is useless if the
operator only reads Hangul). The `DropdownMenu` primitive does not
model radio groups, so the active row uses the existing `hint` slot
to render a `Check` icon (lucide). A `data-testid` of
`locale-check-<code>` is attached to the icon for stable test
selectors. The inactive row's hint slot stays undefined so the
primitive's existing layout takes over.

### Selecting the active row is a no-op

`onSelect` short-circuits with `if (active) return;` before calling
`setLocale`. Without the guard, picking the current locale would
re-fire `c4:locale-changed` and force every `useLocale()` subscriber
to re-render for no observable change. The guard is local to the
component; `setLocale` semantics are unchanged.

### No page reload, no bundle reload

`web/src/lib/i18n.ts` already writes `localStorage[LOCALE_KEY]` and
dispatches the `c4:locale-changed` `CustomEvent` from `setLocale`,
and the `useLocale()` hook subscribes to the event via
`onLocaleChange`. Every mounted component that reads through `t()`
inside a `useLocale()` consumer (the AppHeader does this, AccountMenu
does this, DropdownMenu itself calls `useLocale()`) re-renders on
the event with the new bundle. No `window.location.reload()` is
required and none is wired -- the design note in the task is the
"document and leave alone" branch.

### AppHeader adoption

The actions slot previously rendered the `Languages` `IconButton`
inside a `Tooltip` block plus a `getLocale()` cursor for the inline
display. Both go away. `<LocaleSwitcher />` is mounted next to
`<ThemeToggle />` -- same visual cluster, one fewer cursor binding.
The unused `getLocale`, `setLocale`, and `Languages` imports are
removed. `useLocale()` and `t` remain because other surfaces in the
header (the sidebar toggle aria-label, the help-center tooltip, the
title) still go through them.

### Existing AppHeader test

`web/src/components/layout/AppHeader.test.tsx > flips locale en->ko`
is updated, not deleted, to keep coverage on the bundle-swap path:
the test opens the dropdown via the trigger (still labeled
`Language`) and clicks the Korean menuitem (matched by the native
label regex). The downstream assertion -- that the `Workers` tab
label no longer renders in English after the swap -- is unchanged.

## Test plan

`web/src/components/LocaleSwitcher.test.tsx` ships 6 vitest + RTL
cases:

1. Trigger renders with the Globe icon + uppercase locale code +
   `data-locale="en"` on first paint.
2. Clicking the trigger opens the menu and both locale menuitems
   render (English + native Korean label, regex-matched so the
   case is locale-bundle-independent).
3. The active locale row carries a `Check` icon
   (`locale-check-en`) and the inactive row does not.
4. Selecting the inactive row calls `setLocale`, flips
   `getLocale()` to `ko`, and writes `ko` to
   `localStorage[LOCALE_KEY]`.
5. Selecting the already-active row is a no-op: no
   `c4:locale-changed` event fires and `getLocale()` does not
   move.
6. Keyboard navigation: opening via click then `{ArrowDown}`
   highlights row 0 (rAF-deferred focus is awaited via `waitFor`),
   the next `{ArrowDown}` advances to row 1, and `{Enter}` on the
   focused menuitem activates `onSelect` and flips the locale.

Test command:

```
env -C /root/c4-worktree-auto-w185/web /root/c4/web/node_modules/.bin/vitest run src/components/LocaleSwitcher.test.tsx
```

Result: 6 / 6 passing. `AppHeader.test.tsx` regression check: 21 /
21 passing under the same command pointed at the layout test path.
`web/node_modules` is shared with the primary checkout via a symlink
in this worktree.

## Out of scope

- New i18n keys (the existing `common.language` is reused).
- Reload on switch (the existing event-driven `useLocale()` already
  reflows mounted components).
- More than two locales (the component iterates `LOCALES` so adding
  a third locale only touches `web/src/lib/i18n.ts`).
- Touching `docs/autonomous-queue-v10.md` (manager owns the queue).
- Touching `package-lock.json`.
