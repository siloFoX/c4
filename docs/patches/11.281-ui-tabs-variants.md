# 11.281 -- Tabs primitive: variants + scroll-into-view (v1.11.299)

Component-scope-only enhancement of the existing `Tabs` primitive.
The current `pill` look is preserved byte-for-byte (via the default
`variant`) and the existing `TopTabs` adapter continues to work
unchanged.

## Why

The existing Tabs ships exactly one visual variant: rounded-border
tablist + filled-background active tab (the "pill" look). For
in-page section navigation a thinner Material / iOS-style underline
indicator reads better than the chunky pill. Patch 11.281 adds the
second variant + brings the overflow handling under explicit props
so callers can opt into horizontal-scroll OR flex-wrap depending on
the host layout.

## What changed

### `web/src/components/ui/tabs.tsx`

- `variant?: 'pill' | 'line'` prop. `pill` (default) preserved
  byte-for-byte. `line` swaps in a transparent tablist + bottom-
  border divider + 2 px primary-color underline drawn beneath the
  active tab. Underline is `<span aria-hidden>` so SR users still
  hear `aria-selected`.
- `overflow?: 'scroll' | 'wrap'` prop. `scroll` (default) keeps
  the existing horizontal scroll + invisible scrollbar. `wrap`
  drops scroll classes and adds `flex-wrap`.
- `scrollOnFocus?: boolean` (default true). On `value` change,
  the new active tab is brought into view via
  `scrollIntoView({ block: 'nearest', inline: 'nearest' })`.
  Guarded against jsdom (function undefined there).
- `data-section="tabs"` on tablist + `data-variant` +
  `data-overflow` attrs + per-tab `data-tab-active="true|false"`.
- `data-section="tab-underline"` on the line-variant underline.
- `CSS.escape(value)` when available so a tab value with reserved
  selector characters still resolves to a unique
  `[data-tab-value="..."]` selector.

### Adopted

- `web/src/pages/DesignSystem.tsx` -- new `<TabsDemo>` rendering
  the pill + line variants side by side with a 5-item
  `TABS_ITEMS` array so designers can scan the visual difference
  and the keyboard nav at the same time.

### Deferred

- Settings sections: currently a stack of `<Panel>` components
  (Appearance / Layout) with inline ChoiceGroup widgets.
  Converting to a Tabs layout requires either folding the
  panels into a single tab-switch or duplicating panel chrome
  inside tab panels.
- Health categories: page already adopted Accordion in 11.272.
  A Tabs-vs-Accordion swap needs a dedicated UX pass.
- Profiles flat groupings: page renders a flat searchable list,
  not grouped tabs. Adding Tabs needs new grouping logic on the
  data shape.

### Tests

- 11 new vitest cases: default variant=pill + rounded border,
  variant=line + border-b + no rounded, line variant renders
  underline only under active tab, pill variant has no
  underline, default overflow=scroll + overflow-x-auto,
  overflow=wrap + flex-wrap, data-tab-active flip, data-section

  scrollOnFocus=false skips. 35 (24 + 11) green.
