## Goal

The dispatch lists:

- Build `web/src/lib/shortcut-conflicts.ts` -- at
  startup, validate that registered shortcuts don't
  conflict (e.g., same key with same modifiers in
  same scope).
- Report violations in dev console + add to
  KeyboardShortcutsModal.

This patch ships the detector + 24 unit/integration
tests + the modal banner integration.

## Public API

```ts
export type ConflictKind = 'conflict' | 'duplicate';

export interface ShortcutSpec {
  keys: string;
  action: string;
  scope?: string;
}

export interface ShortcutConflict {
  combo: string;
  scope: string;
  kind: ConflictKind;
  specs: ShortcutSpec[];
}

export function normalizeCombo(keys: string): string;
export function detectShortcutConflicts(specs: readonly ShortcutSpec[]): ShortcutConflict[];
export function formatShortcutConflicts(conflicts: readonly ShortcutConflict[]): string;
export function reportShortcutConflicts(conflicts: readonly ShortcutConflict[]): number;
```

## Normalisation

`normalizeCombo` produces a canonical string suitable
as a map key. Rules:

1. Lower-case everything.
2. Strip whitespace around `+`.
3. Collapse synonym modifier names:
   `Cmd`/`Command`/`Win` -> `meta`,
   `Control` -> `ctrl`, `Option` -> `alt`.
4. Order modifiers as `ctrl, alt, shift, meta`.
5. Trailing single token is the key (lower-cased).
6. Chord combos (`g g`, `Ctrl+K P`) split on
   whitespace into ordered steps; each step is
   normalised independently and joined with a single
   space.

Examples:

| input | output |
| --- | --- |
| `Ctrl+B` | `ctrl+b` |
| `Shift+Ctrl+B` | `ctrl+shift+b` |
| `Cmd+K` | `meta+k` |
| `g g` | `g g` |
| `Ctrl + B` | `ctrl+b` |
| `Ctrl++B` | `ctrl+b` |

## Conflict vs duplicate vs synonym

| pattern | combo | action | report |
| --- | --- | --- | --- |
| same combo + same action (twice) | match | match | `duplicate` |
| same combo + different action | match | differ | `conflict` |
| different combo + same action | differ | match | none (SYNONYM, e.g. `?` + `Shift+/` -> help) |
| different combo + different action | differ | differ | none |

Scope isolation: two specs sharing `(combo, action)`
but with different `scope` values never conflict --
they live in independent surfaces (modal vs editor,
worker pane vs sidebar).

## Modal integration

`KeyboardShortcutsModal` runs the detector once via
`useMemo` (the list is module-level constant so the
memo only fires once). When `conflicts.length > 0`
the modal renders a warning banner above the
search-empty state:

```tsx
<div role="alert" data-testid="shortcuts-conflicts-banner" className="...">
  <AlertTriangle />
  <p>{conflicts.length} shortcut{plural} conflict</p>
  <pre>{formatShortcutConflicts(conflicts)}</pre>
</div>
```

The banner is invisible on the happy path. When a
new binding lands that collides with an existing
one, the operator sees the warning the moment they
open the modal, and the integration test in CI fails
loudly.

integration test locks this baseline.

## Out of scope

- **Per-component shortcut registry.** The hook
  ecosystem (`useKeyboardShortcut`,
  `useShortcutSequence`) has multiple sources of
  truth. A central registry that all of them feed
  into would let the detector cover dynamic
  registrations too. That refactor is a separate
  patch.
- **Conflict-resolution UI.** The modal surfaces the
  count + the summary. A future patch could add a
  "Disable conflicting binding" affordance, but most
  adopters fix the conflict in code rather than at
  runtime.
