## Goal

Dispatch:

- Build `web/src/lib/form-validation.ts` -- Zod
  schema-based validation with field-level
  errors, async validators, Tailwind error
  styling, ARIA error association.

`lib/form-validation.ts` already shipped in
11.168 (v1.11.186) with the synchronous
`Validator<T>` primitives (`required`,
`minLength`, `maxLength`, `pattern`, `email`,
`custom`, `compose`). This patch layers the
schema + async + ARIA + Tailwind surface on top
and adds a `useForm` hook so adopters get a
zod-shaped API without the zod runtime
dependency.

## Why not zod?

The project carries no `zod` dependency today.
Adding one for a form-validation surface that
the existing `Validator<T>` shape already covers
would be disproportionate. The new schema API
mirrors the zod ergonomics
(`{ field: validator }`, `{ ok, errors }`
result) without the bundle hit; future migration
to zod is a drop-in once an adopter actually
needs the richer parsing.

## Schema validators

```ts
type AsyncValidator<T> = (value: T) => Promise<ValidationResult>;
type AnyValidator<T>   = Validator<T> | AsyncValidator<T>;
type Schema<T>         = { [K in keyof T]?: AnyValidator<T[K]> };
```

```ts
validateSchema(schema, values): { ok, errors, valid }
validateSchemaAsync(schema, values, { onAsyncError? }): Promise<{ ok, errors, valid }>
```

- The sync pass invokes every validator and
  skips any that return a thenable (async).
- The async pass runs everything in parallel via
  `Promise.all`; async rejections surface as
  `'Validation failed'` (or a custom message via
  `onAsyncError`).
- Sync throws inside a validator body are caught
  and surfaced the same way.

## ARIA + Tailwind helpers

```ts
ariaErrorProps(error, errorId, describedByExtra?):
  { 'aria-invalid'?: true; 'aria-describedby'?: string }

fieldErrorClass(hasError): string
hasAnyError<T>(errors): boolean
```

`ariaErrorProps` returns an empty object when
the field has no error so a spread does not
toggle the attributes. When `describedByExtra`
is set the helper appends to `aria-describedby`
(lets adopters chain a hint id + an error id).

`fieldErrorClass` is the canonical Tailwind
error border + ring class for a bespoke input
that does not flow through the Input primitive.

## `useForm`

```tsx
const form = useForm({
  initialValues: { name: '', email: '' },
  schema: {
    name: required(),
    email: compose(required(), email()),
  },
});
```

| key | shape |
| --- | --- |
| `values: T` | current values |
| `errors: SchemaErrors<T>` | per-field error map |
| `touched: { [K]?: boolean }` | per-field touched flag |
| `isValid: boolean` | derived from `errors` |
| `isSubmitting: boolean` | submit-in-flight |
| `setValue(field, value)` | mutator |
| `setValues(next)` | bulk mutator |
| `setError(field, error)` | mutator |
| `setErrors(next)` | bulk mutator |
| `setTouched(field, isTouched)` | mutator |
| `validate(): SchemaErrors<T>` | sync pass |
| `validateAsync(): Promise<SchemaErrors<T>>` | full pass |
| `reset(next?)` | clears errors + touched |
| `field(name): FormFieldHandle` | bundled handle |
| `handleSubmit(onValid, onInvalid?)` | submit wrapper |

`field(name)` returns:

```ts
{
  value, onChange, onBlur,
  error, touched,
  id, errorId,
  props,          // ariaErrorProps bundle
  errorClass,     // Tailwind class
}
```

Stable ids: each field gets a stable
`${baseId}-${name}` so `aria-describedby`
references resolve across re-renders. `useId()`
from React provides the SSR-safe base id.

`handleSubmit` touches every field, runs the
full async pass, calls `onValid(values)` only
when validation succeeds, otherwise
`onInvalid(errors)`. `isSubmitting` flips
true / false around the callback.

`validateOnChange` (default `true`) gates the
per-keystroke validation so only touched fields
re-validate. `validateOnBlur` (default `true`)
runs validation on every blur even if the field
was clean. Both are tunable.

## Pairs with existing primitives

- `lib/form-validation.ts` legacy validators
  unchanged (backward compatible).
- `components/ui/input.tsx` /
  `components/ui/textarea.tsx` /
  `components/ui/select.tsx` -- each already
  accepts `error` + auto-renders the
  `aria-describedby` error span. The new
  helpers ride along.
- `lib/i18n.ts` -- adopters wrap the validator
  messages through `t(key)` for locale-aware
  errors.

## Out of scope

- Per-page adoption (Login / Settings /
  WikiPage forms still use direct state +
  inline validation).
- Field array / nested object validation.
- File / blob validation integration with
  `lib/file-upload.ts::validateFile`.
- Cross-field rules (confirm-password). The
  schema is per-field today; an additional
  `crossFieldValidators` slot is a follow-up.
- zod adoption (deferred until an adopter
  needs richer parsing).
