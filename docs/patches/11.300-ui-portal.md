# 11.300 -- Portal primitive (v1.11.318)

New `Portal` primitive at `web/src/components/ui/portal.tsx`.
Component-scope only, no daemon-side change and no `c4` CLI
surface change.

## Why

The c4 web app currently calls `createPortal` from
`react-dom` directly in seven places (Toast container,
CommandPalette, Drawer, Tooltip wrapper, Dialog, Popover,
ContextMenu). Each call site repeats roughly the same
boilerplate:

1. Import `createPortal` from `react-dom`.
2. Probe `typeof document !== 'undefined'` to avoid the
   classic SSR `document is not defined` crash.
3. Reach for a portal target -- usually
   `document.body`, sometimes a hand-rolled lookup of a
   well-known div id.
4. Sometimes guard the first client render so the SSR
   markup and the first client render match (avoiding the
   React hydration mismatch warning).

Steps 2 + 4 are silently inconsistent across the codebase --
some surfaces are SSR-safe, others are not, and the hydration
contract is enforced only where it was remembered. The
pre-existing `getPortalRoot` helper at
`web/src/lib/portal-root.ts` centralises step 3 (the named
mount-point lookup), but each caller still has to write its
own `createPortal` + SSR guard wrapper around it.

Patch 11.300 ships the canonical `Portal` primitive so every
overlay surface (and any new one) reaches for the same
abstraction.

## What changed

### `web/src/components/ui/portal.tsx` (new file)

- `Portal` function component. Three guarantees:
  1. **SSR-safe lazy mount.** The first synchronous render
     (SSR pass and the first client render) returns `null`.
     The portal target is resolved inside a `useEffect`, so
     the second client render is the first time the children
     are materialised. This avoids both the SSR
     `document is not defined` crash and the hydration
     mismatch warning when the server output (no portal
     child) is compared against the client output (portal
     mounted).
  2. **Idempotent named mount.** The default
     `containerId='app-portal-root'` is resolved via the
     existing `getPortalRoot` helper, which creates the
     `<div id="app-portal-root" data-portal-root="true">`
     under `document.body` lazily on first call and reuses it
     on subsequent calls. Other well-known ids
     (`'toast-root'`, `'dialog-root'`, `'popover-root'`,
     `'dropdown-root'`) are supported byte-for-byte
     identically.
  3. **`disabled` opt-out.** When `disabled={true}` the
     children render inline (a transparent Fragment at the
     Portal element's location). Useful for tests that want
     to assert against a known container, and for
     progressive enhancement where the portal target is
     intentionally unavailable.
- Props:
  - `children: ReactNode` -- required, the subtree to portal.
  - `containerId?: PortalRootId` -- named portal target.
    Defaults to `'app-portal-root'`.
  - `container?: HTMLElement | null` -- explicit DOM target,
    takes precedence over `containerId`. `null` is honoured
    as a "not yet" state (defers the mount) so callers can
    bind a ref before the target is live.
  - `disabled?: boolean` -- inline-render opt-out (default
    false).
- Stable `displayName = 'Portal'` for devtools.
- Re-exported from `web/src/components/ui/index.ts`.

### Adopted

- Primitive only this commit. The dispatch listed Toast
  container, CommandPalette, Drawer, and Tooltip as
  adoption sites. Threading `<Portal containerId="...">`
  into those call sites is a follow-up per-component patch
  so the blast radius of this commit stays small and the
  per-site test refresh lands with each adoption.

## Deferred (dispatch follow-ups)

Each of the following call sites currently calls
`createPortal` from `react-dom` directly and can be swapped
to the new primitive once the per-page test suite is
updated:

- `web/src/components/Toast.tsx` -- the legacy toast root
  (mounts at `getToastRoot()`).
- `web/src/components/ui/toast.tsx` -- the v1.11.298 toast
  provider.
- `web/src/components/ui/command-palette.tsx` -- the
  v1.11.295 command palette.
- `web/src/components/ui/drawer.tsx` -- the drawer.
- `web/src/components/ui/dialog.tsx` -- the dialog.
- `web/src/components/ui/popover.tsx` -- the popover (uses a
  computed `target` ref).
- `web/src/components/ui/context-menu.tsx` -- the context
  menu (inline createPortal call).

A representative swap looks like:

```tsx
// Before
import { createPortal } from 'react-dom';
// ...
return createPortal(node, getPortalRoot('toast-root'));

// After
import { Portal } from './portal';
// ...
return <Portal containerId="toast-root">{node}</Portal>;
```

The Portal's SSR guard subsumes the
`typeof document !== 'undefined'` checks that some call sites
duplicate today.

## Tests

`web/src/components/ui/portal.test.tsx` -- 14 vitest cases:

- Returns `null` on the first synchronous render before
  effects flush (SSR-safe).
- Mounts children into the canonical `app-portal-root` by
  default.
- Reuses the same portal root across multiple Portal
  instances (idempotent via `getPortalRoot`).
- `containerId="toast-root"` routes to a distinct DOM
  target; the canonical `app-portal-root` is not created.
- Explicit `container` HTMLElement escape hatch mounts into
  the supplied node; canonical `app-portal-root` is not
  created.
- `container={null}` defers the mount; children are not
  yet rendered.
- `disabled={true}` renders children inline at the Portal's
  location; no portal root is created.
- `disabled` flip from true -> false moves children from
  inline to the portal target.
- Props update propagation (children content changes).
- `containerId` switch moves the subtree between named
  targets.
- Clean unmount: subtree removed from the portal root.
- Stable `displayName = 'Portal'` for devtools.
- Multiple children via implicit fragment.
- `data-portal-root="true"` attribute selector for e2e.

All 14 green via
`vitest run --project unit src/components/ui/portal.test.tsx`.

## Notes

- The Portal primitive is intentionally a function component
  (not `forwardRef`). `createPortal` returns a React element,
  not a DOM node -- there is nothing useful to forward a ref
  to from the call site's perspective. Callers that need a
  ref to the rendered content should attach it to a child
  element inside the portal.
- The SSR-safe lazy mount is implemented via a
  `useState<HTMLElement | null>(null)` + `useEffect` pair.
  The state defaults to `null`, so:
  - SSR: `useEffect` does not run, render returns `null`,
    server emits no portal child.
  - First client render: state is still `null`, render
    returns `null`, hydration succeeds (matches the server
    output).
  - Post-effect render: state is the resolved target, the
    portal mounts and the children land in their final DOM
    location.
- The `disabled` opt-out is a transparent `Fragment` wrapper,
  not a `<div>`. The Portal element itself contributes zero
  DOM at its tree location regardless of the `disabled`
  value -- the children are either inlined verbatim or moved
  to the portal target.
