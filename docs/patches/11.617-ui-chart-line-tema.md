## Summary

New `ChartLineTema` UI primitive at
`web/src/components/ui/chart-line-tema.tsx` -- a pure-SVG line chart
with a Triple Exponential Moving Average overlay drawn on the same
axes as the price, for very low lag smoothing.

## Indicator math

Patrick Mulloy's Triple Exponential Moving Average:

```
ema1 = EMA(value, period)
ema2 = EMA(ema1, period)
ema3 = EMA(ema2, period)
TEMA = 3 * ema1 - 3 * ema2 + ema3
```

A plain EMA lags the price. Nesting three EMAs and combining them
with the `3 / -3 / 1` coefficients cancels the lag to third order,
driving it lower than a double EMA can while keeping the line
smooth.

The EMA helper is the project's null-tolerant, SMA-seeded EMA: it
collects the defined values, seeds with the simple mean of the first
`period` of them at that value's index, then folds each later value
in at weight `2 / (period + 1)`. EMA1 is therefore defined from
index `period - 1`; EMA2 needs another `period` defined values so it
begins at `2 * period - 2`; EMA3 begins at `3 * period - 3`; and the
**TEMA is defined from index `3 * period - 3` onward**.

Because the TEMA is a third-order lag corrector, it reproduces a
linear price ramp with **zero lag** once warmed up.

### Worked fixture

`values = [3, 6, 9, 12, 15, 18, 21, 24, 27, 30]` (a perfect linear
ramp), `period 3`. The EMA multiplier is exactly `2 / (3 + 1) = 0.5`
and every window sum divides cleanly, so the whole three-stage
pipeline stays bit-exact:

```
ema1 = [.,., 6, 9, 12, 15, 18, 21, 24, 27]
ema2 = [.,.,.,., 9, 12, 15, 18, 21, 24]
ema3 = [.,.,.,.,.,., 12, 15, 18, 21]
tema = 3*ema1 - 3*ema2 + ema3 = [.,.,.,.,.,., 21, 24, 27, 30]
```

The TEMA reproduces the ramp value exactly from index 6 (`3*3-3`)
onward: `tema[6] = 21 = value[6]`, `tema[9] = 30 = value[9]`. Every
value is exact, so the tests assert the series with `toEqual` rather
than `toBeCloseTo`.

## API

- `getLineTemaFinitePoints(points)` -- drop points with a non-finite
  `x` or `value`.
- `normalizeLineTemaPeriod(period, fallback)` -- coerce to a
  positive integer; non-finite or sub-1 falls back, fractional
  floors.
- `computeLineTemaEma(src, period)` -- the null-tolerant SMA-seeded
  EMA series.
- `computeLineTema(values, period)` -- returns `{ ema1, ema2, ema3,
  tema }`.
- `runLineTema(points, options?)` -- sort finite points by `x`,
  compute the series, return `{ series, period, ema1, ema2, ema3,
  tema, samples, temaFinal, temaMin, temaMax, aboveCount,
  belowCount, ok }`. Each sample carries the price's `above` /
  `below` / `on` position versus the TEMA. `ok` is false for fewer
  than two points.
- `computeLineTemaLayout(options)` -- project the price and TEMA
  onto one shared panel. Returns the panel, ticks, the price path,
  the TEMA path, and TEMA markers.
- `describeLineTemaChart(data, options?)` -- ARIA description.

## Component

`ChartLineTema` is a `forwardRef` region (`role="region"`) with an
ARIA-linked description, a `role="img"` SVG, axis ticks and grid, a
config badge, a two-series legend (Price / TEMA) with toggle, a
hover/focus tooltip exposing the value, TEMA and position, and a
`motion-safe:animate-fade-in` entrance. Every node carries a
`data-section` attribute.

## Distinctness

- **vs chart-line-dema** -- the DEMA stops at two EMA stages
  (`2 * EMA - EMA(EMA)`, a second-order lag corrector). The TEMA
  adds a third nested EMA and uses the `3 / -3 / 1` coefficient
  combination for a third-order correction -- lower lag, but a
  longer `3 * period - 3` warm-up against the DEMA's
  `2 * period - 2`.
- **vs chart-line-zlema** -- the ZLEMA removes lag by feeding a
  *de-lagged input* (the price plus its own recent momentum)
  through a single EMA. The TEMA leaves the input untouched and
  removes lag through an *algebraic combination of three nested
  EMAs*.
- **vs chart-line-kama** -- the KAMA *adapts* its smoothing every
  bar from an efficiency ratio. The TEMA applies a *fixed*
  three-EMA formula regardless of market conditions.
