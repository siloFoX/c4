# 11.189 -- Snapshots (daemon + UI)

Snapshots let an operator save the current daemon config plus the
autonomous queue markdown to disk, then restore that combined state
later (e.g. before a risky migration of `config.json` or a bulk
rewrite of `docs/autonomous-queue-v10.md`).

## Daemon

New self-contained module `src/snapshots.js`, registered from
`src/daemon.js` with four routes:

- `GET  /api/snapshots`                  -- list saved snapshots
- `POST /api/snapshots`                  -- take a new snapshot
- `POST /api/snapshots/:id/restore`      -- restore a snapshot
- `DELETE /api/snapshots/:id`            -- delete a snapshot

The routes all live behind the existing `/api/*` JWT auth gate and
require the `CONFIG_RELOAD` RBAC action (same gate as the queue
editor write path).

Storage layout: each snapshot is a single JSON file under
`<repoRoot>/.c4/snapshots/<timestamp>-<short-id>.json` with the
shape

```json
{
  "id": "2026-05-14T16-30-00-123Z-ab12cd34",
  "label": "before-config-rewrite",
  "createdAt": "2026-05-14T16:30:00.123Z",
  "config": { ... parsed config.json ... },
  "queue":  "## raw queue markdown\n| 1.1 | ... |\n",
  "version": 1
}
```

The list endpoint reads the directory and returns metadata only
(`id`, `label`, `createdAt`, `configBytes`, `queueBytes`) sorted
newest first so the UI never has to fetch every blob just to render
the list.

Safety:

- All writes go through `writeAtomic` (tmp + rename, EXDEV fallback)
  so a half-written file never lands on disk.
- Config payloads are capped at 10 MB and queue payloads at 5 MB to
  bound memory.
- Snapshot ids are validated against `^[0-9A-Za-z._-]+$` before
  any filesystem access -- callers cannot pass `../../etc/passwd`
  style ids.
- Each mutating route emits a single audit line
  (`snapshot.create` / `snapshot.restore` / `snapshot.delete`)
  through the shared `_safeAudit` helper, so the trail survives
  daemon restarts.

The handler functions are pure (`fs` is injected) so
`tests/snapshots.test.js` drives them against a tmp repo without
spinning up HTTP -- same DI pattern as `queue-editor.js`.

## Web

New page `web/src/pages/Snapshots.tsx` in the Config category
(Archive icon). Layout:

- Header: `Snapshots` + `Take snapshot` button (right).
- Table: Label / ID / Created / Size / Actions
  (Restore + Delete, both `Dialog`-confirmed).
- Empty state: `Archive` icon + copy when zero snapshots come back.
- Errors: `Alert` for action failures, `ErrorState` for load failures.

Restore dialog warns about overwriting the live config + queue;
Delete dialog clarifies that only the snapshot file is removed.

Registry entry added with `feature.snapshots.label` /
`feature.snapshots.description` keys in both `en.json` and
`ko.json`.

  covering page header rendering, list rendering, empty state,
  Restore confirm dialog opening, and the full Take-snapshot
