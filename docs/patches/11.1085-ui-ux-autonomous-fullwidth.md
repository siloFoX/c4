# 11.1085 -- UX FIX: Autonomous dashboard should fill width (v1.11.1103)

## Scope

The Autonomous view's status stat-grid and escalation queue were
capped near ~630px and left-aligned at 1440, leaving ~810px empty on
the right.

## Root cause

The cap was NOT in AutonomousView itself (its cards are full-width
within their parent and the digest grid is already `md:grid-cols-4`).
Two flex items in the route chain failed to grow:

1. `PageTransition`'s **incoming layer** `<div>` is a flex item of the
   outer flex-row (`relative flex min-h-0 flex-1`) but had no
   `flex-1`/`w-full`, so it collapsed to its content's max-content
   width (~630px for AutonomousView) and left-aligned -- capping every
   route's content and leaving wide viewports empty on the right.
2. `AutonomousView`'s **root** `<div>` was `flex flex-col` with no
   `flex-1`/`w-full`, so even once its wrapper was full-width it would
   have stayed content-width.

## Change

- `web/src/components/PageTransition.tsx` -- the incoming layer gets
  `min-w-0 flex-1` so it fills the outer flex-row instead of shrinking
  to content width. Routes whose content does not opt into filling
  (no `flex-1`/`w-full` root) are visually unaffected -- the layer just
  no longer imposes a content-width cap.
- `web/src/components/AutonomousView.tsx` -- the root gets
  `w-full min-w-0 flex-1` (+ `data-section="autonomous-view"`) so it
  fills its wrapper. The cards then span the content width, the 4-col
  digest grid spreads across it, and the escalation list uses the full
  width.
- `web/src/lib/preferences.ts` -- added `'autonomous'` to
  `TOP_VIEW_VALUES`. It was missing, so a reload while on the
  Autonomous tab was rejected by `readEnum` and bounced the operator
  back to `workers` (a real reload-persistence bug). This also lets the
  e2e boot directly into the view. (`'settings'` stays excluded per the
  existing note; the other missing tabs were left out of this todo's
  scope.)

## Verification (mandatory visual)

`web/e2e/autonomous-fullwidth.spec.ts` (new) seeds
`localStorage c4.topView='autonomous'` (dismissing the onboarding
tour), stubs the autonomous status/digest/escalations endpoints (plus
auth/list/events), and at 1440 / 768 / 375 asserts the AutonomousView
root and the digest stat grid fill most of the viewport width
(> 85% / > 80% at 375). Screenshots captured to
`test-results/autonomous-fullwidth-{1440,768,375}.png`.

Red/green confirmed the diagnosis: before the fix the root measured
654px at 1440 (the ~630px cap) -- the test failed; after the
PageTransition + AutonomousView change all three widths pass
(`3 passed`).

Verified against a staging copy in `/root/c4/web` (real
`node_modules`) on a throwaway vite server (port 5192) so the live dev
server was never touched; the staging copy was reverted with
`git checkout` after the run.

## Design system

Layout-only change reusing the project's existing token-backed
semantic Tailwind utilities. No new colours, spacing, or radii.

## Risk

- `PageTransition` is shared across routes. The change only *enables*
  filling (adds `flex-1` to a layer that previously shrank to content);
  it does not force narrow content wider, so views without a
  filling root are unchanged. Views that already filled stay filled;
  views that were content-capped (e.g. the workers sidebar+main, which
  had the same empty-right-gutter) now use the available width, which
  is the desired direction.
- The `preferences.ts` change is additive (one enum value); the default
  remains `workers` for unset/invalid values.
