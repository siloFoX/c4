## Summary

New `ChartLineTrima` UI primitive at
`web/src/components/ui/chart-line-trima.tsx` -- a pure-SVG line chart
with a Triangular Moving Average overlay drawn on the same axes as
the price.

## Indicator math

The Triangular Moving Average is a doubly-smoothed average:

```
inner = SMA(values, p1)
TRIMA = SMA(inner, p2)
```

The two simple-moving-average sub-periods depend on the parity of
`period`:

- **odd period** -- both sub-averages span `(period + 1) / 2`.
- **even period** -- the first spans `period / 2 + 1`, the second
  `period / 2`.

Either way the two sub-periods sum to `period + 1`, so the combined
warm-up is `period - 1` -- the same as a plain simple moving
average. Composing the two box-car averages makes the effective
weights form a **triangle** peaking in the middle of the window, so
the TRIMA emphasises the centre of the window and produces an
exceptionally smooth line.

### Worked fixture

`values = [4, 8, 16, 8, 4, 8, 16]`, `period 3`. An odd period splits
into two sub-averages of 2, so the TRIMA is `SMA(SMA(values, 2), 2)`
and -- with these window values dividing cleanly -- stays bit-exact:

```
inner = SMA(values, 2) = [null, 6, 12, 12, 6, 6, 12]
trima = SMA(inner, 2)  = [null, null, 9, 12, 9, 6, 9]
```

The price runs above the TRIMA on the spikes and below it in the
troughs (aboveCount 3, belowCount 2). Every value is exact, so the
tests assert the series with `toEqual`.

## API

- `getLineTrimaFinitePoints(points)` -- drop points with a
  non-finite `x` or `value`.
- `normalizeLineTrimaPeriod(period, fallback)` -- coerce to a
  positive integer; non-finite or sub-1 falls back, fractional
  floors.
- `lineTrimaSubPeriods(period)` -- the two SMA sub-periods
  `{ first, second }` whose composition makes the TRIMA; they sum
  to `period + 1`.
- `computeLineTrimaSma(src, period)` -- a null-tolerant windowed
  simple moving average; a window containing any `null` reads
  `null`.
- `computeLineTrima(values, period)` -- returns `{ inner, trima }`.
- `runLineTrima(points, options?)` -- sort finite points by `x`,
  compute the series, return `{ series, period, firstPeriod,
  secondPeriod, inner, trima, samples, trimaFinal, trimaMin,
  trimaMax, aboveCount, belowCount, ok }`. Each sample carries the
  price's `above` / `below` / `on` position versus the TRIMA. `ok`
  is false for fewer than two points.
- `computeLineTrimaLayout(options)` -- project the price and TRIMA
  onto one shared panel. Returns the panel, ticks, the price path,
  the TRIMA path, and TRIMA markers.
- `describeLineTrimaChart(data, options?)` -- ARIA description.

## Component

`ChartLineTrima` is a `forwardRef` region (`role="region"`) with an
ARIA-linked description, a `role="img"` SVG, axis ticks and grid, a
config badge calling out the period and the two sub-periods, a
two-series legend (Price / TRIMA) with toggle, a hover/focus tooltip
exposing the value, TRIMA and position, and a
`motion-safe:animate-fade-in` entrance. Every node carries a
`data-section` attribute.

## Distinctness

- **vs chart-line-wma** -- the WMA weights the window with a linear
  ramp peaking at the *newest* bar. The TRIMA's triangular weights
  peak in the *middle* of the window, so it lags more but smooths
  more.
- **vs a simple moving average** -- the SMA weights every value in
  the window equally. The TRIMA is the SMA *of an SMA*, giving the
  triangular centre-weighted profile -- it smooths twice.
- **vs chart-line-dema / chart-line-tema** -- those combine
  recursive exponential averages to *cut* lag. The TRIMA
  deliberately *adds* smoothing -- its goal is a maximally smooth
  line, the opposite trade-off.
