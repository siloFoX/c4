# 11.207 -- UI: confirm-dialog hook

Promise-based `useConfirm()` hook for the web UI. Replaces the native
`window.confirm` blocking modal with a `Dialog`-rendered prompt that
fits the ARPS design system surfaces and is themable / mockable in
tests.

## Files

- `web/src/hooks/use-confirm.tsx` (new, ~140 lines) -- `ConfirmProvider`,
  `useConfirm`, `ConfirmOptions`, `ConfirmTone`.
- `web/src/main.tsx` -- mounts `<ConfirmProvider>` between
  `<ErrorBoundary>` and `<App />` so the hook is available app-wide.
- `web/src/lib/use-worker-action-strip.ts` -- example callsite. Swaps
  `window.confirm(action.confirm)` for `await confirm({ title, message,
  confirmLabel, tone })`.
- `web/src/lib/use-worker-action-strip.test.ts` -- mocks `useConfirm`
  to defer to `window.confirm` so the existing test bed keeps its
  legacy spy-based assertions intact.
- `web/src/hooks/use-confirm.test.tsx` (new, 12 cases) -- dedicated
  unit coverage for the new hook.

## API

```ts
interface ConfirmOptions {
  title: string;
  message?: string;
  confirmLabel?: string;   // default 'Confirm'
  cancelLabel?: string;    // default 'Cancel'
  tone?: 'default' | 'destructive'; // default 'default'
}

function useConfirm(): (opts: ConfirmOptions) => Promise<boolean>;
```

Resolves `true` on Confirm button click, `false` on Cancel button
click, Escape key, or backdrop click.

## Behaviour

- **Rendering** -- the provider renders the active request through the
  existing `Dialog` primitive (`web/src/components/ui/dialog.tsx`).
  Confirm button is `autoFocus`-marked so keyboard users can answer
  the prompt without reaching for the mouse.
- **Destructive tone** -- `tone='destructive'` maps to the existing
  `Button` `variant='destructive'` (red surface). Default tone maps to
  `variant='default'` (primary). The Cancel button is always
  `variant='outline'`.
- **Concurrency: last-wins** -- if a second `confirm()` is called
  while one is pending, the prior promise resolves `false` and the
  new request becomes the active dialog. Picked over a FIFO queue
  because the typical surface (a single destructive action button)
  cannot realistically produce a meaningful queue and a stale
  pending prompt would be confusing.
- **Single-resolve** -- internal id ref guards every settle so repeat
  clicks after the first answer are no-ops (the promise resolves
  exactly once).
- **Unmount safety** -- the provider's unmount effect resolves any
  in-flight promise as `false`, so a route change cannot leave a
  dangling pending value awaiting an answer.
- **Without provider** -- `useConfirm()` throws with the message
  `useConfirm() must be used inside a <ConfirmProvider>.` so misuse
  is loud at development time.

## Example callsite

`web/src/lib/use-worker-action-strip.ts` is the row-action runner
behind the per-worker toolbar's Merge / Approve / Interrupt / Close
buttons. Old gate (`window.confirm`):

```ts
if (!window.confirm(action.confirm)) return;
```

New gate (`useConfirm`):

```ts
const ok = await confirm({
  title: action.label,
  message: action.confirm,
  confirmLabel: action.label,
  tone: action.variant === 'destructive' ? 'destructive' : 'default',
});
if (!ok) return;
```

The Close action carries `variant: 'destructive'` in its
`ActionConfig`, so it picks up the red destructive button styling
automatically. Merge / Approve / Interrupt all flow through with the
default tone.

## Test coverage

12 vitest + RTL cases in `web/src/hooks/use-confirm.test.tsx`:

1. confirm() opens a Dialog with the given title and message
2. Confirm button click resolves true
3. Cancel button click resolves false
4. Escape key resolves false
5. Backdrop click resolves false
6. Destructive tone applies the `bg-destructive` Button variant
7. Default tone applies `bg-primary` and not `bg-destructive`
8. Custom `confirmLabel` / `cancelLabel` reach the rendered button text
9. Last-wins concurrency (first promise resolves false, second
   stays active and resolves true on Confirm)
10. Promise resolves exactly once even when followed by a stale click
11. Pending promise resolves false on provider unmount
12. `useConfirm()` throws outside a `<ConfirmProvider>`
