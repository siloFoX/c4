# 11.1086 -- UX FIX: console error sweep on main routes (v1.11.1104)

## Scope

Sweep uncaught console errors on the main routes and fix the top
offender. Per the functional audit (all 12 views) the dominant console
error was the MetricsBar `/api/metrics` poll returning 401 on EVERY
view even with a valid admin session.

## Fixed error: /api/metrics 401 for signed-in operators

### Symptom

On every view, the browser logged a failed-resource console error for
`GET /api/metrics` (401), and signed-in operators saw the
"Metrics paused -- sign in to resume" strip (the 11.1082 needs-login
state) even though their session was valid.

### Root cause

`web/src/lib/use-metrics.ts` polled with a bare
`fetch('/api/metrics')` -- no `Authorization` header. Every other
`/api/*` call goes through `apiFetch` (`lib/api.ts`), which attaches
`Authorization: Bearer <token>` from `localStorage['c4.authToken']`.
`/api/metrics` is auth-gated like the rest, so the header-less request
was rejected with 401. (11.1082 stopped the *flood* -- a 401 now stops
polling -- but did not address why the very first poll 401s for a
logged-in user.)

### Fix

`use-metrics.ts` now attaches the session token to the metrics request
(same source as `apiFetch`, via `getToken()`), so the poll succeeds for
signed-in operators. It deliberately does NOT route through `apiFetch`:
that helper's 401 path clears the token and fires `AUTH_EVENT` (a global
logout), which is too aggressive for a decorative, self-pausing poll --
a metrics 401 should not sign the operator out. The local 11.1082
needs-login behaviour on 401 is preserved. On any failing response the
failing URL is logged (`console.warn`, with `res.url`) so the gated
endpoint is easy to confirm.

## Verification (mandatory)

`web/e2e/console-clean.spec.ts` (new) signs in as an admin (seeds
`c4.authToken` so the app resolves to `authed` and `apiFetch` attaches
the header), then visits the root (Workers) plus two main navigation
destinations (History, Autonomous), collecting console errors AND
uncaught page errors across all three, and asserts the count is zero.

The `/api/metrics` stub is the regression guard: it returns 401 UNLESS
the request carries the `Authorization` header. So the pre-fix bare
fetch would 401 (failed-resource console error) and fail the test; the
fixed poll attaches the token -> 200 -> clean. The spec also asserts
directly that at least one captured `/api/metrics` request carried the
`Authorization` header.

Result: `1 passed` headless (vite throwaway server). The pre-existing
`use-metrics` unit suite still passes (`9 passed`) -- the fetch change
is transparent to its MSW interception.

(The gallery `ChartLineMomentum` crash is tracked separately as
11.1100 and is out of scope here; the sweep covers the root + two main
nav destinations, which are clean after this fix.)

## Risk

- The metrics poll now sends the token. For an authed operator it
  succeeds (no 401). A genuine expiry still 401s once, logs one warn,
  and pauses (needs-login) -- unchanged from 11.1082, no flood.
- `use-metrics` imports `getToken` from `lib/api` (no new dependency;
  same store the rest of the app uses).
