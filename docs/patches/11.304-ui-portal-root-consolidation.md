# 11.304 -- portal-root helper consolidation (v1.11.322)

Consolidates the inline "lazy-create + idempotent
decoration" pattern that previously lived in each portal
call site into a single canonical implementation in
`web/src/lib/portal-root.ts`. Component-scope only, no
daemon-side change and no `c4` CLI surface change.

## Why

The v1.11.318 `<Portal>` primitive (TODO 11.300) introduced
a clean React API for the basic "render children into a
named mount node" case. But two related concerns were
still scattered:

1. **Per-root decoration.** Some named portal layers need
   persistent CSS classes or `data-*` markers on their
   mount node (the toast layer needs
   `pointer-events-none fixed right-4 top-4 z-50 ...` for
   positioning; future Dialog / Drawer overlays will want
   `data-overlay-root="true"` for theming hooks). These
   decorations have to be applied exactly once -- on
   first creation -- and survive subsequent `getRoot()`
   calls. Today `Toast.tsx::getToastRoot` re-implements
   this pattern inline:

   ```ts
   export function getToastRoot(): HTMLElement {
     const root = getPortalRoot('toast-root')!;
     if (!root.hasAttribute('data-toast-root')) {
       root.setAttribute('data-toast-root', 'true');
       root.className =
         'pointer-events-none fixed right-4 top-4 z-50 flex flex-col gap-2 ...';
     }
     return root;
   }
   ```

   Each future named layer (`getDialogRoot`,
   `getPopoverRoot`, etc) would have to copy this pattern.
   Drift hazards: the `hasAttribute` guard is easy to
   forget; the className stamp is easy to do unconditionally
   (re-stamping every render); the sentinel marker name is
   bespoke at each site.

2. **Canonical-pattern documentation.** Three different
   call shapes exist today (`<Portal>` primitive,
   `getPortalRoot(id)` raw, and the inline Toast.tsx
   pattern), and there is no single place that documents
   when to reach for which. New adopters have to read
   three files to figure out the policy.

Patch 11.304 fixes both:

- Adds a `definePortalRoot(id, descriptor)` factory that
  bundles the idempotent decoration into a typed getter.
- Extends `getPortalRoot(id, descriptor?)` with an optional
  descriptor for ad-hoc decoration without a factory.
- Refactors `Toast.tsx::getToastRoot` to delegate to the
  new factory.
- Documents the canonical pattern in the module-level
  JSDoc of `portal-root.ts`.

## What changed

### `web/src/lib/portal-root.ts`

Module-level JSDoc rewritten to document the three call
shapes and when to reach for each:

1. `<Portal>` primitive (preferred for new component code).
2. `getPortalRoot(id)` raw (for legacy `react-dom`
   `createPortal` call sites or imperative DOM ops).
3. `definePortalRoot(id, descriptor)` (when the named
   layer needs persistent decoration).

New exported type `PortalRootDescriptor`:

```ts
interface PortalRootDescriptor {
  className?: string;
  attributes?: Record<string, string>;
  decorationMarker?: string;  // sentinel, default `data-decorated-<id>`
}
```

New function `definePortalRoot(id, descriptor)` returns a
typed getter `() => HTMLElement | null`. The getter is
idempotent in both senses: same id always returns the same
element, AND the decoration step is applied exactly once
per page (the sentinel marker is checked before
re-stamping).

`getPortalRoot` extended with an optional second parameter
of the same `PortalRootDescriptor` type. Call sites that
don't need decoration still call `getPortalRoot(id)` with
no behaviour change.

`PortalRootId` union extended to include the
`'app-portal-root'` canonical default id used by the
`<Portal>` primitive. The `(string & {})` open-end is
preserved.

### `web/src/components/Toast.tsx`

`getToastRoot()` now delegates to
`definePortalRoot('toast-root', { className, attributes })`
instead of re-implementing the inline pattern. The
exported signature is unchanged (still returns
`HTMLElement`) so callers don't move.

```ts
const TOAST_ROOT_CLASSES =
  'pointer-events-none fixed right-4 top-4 z-50 ...';

const getToastRootInternal = definePortalRoot('toast-root', {
  className: TOAST_ROOT_CLASSES,
  attributes: { 'data-toast-root': 'true' },
});

export function getToastRoot(): HTMLElement {
  return getToastRootInternal()!;
}
```

All 55 existing `Toast.test.tsx` cases pass against the
refactored implementation -- the DOM shape (id, sentinel,
className) is byte-identical to the previous inline
implementation.

## Tests

`web/src/lib/portal-root.test.ts` gains 8 new vitest
cases on top of the existing 10 (18 total):

- `getPortalRoot(id, descriptor)` stamps className on first
  creation.
- `getPortalRoot(id, descriptor)` stamps custom attributes
  on first creation.
- Decoration is idempotent across repeated calls -- a
  mutated className survives the next `get()` call (the
  decorator does NOT re-stamp).
- Default `data-decorated-<id>` sentinel marker.
- Custom `decorationMarker` override.
- `definePortalRoot` returns a typed getter that reuses
  the same element across calls.
- `definePortalRoot` preserves `data-portal-root="true"`
  on the underlying element.
- `definePortalRoot` returns null in SSR.

Test cleanup tightened to purge by known id even when the
element lacks the `data-portal-root="true"` sentinel (the
"reuses pre-existing element" case left an undecorated div
behind).

## Notes

- The decoration sentinel (`data-decorated-<id>`) is
  intentionally a separate attribute from `data-portal-root`
  so the two contracts compose: `data-portal-root` marks
  "this is a portal mount node" (used by e2e selectors and
  cleanup helpers), while `data-decorated-<id>` marks
  "this node has been decorated by `definePortalRoot`"
  (used by the idempotency guard).
- `definePortalRoot` is called once at module load (the
  getter is stored on the module level), so the
  decoration descriptor itself is captured in the
  closure. Callers that want runtime-tunable decoration
  should pass the descriptor to `getPortalRoot(id, ...)`
  inline instead of using the factory.
- The Portal primitive (v1.11.318) does NOT need the new
  descriptor mechanism -- its responsibility is
  `createPortal` indirection + SSR-safe lazy mount, not
  decoration. Callers that need both can wrap the
  Portal's `containerId` in a `definePortalRoot` factory
  on the side and call the factory once at app startup
  to pre-stamp the className. Future migrations may add a
  `descriptor` prop to `<Portal>` if a cleaner one-line
  API is wanted.
