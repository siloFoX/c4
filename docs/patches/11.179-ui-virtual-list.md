# patch 11.179 -- UI: VirtualList primitive

Released in `c4-web` v1.11.197.

## What

New `<VirtualList>` primitive at
`web/src/components/ui/virtual-list.tsx`. Component-scope only; no
new dependencies. The component windows a long list by reading
`scrollTop` + measured `clientHeight` and only rendering the rows
in the visible range plus `overscan` extra rows above and below.
The inner spacer is sized to `items.length * itemHeight` so the
native scrollbar still reflects the full list. A
`ResizeObserver` watches the container so resizes recompute the
window; an optional `IntersectionObserver` sentinel at the bottom
of the spacer fires `onEndReached` for lazy-load callers.

Props:

| name              | type                                            | default     | notes |
|-------------------|-------------------------------------------------|-------------|-------|
| `items`           | `T[]`                                           | (required)  | data array |
| `itemHeight`      | `number`                                        | (required)  | fixed-height assumption, px |
| `renderItem`      | `(item: T, index: number) => ReactNode`         | (required)  | per-row renderer |
| `overscan`        | `number`                                        | `5`         | extra rows above/below viewport |
| `height`          | `number \| string`                              | `'100%'`    | container height |
| `estimatedHeight` | `number`                                        | `itemHeight`| spacer floor while items load lazily |
| `className`       | `string`                                        | -           | merged onto outer |
| `ariaLabel`       | `string`                                        | -           | forwarded as `aria-label` |
| `getKey`          | `(item: T, index: number) => string \| number`  | index       | key for each row |
| `onEndReached`    | `() => void`                                    | -           | sentinel-based lazy load signal |

Outer container has `role='list'`; each visible row wraps the
caller's rendered node in a div with `role='listitem'` so
assistive tech still sees a list. The component forwards its
ref to the scroll container itself.

## Adoption (1 site this patch; 2 attempted)

1. **`web/src/components/HistoryView.tsx`** -- left sidebar worker
   summary list (`itemHeight=64`, `height='60vh'`). Replaces a
   plain `<ul className="space-y-1">` over `summary` so a workspace
   with hundreds of historical workers renders O(viewport) rows
   instead of O(N). Tests stay green.

### Sites considered but not landed

- **`web/src/components/ChatMessageLog.tsx`** -- chat-message log.
  Ruled out because the parent owns a scroll ref + `onScroll`
  handler that detects scroll-to-top and triggers older-message
  backfill; VirtualList takes ownership of scrolling and exposes
  no `onScroll` passthrough today, so adoption would break the
  backfill flow. Marked as a follow-up once VirtualList grows an
  optional `onScroll` prop.
- **`web/src/components/SessionsListSection.tsx`** -- per-group
  session rows. Wiring was implemented (`itemHeight=76`,
  `height=min(items.length * 76, 480)`), but the existing
  `SessionsListSection.test.tsx` queries rely on the section
  rendering a single flat DOM subtree per group. Threading
  VirtualList in changes the cell layout (absolute positioning
  inside a spacer div) enough that the test suite needs an
  update before adoption is safe. Held back to keep this patch
  green; tracked for the follow-up that touches the test fixtures.

## Fixed-height tradeoff

VirtualList assumes uniform `itemHeight`. The landed adoption
honours that assumption:

- The HistoryView sidebar row is very nearly uniform: a name +
  badge line and a single muted meta line. `64px` matches the
  rendered height with margin to spare; no visible clipping at
  the typical content lengths.
For variable-height adopters like the deferred SessionsListSection
row (`ListItem` with optional `lastAssistantSnippet`) and the
deferred ChatMessageLog (free-form message bubbles), the same
fixed-height compromise applies: pick an `itemHeight` close to
the typical row and accept clipping on outliers, since the row
is a click target rather than the primary content surface.

If we later need true variable-height windowing, the natural
follow-up is a `getItemHeight?: (index: number) => number`
escape hatch plus a measured-offset prefix-sum cache, gated
behind the existing `itemHeight` prop so adopters keep their
current behaviour.

## What we deliberately did **not** do

- No new dependencies (no `react-window`, no `react-virtual`).
- No changes to other UI primitives.
- No edit to `docs/autonomous-queue-v10.md` (manager owns the queue).
- No `package-lock.json` modification.
