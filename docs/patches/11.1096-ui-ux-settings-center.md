# 11.1096 -- UX FIX: Settings column should be centered (v1.11.1114)

## Finding

The dispatch reported the SettingsView card left-aligned with a large
right dead zone at 1440. A red-check against the current code showed the
opposite: SettingsView is ALREADY centered. Its root container is
already `mx-auto flex w-full max-w-3xl ...` (the narrow reading column
with `mx-auto`), and in the App.tsx settings flex-row wrapper the
auto-margins center the 768px column on wide viewports. Measured at
1440: left margin ~= right margin (symmetric), width capped < 900px;
screenshot confirms the card sits in the middle. So the "add mx-auto"
fix was already in place -- there was nothing to add (a second mx-auto
would twMerge to a no-op).

## Change

Rather than a redundant no-op, this locks the behaviour in:

- `src/components/SettingsView.tsx`: added
  `data-section="settings-view"` to the root container (no visual
  change) so the centering is addressable from e2e.
- `web/e2e/settings-center.spec.ts` (new): a regression-guard spec that
  navigates to Settings (via the Settings tab at a wide viewport, then
  resizes) and asserts the column is centered at 1440 (symmetric
  margins, capped width), fits within the viewport at 768, and is
  full-width on mobile (375).

No layout/markup change to the settings column itself -- the existing
`mx-auto max-w-3xl` is what centers it, now verified and guarded.

## Verification

`web/e2e/settings-center.spec.ts`: `3 passed` headless. 1440 screenshot
(`test-results/settings-center-1440.png`) confirms the centered column
with symmetric side margins and the narrow reading width preserved.

## Risk

- No functional/layout change; only a test + a data attribute. Cannot
  regress the settings column. The new spec will catch any future
  change that breaks the centering.
