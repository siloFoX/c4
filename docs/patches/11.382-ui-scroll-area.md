## Goal

Dispatch:

- Build `web/src/components/ui/scroll-area.tsx`
  -- custom-styled scrollbars with native
  scroll behavior, max-height variants,
  scroll-position tracking via callback.

`<ScrollArea>` (11.164 + 11.227 + 11.297)
already shipped most of the contract:

| dispatch item | shipped previously |
| --- | --- |
| custom-styled scrollbars | `.c4-scroll` CSS hooks (`thin` / `default` / `wide` + responsive `auto`) |
| native scroll behavior | wraps a single `<div>` with `overflow-*` classes |
| max-height variants | freeform `maxHeight` (number | string) |
| safe-area padding | `safeArea` opt-in |
| shadow indicators | `shadows` opt-in with `data-at-top` / `data-at-bottom` |
| axis | `axis: 'y' | 'x' | 'both'` |

The remaining dispatched bullets were:
1. **Named max-height variants** -- the
   freeform value was supported but
   adopters wanted a consistent rhythm
   across surfaces.
2. **Scroll-position tracking via
   callback** -- the internal shadow
   recompute existed but was not
   exposed for hosts.

This patch closes both.

## API additions

### `maxHeightPreset`

```ts
type ScrollAreaMaxHeightPreset =
  | 'sm'       // 16rem (256px)  -- dropdown / popover body
  | 'md'       // 24rem (384px)  -- sidebar panel
  | 'lg'       // 32rem (512px)  -- mid-page block
  | 'xl'       // 48rem (768px)  -- modal body
  | 'screen';  // 100vh           -- full-viewport scroller
```

Freeform `maxHeight` (number | string)
wins when both are supplied. Caller can
mix-and-match: `maxHeight={500}
maxHeightPreset="md"` resolves to
`500px`.

`SCROLL_AREA_MAX_HEIGHT_PRESET` map is
exported so adopters can pull the
matching CSS value (e.g., to match the
rhythm in a non-ScrollArea surface).

### `onScrollPositionChange`

```ts
interface ScrollAreaPosition {
  scrollTop: number;
  scrollLeft: number;
  scrollHeight: number;
  scrollWidth: number;
  clientHeight: number;
  clientWidth: number;
  atTop: boolean;
  atBottom: boolean;
  atLeft: boolean;
  atRight: boolean;
}

interface ScrollAreaProps {
  // ... existing
  onScrollPositionChange?: (pos: ScrollAreaPosition) => void;
}
```

Fires on mount + every scroll +
ResizeObserver tick + window resize.
The snapshot carries pre-computed
edge-flags so callers do not redo the
math.

### Pure helper

```ts
export function snapshotScrollPosition(
  el: HTMLElement,
): ScrollAreaPosition;
```

Exported alongside the component for
tests + alternate hosts that want their
own position-aware chrome.

## Data attributes

- `data-section="scroll-area"` (legacy)
- `data-axis` -- new; mirrors the axis
  prop.
- `data-max-height-preset` -- new;
  mirrors the named preset; omitted
  when no preset is supplied.
- `data-shadows` + `data-at-top` +
  `data-at-bottom` -- retained from
  11.297.

## Pairs with existing primitives

- `<VirtualizedList>` (11.333) /
  `<VirtualTable>` (11.375) -- own their
  scroll machinery; do NOT wrap with
  ScrollArea.
- `<Drawer>` / `<Sheet>` -- internal
  body scroll handled separately; can
  host a `<ScrollArea>` inside their
  body slot.
- `<Card>` / `<Panel>` -- common host
  surfaces for max-height-preset.

## Out of scope

- Per-page adoption (`maxHeightPreset`
  rollout). All five presets are
  additive; every existing call site
  stays byte-identical at the legacy
  freeform `maxHeight`.
- Throttling / debouncing the
  `onScrollPositionChange` callback.
  Host owns rate-limiting (typical
  pattern: wrap the callback in
  `requestAnimationFrame` or a
  debounce helper).
- Imperative `scrollTo(top, left)`
  method. Adopters use the forwarded
  `ref` to the underlying scroll
  element and call native `scrollTo`
  directly.
- Horizontal-only shadow indicators.
  Fade-shadow contract is vertical-axis
  only by design.
- Custom preset extension. The five
  preset values are fixed; adopters
  with non-standard heights use the
  freeform `maxHeight` prop.
