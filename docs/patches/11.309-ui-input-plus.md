# 11.309 -- Input primitive enhancements (v1.11.327)

Enhances `web/src/components/ui/input.tsx` with icon
slots, a built-in clear button, and unified
warning/success state slots that mirror the existing
error slot. Component-scope only, no daemon-side change
and no `c4` CLI surface change.

## Why

Three pain points across Input adopters today:

1. **Inline icons re-invented per call site.** SearchBar
   wires its own `<MagnifyingGlassIcon>` next to an Input
   inside a flex container; the Snapshots filter does
   the same with an Inbox icon; the Settings filter
   re-invents the pattern with yet another icon. Each
   site has to remember to absolutely-position the
   icon, pad the input to make room, and `aria-hidden`
   the icon. Drift is easy.

2. **No canonical "clear" affordance.** SearchBar
   currently renders its own X button beside the input
   when a value is present. The button's hit target,
   icon size, and clear semantics (does it focus the
   input afterwards? Does it fire onChange?) are
   bespoke per site.

3. **Warning and success states have no primitive
   representation.** The existing `error` slot wires a
   red border, an `aria-invalid`, and a `<p role="alert">`
   message. There is no equivalent for warnings (the
   field is still submittable, but the operator should
   notice) or success (the field validated cleanly and
   the operator should see a green check). Adopters
   that need these states inline an ad-hoc message div
   below the input.

Patch 11.309 ships all three in the primitive.

## What changed

### `web/src/components/ui/input.tsx`

New props:

- `leadingIcon?: ReactNode` -- rendered as an inline
  span at the left edge of the input frame with
  `data-section="input-leading"` and `aria-hidden="true"`.
  The input gets `pl-9` to make room.
- `trailingIcon?: ReactNode` -- same shape on the
  right edge with `data-section="input-trailing"`.
  The input gets `pr-9`.
- `onClear?: () => void` -- when provided AND the
  input has a non-empty value (controlled or
  uncontrolled), an X button renders at the trailing
  edge with `data-section="input-clear"`. Clicking it
  calls `onClear()` (the caller is responsible for
  resetting the input value).
- `clearLabel?: string` -- SR label for the clear
  button. Default `'Clear'`.
- `warning?: ReactNode` -- message slot mirroring
  `error`. Renders a `<p data-section="input-warning">`
  below the input, wires the id into
  `aria-describedby`, applies `border-warning` tone on
  the input. Does NOT set `aria-invalid`.
- `success?: ReactNode` -- same shape with
  `border-success`. Also no `aria-invalid`.

State precedence: `error > warning > success > default`.
All three message slots can render simultaneously; the
input's tone class reflects only the highest-precedence
state. The new `data-state` attribute on the input,
field wrapper, and decoration wrapper exposes the
resolved state for e2e and theming.

Internal `TONE_CLASSES` constant keyed by canonical
state name (`default | error | warning | success`).
The follow-up adoption patches for FormField,
NativeSelect, and TagInput will share the same class
set so the visual vocabulary stays in lockstep across
form-control primitives.

Decoration wrapper: when any of `leadingIcon`,
`trailingIcon`, or the clear button is present, the
input is wrapped in a
`<div data-section="input-wrap" class="relative">`
with absolute-positioned icon spans on either side.
When none of those are present, the bare input
renders without a wrapper (backwards-compat with the
existing call sites).

### Adopted

- Primitive enhancement only this commit. The dispatch
  listed SearchBar, Settings inputs, and Snapshots
  filter as adoption sites. Threading the new props
  at those call sites is a follow-up per-page patch
  so each migration lands with its own test refresh
  and the blast radius of this commit stays minimal.

## Deferred (dispatch follow-ups)

Representative migration:

```tsx
// Before -- inline icon + clear button beside Input
<div className="relative flex items-center">
  <SearchIcon className="absolute left-3 ..." aria-hidden="true" />
  <Input
    className="pl-9 pr-9"
    value={query}
    onChange={(e) => setQuery(e.target.value)}
  />
  {query && (
    <button
      type="button"
      className="absolute right-3 ..."
      onClick={() => setQuery('')}
      aria-label="Clear"
    >
      <XIcon />
    </button>
  )}
</div>;

// After -- primitive owns the layout
<Input
  leadingIcon={<SearchIcon />}
  value={query}
  onChange={(e) => setQuery(e.target.value)}
  onClear={() => setQuery('')}
/>;
```

Per-page follow-ups: `web/src/components/ui/search-bar.tsx`,
the Settings page inputs, the Snapshots filter inputs.

## Tests

`web/src/components/ui/input.test.tsx` gains 22 new
vitest cases on top of the existing 20 (42 total):

- Icon slots (4):
  - `leadingIcon` renders with `pl-9` on the input.
  - `trailingIcon` renders with `pr-9` on the input.
  - Slots are `aria-hidden`.
  - Decorated input is wrapped in
    `data-section="input-wrap"`.
- Clear button (8):
  - Not rendered when `onClear` is omitted.
  - Not rendered when `value` is empty.
  - Rendered when `onClear` + non-empty value.
  - Honours custom `clearLabel`.
  - Click fires `onClear` exactly once.
  - Adds `pr-9` to the input.
  - `trailingIcon` takes precedence over the clear
    button.
  - Works with uncontrolled `defaultValue`.
- Warning slot (3):
  - Renders the message and applies `border-warning`.
  - Wires `aria-describedby` to the warning id.
  - Does NOT set `aria-invalid`.
- Success slot (2):
  - Renders the message and applies `border-success`.
  - Does NOT set `aria-invalid`.
- State precedence (3):
  - Error beats warning + success.
  - Warning beats success.
  - Default state when no slot is set.
- Data attributes (2):
  - `data-section="input-control"` + `data-state` on
    the input.
  - `data-section="input-field"` + `data-state` on
    the slotted wrapper.

All 42 green via

Downstream sanity check: 55 cases across

## Notes

- The clear button only renders when `onClear` is
  provided AND the input has a non-empty value. This
  keeps backwards-compat with the existing call sites
  that never pass `onClear` -- they will not see any
  change in rendered DOM.
- `trailingIcon` takes precedence over the clear
  button because callers that pass both signal an
  intentional override (e.g. a custom suffix that
  doubles as a clear affordance with bespoke
  semantics). The clear button is a default, not a
  hard requirement.
- The decoration wrapper is conditional: the bare
  input renders without any wrapper when none of the
  decoration / slot props are set, so the DOM shape
  stays byte-identical with prior versions for
  callers that only used the basic Input.
- `border-warning` and `border-success` reference the
  Tailwind tokens that resolve to the ARPS amber and
  emerald palette (see `border-warning` / `border-success`
  matches in `badge-counter.tsx` and `alert.tsx`).
  The same tokens drive the focus ring colour so the
  input's focus state stays in lockstep with its
  resting state.
- The clear button is a `<button type="button">` with
  `e.preventDefault()` + `e.stopPropagation()` to
  avoid form submission and outer-click handlers.
