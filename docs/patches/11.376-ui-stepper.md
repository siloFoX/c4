## Goal

Dispatch:

- Build `web/src/components/ui/stepper.tsx`
  -- horizontal / vertical wizard stepper
  with completed / active / pending states,
  click-to-jump, ARIA progressbar.

`<Stepper>` (11.252 / v1.11.270) already
shipped the dispatched core:

| dispatch item | shipped at |
| --- | --- |
| horizontal / vertical orientation | `orientation: 'horizontal' \| 'vertical'` |
| completed / active / pending states | `stateOf(idx, currentIndex)` -> `complete \| current \| pending \| error` |
| click-to-jump | `onStepClick(idx)` + `allowFuture` |
| per-step error | `step.error` flag |
| connectors per orientation | `data-stepper-connector` |
| `aria-current="step"` on active | yes |
| size variants (sm / md) | yes |

The single remaining gap was **"ARIA
progressbar"** -- the existing markup used
`role="list"` on the `<ol>` and per-item
`aria-current="step"`, but there was no
discrete `role="progressbar"` surface for
assistive tech to anchor on. This patch
adds one.

## What changed

### Companion progressbar

A visually-hidden `<span role="progressbar">`
sibling now renders inside the wrapper:

```tsx
<span
  role="progressbar"
  data-section="stepper-progressbar"
  aria-valuemin={0}
  aria-valuemax={steps.length}
  aria-valuenow={completedCount}
  aria-valuetext="Step <current+1> of <total>"
  aria-label="Step <current+1> of <total>"
  className="sr-only"
/>
```

- `aria-valuenow` counts only rows with
  `state === 'complete'`. Retroactive
  errors do NOT count even when they sit
  before `currentIndex` (an "error" row
  re-opens the wizard at that step).
- When `currentIndex >= steps.length` the
  caption flips to "Wizard complete" and
  `aria-valuenow` becomes `steps.length`.
- When `steps.length === 0` the
  progressbar is suppressed (no progress
  to surface).

### Visible caption (opt-in)

`showVisibleProgress={true}` renders a
visible `<p>` above the list carrying the
same text as `aria-valuetext`. Useful for
wizards that want a separate
"Step X of N" callout. Default `false`
keeps the legacy markup byte-identical.

### `progressLabel` override

```ts
progressLabel?: string | null;
```

- `string` -> replaces the auto caption
  in BOTH the visually-hidden progressbar
  and the visible callout (when on).
- `null` -> suppresses the visually-hidden
  progressbar entirely. Rare; assistive
  tech still needs SOME progress hook.
- `undefined` (default) -> auto caption.

### Wrapper structure

The component now returns a wrapper `<div
data-section="stepper-root">` so the
progressbar + visible caption can sit
alongside the `<ol>`:

```tsx
<div data-section="stepper-root" data-orientation={orientation}>
  {showVisibleProgress && <p data-section="stepper-visible-progress">{caption}</p>}
  {progressLabel !== null && (
    <span role="progressbar" ... className="sr-only" />
  )}
  <ol ...> ... </ol>
</div>
```

The `<ol role="list">` markup + per-item
`<li>` + badge + connector chrome is
unchanged. The `ref` still forwards to the
`<ol>` so existing imperative callers stay
compatible.

## Pairs with existing primitives

- `<Progress>` / `<ProgressBar>` -- linear
  progress bar for non-step measurement.
- `<CircularProgress>` -- circular variant.
- `<Tabs>` -- one-of-N selection without
  the "must complete in order" semantic.
  Reach for Tabs when steps are
  reorderable; reach for Stepper when the
  ordering carries meaning.

## Out of scope

- Per-page adoption of `showVisibleProgress`.
  Every existing call site stays
  byte-identical at the default.
- Smooth-animated progress transition. The
  progressbar attrs are read snapshots;
  any visible motion comes from the badge
  ring + connector classes, which the
  caller's CSS owns.
- A `partialProgress` semantic where the
  current step counts as 0.5. The
  canonical WAI-ARIA reading uses integer
  step counts; partial progress on a
  multi-step wizard belongs in a separate
  primitive (linear `<Progress>`).
- Wizard-state machine integration. The
  primitive stays presentational; state
  belongs in the host (or a wizard hook
  follow-on).
