# 11.203 - UI: worker resource mini-graphs

## Summary

Per-worker resource mini-graphs surfaced inline on every row of the
`WorkerList` sidebar. Each row now carries a `<WorkerResourceGraph
workerName={worker.name} />` next to its status badge, rendering two
side-by-side SVG sparklines (CPU% and RSS bytes) plus the current
value as a small mono-font text label on the right of each sparkline.
The graph is component-scoped: it self-polls `/api/metrics` on its
own cadence (default 5 s) so each row picks its own
`sampleIntervalMs` / `windowMs` and the existing shared `useMetrics()`
consumers (MetricsBar) keep their unrelated 5 s shared poll without
contention.

## Files

- `web/src/components/WorkerResourceGraph.tsx` (new, ~160 lines)
  - Props: `{ workerName: string; sampleIntervalMs?: number = 5000;
    windowMs?: number = 60000; height?: number = 28 }`.
  - Ring buffer of the last `N` samples where
    `N = Math.max(1, Math.floor(windowMs / sampleIntervalMs))`.
    Older entries are evicted by slicing off the head once the
    buffer exceeds capacity (the cap is captured in a ref so it
    stays in sync if props change without remount).
  - Polls `fetch('/api/metrics')`, finds the row whose `name ===
    workerName`, extracts `{ cpuPct, rssKb }` from the existing
    daemon payload (already exposed by `web/src/lib/use-metrics.ts`),
    and converts `rssKb -> rss bytes` for the RSS sparkline.
  - Renders two inline `<svg>` polylines (`stroke-width=1.5`,
    Tailwind `stroke-primary` class, `bg-muted/40` background) with
    `viewBox` driven by `SVG_WIDTH=60` and the prop `height`. CPU
    peak is fixed at 100; RSS peak is normalised to the in-window
    max so the line is always visible. The current value lands in
    a `font-mono tabular-nums` `<span>` (`73%`, `2.4MB`).
  - 'no data' placeholder (`<span data-testid="wrg-empty">no
    data</span>`) renders until the first poll resolves.
  - Cleans up the `setInterval` plus an `alive` guard on unmount so
    in-flight fetches cannot setState past teardown.
- `web/src/components/WorkerList.tsx`
  - Imports `WorkerResourceGraph`.
  - In the existing `CardHeader` right slot, the worker status
    `Badge` is wrapped together with `<WorkerResourceGraph
    workerName={w.name} />` inside a `flex shrink-0 items-center
    gap-2` div so the mini-graph sits inline with the badge on
    every row.

## Tests

`web/src/components/WorkerResourceGraph.test.tsx` (7 vitest+RTL
cases):

1. No-data placeholder renders before the first sample resolves (a
   release-gate MSW handler keeps the fetch pending).
2. After the first tick (vi.useFakeTimers + advanceTimersByTimeAsync),
   both `wrg-cpu-svg` and `wrg-rss-svg` are in the DOM and each
   polyline has a non-empty `points` attribute.
3. Current value text labels render (CPU `73%`, RSS matching the
   `B|KB|MB|GB` formatter).
4. Ring buffer caps at `windowMs / sampleIntervalMs` (4 with the
   chosen 20 000 / 5 000 split, after 7 server hits).
5. Cleanup on unmount stops further fetches (call counter stays at
   1 after a 60 s advance past unmount).
6. Rows whose `name` does not match `workerName` are ignored (graph
   stays in the no-data state).
7. The CPU value label updates across consecutive samples (10 % ->
   80 % across two ticks).

## Notes

- The mini-graph does not introduce a new `/api/metrics` route or
  payload shape; it consumes the existing `workers[].cpuPct` /
  `workers[].rssKb` columns from `MetricsResponse`.
- `useMetrics` was inspected for reuse but its 5 s cadence is
  hard-coded and globally shared, so the component owns its own
  `setInterval` to honour the `sampleIntervalMs` prop without
  refactoring the shared hook.
- ARPS design system tokens (`text-foreground`, `stroke-primary`,
  `bg-muted/40`) are used directly so the mini-graph picks up theme
  changes automatically.
- `web/package.json` bumped from `1.11.220` to `1.11.221`.
- `package-lock.json` untouched. `docs/autonomous-queue-v10.md`
  untouched (manager owns the queue).
