## Helper

`web/src/test-utils/axe.ts` exposes:

```ts
export async function pageA11yCheck(
  container: Element,
  opts?: { runOptions?: axe.RunOptions; skipRules?: readonly string[] },
): Promise<A11yResult>;

export function expectNoA11yViolations(result: A11yResult): void;
export function formatA11ySummary(violations: readonly A11yViolationDetail[]): string;

export interface A11yResult {
  violations: A11yViolationDetail[];
  ok: boolean;
  summary: string;
}
```

`pageA11yCheck()` wraps `axe.run()`, maps the result
into a stable shape (`id`, `impact`, `description`,
`helpUrl`, `nodes`, `selectors`), and applies a default
jsdom skip list:

| rule | reason |
| --- | --- |
| `color-contrast` | jsdom does not resolve Tailwind utility-class colors -- contrast cannot be evaluated without a real browser render. |
| `landmark-one-main` | per-component tests render snippets, not full pages. |
| `region` | same scope mismatch. |
| `page-has-heading-one` | primitives are exercised in isolation; an h1 in every test page is unrealistic. |

The skip list is overridable via
`{ skipRules: [...] }` so an integration test can
re-enable any rule against a richer fixture.

`expectNoA11yViolations()` throws a single
human-readable Error when violations exist so vitest
renders a concise failure report instead of an axe JSON
dump.

`formatA11ySummary()` is the same formatter the throw
uses; exported for callers that want to compose their
own assertion.

## UIDemoRoute audit

`web/src/pages/UIDemoRoute.a11y.test.tsx` runs two
cases against the storybook gallery (with the
`uiDemoRoute` feature flag enabled):

1. dark theme (`applyTheme('dark')`);
2. light theme (`applyTheme('light')`).

Both render the full primitive gallery (~9 sections,
every variant of every primitive) and pipe the
rendered container through `pageA11yCheck`. The harness
catches `aria-progressbar-name`,
`aria-tooltip-name`, `button-name`, `image-alt`,
`heading-order`, and the other dispatch-target rules
that jsdom can evaluate.

## Violations fixed

### `aria-progressbar-name`

Before:

```tsx
<div role="progressbar" aria-valuemin={0} aria-valuemax={100}>...</div>
```

Two `Progress` primitives in the demo gallery
rendered without an accessible name. axe rule
`aria-progressbar-name` requires `aria-label` /
`aria-labelledby` / `title` on every progressbar.

After:

```tsx
<div
  role="progressbar"
  aria-label={ariaLabel ?? (typeof labelText === 'string' ? labelText : 'Progress')}
  aria-valuemin={0}
  aria-valuemax={100}
>...</div>
```

Precedence: explicit `ariaLabel` -> string `labelText`
-> fallback `"Progress"`. The fallback keeps the
surface accessible even when neither prop is set.

### `aria-tooltip-name`

Before:

```tsx
<span role="tooltip" id={id} data-visible={visible}>{label}</span>
```

A single tooltip in the demo gallery rendered with a
non-text ReactNode label; the body's text content was
empty, so axe flagged it.

After:

```tsx
<span
  role="tooltip"
  id={id}
  data-visible={visible}
  aria-label={typeof label === 'string' ? label : 'Tooltip'}
>{label}</span>
```

When the label is a string the aria-label echoes it
(satisfies the accessible-name computation deterministically);
non-string labels fall back to `"Tooltip"`.

## Out of scope

- **Color-contrast rule.** jsdom returns empty strings
  for utility-class backgrounds, so axe cannot resolve
  computed colors. A Playwright + real-browser
  harness can re-enable the rule against a live
  render; that is a separate patch.
- **Per-page audit harness.** Queue, Workers,
  Sessions, Notifications, etc. each have their own
  data stubs and would balloon this patch. The new
  `pageA11yCheck` helper is ready to drop into any
  existing page test file; follow-up patches can
  incrementally add per-page audits.
- **Existing baseline failures.** The full suite has
  ~55 failing cases in unrelated files (ErrorBoundary,
  SessionsListSection, SpecialistsAuditPanel, etc.).
  None are tooltip- or progress-related and none
  regressed because of this patch.
