## Goal

The dispatch lists:

- Audit all transitions/animations across UI primitives.
- Replace any ad-hoc CSS transitions with `motion.ts`
  helpers.
- Add stagger animation for list items (motion-safe).
- Verify all animations respect
  `prefers-reduced-motion`.

This patch lands the audit infrastructure (regex
scanner + integration ratchet) and the stagger
primitives. The findings: every `web/src/components/
ui/*.tsx` file that emits `animate-in` / `animate-out`
either prefixes with `motion-safe:` or imports
`useReducedMotion` / `useMotionClass` to gate the
class manually. The scanner now guards against
regressions.

## Stagger helpers

`web/src/lib/motion.ts` exposes:

```ts
export const MOTION_STAGGER_STEP_MS: 40;
export const MOTION_STAGGER_MAX_MS:  320;

export function motionStaggerStyle(
  index: number,
  reducedMotion: boolean,
  step?: number,
): { animationDelay: string } | {};

export function useMotionStaggerStyle(
  index: number,
  step?: number,
): { animationDelay: string } | {};
```

Adoption pattern:

```tsx
function List({ items }) {
  return (
    <ul>
      {items.map((item, i) => {
        const delayStyle = useMotionStaggerStyle(i);
        return (
          <li
            key={item.id}
            style={delayStyle}
            className="motion-safe:animate-in motion-safe:fade-in"
          >
            {item.label}
          </li>
        );
      })}
    </ul>
  );
}
```

The hook returns an empty object when
`prefers-reduced-motion` is on, so reduced-motion
users see the items snap in without delay. Index 0
also returns an empty object (no delay on the first
item; the cascade starts at index 1). The delay is
clamped to `MOTION_STAGGER_MAX_MS` so a 200-row list
does not delay the last entrance by 8 seconds.

## Audit scanner

`web/src/lib/motion-audit.ts` flags `animate-in` and
`animate-out` Tailwind classes that bypass the
reduced-motion gate. Rules:

| pattern | accepts | rejects |
| --- | --- | --- |
| `motion-safe:animate-in` | yes | -- |
| `animate-in` in a file that imports `useReducedMotion` | yes | -- |
| `animate-in` in a file that uses `useMotionClass` / `motionClass(...)` | yes | -- |
| `animate-spin` / `animate-pulse` | yes (essential loading motion) | -- |
| bare `animate-in` / `animate-out` | -- | flagged |

Comments (line + block) are stripped before regex
matching. The scanner mirrors the v1.11.324
`theme-tokens-validator` + v1.11.346 `i18n-coverage`
pattern: regex-based, with an integration test that
walks `components/ui/*.tsx` and asserts the violation
count stays at or below a baseline constant.

### Integration ratchet

`MAX_KNOWN_UI_MOTION_VIOLATIONS = 0`. The components
tree currently has zero unsafe `animate-in/out`
usages. New code that introduces one will fail the
integration test loudly.

The ratchet is set to zero (not the i18n's 194-style
"acknowledged debt" non-zero baseline) because the
animation surface IS already in compliance -- the
audit found no remediation work to do.

## What the audit accepts

Three patterns satisfy the reduced-motion contract:

1. **`motion-safe:` modifier.** The CSS-layer gate
   that Tailwind expands to a `@media (prefers-
   reduced-motion: no-preference)` query. The
   browser drops the animation when the OS-level
   flag is on. Used by Panel, Card, Drawer, Toast,
   AlertBanner, EditModal, and other entrance
   animations.
2. **`useReducedMotion()` import.** The hook returns
   the current preference; the component conditionally
   composes the class string. Used by Switch (which
   gates the thumb-translation transition), Tooltip,
   and any primitive that needs to compute a class
   from the preference at render time.
3. **`useMotionClass(key)` helper.** The
   one-liner wrapper around #2. Used by Dialog,
   Popover, and other recent primitives.

The scanner accepts all three by design. The
`motion.ts` documentation lists them in order of
preference (most concise first) and the audit
treats them as equivalent.

## Out of scope

- **Migrating every `motion-safe:animate-*` site to
  `useMotionClass`.** Stylistic refactor with no
  functional payoff; both forms are motion-safe.
- **Per-page stagger adoption.** The
  `useMotionStaggerStyle` helper is exported and
  documented; per-list adoption is a follow-up that
  per-page patches can pick up.
- **CSS transition audit.** Tailwind's `transition-*`
  utilities honour `prefers-reduced-motion` at the
  CSS layer automatically. The scanner does NOT flag
  them; the audit treats them as compliant by
  default.
