## Goal

Dispatch:

- Build `web/src/components/ui/tabs.tsx` --
  accessible tabs with
  horizontal/vertical orientation, keyboard
  navigation (arrow keys), lazy-mount panels,
  controlled/uncontrolled.

Existing primitive already covers most of the
contract:

- `<Tabs>` (11.228 / v1.11.246 + 11.281 /
  v1.11.299) ships role-based markup,
  ArrowLeft / ArrowRight + Home / End nav (via
  `useFocusCycle`), the `pill` / `line`
  variants with `scroll` / `wrap` overflow,
  and lazy-mount panels via
  `{active ? children : null}` in
  `<TabsPanel>`.

The gaps were:

1. **Vertical orientation.**
   `useFocusCycle` already accepts
   `orientation: 'horizontal' | 'vertical' |
   'both'`, but `<Tabs>` hard-coded
   `'horizontal'`. Vertical layout also needs
   a flex-col tablist, `aria-orientation`
   announcement, side-bar indicator for the
   `line` variant, and a `border-r` tablist
   edge.
2. **Uncontrolled mode.** `<Tabs>` previously
   required `value` + `onChange`. Many
   adopters want an "uncontrolled" form where
   the component manages its own state.

This patch closes both gaps.

## API

```tsx
import { Tabs, TabsPanel } from './components/ui/tabs';

// Controlled (legacy, byte-identical)
<Tabs value={v} onChange={setV} items={ITEMS} />

// Vertical orientation
<Tabs
  value={v}
  onChange={setV}
  items={ITEMS}
  orientation="vertical"
/>

// Uncontrolled with defaultValue
<Tabs items={ITEMS} defaultValue="two" />

// Uncontrolled, observable
<Tabs
  items={ITEMS}
  defaultValue="two"
  onChange={(v) => log(v)}
/>

// Uncontrolled, no seed (falls back to first item)
<Tabs items={ITEMS} />
```

## Orientation behaviour

| concern | horizontal | vertical |
| --- | --- | --- |
| `aria-orientation` | `horizontal` | `vertical` |
| `data-orientation` | `horizontal` | `vertical` |
| Flex direction | row | `flex-col` |
| Arrow keys | Left / Right | Up / Down |
| `line` variant tablist border | `border-b` | `border-r` |
| `line` variant active indicator | `inset-x-0 -bottom-px h-0.5` | `inset-y-0 -right-px w-0.5` |
| `overflow="scroll"` axis | `overflow-x-auto` | `overflow-y-auto` |

Home / End behaviour is unchanged in both
orientations: they jump to the first / last
tab respectively.

The `pill` variant works identically across
orientations -- only the tablist flex
direction changes.

## Controlled-vs-uncontrolled contract

| Props | Mode |
| --- | --- |
| `value` + `onChange` | controlled (legacy) |
| `defaultValue` only | uncontrolled, internal state |
| `defaultValue` + `onChange` | uncontrolled, observable |
| `value` + `defaultValue` | controlled (`value` wins) |
| neither | uncontrolled, falls back to `items[0].value` |

Internal state uses `useState` seeded once on
mount. Re-rendering with a different
`defaultValue` does NOT re-seed the state --
that mirrors the React convention for
uncontrolled inputs (e.g.,
`defaultValue` on `<input>`).

## Tests

`tabs.test.tsx` -- +14 new cases (49 total):

- default `orientation="horizontal"` sets
  aria-orientation + data-orientation.
- `orientation="vertical"` sets
  aria-orientation + data-orientation +
  flex-col class.
- vertical ArrowDown moves to the next tab +
  fires onChange.
- vertical ArrowUp moves to the previous tab.
- vertical + line variant draws the side
  indicator on right edge.
- vertical + line variant tablist has
  `border-r` (and NOT `border-b`) -- uses
  classList for exact match (`border-border`
  would otherwise substring-match
  `border-b`).
- vertical + overflow="scroll" uses
  `overflow-y-auto` (not
  `overflow-x-auto`).
- uncontrolled: defaultValue seeds the
  initial active tab.
- uncontrolled: clicking another tab updates
  internal state without a parent.
- uncontrolled: onChange still fires when
  provided.
- uncontrolled: falls back to first item
  when defaultValue is absent.
- controlled: parent `value` wins over
  `defaultValue`.
- controlled: clicking a tab does NOT mutate
  internal state when parent ignores
  onChange.
- uncontrolled: keyboard nav updates the
  active tab through internal state.

## Pairs with existing primitives

- `useFocusCycle` (hooks/use-focus-cycle.ts)
  -- already supported all three orientations;
  this patch finally wires the `vertical`
  branch.
- `<TabsPanel>` -- unchanged; lazy-mount stays
  `{active ? children : null}`.

## Out of scope

- Per-page swap of horizontal `<Tabs>` to
  vertical. The new orientation is additive;
  every existing call site stays
  byte-identical at `orientation="horizontal"`.
- Adaptive orientation (e.g., flip to vertical
  below a breakpoint). Callers can swap
  orientation themselves via a media-query
  hook.
- Drag-to-reorder. The tablist stays in input
  order; reorder UX is a separate primitive.
- A "fully controlled" mode where the parent
  also drives focus. Focus management stays
  internal -- the tablist still manages roving
  tabindex.
