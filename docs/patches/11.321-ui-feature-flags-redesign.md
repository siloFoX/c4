# 11.321 -- UI: FeatureFlags page redesign

- TODO id: 11.321
- Version: 1.11.339
- Date: 2026-05-18
- Scope:
  - `web/src/lib/feature-flags.ts`
  - `web/src/pages/FeatureFlags.tsx`
  - `web/src/pages/FeatureFlags.test.tsx` (new)

## Goal

Bring the component-scoped feature-flag admin up to the
same UX rhythm as the rest of the page hero set (Snapshots,
Templates, Queue). Specifically:

- Category Tabs strip with live count chips.
- Debounced SearchBar narrowing flags by key / label /
  description.
- FormField wrappers around each Switch with the
  description as helperText so the field rhythm matches
  the rest of the admin pages.
- Tooltip on the Switch surfacing the description +
  default value on hover / focus.
- Keep the v1.11.291 TagInput chip filter so the legacy
  surface still works.

Component-scope plus a typed library change. No daemon
endpoint or CLI surface change. The localStorage key
(`c4:feature-flags`) and the `feature-flag-changed`
CustomEvent contract are byte-identical -- only the page's
visual shell and the type of `FeatureFlagDef` change.

## Library change

`web/src/lib/feature-flags.ts` adds:

```ts
export type FeatureFlagCategory =
  | 'motion'
  | 'navigation'
  | 'developer';

export const CATEGORY_LABELS: Record<FeatureFlagCategory, string> = {
  motion: 'Motion',
  navigation: 'Navigation',
  developer: 'Developer',
};

export interface FeatureFlagDef {
  key: FeatureFlagKey;
  label: string;
  description: string;
  defaultValue: boolean;
  category: FeatureFlagCategory; // NEW, required
}
```

All six shipped flags are annotated:

| key | category |
| --- | --- |
| `gridDebug` | `developer` |
| `routeProgress` | `navigation` |
| `pageTransitions` | `motion` |
| `motion` | `motion` |
| `reducedMotion` | `motion` |
| `uiDemoRoute` | `developer` |

Making `category` required is intentional: it lets the
Tabs filter always sort every flag into a bucket and lets
the page exhaustively render per-category counts. New
flags must pick an existing bucket (or extend the union if
none fit).

## Page behaviour

### Category Tabs

Five-option tab strip (`All`, `Motion`, `Navigation`,
`Developer`) above the toolbar. Each label carries a
trailing count chip with the per-category total. Selecting
a category restricts the list to flags whose
`category === activeTab`. Tab semantics follow the
existing `Tabs` primitive (arrow keys cycle, Enter
activates, `aria-label="Filter feature flags by category"`).

### Debounced search

A `SearchBar` (`size="sm"`, `debounceMs=200`) sits below
the Tabs. Matches are case-insensitive substrings across
`key`, `label`, and `description`. Composes with the
active tab AND the TagInput chips -- all three filters
apply together (AND).

### TagInput chips (preserved)

The v1.11.291 chip filter survives unchanged. Each chip is
an OR-substring against the flag's key + label. Composes
with the new filters per the rule above.

### FormField rows

Each row's Switch is now wrapped in a
`<FormField label={flag.label} helperText={flag.description}
layout="horizontal">`. The horizontal layout puts the label
on the left (`w-[30%] shrink-0`) and the Switch on the
right, with the description as the helper text below the
control. The helper element is queryable via
`data-section="form-field-helper"`.

### Tooltip on Switch

`<Tooltip label={`${flag.description} Default ${flag.defaultValue}.`}>`
wraps a span containing the override Badge + the Switch.
On hover / focus the operator sees the full description
plus the default value so they can predict the effect
without scanning the helper text. The Tooltip primitive
already renders the body in the DOM (opacity-0 until
visible) so the content is testable without simulating a
hover.

### Empty-filter placeholder

When the composed filter eliminates every row, a
placeholder paragraph
(`data-testid="feature-flags-empty-filter"`) renders in
place of the list with the message "No flags match the
current filter." Distinguishes a filter miss from "no
flags exist" (which the runtime currently does not
support since `FLAGS` is a typed constant).

## Data attributes

| attr | element | purpose |
| --- | --- | --- |
| `data-section="feature-flags-toolbar"` | toolbar wrapper | e2e selector for the toolbar block |
| `data-section="feature-flags-list"` | `<ul>` of rows | e2e selector for the list |
| `data-testid="feature-flags-category-tabs"` | Tabs wrapper | direct query |
| `data-testid="feature-flags-tab-count-{key}"` | count chip on each tab | per-category total |
| `data-testid="feature-flags-search"` | SearchBar input | direct query |
| `data-testid="feature-flags-empty-filter"` | placeholder `<p>` | filter-miss assertion |
| `data-testid="feature-flag-row-<key>"` | each row `<li>` | per-row e2e |
| `data-testid="feature-flag-control-<key>"` | Tooltip child span | hover / focus target |
| `data-testid="feature-flag-override-<key>"` | override Badge | override-state assertion |
| `data-testid="feature-flag-meta-<key>"` | mono meta line | key / default / category query |
| `data-flag-category="<category>"` | `<li>` row | filter-state assertion |

## Removed

- Per-row `HelpTip`. The Tooltip on the Switch +
  FormField helperText already surface the description.
- Inline `RichText` description block. The helper text
  uses plain text; the Tooltip body uses plain text. If
  the markdown-lite syntax becomes important again, the
  Tooltip body accepts ReactNode so a future patch can
  wrap the description in RichText without changing the
  page shape.

## Tests

`web/src/pages/FeatureFlags.test.tsx` is new. Thirteen
cases:

1. renders one row per declared flag under the default
   `all` tab;
2. renders the experimental AlertBanner at the top;
3. clicking a category tab filters rows by category;
4. per-category count chips render correct totals;
5. debounced search filters by key / label / description;
6. composed tab + search filter narrows the list;
7. "no flags match" placeholder appears when the filter
   eliminates everything;
8. FormField wraps each row with `data-layout="horizontal"`
   and the helper text mirrors the description;
9. each Switch carries the motion-aware
   `data-reduced-motion` attribute;
10. clicking the Switch flips the override badge and
    writes to `localStorage`;
11. Reset clears overrides and restores defaults;
12. Switch is wrapped in a Tooltip whose body contains
    the description + default value;
13. the legacy TagInput chip filter still narrows the
    list.

Cases 5, 6, and 7 use
`vi.useFakeTimers({ shouldAdvanceTime: true })` paired
with `userEvent.setup({ advanceTimers: ... })` so the
200ms debounce window is stepped past without sleeping.

All 13 cases pass against vitest 4.1.5 + jsdom 29.1.1.
