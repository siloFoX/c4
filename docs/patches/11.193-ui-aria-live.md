# 11.193 -- UI: aria-live announcer (v1.11.211)

## Summary

Component-scoped aria-live announcer for screen-reader notifications.
Adds a `useAnnounce()` hook + `<AnnounceRegion>` provider, then wires
three consumer sites (Toast, AutonomousStatusBanner, useForm) so SR
users hear toast messages, autonomous-loop lifecycle transitions, and
form submit / validation outcomes without polling the DOM.

## Files

- `web/src/hooks/use-announce.ts` (new)
  - `AnnouncePriority` type: `'polite' | 'assertive'`.
  - `AnnounceFn` interface: `(message, priority?) => void`.
  - `AnnounceContext`: `Context<AnnounceFn | null>`.
  - `useAnnounce(): AnnounceFn` -- consumes the context, throws when
    used outside an `<AnnounceRegion>` provider.

- `web/src/components/AnnounceRegion.tsx` (new)
  - Provider component. Mounts two visually-hidden live regions in
    its own subtree:
    - polite: `role='status' aria-live='polite' aria-atomic='true'`.
    - assertive: `role='alert' aria-live='assertive' aria-atomic='true'`.
  - Both regions use the Tailwind `sr-only` utility via `cn()`.
  - Exposes `announce(message, priority='polite')` through
    `AnnounceContext.Provider`. Empty / non-string messages are
    silently dropped.
  - To force re-announce of duplicate text, the region is cleared for
    one tick (`setTimeout(..., 30)`) then re-written. AT detects the
    change and reads the message again. Per-priority timers are
    cleared on the next call and on unmount.
  - `children` pass-through so the provider can wrap the existing
    tree without an extra DOM box.

- `web/src/App.tsx`
  - `<AnnounceRegion>` wraps the authed App tree (after the Login /
    loading early returns). Login and the loading splash do not
    consume `useAnnounce`, so the early returns stay simple.

- `web/src/components/Toast.tsx`
  - Reads `AnnounceContext` directly with `useContext`. On mount,
    fires `announce(message, type === 'error' ? 'assertive' : 'polite')`.
  - Empty-deps effect on purpose: each new toast is a new mount, so
    a single fire per toast is correct. No re-announce when props
    change within the same instance.
  - Direct context read (not `useAnnounce()`) so the existing
    `Toast.test.tsx` cases that render Toast without a provider
    continue to pass -- announce is a progressive enhancement.

- `web/src/components/AutonomousStatusBanner.tsx`
  - Reads `AnnounceContext` directly. Computes a lifecycle kind from
    the current status snapshot (`halt` / `escalation` / `dispatch` /
    `complete`) and announces once per kind transition in the format
    `[<kind>] <title>` (`halt` / `escalation` -> `assertive`,
    everything else -> `polite`).
  - Last-kind ref guards against repeating the same announcement
    when `/api/autonomous/status` polls return the same state.

- `web/src/hooks/use-form.ts`
  - Reads `AnnounceContext` directly. `handleSubmit` announces
    `Form has errors: <first-error>` assertively when validation
    fails or `Saved` politely when validation succeeds.

## Tests

`web/src/hooks/use-announce.test.tsx` -- 8 cases (vitest + RTL +
fake timers):

1. `useAnnounce` throws when no provider is mounted.
2. `useAnnounce` returns a function inside a provider.
3. AnnounceRegion mounts both polite + assertive regions with the
   correct `role` / `aria-live` / `aria-atomic` attributes.
4. `announce(msg, 'polite')` writes to the polite region only.
5. `announce(msg, 'assertive')` writes to the assertive region only.
6. Duplicate-message re-announce: the region clears for one tick
   then re-populates the same text so AT reads it twice.
7. Default priority is `polite` when omitted.
8. Empty-string messages are dropped.

Run:

```bash
env -C /root/c4-worktree-auto-w177/web \
  /root/c4/web/node_modules/.bin/vitest run \
  src/hooks/use-announce.test.tsx
```

Existing `useForm` / `Toast` suites still pass because the new
consumers degrade silently outside a provider.
