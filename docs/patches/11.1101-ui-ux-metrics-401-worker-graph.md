# 11.1101 -- UX FIX: /api/metrics 401 flood from the per-row WorkerResourceGraph (v1.11.1119)

## Scope

c4-qa verified v1.11.1104 made the MetricsBar resilient (it shows live
data; the needs-login strip never appears for a signed-in admin), but
14 of 18 /api/metrics responses were STILL 401 in the console.

## Root cause (the dispatch's service-worker hypothesis was wrong)

The dispatch blamed the service worker (web/src/sw.ts) for re-issuing
/api/metrics without the auth header. That is NOT the cause: **sw.ts is
never registered.** `registerServiceWorker()` (web/src/lib/sw-register.ts)
is not called from main.tsx, there is no vite-plugin-pwa / workbox, and
nothing else invokes it -- so the service worker never runs and cannot
reissue anything. (Even if it were registered, `networkFirst` forwards
`event.request`, which already carries the page's header; and the SW has
no localStorage, so it could not add a token the page didn't.)

The real cause is a SECOND /api/metrics consumer that 11.1086 missed.
11.1086 attached the session token to the MetricsBar poll
(use-metrics.ts), but **WorkerResourceGraph.tsx** -- the per-worker
CPU/RSS sparkline mounted once per row in the sidebar (WorkerList.tsx) --
kept doing a bare `fetch('/api/metrics')` with no Authorization header.
/api/metrics is auth-gated, so for a signed-in admin every row's 5s poll
401'd. And because it polled on a plain setInterval that just `return`ed
on failure, it never stopped -- N worker rows x every 5s = the 14/18
401 flood. (This is also why 11.1086's console-clean.spec.ts did not
catch it: that spec returns `workers: []`, so no row -- and no
WorkerResourceGraph -- ever mounted.)

## Fix

web/src/components/WorkerResourceGraph.tsx:
- Attach `Authorization: Bearer <token>` from the same source as
  use-metrics / apiFetch (`getToken`). The init is built so `headers`
  is only set when a token exists (no `headers: undefined`, which trips
  exactOptionalPropertyTypes).
- STOP polling on a 401 (clearInterval) so an expired/absent session
  never floods the console -- mirrors the 11.1082/11.1086 discipline in
  use-metrics.

Deliberately did NOT touch the dormant sw.ts: a no-op change there would
misrepresent the fix.

## Verification (mandatory visual)

- Unit (web/src/components/WorkerResourceGraph.test.tsx): two new cases
  -- the poll attaches `Bearer <token>` from getToken, and a 401 stops
  the interval (no further fetches). 9 passed.
- e2e (web/e2e/worker-graph-metrics-auth.spec.ts, new): signed-in admin
  (token seeded), /api/list returns one worker so a row + its graph
  mount, /api/metrics gated on the header (401 without, 200 with).
  Dwells past 3+ of the graph's 5s poll intervals and asserts ZERO
  /api/metrics 401 responses AND that the polls carried the session
  header. 1 passed (23.9s) + screenshot
  test-results/worker-graph-metrics-auth.png.
- tsc: the fix adds zero new type errors (the 3 pre-existing
  index-access warnings already in this file are unchanged).

## Risk

- Behaviour change is limited to the metrics poll: it now carries the
  token (so it 200s for a signed-in admin) and stops on a 401 (so an
  expired/absent session shows "no data" instead of flooding). No change
  to rendering or the SVG sparklines.
