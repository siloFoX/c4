# 11.1097 -- UX FIX: Features content panel width (v1.11.1115)

## Finding

The dispatch reported the FeatureView feature-panel card left-aligned
near ~560px next to the rail with a wide right dead zone at 1440. A
red-check against the current code showed it is ALREADY balanced: the
panel `<main>` is `flex-1` (fills the area right of the rail) and the
PageFrame card inside is full-width (no max-width cap), so the card
spans the whole panel. Measured at 1440: the card fills >= 80% of the
panel width; screenshot confirms the card runs from just past the rail
to the right edge with no dead zone.

This is almost certainly resolved by 11.1085 (v1.11.1103): before that,
the shared PageTransition incoming layer was content-width (~630px),
which capped every route's content -- including the Features panel --
near the left. 11.1085 gave that layer `min-w-0 flex-1` so routes fill
the available width, which fixed this panel as a side effect. The
~560px state the audit saw no longer exists.

## Change

Rather than a redundant no-op, this locks the behaviour in:

- `src/components/layout/FeatureView.tsx`: added
  `data-section="feature-panel"` to the panel `<main>` (no visual
  change) so the panel is addressable from e2e.
- `web/e2e/features-panel-width.spec.ts` (new): asserts at 1440 the
  PageFrame card fills (>= 80%) or is centered within the panel (no
  left-clinging / right dead zone), and at 375 the panel is a single
  full-width column.

No layout/markup change to the panel itself.

## Verification

`web/e2e/features-panel-width.spec.ts`: `2 passed` headless. 1440
screenshot (`test-results/features-panel-width-1440.png`) confirms the
panel card fills the area right of the rail.

## Risk

- No functional/layout change; only a test + a data attribute. The new
  spec will catch any future regression that narrows or left-aligns the
  feature panel.
