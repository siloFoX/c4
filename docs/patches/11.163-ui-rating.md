# 11.163 -- UI Rating primitive

New `<Rating>` UI primitive in `web/src/components/ui/rating.tsx`.
Shipped in `web/package.json` v1.11.181.

## Shape

- 5-star widget (configurable via `max`) with hover preview, optional
  half-star granularity, readonly mode, and slider-style ARIA.
- Each star renders a stroked empty `Star` (lucide-react) plus an
  overlay sized at 0% / 50% / 100% of the star's width with
  `fill='currentColor'` in `text-primary`. Half-overlay is gated by
  `allowHalf`.
- Hover preview: `mouseEnter` / `mouseMove` over a star sets local
  `hoverValue` (rounded to the nearest 0.5 when allowHalf, else 1),
  `mouseLeave` clears. Render falls back to `value` when not hovered.
- Click: when `allowHalf` and the click X lies in the left half of
  the star's bounding box, commits `(index + 0.5)`; otherwise
  `(index + 1)`. Values are clamped to `[0, max]`.
- Keyboard (root slider div is focusable when interactive):
  - `ArrowRight` / `ArrowUp`: increment by 0.5 (allowHalf) or 1.
  - `ArrowLeft`  / `ArrowDown`: decrement by the same step.
  - `Home`: 0. `End`: `max`.
- `readonly` (auto-true when `onChange` is omitted) renders stars as
  `<span>` instead of `<button>`, sets `aria-readonly='true'`, and
  skips all keyboard mutation.
- `role='slider'` with `aria-valuenow={value}` /
  `aria-valuemin={0}` / `aria-valuemax={max}`. Default `aria-label`
  is `'Rating'`; a string `label` overrides it. A non-string
  `label` is also rendered as `<span className='sr-only'>`.
- `forwardRef` exposes the wrapper `<div>`.

## Props

```ts
interface RatingProps {
  value: number;
  max?: number;                 // default 5
  onChange?: (v: number) => void;
  size?: 'sm' | 'md' | 'lg';    // 16 / 20 / 24 px - default 'md'
  readonly?: boolean;           // default !onChange
  allowHalf?: boolean;          // default true
  label?: ReactNode;            // sr-only when non-string
  className?: string;
}
```

## Adoption (v1.11.181)

- `pages/Health.tsx` -- 'Was this dashboard helpful?' Rating widget in
  a corner under the StatCards. `onChange` logs to `console.log`
  (no backend, no persistence). The site documents the primitive in
  use until a real feedback collector lands.

## Future feedback-form plan

The Health adoption is intentionally minimal so the same `<Rating>`
can drop into the planned in-app feedback form without API churn:

- Add a `FeedbackForm` shell that composes `<Rating>` + `<Textarea>`
  and POSTs `{ rating, comment, source }` to a forthcoming
  `POST /api/feedback` endpoint (one row per submission, written
  through the existing audit log so the entry shows up in `c4 audit`).
- Surface the form from a `<HelpDrawer>` slot so any page can open it
  via `openHelpDrawer({ panel: 'feedback' })` without a route change.
- The default 5-star, allow-half preset keeps parity with the
  Health corner so muscle memory transfers between contexts.

## Tests

`web/src/components/ui/rating.test.tsx` covers 20 cases with vitest
+ RTL: default 5 stars, `max=10`, full-star overlay widths at
`value=3`, half overlay on the 4th star at `value=3.5`, readonly
removes buttons and sets `aria-readonly`, readonly is inferred from
omitted `onChange`, click with `allowHalf=false`, geometric left-half
click commits `x.5`, keyboard increment / decrement / Home / End,
hover sets `data-hover` and clears on leave, `aria-valuenow` /
`aria-valuemax` reflect props, `className` merges, `forwardRef`
points at the wrapper, readonly keyboard is a no-op, increments
clamp at `max`, `label` string surfaces as `aria-label`, and a
stable `displayName`.

Run only this suite: `npx vitest run web/src/components/ui/rating.test.tsx`.
(Skip if `node_modules/` is not installed in the worktree.)
