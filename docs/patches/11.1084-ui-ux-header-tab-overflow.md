# 11.1084 -- UX FIX: top tab bar overflow + title overlap (v1.11.1102)

## Scope

The AppHeader top tablist (the 12 `TopTabs`) overlapped the
"C4 Dashboard" wordmark on the left and collided with the action
icons on the right at wide and mid widths (1440 / 768).

## Root cause

`AppHeader` lays its content out through the shared `Navbar`
primitive, a `grid grid-cols-[auto_1fr_auto]` with the tablist in the
centered `1fr` cell. The shared `Tabs` primitive marks the tablist
`flex shrink-0 overflow-x-auto`, but nothing capped its width. Because
`shrink-0` stops it from shrinking and there was no `max-width`, the
tablist rendered at its full intrinsic width (all 12 tabs) and -- being
centered by the cell's `justify-center` -- spilled symmetrically over
the brand on the left and the actions on the right. The
`overflow-x-auto` never engaged because the element was never
constrained narrower than its content.

## Change (scoped to TopTabs.tsx + AppHeader.tsx)

- `src/components/layout/TopTabs.tsx`
  - Pass `className="min-w-0 max-w-full"` to `<Tabs>`. `max-w-full`
    clamps the tablist to its Navbar center cell, so the overflow now
    scrolls inside the tablist's own isolated horizontal track instead
    of painting over the neighbours. `min-w-0` lets it shrink. It stays
    centered while it fits and only scrolls once the tabs exceed the
    cell. (twMerge appends these cleanly; no conflict with the
    primitive's base classes.)
  - Collapse labels to icons earlier: the per-tab label span moves from
    `hidden sm:inline` to `hidden lg:inline`, so below 1024 the strip is
    icon-only (each icon keeps its `title` / `aria-label`), which keeps
    the 12-tab row compact at 768/1024.
- `src/components/layout/AppHeader.tsx`
  - The "C4 Dashboard" wordmark hides below `sm` (`hidden sm:block`), so
    the narrowest viewport (375) shows just the logo + tab strip +
    actions with no title to truncate or crowd. `truncate` stays as an
    in-between-width safety net.

No change to the shared `Navbar` or `Tabs` primitives -- the fix is
entirely in the two header files via the existing `className` /
breakpoint hooks, so other tablists (DetailTabs, in-page tabs) are
unaffected.

## Verification (mandatory visual)

`web/e2e/header-tab-overflow.spec.ts` (new) loads the dashboard
headless at 1440 / 768 / 375 (auth stubbed to `{ enabled: false }` so
the dashboard chrome renders) and, for each width, reads the three
Navbar grid cells via the tablist's ancestry and asserts:

- the tablist's left edge >= the brand cell's right edge (no left
  overlap with the wordmark), and
- the tablist's right edge <= the actions cell's left edge (no right
  overlap with the icons), and
- the tablist's `clientWidth` fits within the viewport (overflow is
  scrolled internally, never painted on top of the neighbours).

Result: `3 passed`. Screenshots captured to
`test-results/header-tab-overflow-{1440,768,375}.png` and visually
confirmed: 1440 shows the wordmark + labeled (scrolling) tabs +
actions with clear gaps; 768 shows the wordmark + icon-only tabs +
actions; 375 shows logo-only brand + icon tabs + actions. No overlap
or truncation at any width.

Verified against a staging copy in `/root/c4/web` (real `node_modules`)
on a throwaway vite server (port 5191) so the live dev server was never
touched; the staging copy was reverted with `git checkout` after the
run.

## Design system

Layout-only change reusing the project's existing token-backed
semantic Tailwind utilities (the ARPS design tokens back those classes
via the Tailwind config). No new colours, spacing, or radii were
introduced.

## Risk

- Purely additive utility classes on the tablist plus two responsive
  breakpoint tweaks. The scroll behaviour is the primitive's existing
  `overflow-x-auto` + `scrollIntoView`-on-select, now actually engaged
  because the width is capped.
- Labels are hidden below `lg` rather than `sm`; the icons retain
  `title` + `aria-label`, so the tabs stay identifiable and accessible
  at mid widths.
