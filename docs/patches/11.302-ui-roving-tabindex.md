# 11.302 -- useRovingTabindex helper (v1.11.320)

New `useRovingTabindex` hook at
`web/src/hooks/use-roving-tabindex.ts`. Component-scope only,
no daemon-side change and no `c4` CLI surface change.

## Why

Several composite-widget primitives in the c4 web app
implement the WAI-ARIA roving-tabindex pattern by hand:

- `web/src/components/ui/tabs.tsx` -- the tab strip stamps
  `tabIndex={active ? 0 : -1}` on each `<button role="tab">`
  and routes Arrow keys through a `useFocusCycle`
  side-call.
- `web/src/components/ui/accordion.tsx` -- the accordion
  trigger row uses the same `tabIndex` + Arrow nav pattern.
- `web/src/components/ui/segmented-control.tsx` -- the
  segmented control's pressed segment is the
  `tabIndex=0` one.
- `web/src/components/ui/list-action-menu.tsx` (and any
  future `<Toolbar>`) -- same shape.

Each call site has to remember three things:
1. Track the "active" index (often derived from a
   prop value like `selectedTabValue`).
2. Stamp `tabIndex=0` on the active item and `-1` on the
   rest.
3. Wire a per-component key handler that drives Arrow keys
   + Home/End -> focus the right element.

Step 3 already has a shared helper (`useFocusCycle` in
`web/src/hooks/use-focus-cycle.ts`), but that hook is
intentionally stateless -- the active-index bookkeeping
(step 1) and the `tabIndex` stamping (step 2) still live in
each composite.

Patch 11.302 ships `useRovingTabindex` so any composite can
opt into all three steps in one call.

## What changed

### `web/src/hooks/use-roving-tabindex.ts` (new file)

- `useRovingTabindex(opts)` returns
  `{ activeIndex, setActiveIndex, getItemProps, itemRefs }`.
- Options:
  - `itemCount: number` -- required.
  - `orientation?: 'horizontal' | 'vertical' | 'both'`
    (default `'horizontal'`).
  - `wrap?: boolean` (default `true`).
  - `initialIndex?: number` (default `0`, clamped to a
    valid range).
  - `onChange?: (idx: number) => void` -- fires on Arrow /
    Home / End / direct focus / programmatic
    `setActiveIndex`. Does NOT fire when the new index equals
    the current one.
  - `isItemDisabled?: (idx: number) => boolean` -- the
    Arrow step skips disabled items; Home and End jump to
    the nearest non-disabled end.
- `getItemProps(idx)` returns the bag of props each item
  needs to spread:
  - `tabIndex: 0 | -1` -- `0` on the active item, `-1` on
    the others.
  - `onKeyDown` -- the shared Arrow / Home / End handler.
  - `onFocus` -- promotes the focused item to active so the
    next Tab-out exits from the same item the user just
    interacted with.
  - `ref` -- a ref callback that registers the item's DOM
    node in `itemRefs` so the hook's internal focus logic
    can call `.focus()` on the right element.
  - `data-roving-index` -- a string-form item index for
    e2e selectors.
- `setActiveIndex(idx)` is the programmatic setter (no
  focus side-effect). For callers that want to focus the
  item after setting it, they can read `itemRefs.current[idx]`
  and call `.focus()` directly.

### Adopted

- Hook only this commit. The Tabs / Accordion /
  SegmentedControl / ActionMenu adoption swaps are
  documented as follow-ups so each migration can land with
  its own test refresh.

## Deferred (dispatch follow-ups)

A representative migration looks like:

```tsx
// Before -- inline roving tabindex in tabs.tsx
const { handleKeyDown } = useFocusCycle({ containerRef: tablistRef, orientation: 'horizontal' });
// ...
<button
  role="tab"
  tabIndex={active ? 0 : -1}
  onKeyDown={handleKeyDown}
  // ...
/>

// After -- using useRovingTabindex
const roving = useRovingTabindex({
  itemCount: items.length,
  orientation: 'horizontal',
  initialIndex: items.findIndex((i) => i.value === value),
  onChange: (idx) => onChange(items[idx].value),
});
// ...
<button
  role="tab"
  {...roving.getItemProps(itemIndex)}
  // ...
/>
```

Per-primitive follow-ups: `web/src/components/ui/tabs.tsx`,
`web/src/components/ui/accordion.tsx`,
`web/src/components/ui/segmented-control.tsx`,
`web/src/components/ui/list-action-menu.tsx`.

## Tests

`web/src/hooks/use-roving-tabindex.test.tsx` -- 20 vitest
cases covering:

- Initial tabIndex distribution (0 on active, -1 on rest).
- `initialIndex` prop honored.
- Out-of-range `initialIndex` clamped to the last valid
  index.
- ArrowRight moves active forward (horizontal default).
- ArrowLeft moves active backward.
- ArrowRight wraps from last to first when `wrap=true`.
- ArrowRight stays put at last when `wrap=false`.
- Home jumps to the first item.
- End jumps to the last item.
- Vertical orientation uses ArrowDown/Up.
- Horizontal orientation ignores ArrowDown/Up.
- `'both'` orientation responds to all four arrows.
- `onChange` fires exactly once per index change.
- Disabled items are skipped during Arrow nav.
- Home/End respect disabled items at the ends.
- Direct focus on an item promotes it to active.
- Programmatic `setActiveIndex` updates tabIndex without
  focusing.
- `data-roving-index` attribute selector for e2e.
- `itemCount=0` no-crash, `activeIndex=0`.
- Unrelated keys (Tab) do not preventDefault.

All 20 green via
`vitest run --project unit src/hooks/use-roving-tabindex.test.tsx`.

## Notes

- The hook lives in `web/src/hooks/` (not
  `web/src/components/ui/`) because it is a behavioural
  hook, not a renderable primitive. The `index.ts` barrel
  under `components/ui/` is intentionally NOT touched.
- `setActiveIndex` is deliberately a no-focus setter -- the
  caller may want to change the active item without moving
  focus (e.g. when a controlled prop drives the active
  state and the user has moved focus elsewhere). Callers
  that DO want the focus side-effect can grab
  `itemRefs.current[idx]?.focus()` after the setter.
- `onChange` is invoked synchronously inside the setState
  updater because React state updates inside event handlers
  are batched and the caller often needs the new index
  before the next render (e.g. to fire an analytics event).
  The hook guarantees `onChange(idx)` is called exactly once
  per index transition.
- The internal skip logic for disabled items uses a bounded
  loop (`for (let i = 0; i < count; i += 1)`) so a group
  with every item disabled returns the current index
  instead of looping forever.
