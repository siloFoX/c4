## Goal

The dispatch lists:

- Build `web/src/components/ui/virtualized-list.tsx`
  with fixed-row-height virtualization, overscan,
  scroll restoration.
- Adopt at HistoryView rows, Sessions list, Workers
  list (when >50 items).

This patch ships the primitive + 20 unit tests + the
adoption pattern documentation. The three adoption
sites use grouped / variable-height row rendering that
doesn't slot into the primitive's flat-fixed-height
shape without preparatory work; their adoption is
deferred to per-page follow-up patches.

## Public API

```ts
export function VirtualizedList<T>(props: {
  items: readonly T[];
  rowHeight: number;
  renderRow: (item: T, index: number) => ReactNode;
  overscan?: number;                                 // default 4
  scrollRestorationKey?: string;
  keyFor?: (item: T, index: number) => string;
  onVisibleRangeChange?: (r: { start: number; end: number }) => void;
  onReachBottom?: () => void;
  emptyContent?: ReactNode;
  className?: string;
  ariaLabel?: string;
  style?: CSSProperties;
}, ref?: Ref<VirtualizedListHandle>): JSX.Element;

export interface VirtualizedListHandle {
  scrollToIndex(index: number): void;
  getScrollTop(): number;
}

export function computeVisibleRange(args: {
  scrollTop: number;
  viewportHeight: number;
  rowHeight: number;
  itemCount: number;
  overscan: number;
}): { start: number; end: number };
```

## Behaviour

### Windowing math

`computeVisibleRange` is a pure helper exported for
unit testing:

```
rawStart = floor(scrollTop / rowHeight)
visibleCount = ceil(viewportHeight / rowHeight) + 1
start = max(0, rawStart - overscan)
end = min(itemCount, rawStart + visibleCount + overscan)
```

The math is bounded: `start >= 0`, `end <= itemCount`,
and `start <= end` for all inputs. Zero items returns
`{ 0, 0 }`.

### Render shape

```tsx
<div data-section="virtualized-list" onScroll={...}>
  <div data-section="virtualized-list-spacer" style={{ height: totalHeight }}>
    <div
      data-section="virtualized-list-rows"
      style={{ transform: `translateY(${offsetY}px)` }}
    >
      {/* only items[range.start..range.end] rendered */}
    </div>
  </div>
</div>
```

The spacer carries the full `itemCount * rowHeight`
height so the native scrollbar reflects the real
content size. The rows group is transform-translated
so the visible window paints at the right offset.
This is the canonical react-window / react-virtuoso
pattern.

### Scroll restoration

When `scrollRestorationKey="foo"` is set:

1. On mount, the primitive reads
   `sessionStorage['c4:virtualized-list:foo']` and
   assigns the value to `scroller.scrollTop`.
2. On every scroll event, it writes the current
   `scrollTop` back. Throttled to one write per
   render tick.

The `c4:virtualized-list:` namespace is shared so
unrelated hosts cannot collide.

### Visibility callback

`onVisibleRangeChange({ start, end })` fires every
time the visible window changes. Skipped on the
initial `{ 0, 0 }` placeholder render so adopters do
not have to filter it. Useful for pre-fetching detail
data for currently visible rows.

### Bottom-edge callback

`onReachBottom()` fires once per "reached bottom"
transition. The transition is computed as
`scrollTop + viewportHeight >= totalHeight - 1`.
Subsequent scroll events while still at the bottom
do NOT re-fire; the next fire requires scrolling
away from the bottom and back.

### Imperative ref

`VirtualizedListHandle.scrollToIndex(i)` clamps `i`
to `[0, itemCount - 1]` and sets
`scrollTop = i * rowHeight`. `getScrollTop()` reads
the live value (useful for tests + scroll-anchor
flows).

## jsdom adoption notes

jsdom does not compute layout. The primitive handles
this by:

- Using a scroll-listener path (not IntersectionObserver)
  for the row windowing math. IntersectionObserver
  cannot efficiently emit "currently visible row range"
  for thousands of rows.
- Re-measuring `clientHeight` on every scroll event so
  jsdom-based tests can drive the viewport height by
  patching `Object.defineProperty(el, 'clientHeight',
  { value: N })` and firing a scroll.
- Falling back to a single mount-time measurement when
  ResizeObserver is unavailable (jsdom).

Tests drive scroll via:

```ts
Object.defineProperty(el, 'scrollTop', { writable: true, value: 200 });
fireEvent.scroll(el);
```

## Why the dispatch's adoptions are deferred

The dispatch lists three adoption sites:

- **WorkerList** -- renders two grouped lists
  (`managers.map` + `regular.map` with `renderRow`
  helpers). Virtualizing requires either:
  - Flattening into a single
    `{ kind: 'header' | 'row' }` list.
  - Or wrapping each group in its own VirtualizedList
    instance (loses cross-group scroll context).
  Neither is a quick swap.
- **SessionsView** -- list grouping by project +
  collapsible group affordance. Same shape mismatch as
  WorkerList.
- **HistoryView** -- rows include inline detail panes
  with variable height (filter expansion, inline
  preview). The v1.11.351 primitive specifies
  fixed-row-height; variable-height virtualization is
  a separate primitive.

Adoption is meaningful only when the per-page data
shape is reshaped into a flat fixed-row array. That
reshaping is the right scope for per-page follow-up
patches; landing it in this patch alongside the
primitive would make a 5-file diff into a 15-file
diff and risk regressions in three different views.

The primitive's contract is stable. Adoptions can
land incrementally as:

1. Per-page patch reshapes the row data into
   `{ kind: 'header' | 'row', ... }`.
2. Sets `rowHeight` to the largest row variant + a
   shared height for the header.
3. Renders via `<VirtualizedList items={flat}
   rowHeight={48} renderRow={(item) => item.kind ===
   'header' ? <Header ... /> : <Row ... />} />`.

## Tests

`virtualized-list.test.tsx`:

- 5 `computeVisibleRange` unit cases (empty, top,
  mid-scroll, clamp at bottom, clamp at top with
  overscan).
- 15 component cases (empty content, overscan-bounded
  window, window shifts on scroll, spacer height,
  keyFor / default keying, scroll persistence, scroll
  restoration on mount, onVisibleRangeChange,
  onReachBottom firing + non-re-fire, scrollToIndex +
  clamping, data attribute surfaces, ariaLabel
  forwarding).

20/20 cases pass against vitest 4.1.5 + jsdom 29.1.1.

## Out of scope

- **Variable-height virtualization** -- the dispatch
  specifies fixed-row-height. Adapters that need a
  variable contract should compose with a per-row
  measurement pass (or wait for a follow-up primitive).
- **Horizontal virtualization** -- the primitive
  handles vertical only. The Tailwind class
  `overflow-y-auto` is hard-coded.
- **Adoption at the three dispatch sites** -- see
  above. Per-page follow-up patches.
