Extends 8.45 with per-worker, user-customizable persistent rules.

## Why

8.45's post-compact hook re-injects a daemon-side default rule block whenever
Claude Code auto-compacts a worker. That covers the "never run compound
commands" / "never push to main" class of invariant but gives operators no
way to pin a custom rule to a specific worker - e.g. "always use Yarn, never
npm" for a frontend worker, or "never touch src/migrations" for a DB worker.

8.46 lets the operator attach a per-worker rule set that survives auto-compact
and daemon restart.

## Surface

### CLI

```
c4 new <name> [--pin-memory <file>]... [--pin-rules "<text>"]... [--pin-role <role>]
c4 pinned-memory get <name>
c4 pinned-memory set <name> [--file <path>] [--rule "<text>"]... [--role <role>] [--refresh]
```

- `--pin-memory <file>` reads a file and appends its content as one rule
  entry. Repeatable. File read happens client-side so the daemon only sees
  a flat string[].
- `--pin-rules "<text>"` adds a literal rule. Repeatable.
- `--pin-role <manager|worker|attached>` picks a default template from
  `docs/rules/role-<role>.md`. Tier-based fallback: `tier: 'manager'` ->
  `role-manager`, everything else -> `role-worker`.
- `c4 pinned-memory set ... --refresh` forces an immediate re-injection
  without resetting the interval.

### Daemon API

- `GET /api/workers/:name/pinned-memory` - returns
  `{ worker, pinnedMemory: { userRules, defaultTemplate }, lastRefreshAt, intervalMs }`.
- `POST /api/workers/:name/pinned-memory` - body
  `{ userRules?: string[], defaultTemplate?: string|null, refresh?: boolean }`.
  Gated by the `worker.task` RBAC action because the write ultimately writes
  into the worker's PTY.

### Web UI

`WorkerDetail` now mounts `PinnedRulesEditor` below the terminal card. The
editor exposes a `Persistent Rules` textarea, a role-template select, and
both `Save` and `Save and refresh now` buttons. Rules are fenced with `---`
on save so the round-trip back into the textarea preserves entries.

## Scheduling

`src/pinned-memory-scheduler.js` owns a single `setInterval` per worker.
- `config.pinnedMemory.intervalMs` - default 5 minutes, floored at 10s.
- Each tick resolves the effective rule set (role template + userRules) and
  sends it via `manager.send(name, 'PINNED RULES REFRESHED:\n...')`. The
  `PINNED RULES REFRESHED:` prefix is deliberately greppable so operators
  can spot re-injections in scrollback and tests can pin the format.
- The scheduler subscribes to two manager events:
  - `post-compact` - fired from `compactEvent()` when 8.45's PostCompact
    hook (or on-data detector, once merged) reports a compaction. Triggers
    an immediate refresh with trigger label `(post-compact)`.
  - `pinned-memory-updated` - fired when `setPinnedMemory()` mutates the
    rule set. Triggers an immediate refresh with trigger label `(updated)`.

## Metadata persistence

`pty-manager` stores `pinnedMemory` on the worker struct as
`{ userRules: string[], defaultTemplate: string|null }`. `_saveState`
writes the field to `state.json`; `_loadState` forwards it into the
`lostWorkers` snapshot so operators can recover the rule set after a daemon
restart.
