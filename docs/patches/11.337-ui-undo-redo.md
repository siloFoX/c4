## Public API

```ts
export function useUndoHistory<T>(
  initial: T,
  options?: { maxHistory?: number; shortcuts?: boolean },
): {
  state: T;
  set: (next: T | ((prev: T) => T)) => void;
  undo: () => void;
  redo: () => void;
  canUndo: boolean;
  canRedo: boolean;
  reset: (next: T) => void;
  past: readonly T[];
  future: readonly T[];
};

export const UNDO_HISTORY_DEFAULT_MAX: 100;
```

## State model

The hook stores state as the classic triple:

```
past:    [s0, s1, s2]
present: s3
future:  [s4, s5]
```

Transitions:

| operation | past | present | future |
| --- | --- | --- | --- |
| initial | `[]` | `s0` | `[]` |
| `set(s1)` | `[s0]` | `s1` | `[]` |
| `set(s2)` | `[s0, s1]` | `s2` | `[]` |
| `undo()` | `[s0]` | `s1` | `[s2]` |
| `redo()` | `[s0, s1]` | `s2` | `[]` |
| `set(s3)` after undo | `[s0, s1]` | `s3` | `[]` (future cleared) |

The "new `set` clears `future`" rule is the canonical
undo-redo semantics. Every text editor / drawing tool
does it; users expect the redo path to die the
moment they make a fresh edit after an undo.

## Keyboard shortcuts

Opt in via `{ shortcuts: true }`:

| Keys | Action |
| --- | --- |
| `Cmd+Z` / `Ctrl+Z` | undo |
| `Cmd+Shift+Z` / `Ctrl+Shift+Z` | redo |
| `Cmd+Y` / `Ctrl+Y` | redo (Windows alt-form) |
| `Meta+Z` | undo (mac form) |

The listener attaches to `window.keydown`. It skips
when the keydown target is a text-entry surface:

- `<input>`, `<textarea>`, `<select>`
- any element with `el.isContentEditable === true`

so the browser's native undo-in-text-input keeps
working. The hook calls `event.preventDefault()` only
when it actually invokes undo/redo (i.e., the
relevant stack is non-empty); otherwise the event
bubbles untouched.

## Adoption pattern

### Drag-reorder page (e.g., Queue)

```tsx
const {
  state: rows,
  set: setRows,
  undo,
  redo,
  canUndo,
  canRedo,
} = useUndoHistory<Row[]>(initialRows, { shortcuts: true });

const onDrop = (event: DragEvent, targetId: string) => {
  setRows((prev) => moveRow(prev, dragId, targetId));
};

<Button disabled={!canUndo} onClick={undo}>Undo</Button>
<Button disabled={!canRedo} onClick={redo}>Redo</Button>
```

Every drag commits to the history; the operator can
undo a misclick + redo if they change their mind.
Server commits compose with the v1.11.354
`useOptimisticMutation` hook -- pass
`{ state: rows, setState: setRows }` into the
optimistic hook and the rollback path lands on the
history's previous state automatically.

### Form-editor page (e.g., Settings, Templates)

```tsx
const {
  state: form,
  set: setForm,
  undo,
  canUndo,
  reset,
} = useUndoHistory<FormState>(initialForm);

useEffect(() => {
  if (loaded) reset(loaded);
}, [loaded, reset]);

<input
  value={form.title}
  onChange={(e) => setForm((prev) => ({ ...prev, title: e.target.value }))}
/>
```

`shortcuts: false` here -- the native Cmd+Z inside
`<input>` should win for character-level undo.
Toolbar-level undo can wire to `undo()` explicitly.

### Cap the past

`maxHistory` defaults to 100. Pass `Infinity` for
unbounded history (drawing tools), pass a smaller
value for memory-sensitive surfaces (rich-text
editors with large per-state payloads).

## Deferred (adoptions)

Three sites: Queue page reorder, Settings edits,
Templates edits. Each adoption is a one-line state
swap (the page's `useState` becomes
`useUndoHistory`), BUT the existing per-page tests
assert direct `setRows(...)` calls against the
page's own state, not the hook's shape. Adopting
needs paired test updates that re-point assertions
at the hook's `state` / `set` / `past` / `future`
exposed via the controlled-form contract.

The hook is stable. Per-page adoption is a routine
swap that should land with its test update.
