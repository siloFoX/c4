# 11.1082 -- UX FIX: MetricsBar 401 flood + polling backoff (v1.11.1100)

## Scope

`MetricsBar` polled `/api/metrics` every 5s and, on a
persistent 401 (auth expired / required), kept hammering
the daemon -- and spamming the console -- every interval
forever. This fix:

1. **Stops polling after a 401** and surfaces a quiet
   `needs-login` strip instead of looping.
2. **Raises the healthy poll interval from 5s to 20s.**
3. **Exponential backoff on transient errors** (network
   blips, 5xx): 20s -> 40s -> 80s ... capped at 160s,
   resetting to 20s after the next success.

## Files

- `web/src/lib/use-metrics.ts` -- rewrote the poll loop.
  New return shape `{ data, status }` (was bare
  `MetricsResponse | null`); `status` is one of
  `loading | ok | needs-login | error`. Scheduling moved
  from `setInterval` to a self-rescheduling `setTimeout`
  so the delay can vary per tick. A 401 sets
  `needs-login` and does NOT reschedule.
- `web/src/components/MetricsBar.tsx` -- consume
  `{ data, status }`; render a quiet needs-login strip
  (`data-section="metrics-bar-needs-login"`) when
  `status === 'needs-login'`, else the existing live bar
  (or null while loading / no data).
- `web/src/i18n/en.json` / `ko.json` -- `metrics.needsLogin`
  label.
- `web/src/lib/use-metrics.test.ts` -- rewritten for the
  new shape + behavior (20s interval, 401-stop, backoff,
  reset-after-success, keep-last-value).
- `web/src/components/MetricsBar.test.tsx` -- mock updated
  to the `{ data, status }` shape; added needs-login
  render tests.
- `web/e2e/metrics-bar-401.spec.ts` -- new Playwright spec
  (visual + request-count verification).
- `package.json` / `web/package.json` /
  `package-lock.json` / `web/package-lock.json` ->
  1.11.1100.

## Poll-loop logic

```
delay = HEALTHY_INTERVAL_MS (20000)
tick():
  res = fetch('/api/metrics')
  if res.status === 401:
    status = 'needs-login'      // STOP -- no reschedule
    return
  if !res.ok:                   // transient 5xx
    if no data yet: status = 'error'
    delay = min(delay * 2, MAX_BACKOFF_MS)   // backoff
    schedule(delay)
    return
  // success
  data = json; status = 'ok'
  delay = HEALTHY_INTERVAL_MS   // reset backoff
  schedule(HEALTHY_INTERVAL_MS)
catch (network blip):
  if no data yet: status = 'error'
  delay = min(delay * 2, MAX_BACKOFF_MS); schedule(delay)
```

`HEALTHY_INTERVAL_MS = 20000`, `MAX_BACKOFF_MS = 160000`,
both exported for the tests. The `alive` guard + a single
tracked `timer` ref keep the cleanup (clearTimeout on
unmount) race-free, and a successful poll resets the
backoff so a recovered daemon returns to the 20s cadence.

## Visual verification (mandatory)

`web/e2e/metrics-bar-401.spec.ts` drives the running dev
server headless:

- Intercepts `/api/metrics` -> 401 (counting each request),
  stubs `/api/auth/status` -> `{ enabled: false }` so the
  dashboard renders, serves empty `/api/list` +
  `/api/events`, aborts the rest.
- Asserts the `metrics-bar-needs-login` strip renders and
  contains the sign-in copy.
- Waits a full 10s window and asserts **no more than 2**
  `/api/metrics` requests fired (one mount tick plus at
  most one React StrictMode double-invoke in dev -- the
  loop must NOT keep firing every interval; the old 5s
  loop would have fired ~3 in 10s).

Result: `1 passed (33.2s)`.

Unit tests (`vitest`): `use-metrics.test.ts` +
`MetricsBar.test.tsx` -> `34 passed`, covering the 20s
interval, the 401-stop, exponential backoff (40s then
80s), backoff reset after success, keep-last-value on
transient error, unmount cleanup, and the needs-login
render path.

## Risk

- The hook's public return shape changed from
  `MetricsResponse | null` to `{ data, status }`. The only
  consumer is MetricsBar (grep-confirmed); it is updated
  in lockstep, and both unit suites are rewritten to the
  new shape.
- A 401 now permanently stops the poll loop for the life
  of the mounted hook -- intentional. The operator
  re-authenticating triggers an app re-render path
  (AUTH_EVENT -> Login -> re-mount) that remounts
  MetricsBar with a fresh loop, so recovery does not need
  in-loop 401 retry. (If a future requirement wants
  auto-resume without remount, a focus/visibility listener
  could re-arm the loop; out of scope here.)
- Transient (non-401) errors keep the last value and back
  off, so a flaky daemon degrades gracefully rather than
  blanking the bar.
- Verified against a byte-identical staging copy in
  `/root/c4/web` because the worker worktree's
  `node_modules` is a symlink that blocks Node's ESM
  config resolution for vite / playwright when run from
  the worktree path; the spec runs cleanly from any
  checkout whose `node_modules` is a real directory.
