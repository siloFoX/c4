# 11.181 -- i18n key extractor refresh (v1.11.199)

Branch: `c4/auto-i18n-refresh`
Date: 2026-05-14

## Why

The 11.116 -- 11.180 UI primitive sweep added new components and call
sites without always topping up `web/src/i18n/en.json` /
`web/src/i18n/ko.json`. This patch closes the audit gap.

## Audit method

Static grep over `web/src/**/*.{ts,tsx,js,jsx}` for two call shapes:

- `t('<key>')`        -> 933 unique keys
- `tFormat('<key>')`  -> 182 unique keys

Union: **1114** unique referenced keys. Comparison against the
top-level keys present in `web/src/i18n/en.json` (1346 entries on the
pre-patch tree) yielded the following diff:

| Key                          | Status in en.json | Status in ko.json | Action            |
| ---------------------------- | ----------------- | ----------------- | ----------------- |
| `workerDetail.mergeConfirm`  | missing           | missing           | add to both       |
| `no.such.translation.key`    | missing (test)    | missing (test)    | allowlist (keep)  |
| `totally.absent.key`         | missing (test)    | missing (test)    | allowlist (keep)  |
| `feature.queue.label`        | present           | missing           | add ko            |
| `feature.queue.description`  | present           | missing           | add ko            |

`no.such.translation.key` and `totally.absent.key` are intentional
fixtures consumed by `web/src/lib/i18n.test.ts` to assert the
"missing-key returns the key verbatim" fallback. They must remain
absent from both bundles; the new test allowlists them by name.

## Changes

- `web/src/i18n/en.json`: +1 key (`workerDetail.mergeConfirm`).
- `web/src/i18n/ko.json`: +3 keys (`workerDetail.mergeConfirm`,
  `feature.queue.label`, `feature.queue.description`).
- en.json key count: 1346 -> 1347. ko.json key count: 1344 -> 1347.
  Bundles are now at exact parity.
- New test: `web/src/i18n/i18n-keys.test.ts` walks `web/src` for
  `t('...') / tFormat('...')` references and asserts:
  1. Every referenced key exists in `en.json` (with the two-key
     fixture allowlist documented above).
  2. `ko.json` has the same key set as `en.json` (no en-only or
     ko-only drift).
  3. Allowlisted fixture keys remain absent from both bundles -- a
     guard so a future "fix all missing keys" sweep does not silently
     break `i18n.test.ts`.

## Out of scope

- No changes to UI primitives or call sites; this is i18n+test only.
- `docs/autonomous-queue-v10.md` not touched (manager-owned).
- `package-lock.json` not touched.

## Verification

`env -C /root/c4-worktree-auto-w165/web /root/c4/web/node_modules/.bin/vitest run src/i18n/i18n-keys.test.ts`
could not load the worktree-local vite config (worktree has no
`node_modules` installed; the shared install at `/root/c4/web` cannot
resolve `@vitejs/plugin-react` etc. from this worktree's path). JSON
parity validated out-of-band via `node -e` (en/ko both 1347 keys,
no one-sided keys). The test file uses only `vitest` + `node:fs`
APIs and the JSON bundles, so it will run cleanly once
`web/node_modules` is installed in this worktree or as part of the
post-merge CI.
