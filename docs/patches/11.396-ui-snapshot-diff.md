```ts
interface DiffLine {

interface DiffHunk {

  endIndex: number;
}

  ariaLabel?: string;
  renderText?: (text: string, language: string) => ReactNode;
  activeChangeIndex?: number;
  defaultActiveChangeIndex?: number;
  onActiveChangeIndex?: (index: number) => void;
  defaultExpandedFolds?: boolean;
}
```

## Behaviour

### Diff engine

`computeLineDiff` runs an LCS-based
line diff (O(n*m) DP table with a
linear-pass backtrack). Empty strings
produce no lines (the "both empty"
case yields `[]`, not a single empty
line). Line numbers track the 1-based
position in each source -- `null` for
the side where the line does not
exist.

### Change blocks

`findDiffChangeBlocks` groups
consecutive non-equal lines into a
single block. A run of removes
followed by a run of adds counts as
one block, which matches the
canonical "this change" unit for
review navigation.

### Hunk grouping

`groupDiffHunks(lines, contextLines)`
walks the line array and emits:

- `{ type: 'lines', lines }` -- a
  contiguous run of rendered rows.
- `{ type: 'fold', lines,
  foldedLineCount }` -- a collapsed
  run of unchanged lines.

For each equal run, the algorithm
keeps `contextLines` rows on each
side of a change block and folds the
remainder. At the start / end of the
diff there is no anchor on one side,
so the head / tail context collapses
to zero -- a diff with no changes at
all folds entirely.

### Side-by-side pairing

`pairSideBySide(lines)`:

- equal -> render on both sides.
- run of removes followed by run of
  adds -> pair element-by-element.
  Excess removes get empty right
  cells; excess adds get empty left
  cells. This minimizes row count
  while keeping the canonical
  GitHub-style layout.

### Navigation

| input | action |
| --- | --- |
| `Next change` button | next block, wraps |
| `Prev change` button | previous block, wraps |
| `n` / `Alt+ArrowDown` | next |
| `p` / `Alt+ArrowUp` | previous |
| plain `ArrowUp` / `ArrowDown` | no-op (falls through to scroll) |
| `Ctrl/Cmd + n / p` | no-op |

`activeChangeIndex` prop is
optional: omit for uncontrolled mode
(internal state, seeded by
`defaultActiveChangeIndex`), supply
for controlled mode.
`onActiveChangeIndex(idx)` fires on
every navigation in both modes.

When there are no change blocks, the
nav buttons disable and the counter
shows `0 / 0`.

### Fold rendering

Each fold renders one button with
text "... <N> unchanged lines (click
to expand)" and `aria-label="Expand
<N> unchanged lines"`. Click expands
that fold in place (state is
per-hunk; expanding one fold does not
expand others). `defaultExpandedFolds=
true` mounts with every fold already
expanded -- useful when the host
wants a full snapshot view that the
user can collapse afterwards.

### Syntax highlighting

The component does NOT bundle a
highlighter. Pass `renderText: (text,
language) => ReactNode` to plug in
Prism / Highlight.js / Shiki / etc.
Default behaviour is plain text (the
line content renders as-is in a
monospace pre-wrap cell with
preserved whitespace).

## ARIA + data attributes

Root (`role="region"`, focusable
`tabIndex={0}`, `aria-label`):

- `data-section="snapshot-diff"`
- `data-mode`
- `data-language`
- `data-change-count`
- `data-active-change-index`

Header:

- `data-section="snapshot-diff-header"`
- summary span:
  `data-section="snapshot-diff-summary"`
  text `+N -M`
- nav: `-nav`, `-prev`, `-next`,
  `-nav-counter`

Body:

- `data-section="snapshot-diff-body"`
- inner table:
  `-unified` (mode=unified) or
  `-side-by-side` (mode=side-by-side)
- row: `-row` +
  `data-line-type` (unified) +
  `data-line-index` (unified) +
  `data-active`
- side-by-side cell:
  `-cell` + `data-side` +
  `data-line-type`
- line number: `-line-number` +
  `data-side`
- prefix: `-line-prefix`
  (unified only, " " / "+" / "-")
- content: `-line-content`
  (unified only)
- fold row: `-fold` + `data-fold-id`
  + `data-fold-count`
- fold button: `-fold-button` +
  dynamic `aria-label`

## Out of scope (deferred)

- Bundled highlighter. The
  `renderText` prop is the canonical
  plug; shipping Prism would balloon
  the primitive budget.
- Word / character-level intra-line
  diff. Line-level only for v1;
  inline emphasis is a follow-on.
- Three-way (merge) diff. Two-way
  covers audit + preview; the
  three-way merge UX is a separate
  primitive.
- Patience / Myers-O(ND) diff. LCS
  is correct + simple; faster
  algorithms belong in a follow-on
  for very large files.
- Side-by-side empty-row alignment
  to keep equal lines adjacent
  across multi-line adds. The
  pair-removes-with-adds contract is
  canonical for v1.
- Copy / share-link buttons. Belongs
  to the audit log host, not the
  primitive itself.
