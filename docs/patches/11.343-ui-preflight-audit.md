## Goal

The dispatch lists:

- Audit Tailwind preflight overrides -- ensure no
  unwanted resets break design tokens.
- Document any project-specific overrides in
  `docs/patches/`.

This patch ships a small CI-runnable scanner that
pins the current override catalogue. The project's
`index.css` is well-disciplined; every universal
selector is intentional and documented inline. The
scanner ratchet guarantees that future patches that
introduce a new override surface a CI failure +
require a baseline bump.

## Preflight state

`web/tailwind.config.js` has NO `corePlugins`
override -- Tailwind preflight is ENABLED in full.
The base layer normalises browser defaults the
standard way (margin/padding zeroing, list-style
removal, font inheritance, button reset, etc).

`web/src/index.css` extends the base layer with the
shadcn pattern + a custom scrollbar theme. The
catalogue below is the complete list of universal
overrides as of v1.11.361.

## Override catalogue

### 1. `* { @apply border-border }` (line ~143)

```css
@layer base {
  * {
    @apply border-border;
  }
  ...
}
```

**Intent:** Seeds every element with the design
system's border color (`hsl(var(--border))`) so the
`border` Tailwind utility paints the right color
without re-specifying it on every call site.

**Token impact:** None. Reinforces a token.

### 2. `* { scrollbar-width: thin; scrollbar-color: ... }` (line ~166)

```css
* {
  scrollbar-width: thin;
  scrollbar-color: hsl(var(--muted-foreground) / 0.4) transparent;
}
```

**Intent:** Custom scrollbar theme (Firefox / Edge).
Pulls `--muted-foreground` so the thumb color tracks
the design tokens through theme flips.

**Token impact:** None. Uses tokens.

### 3-7. `*::-webkit-scrollbar*` (lines ~171-188)

```css
*::-webkit-scrollbar { width: 10px; height: 10px; }
*::-webkit-scrollbar-track { background: transparent; }
*::-webkit-scrollbar-thumb { ... background-color: hsl(var(--muted-foreground) / 0.35); ... }
*::-webkit-scrollbar-thumb:hover { background-color: hsl(var(--muted-foreground) / 0.6); ... }
*::-webkit-scrollbar-corner { background: transparent; }
```

**Intent:** WebKit/Chrome scrollbar theme aligned
with the design tokens. Without this rule macOS
hides scrollbars entirely while other OSes show fat
default scrollbars -- the result is visually
inconsistent.

**Token impact:** None. Each color references a
design token; sizes are intentional UX choices
documented in the index.css block comment.

## Why no preflight rules are disabled

`tailwind.config.js` does not set
`corePlugins: { preflight: false }` or any per-rule
opt-out. The project intentionally inherits the full
Tailwind reset because:

- Buttons, form controls, lists, headings, and
  paragraphs need the normalised baseline so design
  tokens compose predictably.
- The shadcn / ARPS design system assumes preflight
  is on (the `--border` token is applied via the
  `border-border` rule above).

A future patch that disables a specific rule would
update this section.

## Scanner

`web/src/lib/preflight-audit.ts` exposes:

```ts
export function scanSourceForPreflightOverrides(
  filePath: string,
  contents: string,
): PreflightOverride[];

export function scanFilesForPreflightOverrides(
  files: readonly string[],
): PreflightOverride[];

export function formatPreflightOverrides(
  overrides: readonly PreflightOverride[],
): string;

export interface PreflightOverride {
  file: string;
  line: number;
  selector: string;
  rule: 'universal-selector' | 'universal-pseudo';
  excerpt: string;
}
```

The regex pattern matches top-level CSS rules whose
selector starts with `*` (optionally followed by a
pseudo-element). Compound selectors like
`.container > *` are NOT flagged because they are
scoped to a specific parent. Block + line comments
are stripped before matching.

## Integration ratchet

```ts
export const MAX_KNOWN_PREFLIGHT_OVERRIDES = 7;
```

The integration test asserts the scanner returns at
most 7 overrides against `web/src/index.css`. When a
new intentional override is added, bump the constant
in the test file AND add an entry to this catalogue.

## Tests

`preflight-audit.test.ts` -- 14 cases:

- Empty CSS / universal flag / universal-pseudo flag.
- Compound-selector skip (`.foo *`, `.foo > *`).
- Attribute-selector skip (`[class*="prefix-"]`).
- Block + line comment skip.
- File/line/column report.
- Multi-override grouping (3 rules -> 3 reports).
- Universal inside `@layer base` block.
- `*:where(.foo)` pseudo-arg matching.
- Formatter empty + populated.
- Integration ratchet against `index.css`.
- Sanity: `@apply border-border` +
  `@apply bg-background text-foreground` lines
  still present.

14/14 pass against vitest 4.1.5.

## Out of scope

- **Disabling specific Tailwind preflight rules.**
  The project enables the full preflight. A future
  patch that opts out of a rule would update the
  catalogue.
- **CSS-in-JS preflight audit.** The project has no
  styled-components / emotion call sites; the
  scanner only walks static `.css` source files.
- **Browser-engine-specific reset audit.** The
  scanner does not distinguish WebKit vs Firefox
  scrollbar pseudo-elements; both are treated as
  intentional overrides. A future patch could add a
  rule-by-rule report (which override targets which
  engine).
