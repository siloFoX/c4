# 11.286 -- DropdownMenu separators + data selectors (v1.11.304)

Component-scope-only enhancement of the existing `DropdownMenu`
primitive (the dispatch asked for an `action-menu.tsx` file but
DropdownMenu already covers the trigger + popover menu + keyboard
nav + item icons + danger variant + prefetch contract; the new
work consolidates the missing pieces there instead of duplicating
the surface). No daemon-side change and no `c4` CLI surface
change.

## Why

DropdownMenu shipped with a flat item list. Real action menus
need to visually separate edit-grade rows (Rename / Duplicate /
Archive) from destructive rows (Delete) so the operator does not
chord the wrong action via type-ahead. Callers worked around it
by mounting a non-clickable item with a styled label, which the
focus-cycle hook still treated as a navigable row. Patch 11.286
puts the separator inside the primitive so the focus + type-ahead
contract excludes it for free.

The patch also widens the data-attribute selector set so e2e
tests can target the menu, the per-row variant, and each
separator without parsing class names.

## What changed

### `web/src/components/ui/dropdown-menu.tsx`

- New exported type `DropdownMenuSeparator`:
  `{ key: string; kind: 'separator' }`. Renders as
  `<li role="separator" aria-orientation="horizontal">` with
  a 1px `bg-border` divider.
- New exported union
  `DropdownMenuEntry = DropdownMenuItem | DropdownMenuSeparator`.
  `DropdownMenuProps.items` now accepts the union so callers
  can interleave separators with action rows. Existing
  `DropdownMenuItem[]` call sites narrow to the union without
  a cast.
- Separators are skipped by:
  - The `useFocusCycle` selector
    (`[role=menuitem]:not([aria-disabled=true])` -- separators
    do not carry `role=menuitem`).
  - The single-letter type-ahead matcher (
    `for entry of items: if isSeparator(entry) continue`).
  - The `aria-activedescendant` resolver on the menu container.
- New data-attribute selectors:
  - `data-section="dropdown-menu"` on the menu container.
  - `data-section="dropdown-menu-item"` per action row.
  - `data-variant="default|danger"` per action row.
  - `data-section="dropdown-menu-separator"` per separator `<li>`.
- exactOptionalPropertyTypes hygiene: tightened the `close()`
  invocation in `handleItemActivate` so a `restoreFocus`
  argument is spread only when truthy. Avoids passing
  `undefined` into the strict optional shape.

### Adopted

- Primitive enhancement only this commit. Existing
  DropdownMenu adopters (AccountMenu in the sidebar /
  AppHeader, ListActionMenu wrapper) inherit the new
  `data-section` attributes without any caller change.

## Deferred (dispatch follow-ups)

- Submenu / nested menu support: the dispatch asks for
  "optional submenu". Building nested menus requires a
  second portal + focus-trap + click-outside scope that does
  not map cleanly onto the existing single-portal pattern.
  Tracked as a follow-up so the submenu primitive can land
  with its own jsdom-friendly test contract.
- Workers / Sessions / History row-actions adoption: those
  rows already use `ListActionMenu` (which wraps
  DropdownMenu). The separator-aware item shape is now
  available; threading separators into the per-page action
  arrays is a follow-up.

## Tests

- `web/src/components/ui/dropdown-menu.test.tsx` -- 6 new
  vitest cases:
  - Separator entry renders as `role="separator" <li>` with
    `aria-orientation="horizontal"`.
  - Separator does NOT appear in the `role="menuitem"` list.
  - ArrowDown nav skips the separator (Edit -> Delete with no
    intermediate stop).
  - Type-ahead skips the separator.
  - `data-section="dropdown-menu"` + `data-section="dropdown-
    menu-item"` + `data-variant` selectors present.
  - `data-section="dropdown-menu-separator"` per divider.
  All 37 (31 original + 6 new) green.
- Dependent suites (ListActionMenu + AccountMenu) re-run
  clean. 93 / 93 across the three touched files.
