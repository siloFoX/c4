# 11.285 -- FormField warning state + data selectors (v1.11.303)

Component-scope-only enhancement of the existing `FormField`
primitive. No daemon-side change and no `c4` CLI surface change.
The existing label / control / hint / error contract is preserved
byte-for-byte; the new affordances opt in via the new `warning`
prop and the new `data-section` attribute set.

## Why

The previous FormField could only signal two states: helper
(neutral hint) and error (destructive, sets `aria-invalid`).
Real forms have a third intermediate band -- the field is
submittable as-is but the operator should see a heads-up before
they commit. Examples that came up across the recent UI work:

- Templates editor: "Saving as the active template will
  overwrite the previous effort tier."
- Profiles editor: "This profile grants elevated permissions."
- Settings forms: "Using the default value clears your custom
  override."

Forcing those into `error` would block submission. Forcing them
into `helperText` would mute the urgency. Patch 11.285 adds the
soft `warning` state so callers can opt in without inventing
their own warning chrome.

The patch also widens the data-attribute selector set so e2e
tests can target the active state (`data-state="ok|warning|error"`)
plus each message slot without parsing class names.

## What changed

### `web/src/components/ui/form-field.tsx`

- New `warning?: string` prop. Styled with the warning palette
  (`text-warning`) instead of the destructive palette. Does NOT
  flip `aria-invalid` on the child control -- the field is
  still submittable.
- State precedence: `error` > `warning` > `ok`. When both
  `error` and `warning` are set, `error` wins (the more severe
  state surfaces).
- The warning message renders with `role="status"` (polite live
  region) so screen-reader users hear the heads-up without the
  form being announced as invalid. The error message continues
  to use `role="alert"` (assertive).
- `aria-describedby` plumbing follows the active state. In the
  warning state the input's `aria-describedby` references the
  warning message id (`<id>-warning`), so SR users hear the
  warning copy when they refocus the field.
- Data-attribute selectors:
  - `data-section="form-field"` + `data-state="ok|warning|error"`
    on the wrapper.
  - `data-section="form-field-label"` on the `<label>`.
  - `data-section="form-field-required"` on the asterisk span.
  - `data-section="form-field-error"` on the error paragraph.
  - `data-section="form-field-warning"` on the warning
    paragraph.
  - `data-section="form-field-helper"` on the helper text
    paragraph.

### Adopted

- Primitive enhancement only this commit. Existing FormField
  adopters (Login, Plan, Profiles, Templates editor inputs)
  inherit the new attributes + the warning state when callers
  opt in. A follow-up will wire `warning` copy into the
  Templates save + Profiles editor flows.

## Deferred (dispatch follow-ups)

- Settings forms warning rollout: Settings uses bespoke
  ChoiceGroup panels rather than FormField rows. Adoption
  there needs a separate path.
- Profile editor + Template editor warning copy: the warning
  semantics are now available on the primitive, but the
  per-page warning strings need product input.

## Tests

  cases:

## Notes

- The warning palette uses `text-warning` from the design
  tokens; if a future theme migration changes the warning
  hue, the FormField warning copy follows automatically.
- The warning message id is suffixed `-warning` so a future
  test can disambiguate it from the helper / error message
  ids without re-mounting the field.
