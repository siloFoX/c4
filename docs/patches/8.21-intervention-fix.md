# TODO 8.21 — Intervention / escalation false-positive fix

**Status:** done
**Branch:** `c4/intervention-fix`
**Scope:** Narrow the `intervention` flag the daemon publishes in
`c4 list` + the monitor cron so that only workers that actually need
human attention are surfaced. The P2 `c4 list` UI surface lands in
the same patch because the table was the most visible entry point of
the false positives.

## Summary

`manager.list()` now emits a narrowed public enum
`'approval_pending' | 'background_exit' | 'past_resolved' | null`
plus two breadcrumb fields (`hasPastIntervention: boolean`,
`lastInterventionAt: string | null`). A new `src/intervention-state.js`
module owns the detection + mapping + auto-clear logic; the daemon's
healthCheck stall-detection loop narrows its notify predicate to
`approval_pending`, and the PostToolUse / repeat-error escalation
set-sites downgrade to `bg_exit` whenever the parent worker is still
alive and no TUI prompt is visible. A 10-minute stall promoter
re-escalates `bg_exit` back to `approval_pending` so a truly stuck
background job still pages — just not on tick zero.

## Reproduction

1. **ux-explorer (2026-04-19).** A helper `vite` process on port
   5174 was spawned during a UI audit. After the audit completed
   `vite` exited 144 (normal shutdown; Puppeteer / Vite graceful
   teardown). Claude Code reported the non-zero exit to the
   PostToolUse hook; `_handlePostToolUse._errorHistory` hit the
   3-retry threshold and set `_interventionState = 'escalation'`.
   The worker was not waiting on anything — the background helper
   had just finished its job.
2. **c4-mgr-auto (2026-04-19~20).** A real escalation landed
   earlier in the day; the operator resolved it by sending the
   correct `c4 key Enter`. No path cleared `_interventionState`,
   so hours later `c4 list` still rendered the intervention column.
3. **Monitor cron.** `PtyManager.healthCheck()` iterates every
   tick (4.14) and calls `notifyStall(worker, 'intervention: ...')`
   when `_interventionState` is truthy. Both (1) and (2) kept
   paging + the autonomous loop polled `c4 read-now` against them.

## Pre-patch code shape (the OR-combination)

Before 8.21 the daemon kept one `_interventionState` string with
legacy values `'question' | 'escalation' | 'critical_deny' |
'routine' | null` and treated every non-null value as "needs human":

- `src/pty-manager.js` list() row: `intervention: w._interventionState || null`
- healthCheck stall-detection:
  ```js
  if (w._interventionState) {
    this._notifications.notifyStall(r.name, `intervention: ${w._interventionState}`);
  }
  ```
- Three set sites all set `'escalation'` directly:
  - PostToolUse hook `_errorHistory` hit `maxRetries`
  - PTY `_detectErrors` hit `maxRetries` against scraped output
  - `StateMachine.update()` returned `escalation` after 3 consecutive
    test fails
- Clear was only hit for `'question'` when `_detectQuestion` no
  longer matched the latest PTY chunk. `'escalation'` / `'critical_
  deny'` / `'routine'` had no clear path.

The set logic was effectively "repeated error OR stale question OR
live TUI prompt OR state-machine escalation" — a broad OR-combination
that did not differentiate "live human-gated prompt" from "background
job teardown" or "stale breadcrumb". This patch splits those.

## P1.1 — Split `approval_pending` from `background_exit` / `past_resolved`

New module `src/intervention-state.js`:

```js
const { detectApprovalPrompt, mapInterventionToPublic,
  clearInterventionIfResolved } = require('./intervention-state');
```

### `detectApprovalPrompt(snapshot) -> boolean`

Tail-regex sweep over the last 60 lines of the screen buffer. Only
the tail counts — a prompt that scrolled off the visible terminal is
not actionable. Patterns:

- `Do you want to (proceed|create|make this edit|run this|make these changes)`
- `Continue?\s*[?y/n]` / `[y/N]` / `[Y/n]` / `(y/n)` / `(yes/no)`
- `trust this folder`
- `Bash command` + numbered `1. Yes` grid within 200 chars
- Korean `계속하시겠습니까`
- `^\s*Proceed\?` (multiline)
- `Accept the changes?`

Empty / non-string input returns `false` without throwing. The
regex list sits at the top of the module so future prompt strings
are easy to add.

### `mapInterventionToPublic(worker) -> public enum`

- `_interventionState === 'question' | 'critical_deny' | 'escalation'`
  → `'approval_pending'`
- `_interventionState === 'bg_exit'` → `'background_exit'`
- `_interventionState === null` + `_hadIntervention === true`
  → `'past_resolved'`
- `_interventionState === null` without breadcrumb → `null`
- Unknown future values fall through to `'approval_pending'`
  (conservative — a caller that thinks "needs human" and is wrong
  is cheaper than missing a real intervention).

### `clearInterventionIfResolved(worker, snapshot) -> boolean`

- Returns `false` for already-null state.
- **`critical_deny` never auto-clears.** `c4 approve` / the TUI
  Down+Enter path is the only thing that flips it. Auto-clearing
  would bypass the deny-list guardrail.
- If `detectApprovalPrompt(snapshot)` is `true`, keeps the flag.
- Otherwise: sets `_interventionState = null`, marks
  `_hadIntervention = true`, and stamps `_lastInterventionAt` (ISO)
  if not already set. Returns `true`.

This is the sticky-state fix — `list()` re-runs the detector per
call so a flag that once fired but no longer has a live prompt is
dropped on the next list cycle.

## P1.2 — Sticky-state clear integration

`src/pty-manager.js` `list()` now:

1. Pulls the current screen snapshot (`this._getScreenText(w.screen)`).
2. Stamps `_lastInterventionAt` when the internal state is still
   set, so the breadcrumb is accurate even if no one has called
   `clearInterventionIfResolved` yet.
3. Calls `clearInterventionIfResolved(w, snapshot)` — cheap
   (tail-regex), so per-list is fine.
4. Builds the row with the mapped public `intervention` + the new
   `hasPastIntervention` + `lastInterventionAt` fields alongside
   the existing shape.

Internal `_interventionState` keeps its legacy values so hot-paths
(`wait --interrupt-on-intervention`, `cancelCriticalCommand`, SSE
events, the existing hook-architecture / parallel-wait tests) do not
need a coordinated rewrite.

## P1.3 — Background-exit downgrade

Both paths that used to set `_interventionState = 'escalation'` on a
non-zero child exit now downgrade:

### `_handlePostToolUse` (hook)

```js
if (existing.count >= maxRetries) {
  const snapshot = worker.screen ? this._getScreenText(worker.screen) : '';
  const hasPrompt = interventionState.detectApprovalPrompt(snapshot);
  const nextState = (hasPrompt || !worker.alive) ? 'escalation' : 'bg_exit';
  worker._interventionState = nextState;
  worker._hadIntervention = true;
  worker._lastInterventionAt = new Date().toISOString();
  // snapshot + SSE uses [HOOK BG EXIT] tag when nextState === 'bg_exit'
  if (nextState === 'escalation' && this._notifications) {
    this._notifications.notifyStall(...);
  }
}
```

### `_detectErrors` (PTY scan)

Same guard — live prompt or dead worker keeps `'escalation'`, alive
+ quiet terminal downgrades to `'bg_exit'`. The vite-exit-144 case
lands here: no human-gated prompt on screen + worker still alive =
informational `bg_exit`. No `notifyStall`, no autonomous-loop read-now.

Both sites unconditionally set `_hadIntervention + _lastInterventionAt`
so the breadcrumb is recorded even if the flag itself is later cleared.

## P1.4 — Monitor cron predicate narrowing

`PtyManager.healthCheck()` stall-detection block:

```js
const mapped = interventionState.mapInterventionToPublic(w);
if (mapped === 'approval_pending') {
  this._notifications.notifyStall(
    r.name,
    `intervention: approval_pending (${w._interventionState || 'unknown'})`
  );
} else if (w._taskText && r.idleMs >= stallThresholdMs && !w._interventionState) {
  this._notifications.notifyStall(r.name, `no output for ${...}min`);
}
```

Before this loop, a bg_exit stall promoter runs:

```js
if (w._interventionState === 'bg_exit' && r.idleMs >= bgExitPromoteMs
    && !interventionState.detectApprovalPrompt(snap)) {
  w._interventionState = 'escalation';
  w._lastInterventionAt = new Date().toISOString();
  w.snapshots.push({ screen: `[STALL PROMOTE] bg_exit -> escalation after ...` });
}
```

Threshold: `bgExitPromoteMs = 600000` (10 min). Honours the spec's
"bg_exit + stall (>10min no output + no prompt) -> THEN escalate"
contract. Still skips the idle-only `no output for Nmin` message
when `_interventionState` is set so we do not double-notify.

## P2 — `c4 list` UI / JSON surface

### `c4 list --json` (daemon `GET /list`)

Row shape gains:

```
{
  ...existing fields,
  intervention: 'approval_pending' | 'background_exit' | 'past_resolved' | null,
  hasPastIntervention: boolean,
  lastInterventionAt: string | null,
  ...
}
```

Existing consumers that check `w.intervention` truthy still see
`background_exit` / `past_resolved` as truthy — but the narrowed
enum means callers that want "needs human" must compare explicitly
to `'approval_pending'`. `src/hierarchy-tree.js` `isInterventionActive`
and `web/src/components/WorkerList.tsx` `isInterventionActive` are
updated to do exactly that for string values so the tree rollup
and the UI badge only fire on `approval_pending`.

### Human-readable table (`c4 list`)

`src/cli.js`:

```js
if (w.intervention === 'approval_pending') {
  cell = useColor ? '\u001b[31mAPPROVAL\u001b[0m' : 'APPROVAL';
} else if (w.intervention === 'background_exit') {
  cell = useColor ? '\u001b[33mbg-exit\u001b[0m' : 'bg-exit';
}
```

ANSI color only when `process.stdout.isTTY` and `NO_COLOR` is unset.
`past_resolved` / `null` render blank (`-`) so the table stays
scannable on long worker lists.

### Web UI

- `web/src/types.ts`: new `PublicIntervention` union type; `Worker`
  grows optional `hasPastIntervention` + `lastInterventionAt`.
- `web/src/components/WorkerList.tsx` `isInterventionActive`: string

  preserved as a fallback for cached payloads.

## Not done in this patch

- The tree rollup label on `bg_exit` / `past_resolved` stays plain
  `status` badge for now — the dashboard HTML renderer still escapes
  the raw `intervention` string when present. Visible in the web
  UI as `[background_exit]` / `[past_resolved]` badges, but no
  dedicated color or icon yet. Cheap follow-up if we care.
- `_interventionState` internal type remains the legacy union. A
  cleaner refactor would retire `'escalation'` entirely in favour
  of `'approval_pending'` but that fans out to many tests; the
  `mapInterventionToPublic` shim keeps the public contract clean
  without that coordination.
- The stall-promote threshold (10 min) is a constant, not config-
  driven. If operators want per-deployment tuning we can wire
  `config.intervention.bgExitPromoteMs` later.

## Design decisions

- **Why not retire `_interventionState` entirely?** The hook /
  state-machine / question paths all touch it, plus `c4 approve`
  relies on `=== 'critical_deny'`, plus half the tests assert
  against the legacy values. A mapping shim in `list()` is strictly
  additive and removes the only broken contract (the public shape).
- **Why tail-regex instead of a proper parser?** The Claude Code
  TUI is ANSI-escaped text; a full parser would have to re-implement
  the terminal emulator. A small regex list over the last 60 lines
  is cheap, catches every documented prompt we have seen in the
  wild, and is trivial to extend as new prompt strings emerge.
- **Why auto-clear on list() and not via event?** The detector is
  ~µs. Running it per list() means the state stays honest even
  when no PTY data is flowing and no set-site fires, which matches
  the user-visible symptom (stale sticky state long after the
  actual prompt is gone).
- **Why keep `critical_deny` immune to auto-clear?** The critical
  deny flow expects the operator to explicitly `c4 approve` or the
  TUI Down+Enter. Auto-clearing would effectively skip the critical
  command guardrail if a later prompt happened to not be visible.
