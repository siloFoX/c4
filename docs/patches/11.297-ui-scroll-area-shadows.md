# 11.297 -- ScrollArea shadow indicators (v1.11.315)

Component-scope-only enhancement of the existing `ScrollArea`
primitive. No daemon-side change and no `c4` CLI surface change.
The default surface (no `shadows`) is preserved byte-for-byte
modulo the new `data-section="scroll-area"` attribute.

## Why

The existing ScrollArea already ships the styled-thumb +
overflow-axis primitive (the `c4-scroll` CSS class lives in
`index.css`). The dispatch's additional ask -- "top/bottom
shadow indicators when content overflows" -- was missing.
Three pain points across the recent scroll surfaces:

1. The History sidebar renders a long worker list that scrolls
   the operator's mental model away from the page header.
   Without a fade-shadow at the top, the operator does not see
   that there is more content above.
2. The Workers panel + Templates list have the same shape --
   long scrollable lists where the edge transitions read as
   "hard cuts" instead of "more content here".
3. Existing surfaces handled this by hand: an absolutely
   positioned div with a custom gradient + scroll listener.
   That pattern fragmented across pages.

Patch 11.297 puts the shadow-indicator contract inside the
primitive so every adopter reaches for one prop.

## What changed

### `web/src/components/ui/scroll-area.tsx`

- New `shadows?: boolean` prop (default false). When true:
  - Wraps the scrollable surface in a relative-positioned
    `<div data-section="scroll-area-shadow-root">` so the
    overlays can be absolutely positioned without leaking
    into the parent layout.
  - Mounts two `<div aria-hidden>` overlays:
    `data-section="scroll-area-shadow-top"` and
    `data-section="scroll-area-shadow-bottom"`. Each uses
    `bg-gradient-to-b from-background/80 to-transparent`
    (mirrored for the bottom) so the shadow blends into the
    page background rather than masking the content.
  - Tracks scroll position via a scroll listener on the
    inner surface + a `ResizeObserver` on the same node + a
    window resize listener (for browsers without RO).
    State is held in two booleans (`atTop`, `atBottom`)
    that toggle the overlays' `opacity-0` class:
    - At top: top overlay hidden, bottom overlay visible
      (when overflowing).
    - At bottom: top overlay visible (when overflowing),
      bottom overlay hidden.
    - Non-overflowing: both flags stay `true`, both
      overlays stay hidden.
- `useImperativeHandle` hop preserves the forwardRef
  contract for callers while letting the primitive's
  internal effect read the same inner node.
- New data-attribute selectors:
  - `data-section="scroll-area"` on the scrollable surface
    (always present, regardless of `shadows`).
  - `data-shadows="true|false"` on the surface.
  - `data-at-top="true|false"` + `data-at-bottom="true|false"`
    on BOTH the surface AND the shadow-root container so
    e2e can target either layer.
  - `data-section="scroll-area-shadow-root"` on the wrapper.
  - `data-section="scroll-area-shadow-top"` on the top
    overlay.
  - `data-section="scroll-area-shadow-bottom"` on the
    bottom overlay.

### Adopted

- Primitive enhancement only this commit. Existing
  ScrollArea call sites inherit the new `data-section`
  attribute without caller changes.

## Tests

- `web/src/components/ui/scroll-area.test.tsx` -- 8 new
  vitest cases:
  - `shadows=false` renders the same single div.
  - `shadows=true` wraps the surface in a shadow-root
    container.
  - `shadows=true` mounts the top + bottom overlays.
  - Non-overflowing content keeps `data-at-top` +
    `data-at-bottom` both `true`.
  - Top shadow has `opacity-0` when at top (default state).
  - Bottom shadow has `opacity-0` when at bottom (default).
  - Overlays carry `pointer-events-none` + `aria-hidden`.
  - `data-section="scroll-area"` on the surface.
  All 28 (20 original + 8 new) green.

## Notes

- `ResizeObserver` is feature-detected -- under jsdom the
  observer is absent and the fallback is the window resize
  listener. The surface still works (the initial recompute
  on mount populates the flags correctly).
- The overlays use `bg-gradient-to-b from-background/80
  to-transparent` so the colour follows the active theme
  token rather than hard-coding a hue.
- The 4-pixel overlay height (`h-4`) is intentionally
  shallow enough that it does not obscure a single line of
  content beneath it; the operator still reads the
  upcoming row clearly while the fade hints at "more
  content".
