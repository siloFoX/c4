# 11.197 -- UI: mobile viewport polish (v1.11.215)

Mobile viewport + safe-area inset polish for the web UI. Component scope only -- no daemon-side changes, no new runtime dependencies.

## Why

iOS Safari and Android Chrome both expose the device safe-area insets through `env(safe-area-inset-*)`, but only when the page declares `viewport-fit=cover` on its viewport meta. Without those insets, the `AppHeader` notch overlaps the status bar / camera cut-out and the bottom-pinned `Sidebar` + `Toast` portal collide with the home-indicator. This patch wires the viewport meta, the Tailwind utility tokens, and the three layout chrome surfaces so the browser-reserved gutters round-trip into the app.

## Surfaces touched

- `web/index.html`
  - `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">`.
  - `<meta name="theme-color" content="#0a0a0a" media="(prefers-color-scheme: dark)">` + a `#ffffff` light variant so the browser chrome blends with the app surface.
- `web/tailwind.config.js`
  - `theme.extend.padding` gains `safe-t` / `safe-b` / `safe-l` / `safe-r` mapped to `env(safe-area-inset-top|bottom|left|right)`.
  - The same set is mirrored into `theme.extend.spacing` so the tokens also resolve through `m-*` / `gap-*` / `inset-*` utilities.
- `web/src/styles/safe-area.css` (new, ~25 lines)
  - `@supports (padding: env(safe-area-inset-top))` -> surface `--safe-area-inset-*` CSS variables on `:root`.
  - `@supports not (padding: env(safe-area-inset-top))` -> pin the variables to `0px` so older engines that drop `env()` keep stable layout.
- `web/src/main.tsx`
  - Import `./styles/safe-area.css` right after `./styles/print.css`.
- Layout chrome adopters:
  - `web/src/components/layout/AppHeader.tsx` -- `<Navbar className>` gains `pt-safe-t pl-safe-l pr-safe-r min-h-[calc(3rem+env(safe-area-inset-top))]`.
  - `web/src/components/layout/Sidebar.tsx` -- root `<aside>` gains `pl-safe-l pb-safe-b`.
  - `web/src/components/Toast.tsx` -- the lazily-created `#toast-root` portal element gains `pb-safe-b pl-safe-l pr-safe-r`.

## Tests

`web/src/styles/safe-area.test.ts` (vitest, ~70 lines, 9 cases). Each case reads a single static surface via `fs.readFileSync` (relative to `WEB_ROOT = resolve(__dirname, '..', '..')`) and asserts the expected string:

1. `index.html` includes `viewport-fit=cover`.
2. `index.html` declares `theme-color` for both `prefers-color-scheme: dark` and `light`.
3. Tailwind config exposes `safe-t` / `safe-b` / `safe-l` / `safe-r` + the `env(safe-area-inset-top|bottom)` strings.
4. `safe-area.css` carries the `@supports (...)` query, the negated fallback, and the `--safe-area-inset-top` variable.
5. `main.tsx` imports `./styles/safe-area.css`.
6. `AppHeader.tsx` mentions `pt-safe-t`, `pl-safe-l`, `pr-safe-r`.
7. `Sidebar.tsx` mentions `pl-safe-l` + `pb-safe-b`.
8. `Toast.tsx` mentions `pb-safe-b` + `pl-safe-l` + `pr-safe-r`.
9. `web/package.json` `version` is `1.11.215`.

Test command:

```bash
env -C /root/c4-worktree-auto-w181/web /root/c4/web/node_modules/.bin/vitest run src/styles/safe-area.test.ts
```

`web/node_modules` is not provisioned in this worktree, so the suite is documented but not executed in this commit. The assertions are all static `readFileSync` + string `.toContain` / `.toMatch` checks -- no JSDOM, no React render -- so they will run anywhere the host `web/node_modules` is available.
