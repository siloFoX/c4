## Design

- The component stays render-driven; no new
  runtime dependencies.
- A small regex-based tokeniser handles the
  four canonical languages
  (`json` / `javascript` / `typescript` /
  `bash`). Adopters that need richer
  grammar plug in shiki / hljs / prism via
  the new `highlight` callback prop.
- Token spans render INSIDE the existing
  `<code data-code-block-code>` element so
  the line-number gutter + copy + wrap
  toggle keep working unchanged.

## API

```tsx
import {
  CodeBlock,
  highlightCode,
  CODE_BLOCK_SUPPORTED_LANGUAGES,
} from './components/ui/code-block';

// Legacy (plain text)
<CodeBlock code={src} language="json" /> // built-in highlighter
<CodeBlock code={src} />                  // no language -> plain
<CodeBlock code={src} language="rust" />  // unsupported -> plain

// Opt out
<CodeBlock code={src} language="json" highlight={false} />

// Plug-in highlighter
<CodeBlock
  code={src}
  language="rust"
  highlight={(code, language) => myShikiRender(code, language)}
/>
```

### `highlight` prop semantics

| value | effect |
| --- | --- |
| `undefined` (default) | built-in tokeniser if language is supported, else plain text |
| `false` | skip highlighting entirely (legacy byte-identical) |
| `(code, language?) => ReactNode` | caller plug-in (bypasses built-in) |

### Pure helper

```ts
export function highlightCode(
  code: string,
  language?: string,
): CodeBlockToken[];

export const CODE_BLOCK_SUPPORTED_LANGUAGES: readonly string[];
```

Both are exported for tests + alternate
hosts that want to render their own
chrome around the tokens.

## Tokeniser

```ts
type CodeBlockTokenType =
  | 'plain'
  | 'keyword'
  | 'string'
  | 'number'
  | 'comment'
  | 'boolean'
  | 'variable';
```

Per-token Tailwind class:

| type | class |
| --- | --- |
| `plain` | `text-foreground` |
| `keyword` | `text-info` |
| `string` | `text-success` |
| `number` | `text-warning` |
| `comment` | `italic text-muted-foreground` |
| `boolean` | `text-info` |
| `variable` | `text-foreground` |

### Language coverage

- **json** -- strings, numbers, `true` /
  `false` / `null`.
- **javascript** -- line + block comments,
  single / double / template strings,
  numbers (incl. float / exponent), ~40
  keywords, `true` / `false` / `null` /
  `undefined`.
- **typescript** -- JS plus `type` /
  `keyof` / `satisfies` / `is` / `never` /
  `unknown` / `infer` / `asserts`.
- **bash** -- `#` line comments, single /
  double strings, `$VAR` / `${VAR}`
  variables, ~25 builtins (if / then /
  else / fi / for / while / done / echo /
  cd / ls / grep / sed / awk / find /
  source / etc), numbers.

### Aliases

`js` / `jsx` -> javascript,
`ts` / `tsx` -> typescript,
`sh` / `shell` / `zsh` -> bash.

## Pairs with existing primitives

- `<RichText>` -- markdown-style body
  content; the new highlighter does NOT
  cover three-backtick fenced code blocks
  inside RichText. RichText callers that
  want highlighted snippets embed
  `<CodeBlock>` themselves.
- `<Snippet>` -- short inline code
  without the header / copy / line-number
  chrome.

## Out of scope

- Per-page adoption of the highlighter.
  Additive; every existing call site
  stays byte-identical when language is
  omitted.
- Theme-aware token palette. Hard-coded
  info / success / warning / muted; light-
  vs-dark re-tunes belong in a plug-in.
- Grammar accuracy beyond the four
  built-in languages. Adopters bring
  shiki / hljs / prism via the callback.
- Line-level highlighting (diff hunk
  markers, error gutters). Separate diff
  primitive.
- Token-level click handlers.
