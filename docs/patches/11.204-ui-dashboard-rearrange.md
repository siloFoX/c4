# 11.204 -- UI: Health hero card rearrange (v1.11.222)

## Summary

Operator-reorderable hero KPI cards on the Health page. The three
existing `StatCard`s (Uptime trend, Workers, Queue trend) are now
draggable via an inline `GripVertical` handle in each card's top-right
corner. The chosen order persists per-browser under
`localStorage['c4:health:layout']` and is broadcast through the
`health-layout-changed` `CustomEvent` so any future subscriber
re-renders without a context provider. A small `Reset layout` text
button below the hero clears the override and restores the default
order.

## Scope

Strictly component-scoped. No new dependencies, no new lucide icon
additions (`GripVertical` already shipped via `Queue.tsx`), no router
changes. The card visuals are untouched -- only the order, the wrapper
`draggable` div, and the grip handle are new.

## Files

- `web/src/lib/use-health-layout.ts` (new, ~125 lines)
  - `HealthLayoutKey = 'uptime' | 'workers' | 'queue'` union derived
    from the three live hero `StatCard`s on `pages/Health.tsx`.
  - `DEFAULT_LAYOUT` const `['uptime', 'workers', 'queue']`.
  - `STORAGE_KEY = 'c4:health:layout'`, `EVENT_NAME = 'health-layout-changed'`.
  - `getLayout()` reads the persisted order, filters out unknown keys,
    de-dupes, then appends any default keys missing from the stored
    order (so adding a fourth hero card in a follow-up does not strand
    operators on an outdated layout).
  - `setLayout(order)` filters + de-dupes the incoming array, writes
    to `localStorage`, and dispatches `health-layout-changed` carrying
    the fully merged order in `event.detail.order`.
  - `resetLayout()` removes the storage key and dispatches the default
    order.
  - `useHealthLayout()` returns `[layout, setLayout, resetLayout]`,
    subscribing to the event so cross-subtree mutations propagate.
  - SSR-safe: every `window` / `localStorage` touch is guarded.

- `web/src/pages/Health.tsx`
  - Imports `useHealthLayout` + `GripVertical`.
  - Hero `DashboardGrid` now maps over `useHealthLayout()[0]` instead
    of hard-coding the three `DashboardGrid.Item`s, with each card
    rendered by a `Record<HealthLayoutKey, () => JSX.Element>` lookup
    so the existing card visuals stay identical.
  - Each card is wrapped in a `<div draggable data-card-key={key}>`
    with HTML5 `onDragStart` / `onDragOver` / `onDrop` handlers that
    splice the dragged key into the target slot and call `setLayout`.
  - A `GripVertical` button sits absolute top-right inside each card
    wrapper (`data-testid="health-hero-drag-<key>"`) so the drag
    affordance is visible without affecting the card's internal
    layout.
  - A small `Reset layout` text button (`data-testid=
    "health-hero-reset-layout"`) sits flush-right below the
    `DashboardGrid` and calls `resetHeroLayout()`.

- `web/src/lib/use-health-layout.test.ts` (new, ~115 lines, vitest + RTL)
  - `getLayout returns DEFAULT_LAYOUT when no override`
  - `setLayout persists order to localStorage`
  - `getLayout merges any new defaults at the end of stored order`
  - `getLayout filters out invalid keys`
  - `resetLayout clears storage and restores default`
  - `CustomEvent fires on setLayout and resetLayout`
  - `useHealthLayout reflects setLayout via event`
  - `setLayout de-duplicates and filters invalid keys before
    persisting`
  - 8 cases total; well above the required 6.

## Notes / limitations

- The hero has exactly three cards today, so the spec's "first 5 only,
  document the limitation if structure is too complex" fallback does
  not apply -- all three cards are draggable.
- Keyboard reordering is not implemented in this patch. The
  `GripVertical` button carries `tabIndex={-1}` to keep the default
  tab order clean; an `aria-label="Drag <key> card"` is set so
  assistive tech can announce the affordance. A keyboard-driven
  reorder surface is a natural follow-up (arrow-keys on focus).
- The `c4:health:layout` storage key is intentionally narrow so a
  future Sessions / History hero can mint its own key without
  collision.
