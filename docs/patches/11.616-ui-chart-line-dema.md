## Summary

New `ChartLineDema` UI primitive at
`web/src/components/ui/chart-line-dema.tsx` -- a pure-SVG line chart
with a Double Exponential Moving Average overlay drawn on the same
axes as the price. The DEMA cuts the lag of a plain EMA.

## Indicator math

Patrick Mulloy's Double Exponential Moving Average:

```
ema1 = EMA(value, period)
ema2 = EMA(ema1, period)
DEMA = 2 * ema1 - ema2
```

A plain EMA lags the price. The EMA of that EMA lags twice as much,
so the gap `ema1 - ema2` is itself a measure of the lag. Adding that
gap back to the EMA -- `2 * ema1 - ema2` -- cancels most of the lag
while keeping the line smooth.

The EMA helper is the project's null-tolerant, SMA-seeded EMA: it
collects the defined values, seeds with the simple mean of the first
`period` of them at that value's index, then folds each later value
in at weight `2 / (period + 1)`. The first EMA is therefore defined
from index `period - 1`; the EMA of the EMA needs another `period`
defined values, so the **DEMA is defined from index `2 * period - 2`
onward**.

Because the DEMA is a second-order lag corrector, it reproduces a
linear price ramp with **zero lag** once warmed up -- the linear lag
is fully cancelled.

### Worked fixture

`values = [3, 6, 9, 12, 15, 18, 21]` (a perfect linear ramp),
`period 3`. The EMA multiplier is exactly `2 / (3 + 1) = 0.5` and
every window sum divides cleanly, so the whole pipeline stays
bit-exact:

```
ema1 = [.,., 6,  9,  12, 15, 18]
ema2 = [.,.,.,., 9,  12, 15]
dema = 2*ema1 - ema2 = [.,.,.,., 15, 18, 21]
```

The DEMA reproduces the ramp value exactly from index 4 (`2*3-2`)
onward: `dema[4] = 15 = value[4]`, `dema[6] = 21 = value[6]`. Every
value is exact, so the tests assert the series with `toEqual` rather
than `toBeCloseTo`.

## API

- `getLineDemaFinitePoints(points)` -- drop points with a non-finite
  `x` or `value`.
- `normalizeLineDemaPeriod(period, fallback)` -- coerce to a
  positive integer; non-finite or sub-1 falls back, fractional
  floors.
- `computeLineDemaEma(src, period)` -- the null-tolerant SMA-seeded
  EMA series.
- `computeLineDema(values, period)` -- returns `{ ema1, ema2,
  dema }`.
- `runLineDema(points, options?)` -- sort finite points by `x`,
  compute the series, return `{ series, period, ema1, ema2, dema,
  samples, demaFinal, demaMin, demaMax, aboveCount, belowCount,
  ok }`. Each sample carries the price's `above` / `below` / `on`
  position versus the DEMA. `ok` is false for fewer than two points.
- `computeLineDemaLayout(options)` -- project the price and DEMA
  onto one shared panel. Returns the panel, ticks, the price path,
  the DEMA path, and DEMA markers.
- `describeLineDemaChart(data, options?)` -- ARIA description.

## Component

`ChartLineDema` is a `forwardRef` region (`role="region"`) with an
ARIA-linked description, a `role="img"` SVG, axis ticks and grid, a
config badge, a two-series legend (Price / DEMA) with toggle, a
hover/focus tooltip exposing the value, DEMA and position, and a
`motion-safe:animate-fade-in` entrance. Every node carries a
`data-section` attribute.

## Distinctness

- **vs chart-line-zlema** -- the ZLEMA removes lag by feeding a
  *de-lagged input* (the price plus its own recent momentum)
  through a single EMA. The DEMA leaves the input untouched and
  removes lag through an *algebraic combination of two EMAs*,
  `2 * EMA - EMA(EMA)` -- no momentum term, just the
  EMA-of-EMA correction.
- **vs chart-line-kama** -- the KAMA *adapts* its smoothing every
  bar from an efficiency ratio. The DEMA applies a *fixed* two-EMA
  formula regardless of market conditions.
- **vs a plain EMA / a triple EMA (TEMA)** -- a plain EMA has no lag
  correction; the DEMA adds one `EMA - EMA(EMA)` correction term. A
  TEMA would add a further `EMA(EMA(EMA))` stage
  (`3*EMA - 3*EMA(EMA) + EMA(EMA(EMA))`); the DEMA stops at the
  double stage.
