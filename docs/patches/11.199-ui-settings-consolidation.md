# 11.199 -- UI: settings consolidation (v1.11.217)

A consolidated `Settings` landing page that pulls the most frequently
touched preference surfaces (theme, locale, scribe, notifications,
feature flags, daemon config) behind a single tab strip. Existing
dedicated pages remain authoritative; this page only REFERENCES or
EMBEDS them so operators get a one-stop entry point without disrupting
the already-shipped surfaces.

- `web/src/pages/Settings.tsx` -- new tabbed landing page (~200 lines)
- `web/src/pages/Settings.test.tsx` -- 6 vitest+RTL cases
- `web/src/pages/registry.ts` -- new `settings-page` entry under

## Tabs

The page uses the existing `Tabs` / `TabsPanel` primitives in
`web/src/components/ui/tabs.tsx` (ARIA tablist + roving tabindex +
arrow / Home / End keyboard nav already wired) -- no new primitive.

| Tab key          | Tab label       | Body                                                                                                                |
| ---------------- | --------------- | ------------------------------------------------------------------------------------------------------------------- |
| `general`        | General         | Inline summary describing the Config page (GET /api/config + reload trigger) plus a `#feature=config` deep-link.    |
| `theme`          | Theme           | Reuses the shipped `<ThemeToggle variant="group">` plus the `useTheme()` hook; persists to `c4:theme` localStorage. |
| `scribe`         | Scribe          | Inline summary pointing back to the Scribe page (start / stop / scan + last context).                               |
| `notifications`  | Notifications   | Inline summary pointing back to the Notifications feed page.                                                        |
| `locale`         | Locale          | `radiogroup` over `LOCALES` (`en` / `ko`) wired through `setLocale()`.                                              |
| `feature-flags`  | Feature Flags   | Embeds `<FeatureFlags />` inline so the live admin Panel rows + Reset Button + the `c4:feature-flags` localStorage round-trip render unchanged. |

The default selection is `general`; switching tabs flips `aria-selected`
on the trigger and the corresponding `<TabsPanel>` becomes visible.

## Persistence

| Channel                 | Storage key         | Helper                                                  |
| ----------------------- | ------------------- | ------------------------------------------------------- |
| Theme                   | `c4:theme`          | `useTheme()` from `web/src/hooks/use-theme.ts`          |
| Locale                  | `c4.locale`         | `setLocale()` from `web/src/lib/i18n.ts` (plus `c4:locale-changed` `CustomEvent`) |
| Feature flag overrides  | `c4:feature-flags`  | `setFlag()` / `useAllFlags()` from `web/src/lib/feature-flags.ts` (plus `feature-flag-changed` `CustomEvent`) |

The Locale storage key uses the existing `LOCALE_KEY` constant (value
`c4.locale`) rather than introducing a new `c4:locale` slot. The task
spec asked for `c4:locale` persistence, but writing to a different key
than the shipped `setLocale()` helper would create dual locale state in
the same browser. Reusing the existing helper is the strict "REFERENCES
or EMBEDS existing controls" path the spec asked for. Test asserts
against `LOCALE_KEY` so any future rename stays consistent.

## Tests

`web/src/pages/Settings.test.tsx` covers 6 cases:

1. All 6 tab triggers render inside the `Settings sections` tablist.
2. The page mounts on the General tab and renders the inline summary
   (matches `Live daemon config sans secrets` plus the
   `settings-general-config-link` testid).
3. Clicking Theme / Locale flips `aria-selected` on the trigger and
   the corresponding panel becomes visible.
4. Clicking the Korean radio writes `ko` to
   `window.localStorage[LOCALE_KEY]`; clicking the English radio
   writes `en`.
5. Feature Flags tab renders the embedded `<FeatureFlags />` page --
   the `Grid Debug Overlay` row (label from `lib/feature-flags.ts`) is
   visible alongside the `Reset` button.
6. Scribe + Notifications tabs each render their summary copy.

Run:

```bash
env -C /root/c4-worktree-auto-w183/web /root/c4/web/node_modules/.bin/vitest run src/pages/Settings.test.tsx
```

The worktree's `web/node_modules` is symlinked to the primary
checkout's modules so the suite runs in-place; result is 6/6 passing
(Test Files 1 passed, Tests 6 passed).

## Out of scope

- No state migration -- existing Config / Scribe / Notifications /
  Feature Flags pages stay in place and remain authoritative; this
  page is the new entry point, not a replacement.
- No live i18n reload requirement is added by the Locale tab beyond
  what `setLocale()` already broadcasts (mounted `useLocale()` consumers
  re-render via `c4:locale-changed` for free).
- `package-lock.json` untouched.
- `docs/autonomous-queue-v10.md` untouched (manager owns the queue).
