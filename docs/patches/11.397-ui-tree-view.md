## Behaviour

### Keyboard (WAI-ARIA tree)

| key | effect |
| --- | --- |
| ArrowDown | next visible row |
| ArrowUp | previous visible row |
| ArrowRight | expand collapsed; if expanded, go to first child |
| ArrowLeft | collapse expanded; if collapsed, go to parent |
| Home | first row |
| End | last visible row |
| Enter / Space | toggle expansion + select |

### Drag-and-drop reorder

- `enableDrag` flips `draggable` on
  every non-disabled row.
- Drop position is computed by the
  pure helper
  `computeDropPosition(y, height,
  canHaveChildren)`:
  - can-have-children -> top 25%
    'before' / middle 'inside' /
    bottom 25% 'after'.
  - leaf -> top half 'before' /
    bottom half 'after'.
- `onReorder(source, target, pos)`
  fires on drop. The host mutates the
  node array (use
  `updateNodeChildren` for nested
  changes).
- Self-drop and descendant-drop both
  blocked at the primitive level.

### Lazy loading

- A node without `children` and
  without `isLeaf: true` is treated
  as lazy.
- Expanding (via click on the
  toggle button or ArrowRight) calls
  `onLoadChildren(node)`. While the
  promise is pending the
  `loadingLabel` shows in the row
  (default "Loading...") and
  `data-tree-loading="true"` flips.
- The host updates the node array
  with the loaded children. Use the
  exported helper
  `updateNodeChildren(nodes, id,
  children)`.

### Selection / focus / expansion

All three are controlled-or-
uncontrolled. Each supports a
`*Id` controlled prop, a
`default*Id` seed for uncontrolled
mode, and a `on*Change` callback
that fires in both modes.

### Disabled rows

- `disabled: true` ->
  `aria-disabled="true"`,
  `draggable="false"`, the click
  handler short-circuits, and a
  visual treatment (`opacity-50
  cursor-not-allowed`) flips.
- Keyboard navigation does NOT skip
  disabled rows -- they are still
  reachable for inspection. Action
  shortcuts (Enter / Space / drag)
  fall through.

- `data-section="tree-view"`
- `data-node-count`

`aria-level`, `aria-expanded`

`aria-disabled`):

- `data-section="tree-view-item"`
- `data-tree-id`
- `data-tree-depth`
- `data-tree-expanded`
  (`true` / `false` / `leaf`)
- `data-tree-selected`
- `data-tree-focused`
- `data-tree-loading`
- `data-drop-position`
  (during drag)

Sub-nodes:

- `data-section="tree-view-toggle"`
  -- expand button
- `data-section="tree-view-spacer"`
  -- leaf placeholder
- `data-section="tree-view-icon"`
- `data-section="tree-view-label"`
- `data-section="tree-view-loading"`
- `data-section="tree-view-empty"`

## Pairs with existing primitives

- `<HierarchyTree>` -- domain-
  specific feature module tree.
  TreeView is the generic primitive
  that can host the same data.
- `<VirtualTable>` (11.375) -- flat
  data; TreeView is the nested
  variant.
- `<CommandBar>` (11.395) -- often
  paired with a multi-select tree
  host (multi-select itself is out of
  scope here).

## Out of scope (deferred)

- Multi-selection. Single-id model
  for v1; a follow-on can ship
  `selectedIds: Set<string>` +
  shift / ctrl click semantics.
- Virtualized rendering for very
  large trees. Pair with the
  existing virtualization primitives
  in a follow-on.
- Custom drag image / ghost
  element. Native drag preview is
  fine for v1.
- Touch-gesture drag. HTML5 drag
  events do not fire from touch
  without polyfills; out of scope
  for v1.
- Inline rename / cut / paste.
  Belongs in the host.
- Cross-tree DnD (drop in one tree
  from another). v1 enforces same-
  tree only.
