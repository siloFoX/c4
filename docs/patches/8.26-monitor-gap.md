# TODO 8.26 - Approval-miss prevention mechanism

**Status:** done
**Branch:** `c4/monitor-gap`
**Scope:** Close the approval-miss gap where `c4 wait
--interrupt-on-intervention` returned on the worker's first idle and
left subsequent approval prompts unattended until the next 30-minute
stall-detection cron.

## Summary

Before this patch a reviewer session babysitting a worker had to
re-arm `c4 wait --interrupt-on-intervention` after every return. When
the worker idled, then raised an approval prompt a few seconds later,
nothing watched for the transition. 2026-04-20 repro: the manager
session sat on a `bash /tmp/run-tests-main.sh` approval for tens of
minutes before the user interjected ("일 안 하는 거 아냐?"). 8.26
closes the gap:

1. **`src/approval-monitor.js` (new).** A pure diff-tracker that
   receives a plain list of workers with their `publicIntervention`
   shape and fires `enter` / `exit` / `slack_alert` / `timeout`
   events on state transitions. No imports of pty-manager internals,
   no timers on construction — the pty-manager owns the tick
   interval, and the monitor can be tested with a fake clock and a
   stub worker list. Thresholds: `slackAlertAfterMs=60000`,
   `approvalTimeoutMs=3600000`, `autoReject=false`. All configurable
   via `config.monitor` so ops can tune without a code change.

2. **PtyManager wiring (`src/pty-manager.js`).** The constructor
   spins up an `ApprovalMonitor` with three collaborators:
   `getWorkers()` returns the current approval-watch rows (live,
   `_interventionState` auto-cleared via the existing
   `intervention-state` helper), `slackEmit()` defers to the 8.15
   shared emitter through the new `setSlackEmitter` bridge, and
   `onAutoReject(name, message)` calls into `_autoRejectApproval`
   which sends a corrective line through the normal `send()` path
   and clears the intervention flag so the next tick fires `exit`.
   A 1-second `setInterval` (unref'd so it doesn't block process
   exit) drives the tick. The daemon reload path picks up new
   thresholds without a restart — configure() re-normalises the
   config in place.

3. **SSE route `/api/approvals/stream` (`src/daemon.js`).** Reviewer
   sessions and the new CLI subscribe once and receive every
   transition for every worker. The route writes an initial
   `connected` frame plus a snapshot of currently pending approvals
   so a mid-approval connect doesn't miss the worker already
   sitting on a prompt. A 25-second SSE keep-alive comment keeps
   reverse proxies / browsers from timing out. A sibling
   `GET /api/approvals` returns the snapshot as JSON for non-stream
   consumers (tests, external dashboards).

4. **`c4 wait --follow` (`src/cli.js`).** New flag on the existing
   wait command. Instead of the one-shot return, `--follow` hands
   off to `runApprovalFollow()` which subscribes to
   `/api/approvals/stream` and prints transitions until `Ctrl+C`.
   Single-name invocations scope the filter to that worker; `--all`
   or no name means every worker. The reviewer no longer needs to
   re-arm after every idle return — the persistent connection
   captures the next approval as soon as the daemon's tick sees it.

5. **`c4 watch-interventions` (`src/cli.js`).** Standalone command
   that shares the same SSE subscriber path. Safe to run outside a
   Claude Code reviewer session (terminal tab, cron-less watchdog,
   SSH from a phone), which the task called out explicitly — the
   reviewer role is a pattern, not a process dependency.
   `--worker <name>` / `--all` flags are recognised for symmetry
   with `wait --follow`.

6. **Slack alert on stale approval (>60s).** The monitor tick fires
   `slack_alert` exactly once per pending span when
   `pendingMs >= slackAlertAfterMs`. The emit reuses the existing
   `approval_request` event type from 8.15 (same webhook, same
   dedup window, same level filter) so operators already subscribed
   to approval webhooks see the stale-approval ping without a new
   event taxonomy. `slackAlertAfterMs: 0` disables the ping.

7. **Per-approval timeout (default 1h) + optional auto-reject.**
   `approvalTimeoutMs` defaults to 1h; when `autoReject: true` the
   monitor hands the worker name + configurable
   `autoRejectMessage` to `PtyManager._autoRejectApproval`, which
   calls the normal `send()` path and clears the intervention
   state. Auto-reject is opt-in so the default shipping behavior is
   "surface, don't mutate" — the reviewer still decides, they just
   get 1h instead of 30 minutes and a Slack ping at 60s.

8. **`.claude/agents/manager.md` rule update.** Reviewer workflow
   guidance now recommends `c4 wait --follow` over cron re-arming
   of `c4 wait --interrupt-on-intervention`. Failure-mode reminder
   stays in place: the manager still inspects the pending content
   before approving (no blind Enter).

## API contract additions

- `GET /api/approvals` — one-shot snapshot.
  ```json
  { "type": "snapshot", "ts": 1700000000000, "workers": [
      { "name": "w1", "enteredAt": 1700000000000,
        "internalState": "question", "pendingMs": 4000,
        "slackAlertedAt": null, "timeoutFiredAt": null }
  ]}
  ```

- `GET /api/approvals/stream` — SSE event stream. Events:
  - `{ type: "connected" }` — initial frame.
  - `{ type: "snapshot", ts, workers: [...] }` — current pending
    set when the client connected.
  - `{ type: "enter", worker, internalState, enteredAt, ts }`
  - `{ type: "exit", worker, enteredAt, resolvedAt, durationMs,
    ts, reason? }` (reason present when the worker was closed
    while pending).
  - `{ type: "slack_alert", worker, enteredAt, pendingMs, ts }`
  - `{ type: "timeout", worker, enteredAt, pendingMs, action, ts }`
    where `action` is `auto_reject` or `none`.

## Config additions

Add under `config.monitor` in `config.json`:

```json
{
  "monitor": {
    "tickMs": 1000,
    "slackAlertAfterMs": 60000,
    "approvalTimeoutMs": 3600000,
    "autoReject": false,
    "autoRejectMessage": "[c4 monitor-gap] Auto-rejecting stale approval after 1h. Please restate the action or pick an alternative approach that does not require this approval."
  }
}
```

`config.example.json` is not modified — operators who never touched
the file keep defaults.

## Tests

`tests/monitor-gap.test.js` (new): 19 assertions across 5 suites.

- **Defaults suite** locks in the stable default thresholds so a
  future cleanup that bumps the 60s / 1h / auto-reject-off numbers
  does so deliberately.
- **Transitions suite** verifies enter on `null -> approval_pending`,
  exit on `approval_pending -> null`, exit with `reason=worker_gone`
  when the worker disappears between ticks, and that a stable
  pending state does not re-fire events on every tick (a regression
  guard: duplicate enter events would make SSE clients notify twice).
- **Slack + timeout suite** exercises the `slackAlertAfterMs` edge
  (fires at exactly the threshold, does not re-fire on the next
  tick), `approvalTimeoutMs` with and without `autoReject`, both
  disable paths (0ms threshold), and confirms that a throwing
  `slackEmit` / `onAutoReject` does not break the tick.
- **Subscription semantics suite** covers multi-subscriber delivery,
  unsubscribe behaviour, snapshot shape with and without prior ticks.
- **Formatter contract suite** locks in the fields the CLI
  `runApprovalFollow` formatter reads — a future rename of
  `durationMs` etc. breaks the test, not a live reviewer session.

Pre-existing failing tests (`cli-api-prefix`, `mcp-hub`, `rbac`,
`session-auth`, `web-control`) require `bcryptjs` which is not
installed in this environment and fail identically on `main`. Full
suite delta: +19 tests, 0 new failures. Full run:
`106 passed, 5 failed, 111 total`.

## Why this shape

Three alternatives were considered:

1. Patch `waitAndRead` to re-arm automatically after every idle
   return. Rejected: the existing `wait-read` endpoint is a one-shot
   contract used by many callers (scripts, `c4 wait w1 w2`, the
   auto-mode loop). Re-arming inside it would change those callers'
   semantics silently.
2. Reduce the 30-minute cron to 1 minute. Rejected: the cron runs
   the full health check (lost-worktree cleanup, dirty warning,
   slack digest, worktree prune) — tightening it 30x would spam
   Slack, beat up git, and still only catch approvals 60s old.
3. The chosen approach: a dedicated 1-second ticker on top of the
   already-computed `publicIntervention` mapping, with a push-based
   SSE stream so reviewers and the Web UI both subscribe. Cheap to
   compute (tail-regex we were already running per list()), cheap
   to serve (one Set of SSE clients), and the transitions drive the
   Slack emitter through the existing 8.15 pipeline so the event
   fabric stays unified.

## Out of scope

- Web UI surface for approvals. The JSON snapshot and SSE stream
  are live; wiring a dashboard pane is a follow-up (future TODO).
- Per-worker timeout overrides. Currently `approvalTimeoutMs` is
  daemon-global. A future patch can accept a per-worker override
  at task submission time.
- Replacing the 30-minute cron's intervention handling. The cron
  still runs for the other health signals; we narrow its approval
  duty by attaching this faster watcher.
