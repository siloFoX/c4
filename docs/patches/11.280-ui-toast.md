# 11.280 -- Toast notification system (v1.11.298)

New `ToastProvider` + `useToast()` primitive at
`web/src/components/ui/toast.tsx`. Fresh companion to the existing
`web/src/components/Toast.tsx` + `lib/use-toast.ts` stack which
keeps driving Profiles / Templates / Settings unchanged.

## Why

The legacy toast stack uses per-page `useToast()` calls that own
their own local queue. App-global pushToast from a non-page surface
(navbar, autonomous loop alerts) was impossible without lifting
state. Patch 11.280 ships a context-based API so any descendant
can push toasts without owning a queue. The two stacks coexist via
the shared `toast-root` portal id; existing call sites do not need
to migrate today.

## What changed

### `web/src/components/ui/toast.tsx` (new file)

- `ToastProvider` -- React context provider. Mount near the app
  root (or scoped to a subtree). Owns the queue + portal layer +
  Esc-to-dismiss listener.
  - `defaultDurationMs?` (default 5000).
  - `visibleLimit?` (default 5) -- older toasts shift out FIFO.
  - `portalId?` (default `toast-root`, shared with the legacy
    stack on purpose).
- `useToast()` hook -- returns
  `{ toasts, pushToast, dismissToast, clearToasts }`.
  `pushToast` accepts
  `{ kind?: 'success' | 'error' | 'info'; message: ReactNode; action?: { label, onClick }; durationMs? }`.
  Throws a clear error when called outside a provider.
- Three visual kinds. `error` uses `aria-live="assertive"`;
  others polite.
- Action button slot -- fires `onClick()` AND dismisses on click.
- Auto-dismiss progress bar pinned to the bottom of the card.
  `durationMs === Infinity` opts out (manual close only).
- Escape dismisses the most recently pushed toast. Window-level
  listener; skips when there are no live toasts so other Esc
  surfaces keep firing.
- Data selectors: `data-section="toast-stack"`, `"toast"`,
  `"toast-dismiss"`, `"toast-action"`, plus `data-toast-kind`,
  `role="progressbar"` on the countdown strip.

### Adopted

- `web/src/pages/DesignSystem.tsx` -- new `<ToastDemo>` with four
  trigger buttons (Success / Error / Info / With action) wrapped
  in a scoped `<ToastProvider defaultDurationMs={4000} visibleLimit={3}>`
  so the demo's toasts do not bleed into the rest of the page.
  The "With action" toast pushes a follow-up info toast when the
  Undo button is clicked, exercising the action button +
  chained-push pattern.

### Deferred

- Settings / Profiles / Templates save flows: already use the
  legacy `useToast()` from `lib/use-toast.ts` + the
  `components/Toast.tsx` stack. Migrating them to the new
  context-based API requires hoisting `ToastProvider` into the
  app shell or wrapping each page subtree, plus rewriting every
  `showToast(...)` call. Both stacks coexist via shared
  portal-id.
- Snapshots delete: already uses `useUndoToast` (bottom-right
  `<UndoToast>` primitive). The new action-button slot would
  enable similar undo flows but the current optimistic-update
  plumbing is outside this TODO's scope.

### Tests

- 22 vitest cases: throw-without-provider, empty-state portal,
  pushToast renders + role / aria, default kind, error ->
  assertive, success/info -> polite, dismiss by id, X button
  dismisses, action button renders + clicking fires onClick +
  dismisses, progressbar present, Infinity drops progressbar
  AND skips auto-dismiss, fake-timer auto-dismiss, per-entry
  duration override, Escape dismisses most recent, clearToasts
  removes every row, visibleLimit FIFO, data-section selectors
  on the stack / row / dismiss / action, Escape no-op when
  empty, unique ids, stable displayName. 28 / 28 across toast +
  DesignSystem.
