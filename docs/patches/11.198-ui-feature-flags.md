## Registry

```ts
export type FeatureFlagKey =
  | 'gridDebug'
  | 'routeProgress'
  | 'pageTransitions'
  | 'motion'
  | 'reducedMotion';

export const FLAGS = [
  { key: 'gridDebug',       label: 'Grid Debug Overlay',     defaultValue: false },
  { key: 'routeProgress',   label: 'Route Progress Bar',     defaultValue: true  },
  { key: 'pageTransitions', label: 'Page Transitions',       defaultValue: true  },
  { key: 'motion',          label: 'Motion / Animations',    defaultValue: true  },
  { key: 'reducedMotion',   label: 'Reduced Motion Override', defaultValue: false },
];
```

## Persistence

- Storage key: `c4:feature-flags`
- Shape: `{[key: FeatureFlagKey]: boolean}` JSON object
- Malformed JSON, missing keys, and non-boolean values fall back to the per-flag `defaultValue`
- Every `setFlag()` rewrites the whole object then fires a `CustomEvent('feature-flag-changed', { detail: { key, value } })` on `window`
- `resetFlags()` removes the storage key and broadcasts the default for every flag so live subscribers re-render

All `window` / `localStorage` calls are guarded for SSR (`typeof window`)
and wrapped in try/catch so private-browsing mode degrades silently.

## Hooks

- `useFeatureFlag(key) -> [value, setValue]` -- subscribes to the
  `feature-flag-changed` event so multiple consumers stay in sync; the
  setter calls `setFlag(key, next)` which both persists and broadcasts.
- `useAllFlags() -> Record<FeatureFlagKey, boolean>` -- table-style
  consumer used by the admin page.

## Admin page

Renders the registry as a `<ul>` of `<Panel>` rows. Each row shows the
label + description, a `key=... default=...` monospace hint, and a
`<Switch>` on the right. An `(override)` chip appears next to the label
when the live value diverges from the default. Header carries a single
`Reset` `<Button>` (disabled when no overrides exist) wired to
`resetFlags()`. An `<Alert variant="info">` at the top reminds operators
the flags are browser-local and never round-trip to the daemon.

## Consumer wiring (deferred)

`GridDebugOverlay` and `RouteProgressBar` already self-gate via their
own localStorage keys (`c4:grid-debug:visible`, etc.) and keyboard
shortcuts. Re-pointing them at `getFlag(...)` would require a small
refactor of their boot path and is out of scope for this patch -- the
admin page is the new authoritative surface and the consumer rewire is
tracked for the next iteration. The registration alone counts per the
task spec.

## Tests

`web/src/lib/feature-flags.test.ts` -- 9 vitest+@testing-library/react
cases:

1. `getFlag` returns the defaultValue for every flag on an empty store
2. `setFlag` persists the value to `c4:feature-flags`
3. `setFlag` dispatches `feature-flag-changed` with `{key, value}`
4. `useFeatureFlag` updates locally when its setter runs
5. `useFeatureFlag` updates when an external `setFlag()` fires the event
6. `useAllFlags` returns every flag and re-renders on update
7. `resetFlags` clears localStorage and restores defaults
8. Malformed JSON in localStorage falls back to defaults
9. Non-boolean stored values are ignored in favor of the default

Test command:

```bash
env -C web /root/c4/web/node_modules/.bin/vitest run src/lib/feature-flags.test.ts
```
