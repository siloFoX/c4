## Goal

The dispatch lists:

- Add a CI-runnable script (`npm run bundle-budget`)
  that asserts the production build stays under
  target byte budgets (main < 500KB gzipped, vendor
  < 800KB gzipped). Fail CI if budget exceeded.
- Add `npm run bundle-budget` to `package.json`.

This patch ships the script + 18 unit tests +
documentation.

## Behaviour

Run AFTER `npm run build`:

```bash
npm --prefix web run build
npm --prefix web run bundle-budget
```

The script walks `web/dist/assets/`, gzips every
`.js` chunk in memory, classifies each chunk
(`main` / `vendor` / `route`), sums the gzipped
bytes per class, and compares to the budget map.

Exit codes:

- `0` = every class within budget.
- `1` = at least one class exceeded OR `dist/`
  missing.
- `2` = usage error.

## Budget map

```js
export const DEFAULT_BUNDLE_BUDGETS = Object.freeze({
  main:   500 * 1024,   // 500 KB gzipped (dispatch)
  vendor: 800 * 1024,   // 800 KB gzipped (dispatch)
  route:  200 * 1024,   // 200 KB gzipped (per-route cap)
});
```

The `route` class is the per-page lazy chunk cap. The
dispatch only specifies `main` + `vendor`; adding a
`route` cap defends against the case where a single
feature page (HistoryView, ChatView, etc.) silently
grows multi-MB without showing up in the main / vendor
totals.

## Classifier rules

| filename | class |
| --- | --- |
| `vendor-*.js` / `vendor.*.js` | `vendor` |
| `index.*.js` / `index-*.js` / `main.*.js` | `main` |
| everything else (e.g., `Queue-abc.js`) | `route` |

The classifier mirrors the manualChunks shape in
`web/vite.config.ts`:

```ts
manualChunks(id: string) {
  if (!id.includes('node_modules')) return;
  if (id.includes('xterm'))        return 'vendor-xterm';
  if (id.includes('lucide-react')) return 'vendor-lucide';
  if (id.includes('react-dom'))    return 'vendor-react-dom';
  if (id.match(/[\\/]react[\\/]/)) return 'vendor-react';
  return 'vendor';
}
```

Every node_module chunk lands as `vendor-*` (counted
as `vendor`); the entry point is `index-*` (counted as
`main`); React's `lazy(() => import(...))` produces
per-page chunks named after the imported file
(counted as `route`).

## Public API

```js
// Pure helpers (tested directly):
export function classifyChunk(filename): 'main' | 'vendor' | 'route';
export function checkBundleBudgets(records, budgets?): Report;
export function formatReport(report): string;

// I/O helper:
export function collectChunkRecords(distDir): Record[];

interface Record {
  name: string;
  rawSize: number;
  gzipSize: number;
}

interface Report {
  ok: boolean;
  perClass: { main: number; vendor: number; route: number };
  byClass:  { main: Record[]; vendor: Record[]; route: Record[] };
  breaches: Array<{ class, actual, budget, delta }>;
  budgets;
}
```

## CI integration

```yaml
# .github/workflows/web.yml (sketch)
- name: Build web
  run: npm --prefix web run build
- name: Check bundle budget
  run: npm --prefix web run bundle-budget
```

The npm script + non-zero exit code is the CI
contract. The project does not yet have a
`.github/workflows/` directory; wiring is left for
the patch that adds CI.

## Tests

`bundle-budget.test.js`:

- `classifyChunk` -- 3 cases (vendor, main, route).
- `DEFAULT_BUNDLE_BUDGETS` -- 2 cases (byte values
  match the spec, object is frozen).
- `checkBundleBudgets` -- 10 cases (empty input,
  within budget, main breach, vendor breach, route
  breach, multi-breach, per-class sum, boundary ==
  budget not flagged, one byte over flagged, byClass
  grouping, caller-supplied budgets).
- `formatReport` -- 3 cases (per-class status lines,
  omits Breaches on success, includes per-chunk
  gz/raw sizes).

18/18 pass against vitest 4.1.5 (node environment).

The test file uses `.test.js` (plain JS) rather than
`.test.mjs` because the project's vitest include
pattern is `src/**/*.{test,spec}.{ts,tsx,js,jsx}`.
`web/`'s `"type": "module"` makes the `.js` file ESM
automatically, so the `.mjs` import resolves
cleanly.

## Out of scope

- **Wiring into a CI workflow.** The project does
  not have a `.github/workflows/` directory yet.
  The npm script is the CI contract; whoever adds
  the workflow wires the step.
- **Per-release history tracking.** A future patch
  could persist `perClass` totals to a JSON file so
  trends across releases are visible. Out of scope
  for the initial gate.
- **Sourcemap-based attribution.** The current
  classifier groups by filename. Attributing bytes
  back to individual source files would need the
  sourcemap; the filename heuristic catches the
  dispatch's contract (main vs vendor) cleanly.
