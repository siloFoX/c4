# 11.184 - UI: RouteProgressBar (v1.11.202)

Slim top-of-page progress bar for route transitions, modeled after
NProgress but written from scratch with no new dependencies.

Files:
- `web/src/components/RouteProgressBar.tsx` (component)
- `web/src/components/RouteProgressBar.test.tsx` (vitest + RTL, 8 cases)
- `web/src/App.tsx` (mounted below `AutonomousStatusBanner`,
  `routeKey={topView}`)

## API

```tsx
import RouteProgressBar, {
  type RouteProgressHandle,
} from './components/RouteProgressBar';

// Auto mode: start->done cycle every time `routeKey` changes
<RouteProgressBar routeKey={topView} color="primary" />

// Imperative mode: manage the cycle yourself
const ref = useRef<RouteProgressHandle>(null);
useEffect(() => {
  ref.current?.start();
  return () => ref.current?.done();
}, [route]);
<RouteProgressBar ref={ref} />
```

Props:

| prop        | type                                              | default     | notes                                                |
|-------------|---------------------------------------------------|-------------|------------------------------------------------------|
| `routeKey`  | `string \| undefined`                             | `undefined` | when it changes, auto-runs start -> settle -> done    |
| `color`     | `'primary' \| 'success' \| 'info' \| 'danger'`    | `'primary'` | maps to `bg-primary` / `bg-success` / `bg-info` / `bg-destructive` |
| `className` | `string \| undefined`                             | `undefined` | merged onto the root via `cn()`                      |

## Behavior

- Renders a `fixed left-0 right-0 top-0` 2px bar (`h-0.5`) with
  `pointer-events-none` and `aria-hidden='true'` (purely decorative).
- The fill uses `transform: scaleX(progress)` against an
  `origin-left` element so animation is GPU-friendly.
- On `start()` the bar jumps to 30% and then ticks every 200ms with
  a diminishing-returns step (`min(0.005, remaining * 0.1)`) until
  it reaches the 90% trickle ceiling. It never hits 100% on its own.
- On `done()` it snaps to 100%, holds for the fade duration, then
  hides via `opacity-0` and resets progress to 0.
- `routeKey` mode does not trigger on the first render -- only when
  the key changes -- so initial mounts do not flash a bar.
- `prefers-reduced-motion: reduce` (resolved via the existing
  `useReducedMotion` hook) skips the trickle: `start()` jumps
  straight to 100% and `done()` uses a much shorter hold before
  fading.

## Integration

`App.tsx` mounts a single instance below `AutonomousStatusBanner`:

```tsx
<RouteProgressBar routeKey={topView} />
```

Because the bar is `fixed top-0 z-50`, it overlays the gradient
header without shifting layout. The `Suspense` fallback for top-level
view chunks remains the `LoadingSkeleton` -- the bar complements it,
it does not replace it.

## Testing

`web/src/components/RouteProgressBar.test.tsx` covers (8 cases):

1. renders hidden by default with `aria-hidden='true'` and zero scale
2. imperative `start()` flips visibility and animates above 30% but
   stays under the 90% ceiling
3. imperative `done()` reaches 100% then hides after the fade
4. `routeKey` change auto-runs the cycle (does not fire on first render)
5. `prefers-reduced-motion` skips the trickle and still hides cleanly
6. `color` prop applies the right color class
7. `className` merges onto the root
8. `forwardRef` exposes a handle with `start()` and `done()`
