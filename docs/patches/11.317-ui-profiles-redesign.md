# 11.317 -- Profiles page redesign (v1.11.335)

Refactors `web/src/pages/Profiles.tsx` to use the Tabs
primitive for source grouping, swap the inline
not-implemented Remove handler for a ConfirmDialog
flow, and add a per-profile role RadioGroup picker
inside the expanded panel. Component-scope only, no
daemon-side change and no `c4` CLI surface change.

## Why

The Profiles page already received most of the
"redesign" treatment in earlier patches:

- v1.11.281: FieldGroup wrapper for the allow/deny
  pattern grid.
- v1.11.285: CopyButton for the profile name.
- v1.11.287: NumberInput-driven budget cap row.
- v1.11.288: FileDrop for the (deferred) profile
  import.
- v1.11.291: TagInput for the labels row.
- v1.11.303: FormField wrappers around the budget +
  labels inputs.

The remaining dispatch items were:

1. **Tabs for grouping.** The flat list mixes
   built-in presets (web, api, plan, etc) with
   operator-defined customs. An operator browsing
   for "the profile I created last week" has to
   eyeball the source badge column. Source-grouping
   Tabs make the split a single click.

2. **RadioGroup for role/permission picker.** No
   role picker existed on the page; an operator
   who wanted to capture "this profile is for
   workers" had to context-switch to the RBAC page.

3. **ConfirmDialog for delete.** The Remove button
   fired the `notImplemented` toast directly. Even
   with the daemon-side write unimplemented, the
   destructive confirmation surface should still
   ship -- a missed click on Remove with the
   daemon endpoint in place would silently fire
   the destructive action.

Patch 11.317 fixes all three.

## What changed

### `web/src/pages/Profiles.tsx`

- **Source-grouping Tabs.** Wraps the row list in
  `<Tabs value={activeTab} onChange={setActiveTab}>`
  with three items: `all` (default-selected),
  `builtin`, `custom`. The active tab filters the
  visible rows in addition to the existing fuzzy
  search filter, so the two compose. Each tab label
  carries a count chip (`All (3)` / `Built-in (1)` /
  `Custom (2)`) so the operator sees the bucket
  sizes at a glance.
- **ConfirmDialog for Remove.** Clicking a row's
  Remove button now opens
  `<ConfirmDialog destructive initialFocus="cancel">`
  with a description that names the profile. Cancel
  closes the dialog without firing the action;
  Confirm continues to fire the existing
  `notImplemented` toast until the daemon's profile
  remove endpoint lands. The `initialFocus="cancel"`
  attribute means a quick Enter on the
  newly-opened modal does not accidentally trigger
  the destructive action.
- **RadioGroup role picker.** Adds a new row inside
  `ProfileBudgetRow` (which now also covers the
  role picker -- a small expansion of the
  helper's scope). The RadioGroup is horizontal,
  three options (admin / manager / worker), with
  `worker` as the default checked value. The
  picker is operator-local until the daemon's
  profile schema accepts a role field.

### Data attributes

- `data-section="profiles-list"` on the row list.
- `data-testid="profiles-tabs"` on the Tabs strip.
- `data-testid="profiles-remove-<name>"` on each
  row's Remove button.
- `data-testid="profiles-role-<name>"` on each
  expanded panel's RadioGroup.

### Imports added

```ts
import { ConfirmDialog } from '../components/ConfirmDialog';
import {
  // ... existing imports ...
  RadioGroup,
  Tabs,
  TabsPanel,
  type TabsItem,
} from '../components/ui';
```

## Notes

- The dispatch listed FormField wrappers and
  TagInput as required items. Both were already
  delivered in earlier patches (FormField in
  v1.11.303, TagInput in v1.11.291) and are
  preserved here -- the budget + labels rows
  inside `ProfileBudgetRow` already use
  FormField + TagInput respectively.
- The Tabs labels include count chips inline
  (`Custom (2)`) rather than as separate Badge
  components. The chip is part of the label
  string so the keyboard navigation announces
  "Custom 2" as a single token. A future patch
  could swap the inline number for a BadgeCounter
  if a distinct visual treatment becomes
  necessary.
- The ConfirmDialog dialog uses the
  `destructive=true` flag so the Confirm button
  renders in the destructive palette. The
  `initialFocus="cancel"` setting matches the
  pattern adopted by every other destructive
  ConfirmDialog in the codebase (see
  Cleanup.tsx, WorkerActions.tsx).
- The role picker's "worker" default reflects the
  most common adoption case (a profile applied to
  a `c4 task <worker> --profile <name>` flow).
  Admin / manager values capture the rarer cases
  (a global allow set for the manager's bash
  invocations, a top-level admin override). The
  picker stays operator-local; when the daemon
  schema accepts a role field, the value will
  flow into the write.
- The new `data-section="profiles-list"` selector
  scopes future e2e tests that want to count or
  iterate rows without picking up the per-profile
  Panel internals.
