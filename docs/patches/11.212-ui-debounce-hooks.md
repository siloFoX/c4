# 11.212 - UI debounce + throttle hooks (v1.11.230)

Component-scope-only extraction. Two small timing
primitives land in `web/src/hooks/`, and three existing
inline `setTimeout` / `clearTimeout` debounce sites adopt
them. Behaviour is preserved at every adoption site; the
swap is purely about removing repeated timer plumbing.

## New surface

### `web/src/hooks/use-debounce.ts`

- `useDebounce<T>(value: T, ms: number): T`
  - Returns the latest `value`, but only after `ms` of
    no changes. Rapid changes restart the timer so the
    most recent value wins.
  - Pending timer is cleared on unmount.
  - SSR-safe (timers scheduled inside `useEffect`).
- `useDebouncedCallback<F>(fn: F, ms: number): (...args) => void`
  - Returns a stable callback (identity is invariant
    while `ms` is unchanged) that calls `fn` once, `ms`
    after the last invocation, with the last set of
    args. Trailing-edge.
  - Always uses the latest `fn` at fire time via a ref.
  - Pending fire is cancelled on unmount.

### `web/src/hooks/use-throttle.ts`

- `useThrottle<T>(value: T, ms: number): T`
  - Leading-edge throttled value. The first change after
    mount and the first change after a full `ms` window
    has elapsed propagate immediately. Intermediate
    changes inside the window are dropped (no
    trailing-edge emission).
  - SSR-safe.
- `useThrottledCallback<F>(fn: F, ms: number): (...args) => void`
  - Leading-edge throttled callback. First call fires
    immediately. Subsequent calls inside `ms` are
    dropped. Calls after unmount are ignored.
  - Stable callback identity while `ms` is unchanged.

## Adoption sites (3)

All three swaps are byte-for-byte behaviour-preserving
relative to the previous inline-timeout implementation:

1. `web/src/components/ui/search-bar.tsx`
   The `onDebouncedChange` emit replaces an inline
   `setTimeout` / `clearTimeout` with
   `useDebouncedCallback`. The pre-existing
   `userInteractedRef` gate still suppresses the very
   first emit when the input has not been touched.

2. `web/src/lib/use-meeting-classify-preview.ts`
   The 250 ms classify GET preview now reads from
   `useDebounce(newTask, 250)` and fires the fetch
   from the dependency-driven effect instead of from an
   in-effect timeout.

3. `web/src/lib/use-meeting-preview-plan.ts`
   The 400 ms `/api/meetings/plan` POST preview now
   reads from `useDebounce(newTask, 400)` and
   `useDebounce(newTrack, 400)`; the in-effect timeout
   is gone.

## Tests

`web/src/hooks/use-debounce.test.tsx` and
`web/src/hooks/use-throttle.test.tsx` (vitest + RTL)
cover at least 5 cases each, including:

- Initial-value pass-through
- Trailing-edge fire after `ms`
- Rapid changes collapsing to the latest (debounce)
- Leading-edge fire + suppression window (throttle)
- Unmount cancellation
- `fn` ref freshness across re-renders

The existing test suites for `SearchBar`,
`useMeetingClassifyPreview`, and `useMeetingPreviewPlan`
continue to pass without modification.

## Out of scope

- No daemon code touched.
- No public CLI surface added.
- No package-lock or root `package.json` changes.
- `docs/autonomous-queue-v10.md` is left for the manager.
