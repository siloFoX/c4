# 11.308 -- Button primitive enhancements (v1.11.326)

Enhances `web/src/components/ui/button.tsx` with a
loading state, icon-only accessibility enforcement, and
tone refinements for the destructive and ghost variants.
Component-scope only, no daemon-side change and no `c4`
CLI surface change.

## Why

Three pain points across the Button adopters today:

1. **No canonical loading state.** Every call site that
   needs a button that flips to "Saving..." with a
   spinner re-implements the boilerplate: track a
   `pending` flag, swap the children, set `disabled`,
   wire an `aria-busy` attribute, find a Spinner import.
   Some sites only do half of those steps (e.g. swap
   children but forget `aria-busy`); others over-rotate
   (replace the children entirely, causing the button
   width to jump and the layout to shift). The
   inconsistency surfaces as accessibility regressions
   on every QA pass.

2. **Icon-only buttons skip the SR label.** `size="icon"`
   trims the visible text to fit a 9x9 square. Without
   an `aria-label`, the button's accessible name is
   "" and axe-core flags it. The current adopters all
   know to pass `aria-label`, but the primitive does
   nothing to enforce it.

3. **Tone drift on dark mode.** The destructive variant
   inherited the default primary focus ring, which is
   the wrong colour against a red surface. The ghost
   variant's `hover:bg-accent` filled too aggressively
   and visually merged the button with neighbouring
   chips when the operator hovered.

Patch 11.308 ships fixes for all three in the primitive
so adopters get them for free.

## What changed

### `web/src/components/ui/button.tsx`

New props:

- `loading?: boolean` (default `false`). When true:
  - Renders an inline `<Spinner size="sm">` with
    `data-section="button-spinner"` and
    `aria-hidden="true"` so screen readers do not
    re-announce the icon.
  - Auto-disables the button (`disabled || loading`)
    so repeat clicks are dropped at the disabled
    boundary; callers do not have to debounce
    separately.
  - Sets `aria-busy="true"` on the `<button>`.
  - Renders a `<VisuallyHidden>{loadingLabel}</VisuallyHidden>`
    inside the button so screen readers narrate the
    pending state.
  - Keeps the children in the DOM wrapped in a
    `<span data-section="button-children" aria-hidden="true">`
    with `opacity-0` while loading, so the button's
    intrinsic width does not jump when the state
    flips.
- `loadingLabel?: string` (default `'Loading'`).
  Override the SR-only text emitted while loading is
  true. Call sites with a more specific message
  ('Saving', 'Deleting', 'Sending request') pass
  their own string.

Dev-only enforcement:

- When `size === 'icon'` and no `aria-label` is
  provided, the component logs a one-time
  `console.warn` in non-production builds. Production
  builds skip the warning entirely. The warning text
  points at the missing `aria-label` so the operator
  can grep for it on the next QA pass.

Tone refinements:

- Destructive variant: added
  `focus-visible:ring-destructive`. The focus ring now
  uses the destructive tone instead of the default
  primary, keeping the variant legible against the
  ARPS red surface.
- Ghost variant: changed `hover:bg-accent` to
  `hover:bg-accent/60` and added explicit
  `bg-transparent text-foreground` so the ghost button
  no longer steals the surface colour from
  neighbouring chips on hover.

Data-attribute selectors added for e2e + theming:
`data-section="button"`, `data-variant`,
`data-size`, `data-loading="true|false"`.

### Adopted

- Primitive enhancement only this commit. The
  dispatch listed Workers row actions, Templates
  save, and Profiles delete confirmation as adoption
  sites. Threading `loading={...}` and
  `size="icon" aria-label="..."` at those call sites
  is a follow-up per-page patch so each migration
  lands with its own test refresh.

## Deferred (dispatch follow-ups)

A representative migration looks like:

```tsx
// Before -- inline pending state in Templates editor
const [saving, setSaving] = useState(false);
// ...
<Button onClick={onSave} disabled={saving}>
  {saving ? 'Saving...' : 'Save'}
</Button>;

// After -- using the loading prop
<Button loading={saving} loadingLabel="Saving" onClick={onSave}>
  Save
</Button>;
```

For icon-only buttons:

```tsx
// Before -- silent a11y violation
<Button size="icon">
  <TrashIcon />
</Button>;

// After -- explicit label, dev warning disappears
<Button size="icon" aria-label="Delete row">
  <TrashIcon />
</Button>;
```

## Notes

- The loading state intentionally keeps the children
  in the DOM (with `opacity-0`) instead of swapping
  them out, so the button's intrinsic width remains
  stable. Width-stability matters because a
  flickering layout while a network request resolves
  is one of the most-commented-on QA issues on the
  current ARPS pass.
- The dev-only `console.warn` for missing
  `aria-label` on icon-only buttons is gated on
  `process.env.NODE_ENV !== 'production'` so a
  production build does not pay the runtime cost.
  The warning is also skipped when the children are
  a plain string (the string itself becomes the
  accessible name and the icon-only-no-label
  assumption no longer applies).
- `aria-busy` is set to `loading || undefined` (not
  `'true' | 'false'`) so the attribute is dropped
  entirely when loading is false; React serialises
  the boolean attribute consistently with the WAI-
  ARIA spec.
- The `Spinner` import uses the v1.11.310 primitive
  rather than re-implementing a spinner inline. This
  keeps the canonical motion behaviour in lockstep
  with the rest of the design system.
