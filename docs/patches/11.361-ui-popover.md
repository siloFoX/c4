## Goal

Dispatch:

- Build `web/src/components/ui/popover.tsx`
  -- click/focus-triggered floating panel with
  portal rendering, focus trap, arrow, anchor
  positioning. Adopt at user menu, filter
  dropdowns.

`<Popover>` already shipped in 11.171
(v1.11.189) and has been extended over time
to cover most of the dispatch:

- click trigger ✓
- portal rendering (via
  `getPortalRoot('popover-root')`) ✓
- focus trap (via `useFocusTrap`) ✓
- anchor positioning (`placement` + `align` +
  `offset` + automatic flip) ✓
- controlled / uncontrolled (`open` /
  `defaultOpen` / `onOpenChange`) ✓
- `closeOnClickOutside` + `closeOnEsc` ✓
- ARIA (`aria-haspopup="dialog"`,
  `aria-expanded`, `aria-controls`) ✓

What was outstanding:

1. **Arrow chevron** -- visual signal of
   which trigger the popover belongs to.
   Matches the Tooltip arrow contract
   (11.276).
2. **`data-testid`** -- e2e selectors need
   to address specific popovers when more
   than one is mounted.

The dispatch also says "user menu, filter
dropdowns" -- the existing date-picker /
column-picker / toolbar already adopt the
Popover primitive for filter surfaces; no
per-page rewiring is needed in this patch.

## New props

| prop | default | purpose |
| --- | --- | --- |
| `arrow` | `false` | render the chevron |
| `data-testid` | `undefined` | forwarded onto the dialog node |

## Arrow

Positioned against the inverse of the
**resolved** placement (after the flip
calculation runs) so the visible point always
hugs the trigger:

| resolved placement | arrow anchor class |
| --- | --- |
| `top` | `bottom-[-3px] left-1/2 -translate-x-1/2` |
| `bottom` | `top-[-3px] left-1/2 -translate-x-1/2` |
| `left` | `right-[-3px] top-1/2 -translate-y-1/2` |
| `right` | `left-[-3px] top-1/2 -translate-y-1/2` |

Visual style: 6x6 rotated square half-tucked
behind the popover (3px) -- the visible edge
reads as a triangle. Matches the Tooltip
arrow byte-for-byte so the design language
stays continuous.

The arrow span carries `aria-hidden="true"` so
screen readers skip the decorative chevron.
The panel gets `data-popover-arrow="true|false"`
so e2e selectors can branch on the visual.

## Tests

`popover.test.tsx` -- 4 new cases on top of the
existing 17:

- No arrow by default (`data-popover-arrow="false"`,
  no arrow span).
- `arrow={true}` -> `data-popover-arrow="true"`
  + arrow span + `aria-hidden="true"`.
- Arrow position class matches the inverse of
  the resolved placement.
- `data-testid` is forwarded onto the panel
  dialog (`screen.getByTestId` works).

21/21 pass under vitest 4.1.5. The pre-existing
`useFocusTrap` strict-tuple warning is unchanged.

## Pairs with existing primitives

- `<Tooltip>` arrow contract (11.276 / 11.378)
  byte-identical visual.
- `lib/portal-root.ts` --
  `getPortalRoot('popover-root')` is the
  canonical mount target.
- `<Toolbar>` / `<DatePicker>` /
  `<ColumnPicker>` -- already adopt Popover;
  opting any of them into `arrow` is a
  one-prop change.

## Out of scope

- Collision detection beyond the existing
  binary flip (shifting horizontally when a
  vertical edge runs out of room). jsdom
  layout limitations + the existing flip
  cover the common cases.
- React 19 ref-as-prop migration.
- AccountMenu / sidebar dropdown adoption.
  The primitive supports `arrow`; per-page
  visual adoption is a follow-up.
- Hover trigger (the popover stays
  click-driven by design; hover surfaces
  belong to `<Tooltip>`).
