# 11.298 -- VisuallyHidden polymorphic + focusable (v1.11.316)

Component-scope-only enhancement of the existing `VisuallyHidden`
primitive. No daemon-side change and no `c4` CLI surface change.
The default `<span>` + `.sr-only` class output is preserved
byte-for-byte modulo the new `data-section` + `data-focusable`
attribute set.

## Why

The dispatch called for a "short export but used widely"
primitive. The existing VisuallyHidden ships as that small
`<span class="sr-only">` wrapper -- 10 adopters in the codebase
(Spinner, NumberInput, Queue refresh button, TopTabs hidden
labels, etc.). Two missing affordances limit broader adoption:

1. The primitive is locked to `<span>`. Some surfaces (a
   skip-link inside the AppHeader, a wrapper around multiple
   SR-only siblings) want a block-level element so phrase-level
   children fit cleanly. Callers were inlining
   `<div className="sr-only">` instead of reaching for the
   primitive, fragmenting the SR-only vocabulary across pages.
2. The canonical skip-link pattern ("Skip to main content")
   needs an element that is SR-only at rest BUT becomes
   visible when a keyboard user tabs to it. Tailwind ships the
   `focus:not-sr-only` modifier for this, but it had to be
   spelled per call site.

Patch 11.298 adds both affordances inside the primitive so
every SR-only surface reaches for the same vocabulary.

## What changed

### `web/src/components/ui/visually-hidden.tsx`

- New exported type `VisuallyHiddenAs = 'span' | 'div'`.
- New `as?: VisuallyHiddenAs` prop (default `'span'`).
  Switches the rendered tag between inline phrasing (span)
  and block-level (div). The `sr-only` class is applied on
  either branch.
- New `focusable?: boolean` prop (default `false`). When
  true the component swaps in:
  - `focus:not-sr-only` -- inverts the SR-only rule when
    the element receives keyboard focus.
  - Visible-on-focus chrome:
    `focus:absolute focus:left-2 focus:top-2 focus:z-50
    focus:rounded focus:bg-background focus:px-2 focus:py-1
    focus:text-foreground focus:ring-2 focus:ring-primary
    focus:ring-offset-2 focus:ring-offset-background`.
  At rest the element stays SR-only (matches the prior
  behaviour byte-for-byte); it becomes visible only when a
  keyboard user tabs to it.
- Data-attribute selectors:
  - `data-section="visually-hidden"`.
  - `data-focusable="true|false"`.
- The ref forwarding is per-branch
  (`React.Ref<HTMLDivElement>` vs `React.Ref<HTMLSpanElement>`)
  so callers reaching for either shape get the correct
  HTMLElement type.

### Adopted

- Primitive enhancement only this commit. The existing
  VisuallyHidden call sites inherit the new data-section
  attribute without any caller change.

## Deferred (dispatch follow-ups)

- Icon-only button SR-label audit: many IconButton call
  sites use `aria-label` rather than VisuallyHidden text.
  Both shapes are accessibility-equivalent; consolidation
  is a follow-up.
- Skip-link adoption: AppHeader currently has no
  skip-to-main-content link. `<VisuallyHidden focusable
  as="a" href="#main">` is a one-line add but needs the
  corresponding `id="main"` anchor on the routed content
  surface.

## Tests

- `web/src/components/ui/visually-hidden.test.tsx` -- 7
  new vitest cases:
  - Default `as` renders `<span>`.
  - `as="div"` renders `<div>`.
  - `as="div"` still carries the `sr-only` class.
  - `focusable=false` does NOT add `focus:not-sr-only`.
  - `focusable=true` adds `focus:not-sr-only` + visible-on-
    focus chrome (`focus:absolute`, `focus:bg-background`,
    `focus:ring-2`).
  - `data-section="visually-hidden"` +
    `data-focusable="true|false"` selectors.
  - Extra HTML attributes (id, role) forwarded onto the
    element.
  All 15 (8 original + 7 new) green.

## Versions

## Notes

- The focusable chrome uses theme tokens
  (`focus:bg-background focus:ring-primary`) so a future
  palette migration follows automatically.
- The skip-link positioning (`focus:left-2 focus:top-2
  focus:z-50`) targets the top-left corner of the viewport
  on focus -- the standard browser convention.
- The primitive's `forwardRef` contract is preserved by
  per-branch ref casts; callers passing a typed ref still
  receive the correct HTMLElement on either render path.
