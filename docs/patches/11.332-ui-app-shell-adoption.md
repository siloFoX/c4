## Goal

The dispatch lists:

- Migrate `web/src/App.tsx + AppHeader + Sidebar` to
  use the v1.11.343 `AppShell` primitive end-to-end.
- Preserve all existing nav model + features.
- Add visual snapshot test for collapsed/expanded
  sidebar states.

This patch migrates App.tsx's OUTER shell to use
`AppShell`, preserving every existing affordance
(gradient header backdrop, MetricsBar / banner /
RouteProgressBar strips, per-view scroll discipline,
the v1.11.295 CommandPalette portal). The body's
existing Sidebar (worker list + hierarchy tree +
account menu) continues to render INSIDE the
default-view branch -- it does not flow through
`AppShell.nav` because its dynamic content does not
fit the static `AppShellNavItem[]` shape. Two visual
snapshot tests lock down the Sidebar's collapsed +
expanded markup.

## AppShell escape hatch

The v1.11.343 `AppShell` primitive wraps its `<main>`
children in a vertical `ScrollArea`. App.tsx's body
needs the opposite -- each topView branch owns its
own `overflow-hidden + min-h-0` chain so terminal /
chat / multi-pane editor surfaces work. To support
both contracts, this patch adds:

```ts
interface AppShellProps {
  // ... existing ...
  mainScroll?: 'auto' | 'inherit';
}
```

- `'auto'` (default) keeps the existing contract.
  Long content scrolls inside the `<main>`'s
  `ScrollArea`.
- `'inherit'` skips the wrapper. Children render
  directly inside `<main>` and manage their own
  scroll.

The rendered `<main>` carries
`data-main-scroll="auto"` or `data-main-scroll="inherit"`
so e2e + styling can branch on the mode.

App.tsx uses `mainScroll="inherit"` so the
per-topView scroll discipline survives byte-for-byte.

## App.tsx migration

Before:

```tsx
<AnnounceRegion>
  <div className="flex h-screen flex-col bg-background text-foreground">
    <HelpUIRoot />
    <CommandPalette />
    <div className="bg-gradient-to-b from-[...] to-[...]">
      <AppHeader ... />
    </div>
    <MetricsBar />
    <AutonomousStatusBanner />
    <RouteProgressBar />
    <GridDebugOverlay />
    <Suspense>
      <PageTransition>
        {topView switcher}
      </PageTransition>
    </Suspense>
  </div>
</AnnounceRegion>
```

After:

```tsx
<AnnounceRegion>
  <HelpUIRoot />
  <CommandPalette />
  <AppShell
    mainScroll="inherit"
    headerClassName="p-0 border-0 bg-gradient-to-b from-[...] to-[...]"
    header={
      <div data-testid="app-top-bar-gradient" className="flex w-full">
        <AppHeader ... />
      </div>
    }
  >
    <MetricsBar />
    <AutonomousStatusBanner />
    <RouteProgressBar />
    <GridDebugOverlay />
    <Suspense>
      <PageTransition>{topView switcher}</PageTransition>
    </Suspense>
  </AppShell>
</AnnounceRegion>
```

### Why HelpUIRoot + CommandPalette sit outside AppShell

Both are portal-based modal mounts. They render their
content inside a `createPortal` target on
`document.body`, not inline in the flex tree. Keeping
them outside `<AppShell>` keeps the shell's `<main>`
landmark free of modal content and matches the
v1.11.295 behaviour where the palette mounts at the
root.

### Why the gradient survives via headerClassName

`AppShell`'s default header carries
`border-b border-border bg-card/40 px-3 py-2`. The
v1.11.134 gradient has no padding and a different
background. The migration uses
`headerClassName="p-0 border-0 bg-gradient-to-b from-[...]
to-[...]"` to override the defaults and keep the
gradient pixel-identical.

The inner gradient `<div data-testid="app-top-bar-gradient">`
remains so existing tests / e2e selectors keep
working.

## Sidebar snapshots

`Sidebar.snapshot.test.tsx` is a new file with four
cases:

1. `matches the expanded-state snapshot` -- writes
   `Sidebar.snapshot.test.tsx.snap` with the rendered
   DOM tree when `collapsed={false}`.
2. `matches the collapsed-state snapshot` -- same
   with `collapsed={true}`.
3. `passes collapsed=true to AccountMenu when the
   sidebar is collapsed` -- assertion that the
   collapse prop wiring survives.
4. `passes collapsed=false to AccountMenu when the
   sidebar is expanded` -- inverse.

The snapshots target the rendered DOM (jsdom does NOT
compute layout). They catch class-string drift,
missing data attributes, and structural reordering.
Pixel-level visual regression stays out of scope for
vitest + jsdom; a Playwright follow-up can take
real-pixel snapshots against a live browser render.

The standard 11-mock pattern from
`Sidebar.test.tsx` is reused (WorkerList +
HierarchyTree + AccountMenu stubbed to marker
elements) so the snapshot is stable across the
inner-panel evolution. Only the Sidebar SHELL
markup (icon rail / handle / aside / footer)
participates in the snapshot.

## Not migrated

- **The Sidebar's nav model.** Sidebar renders
  dynamic content (WorkerList polling + HierarchyTree
  + AccountMenu) that does not fit the static
  `AppShellNavItem[]` shape (`id`, `label`, `icon`,
  `href`). The migration leaves Sidebar as a regular
  component rendering inside the
  `topView === 'workers'` default branch. A future
  patch could either (a) extend `AppShell` with a
  `sidebar?: ReactNode` slot that bypasses the
  `nav[]` shape, or (b) refactor Sidebar's contents
  to drive `AppShell.nav` directly.
- **Status strips (MetricsBar / banner /
  RouteProgressBar / GridDebugOverlay) as part of
  `<header role="banner">`.** Semantically these
  belong with the main content, not the navigation
  region. They sit at the top of `<main>` in the
  migrated layout -- visually identical to before
  but landmark-correct.
