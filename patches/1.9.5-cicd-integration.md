# Patch 1.9.5 — CI/CD pipeline integration (10.4)

## Summary

Bind c4 to external CI/CD systems. A new `CicdManager` accepts GitHub
webhooks (HMAC-verified), fans out pipeline actions to c4 workers or
GitHub Actions `workflow_dispatch`, and exposes a thin REST + CLI
surface for creating/listing/deleting pipelines and firing manual
runs.

## Motivation

The 10.x observability + collaboration track already covers `who can
do what` (10.1 RBAC), `what happened` (10.2 audit log), `what it
costs` (10.5 cost report), `what's planned` (10.8 project board), and
a unified per-project dashboard (10.3). The missing link is
integration with the outside world: when a PR opens on GitHub, a
reviewer worker should spin up automatically; when `main` merges, a
deploy workflow should fire; when a tag lands, the release workflow
should fire. That wire is 10.4.

## What changed

### New module: `src/cicd.js`

Pure storage + dispatch. Worker creation flows through an injected
`dispatchWorker` callback so tests run without a daemon; GitHub API
calls flow through an injected `fetch` so tests assert the request
shape without network.

```js
const cicd = new CicdManager({
  storePath: '~/.c4/cicd.json',
  webhookSecret: 'whsec_...',
  repos: [{ name: 'org/repo', token: 'ghp_...', defaultWorkflow: 'deploy.yml' }],
  dispatchWorker: (spec) => manager.create(...),
  fetch: globalThis.fetch,
});

cicd.registerPipeline({
  repo: 'org/repo',
  workflow: 'deploy.yml',
  triggers: ['pr.opened', 'merge.main', 'tag.created'],
  actions: [
    { type: 'worker.task', template: 'Review PR', profile: 'web' },
    { type: 'workflow.trigger', workflow: 'deploy.yml', inputs: { env: 'prod' } },
  ],
});

// On a webhook:
cicd.handleWebhook('pr.opened', payload);
// -> { event: 'pr.opened', matched: 1, dispatched: [{pipelineId, actions:[...]}] }
```

### Webhook signature verification

`verifySignature(secret, body, header)` hashes the raw body with
HMAC-SHA256, prefixes `sha256=`, then compares timing-safely with the
`X-Hub-Signature-256` header. Pre-length check guards against
`crypto.timingSafeEqual` throwing on mismatched Buffer lengths.

### GitHub event translation

`parseGithubEvent(eventHeader, payload)` narrows:

| GitHub event             | payload condition                 | internal event |
|--------------------------|------------------------------------|----------------|
| `pull_request`           | `action: opened\|reopened`        | `pr.opened`    |
| `pull_request`           | `action: closed`, `merged: true`  | `pr.merged`    |
| `pull_request`           | `action: closed`, `merged: false` | `pr.closed`    |
| `push`                   | `ref: refs/heads/main\|master`    | `merge.main`   |
| `create`                 | `ref_type: tag`                   | `tag.created`  |
| anything else            | —                                  | `null`         |

### GitHub Actions dispatch

`triggerWorkflow(repo, workflow, inputs, {ref})` posts to
`https://api.github.com/repos/<repo>/actions/workflows/<workflow>/dispatches`
with headers:

```
Authorization: Bearer <token-from-config.cicd.repos[].token>
Accept: application/vnd.github+json
X-GitHub-Api-Version: 2022-11-28
Content-Type: application/json
```

Body: `{ref, inputs?}` via `buildGithubPayload`. The actual POST
flies through `fetchImpl` (daemon wraps global `fetch`; tests inject
a recording mock).

### Daemon endpoints

```
POST   /cicd/webhook              HMAC auth, JWT bypass
GET    /cicd/pipelines            RBAC cicd.read
POST   /cicd/pipelines            RBAC cicd.manage
GET    /cicd/pipelines/:id        RBAC cicd.read
DELETE /cicd/pipelines/:id        RBAC cicd.manage
POST   /cicd/trigger              RBAC cicd.manage (replay by id or one-off dispatch)
```

`/cicd/webhook` is the only endpoint in the daemon that bypasses the
JWT gate — the sender is GitHub, not a logged-in user, so the HMAC
check inside the handler is the sole auth. Bad signature -> 401,
missing `X-GitHub-Event` -> 400, unroutable event -> 400, no
configured secret -> 500.

A new `parseBodyRaw(req)` helper in `src/daemon.js` buffers the raw
request body alongside parsed JSON so HMAC hashes exactly what
GitHub hashed (canonicalising via `JSON.stringify` would diverge from
the sender's byte stream and break every verify).

`/config/reload` calls `cicdManager.applyConfig(cfg.cicd)` so a new
webhook secret or repo token takes effect without a daemon restart.

### RBAC additions

```
ACTIONS.CICD_READ   = 'cicd.read'
ACTIONS.CICD_MANAGE = 'cicd.manage'
```

`ALL_ACTIONS.length` is now 14 (up from 12). Default matrix:

- `admin`   -> `['*']` (auto-granted)
- `manager` -> gains both
- `viewer`  -> gains `cicd.read` only (read-only operators can inspect
  pipeline state without being able to mutate it)

### CLI

```
c4 cicd pipeline list
c4 cicd pipeline create \
    --repo org/repo \
    --workflow deploy.yml \
    --trigger pr.opened \
    --trigger merge.main \
    --action worker.task:Review\ PR \
    --action workflow.trigger:deploy.yml \
    --profile web \
    --name 'Main pipeline'
c4 cicd pipeline delete pipe_xxxxxxxx
c4 cicd trigger pipe_xxxxxxxx                # replay
c4 cicd trigger --repo org/r --workflow deploy.yml --ref main --input env=prod
```

`fs.mkdtempSync` paths):

  ACTION_TYPES, isNonEmptyString validator.
- **verifySignature**: valid / wrong secret / altered body / missing
  header / empty secret / malformed hex / Buffer body input.
- **parseGithubEvent**: all five internal events + non-main push +
  non-tag create + unknown header.
- **buildGithubPayload**: default ref / custom ref + inputs copy /
  drops invalid inputs.
- **sanitizers + normalizePipeline**: missing repo rejected, default
  fills, trigger dedup, action type filter, inputs copy.
- **CRUD**: register happy path / missing repo rejection / null input
  rejection / list sorted by createdAt / get missing returns null /
  delete idempotent / explicit id survives roundtrip.
- **storage roundtrip**: save-load across instances, external edit +
  reload, missing file -> empty, malformed JSON -> empty.
- **event routing**: pr.opened -> worker.task with branch from PR
  head; merge.main -> workflow.trigger with `ref=main` + inputs;
  tag.created -> workflow.trigger with tag ref; unknown event ->
  error; missing token -> skipped without throwing; no subscribers ->
  matched=0.
- **triggerWorkflow**: URL shape / Authorization+Accept+X-GitHub-Api-Version
  headers / fetchImpl body is JSON-stringified / throws without token /
  throws on missing repo|workflow.
- **runCheck**: task spec / dispatcher callback / missing input
  rejection.
- **applyConfig**: rebuilds secret+repos while keeping pipelines /
  clears repos / rejects invalid provider.
- **shared singleton**: stability + resetShared.

`tests/rbac.test.js` updates: `ALL_ACTIONS.length` assertion bumped
from 12 to 14; new `CICD_READ` + `CICD_MANAGE` enum checks.

Full suite 86 / 86 pass (up from 85).

## Limitations / follow-ups

- GitLab CI and Jenkins live as provider labels only. The dispatch
  logic covers GitHub Actions; GitLab pipeline trigger API + Jenkins
  build triggers are additive follow-ups.
- Pipeline definitions are stored as canonical JSON. YAML export /
  import (so operators can commit pipeline definitions to a repo) is
  on the 10.x UI track.
- `workflow.trigger` is fire-and-forget: errors are captured into the
  executed-actions record but there is no retry queue or dead-letter
  path. A 10.x reliability follow-up can add exponential backoff and
  a persistent failure queue.
- Web UI CI/CD panel slots on top of the four REST endpoints and
  ships under the 10.x UI track.
