# 1.9.2 - Project Management + TODO.md Sync (TODO 10.8)

## Summary

Per-project task board with milestones, sprints, a backlog list, and
bidirectional `TODO.md` sync. Implemented as a pure storage layer in
`src/project-mgmt.js` (JSON file per project, no shell-outs, no daemon
coupling) plus nine REST endpoints and a `c4 project` CLI subcommand.
Progress is derived on read so there is no separate dashboard table to
keep in sync.

## Why

The daemon already orchestrates workers per branch, but teams still
tracked tasks by hand in `TODO.md` tables. The spec (TODO 10.8) asks for
a kanban/list board with milestones, sprints, and backlog so managers
can see progress across a project, plus bidirectional `TODO.md` sync so
the existing markdown workflow keeps working for developers who prefer
editing tables directly.

Keeping the board as a pure storage module (no node-pty, no network, no
process state) makes the data layer testable against tmpdir fixtures
and reusable: the same `ProjectBoard` drives the CLI, the daemon
endpoints, and any future Web UI kanban view.

## Storage

One JSON file per project at `~/.c4/projects/<projectId>.json`. The
path is configurable via `config.projects.path` for operators who want
the board on a different volume, and the shared `ProjectBoard` is
dropped on `config.reload` so a live path change takes effect without
a daemon restart.

Schema:

```json
{
  "id": "arps",
  "name": "ARPS",
  "description": "...",
  "createdAt": "2026-04-17T00:00:00.000Z",
  "milestones": [
    { "id": "m1", "name": "Launch", "dueDate": "2026-06-01", "status": "open" }
  ],
  "sprints": [
    { "id": "sp1", "name": "Week 1", "startDate": "2026-04-01", "endDate": "2026-04-07", "taskIds": ["task_abc"] }
  ],
  "tasks": [
    {
      "id": "task_abc",
      "title": "Add logging",
      "status": "in_progress",
      "assignee": "shin",
      "estimate": 3,
      "milestoneId": "m1",
      "sprintId": "sp1",
      "description": "...",
      "createdAt": "...",
      "updatedAt": "..."
    }
  ],
  "backlog": ["task_xyz"]
}
```

Internal task status is one of `'backlog' | 'todo' | 'in_progress' |
'done'`. The `backlog` list tracks membership where `task.status ===
'backlog'` and is re-asserted on every mutation. Sprint `taskIds` tracks
`task.sprintId` identically.

## TODO.md sync

`syncTodoMd(projectId, repoPath, opts?)` is bidirectional. It reads
`<repoPath>/TODO.md` (or `opts.todoPath`) as a markdown table of the
form:

```
| # | title | status | description |
|---|-------|--------|-------------|
| 1 | Add auth   | todo        | JWT     |
| 2 | Rate limit | in_progress | bucket  |
| 3 | Docs       | done        | README  |
```

Mapping between MD status and internal status:

| MD status     | Internal status |
|---------------|-----------------|
| `todo`        | `backlog`       |
| `in_progress` | `in_progress`   |
| `done`        | `done`          |

Task IDs derive from `sha1(projectId + '::' + title).slice(0, 10)`
prefixed with `task_`, so re-importing the same title preserves
assignee, estimate, milestone, and sprint metadata across sync cycles.
On export the project tasks are sorted by `createdAt` then written back
as a canonical MD table; a second sync over an unchanged file yields
byte-identical output.

## Methods

- `createProject({id, name, description})` - validates id against
  `/^[A-Za-z0-9._-]+$/`, rejects duplicates, returns the created
  project with empty `milestones`, `sprints`, `tasks`, `backlog` arrays.
- `getProject(projectId)`, `listProjects()` (sorted by `createdAt`).
- `addTask(projectId, {title, status?, assignee?, estimate?,
  milestoneId?, sprintId?, description?})` - dedupes by stable ID so
  repeated adds with the same title are idempotent; auto-maintains
  `backlog` list + sprint membership.
- `updateTask(projectId, taskId, patch)` - patches only provided
  fields, re-asserts backlog/sprint invariants, throws `Invalid
  status: X` on an out-of-set status.
- `moveTaskToSprint(projectId, taskId, sprintId)` - `null` clears
  membership; moving a `backlog` task into a sprint promotes it to
  `todo` so the sprint view picks it up.
- `createMilestone(projectId, {id?, name, dueDate?, status?})`,
  `createSprint(projectId, {id?, name, startDate?, endDate?, taskIds?})`
  - both reject duplicate ids.
- `listTasks(projectId, filter?)` - filter keys: `status` (single or
  array), `milestoneId`, `sprintId`, `assignee`. Combined filters AND
  together.
- `projectProgress(projectId)` returns `{ totalTasks, doneTasks,
  percent, byStatus: {backlog, todo, in_progress, done} }`. Percent is
  rounded to two decimals. Empty project reports 0%.
- `syncTodoMd(projectId, repoPath, opts?)` returns `{ imported,
  exported, path }`.

## CLI

```bash
# Create + inspect
c4 project create arps --name "ARPS" --desc "Retrieval-Augmented Pipeline Service"
c4 project list
c4 project show arps

# Task lifecycle
c4 project task add arps "Add logging"
c4 project task add arps "Add metrics" --status in_progress --assignee shin
c4 project task update arps task_abcd1234 --status done

# Milestone + sprint
c4 project milestone add arps "Launch" --due 2026-06-01
c4 project sprint add arps "Week 1" --start 2026-04-01 --end 2026-04-07

# Progress + sync
c4 project progress arps
c4 project sync arps --repo /home/shinc/arps
```

`c4 project progress <id>` prints a human-readable summary:

```
Progress for arps:
  total:  12 tasks
  done:   5
  percent: 41.67%
  backlog: 2  todo: 3  in_progress: 2  done: 5
```

All other subcommands echo the raw JSON response for scripting.

## Daemon endpoints

```
GET    /projects
POST   /projects                        { id, name, description }
GET    /projects/:id
POST   /projects/:id/tasks              { title, status?, assignee?, ... }
PATCH  /projects/:id/tasks/:taskId      { ...partial patch }
POST   /projects/:id/milestones         { id?, name, dueDate?, status? }
POST   /projects/:id/sprints            { id?, name, startDate?, endDate?, taskIds? }
GET    /projects/:id/progress
POST   /projects/:id/sync               { repoPath, todoPath? }
```

The shared `ProjectBoard` is cached per daemon process and dropped on
`config.reload` so `config.projects.path` changes take effect without
a restart.

## Tests

`tests/project-mgmt.test.js` ships 54 assertions across nine suites,
all writing to `fs.mkdtempSync` paths so the real `~/.c4/projects` is
never touched:

1. Helpers - `defaultProjectsDir` under home, `VALID_TASK_STATUS`
   membership, `MD_TO_TASK` / `TASK_TO_MD` mapping, `stableTaskId`
   determinism.
2. `createProject` - defaults, duplicate rejection, missing/invalid
   id rejection, JSON persistence, `getProject` roundtrip,
   `listProjects` creation-order sort.
3. `addTask` - defaults, append order, backlog invariant for
   backlog vs non-backlog status, dedup-by-title, missing-title
   rejection, unknown-project rejection.
4. `updateTask` - patch only provided fields, `updatedAt` bumps,
   `createdAt` immutable, status backlog-list invariant,
   invalid-status rejection, sprint.taskIds sync on sprintId patch,
   unknown-task rejection.
5. `moveTaskToSprint` - enter sprint, cross-sprint move, null clears
   membership, backlog -> todo promotion, unknown-sprint rejection.
6. `createMilestone` / `createSprint` - defaults, uniqueness.
7. `listTasks` filters - status (single + array), milestoneId,
   assignee, sprintId, combined AND, no-filter returns all.
8. `projectProgress` - empty 0%, all-done 100%, mixed 25%, 1/3
   33.33% (two-decimal rounding).
9. `parseTodoMd` / `serializeTodoMd` roundtrip + header/divider skip
   + bold/mixed-case status tolerance; `syncTodoMd` import + stable-id
   re-sync + status change + write-back + export-reimport-export
   stability.

Full suite: 83 / 83 pass (up from 82).

## Limitations

- Web UI kanban / list view is a 10.x follow-up; this patch ships the
  storage + REST + MD sync layer so the UI can consume it without
  schema churn.
- Workload balancing (spec item "team assignment + workload balancing")
  is not automated yet. `listTasks(filter={assignee})` gives managers
  the per-assignee breakdown they need to balance manually; automated
  balancing lands with the UI work.
- `syncTodoMd` targets a single-project `TODO.md` at
  `<repoPath>/TODO.md` or `opts.todoPath`. Multi-project repos will
  need per-project MD paths or a section-scoped parser, which we can
  add when the use case materialises.
