# 1.8.1 - Claude Code Plugin (TODO 9.5)

## Summary

TODO 9.5 (Claude Code Extension/Plugin): new top-level
`claude-code-plugin/` directory ships a Claude Code native plugin that
exposes the five core c4 worker-lifecycle operations as slash
commands. An operator can now drive the daemon from inside Claude Code
without dropping to the `c4` CLI, the MCP server, or the Web UI - all
four surfaces talk to the same daemon routes.

- `claude-code-plugin/plugin.json` - manifest (name, version, engines,
  commandsDir, five command descriptors with typed arguments)
- `claude-code-plugin/commands/*.md` - five markdown slash command
  files (`c4-new`, `c4-task`, `c4-list`, `c4-merge`, `c4-close`) with
  `allowed-tools: Bash` headers that invoke the sibling handler
- `claude-code-plugin/commands/*.js` - five pure-function handlers +
  `_client.js` (client factory + `MinimalC4Client` fallback) +
  `_argv.js` (tiny argv parser for the CLI entry)
- `claude-code-plugin/README.md` - install guide, env vars,
  prerequisites, limitations, manual smoke-test examples
- `tests/cc-plugin.test.js` - 25 node:test assertions validating the
  manifest + every handler's HTTP shape against a stub fetch

## Install

Claude Code loads plugins from `~/.claude/plugins/<name>/`. Three
install paths:

```bash
# Option 1: symlink (preferred for a source checkout)
mkdir -p ~/.claude/plugins
ln -s /abs/path/to/c4/claude-code-plugin ~/.claude/plugins/c4

# Option 2: copy
mkdir -p ~/.claude/plugins/c4
cp -R /abs/path/to/c4/claude-code-plugin/. ~/.claude/plugins/c4/

# Option 3: per-project .claude/plugins/
cp -R /abs/path/to/c4/claude-code-plugin your-project/.claude/plugins/c4
```

Restart Claude Code (or trigger its plugin reload). The five slash
commands autocomplete as `/c4-new`, `/c4-task`, `/c4-list`,
`/c4-merge`, `/c4-close`.

Older Claude Code releases that do not yet honor `plugin.json` can
still use the commands - symlink the five `commands/*.md` files
individually into `~/.claude/commands/`; the handlers resolve through
`$CLAUDE_PLUGIN_ROOT` when set, otherwise the README covers editing
the invocation path.

## Example usage

Inside Claude Code:

```
/c4-new worker1 --target local
/c4-task worker1 write failing tests for src/foo.js --auto-mode
/c4-list
/c4-merge worker1
/c4-close worker1
```

Each command returns a JSON envelope:

```json
{
  "ok": true,
  "command": "c4-new",
  "name": "worker1",
  "result": { "name": "worker1", "branch": "c4/worker1", "status": "idle" }
}
```

Error responses from the daemon (401 auth, 409 name conflict, 5xx) are
surfaced as thrown errors with `err.status` + `err.body`, so the slash
command reports the failure verbatim instead of silently retrying.

## Prerequisites

- **c4 daemon running**: `c4 daemon start` (default port 3456).
- **Node.js >= 18** on `PATH`.
- **Optional JWT**: if `auth.enabled: true` in `config.json` (8.14), the
  plugin reads the token from `C4_TOKEN` env or `~/.c4-token` file.
- **Optional c4-sdk**: when the `c4-sdk` package is resolvable from
  the plugin directory, handlers route through it; otherwise they use
  a built-in `MinimalC4Client` (zero dependencies, fetch-only).

## Environment variables

| Variable | Default | Purpose |
| --- | --- | --- |
| `C4_BASE` | `http://localhost:3456` | Daemon base URL. `C4_URL` is accepted as an alias. |
| `C4_TOKEN` | (reads `~/.c4-token`) | JWT for `auth.enabled: true`. |

## How it works under the hood

1. Claude Code invokes the slash command markdown (e.g.
   `commands/c4-new.md`). The markdown declares `allowed-tools: Bash`
   and runs `node "$CLAUDE_PLUGIN_ROOT/commands/c4-new.js" $ARGUMENTS`.
2. The JS handler parses argv via `_argv.parseArgv`.
3. The handler calls `_client.getClient({env, fetch})` which:
   - Prefers an injected `ClientClass` (used by tests).
   - Next tries `require('c4-sdk')`, then the sibling `../../sdk`.
   - Falls back to `MinimalC4Client` (built-in, zero deps).
4. Handler calls the matching client method (`createWorker`,
   `sendTask`, `listWorkers`, `merge`, `close`).
5. Client sends `POST`/`GET` with `Authorization: Bearer <jwt>` when a
   token is present.
6. Handler wraps the response in `{ok, command, name?, result}` and
   prints it to stdout.

## Method map

| Slash command | Daemon route | Handler |
| --- | --- | --- |
| `/c4-new <name>` | `POST /create` | `commands/c4-new.js` |
| `/c4-task <name> <task>` | `POST /task` | `commands/c4-task.js` |
| `/c4-list` | `GET /list` | `commands/c4-list.js` |
| `/c4-merge <name>` | `POST /merge` | `commands/c4-merge.js` |
| `/c4-close <name>` | `POST /close` | `commands/c4-close.js` |

## Tests

```
node --test tests/cc-plugin.test.js
```

`tests/cc-plugin.test.js` adds 25 assertions across five concerns:

1. **Manifest** - `plugin.json` parses, `name === 'c4'`, semver
   version, `engines.node >= 18`, exactly five commands, every command
   carries description + handler path + markdown path + arguments
   array with required-boolean, handler + markdown files exist, the
   required positional args match the spec.
2. **Shared client** - `MinimalC4Client` method + URL + body + Bearer
   header wiring, undefined-field stripping, non-2xx throws with
   `err.status` + `err.body`, constructor explicit `fetch:null`
   throws.
3. **getClient factory** - injected `ClientClass` wins, `useSdk:false`
   forces `MinimalC4Client`, `c4-sdk` is picked up when resolvable.
4. **parseArgv** - positional capture, `--flag=value`, `--flag value`,
   `boolFlags`, `--` terminator.
5. **Per-handler HTTP** - every handler's method + URL + body shape,
   synchronous missing-arg rejection without any network call, `_[0]`
   positional fallthrough, task-tail folding for `c4-task`, auth
   header propagation, 401 error surfacing.

Full c4 suite: 77 / 77 pass.

## Limitations

- **No streaming.** The plugin exposes only lifecycle operations.
  Callers who need live PTY output use `c4 watch <name>` or the SDK's
  `watch()` async iterator.
- **No interactive approval.** Critical-deny prompts and permission
  questions still require `c4 approve` or the Web UI; this plugin
  does not render TUI options.
- **Single daemon.** Always talks to the local daemon resolved via
  `C4_BASE`. Fleet routing (9.6) is CLI-only.
- **Older Claude Code releases.** If the loader does not yet honor
  `plugin.json`, symlink the five `commands/*.md` files individually
  into `~/.claude/commands/`. The README covers this fallback.

## Scope / follow-ups

- Add a `/c4-watch <name>` slash command that streams SSE into the
  Claude Code transcript (needs a buffered-flush strategy so the
  transcript is not flooded).
- Add `/c4-approve <name> [option]` once the plugin loader supports
  an interactive picker.
- Publish `c4-sdk` to npm so the plugin can declare it as a formal
  dependency in `plugin.json` (currently it resolves the sibling
  `sdk/` directory from a source checkout).
- A `/c4-fleet status` bridge that proxies the daemon's
  `/fleet/overview` response once 9.7 (dispatcher) lands.
