# Patch 1.9.7 - Schedule / calendar management (10.7)

## Summary

Add cron-driven scheduled tasks on top of project-mgmt (10.8),
org-mgmt (10.6), and rbac (10.1). A schedule stores a 5-field cron
expression, a task template string, optional project / assignee, and
an enable flag; the daemon polls the store once a minute and dispatches
a worker per due schedule. Operators can now say "every night at 2am,
spin up a worker on the backup template" without writing a cron job
outside c4.

## Motivation

10.8 lets a project carry tasks and milestones, but it has no notion
of *when*. 10.6 aggregates budgets per department, but still only
measures completed work. 10.7 fills the gap: a schedule is the input
side of the pipeline - it produces worker tasks on a fixed cadence -
while 10.8 / 10.6 remain the reporting side. The cron engine ships in
this module because pulling a cron library would add an external
dependency for five fields of logic and leave us with no hook for the
c4-specific worker dispatch.

## What changed

### New module: `src/schedule-mgmt.js`

Pure storage + decision layer. Writes to `~/.c4/schedules.json` by
default (`config.schedules.path` override). No shell-outs, no direct
daemon coupling; `scheduleTick(now, dispatch?)` accepts an injected
dispatcher callback so unit tests drive the tick in-process.

```js
const mgr = new ScheduleManager({ storePath: '~/.c4/schedules.json' });
mgr.createSchedule({
  id: 'daily-backup',
  name: 'Daily DGX backup',
  cronExpr: '0 2 * * *',          // 2am every day
  taskTemplate: 'run nightly backup on DGX',
  projectId: 'ops',
  timezone: 'UTC',
});

// Runs ~60s after startup and every minute after.
const summary = mgr.scheduleTick(new Date(), (schedule, ctx) => {
  // dispatch one worker per due schedule
});
// summary = { tickAt, dueIds, schedules }

// Operator override: run now without advancing cadence.
mgr.forceRun('daily-backup');

// ASCII Gantt over the next 4 weeks.
process.stdout.write(mgr.renderGanttText(4, new Date()));
```

### Cron parser

Minimal five-field parser. Supports `*`, numeric literals,
comma-lists, ranges `a-b`, and step `*/N` or `a-b/N`. Standard DOM/DOW
OR semantics: when both fields are restricted, a match on either one
fires. Explicitly out of scope: `L` / `W` / `#` operators, name tokens
(`MON`, `JAN`), and seconds. Errors surface at `createSchedule` time
with a descriptive message so bad input never lands in the store.

`computeNextRun(expr, tz, now)` walks minute by minute with a five-year
cap, which covers leap Feb 29 (`0 0 29 2 *` can land up to four years
out) and still terminates fast on genuinely unreachable expressions.

### RBAC

Two new canonical actions in `src/rbac.js`:

- `schedule.read` - list / show schedules + history
- `schedule.manage` - create / update / delete / force-run

Default matrix: `manager` gets both, `viewer` gets `schedule.read`,
`admin` already has `*`. `ALL_ACTIONS.length` is now 18 (up from 16).

### Daemon endpoints

In `src/daemon.js`:

- `GET    /schedules` (`schedule.read`, filters
  `?enabled=true|false&projectId=&assignee=`)
- `POST   /schedules` (`schedule.manage`)
- `GET    /schedules/:id` (`schedule.read`)
- `PUT    /schedules/:id` (`schedule.manage`)
- `DELETE /schedules/:id` (`schedule.manage`)
- `POST   /schedules/:id/run` (`schedule.manage`, force run)
- `GET    /schedules/:id/history` (`schedule.read`)

A minute-cadence `setInterval` starts in `server.listen` and stops on
SIGINT / SIGTERM. `config.schedules.enabled=false` opts out of the
automatic tick; `config.schedules.tickIntervalMs` overrides the
cadence. The tick calls `_scheduleDispatch` which creates a worker
named `sched-<id>-<tickMinute>` and sends the templated task via
`manager.sendTask`. Dispatcher exceptions are swallowed so a bad
handler never blocks the tick.

The shared `_scheduleManager` singleton is dropped on `/config/reload`
alongside `_projectBoard`, `_orgManager`, and `_projectDashboard` so a
new `config.schedules.path` takes effect without a daemon restart.

Audit events:

- `schedule.created`
- `schedule.updated`
- `schedule.deleted`
- `schedule.forced`

### CLI

In `src/cli.js`:

```
c4 schedule list [--enabled] [--disabled] [--project P] [--assignee A]
c4 schedule create --name N --cron 'EXPR' --template T [--project P]
                   [--timezone TZ] [--assignee A] [--id ID]
c4 schedule show <id>
c4 schedule enable <id>
c4 schedule disable <id>
c4 schedule run <id>
c4 schedule delete <id>
c4 schedule next <id>
c4 schedule history <id>
c4 schedule gantt [--weeks N] [--json]
```

Gantt renders client-side against the shared module so the ASCII
timeline does not need its own daemon endpoint.

## Tests

`tests/schedule-mgmt.test.js` adds 60 assertions across ten suites
covering:

- Helpers: `defaultSchedulesPath`, `FIELD_BOUNDS`, `DEFAULT_TIMEZONE`,
  `HISTORY_LIMIT`, `isId`, `normalizeSchedule` defaults +
  `enabled=false` preserved, `freshState`, `ensureShape` dropping
  invalid cron / missing template / invalid id.
- `parseField` primitives: literal, wildcard expansion, comma list,
  range, step over wildcard, step over range, out-of-bounds rejected,
  inverted range rejected, non-numeric rejected, non-positive step
  rejected.
- `parseCron` / `validateCron`: every minute, every hour, 2am, Monday
  9am, step + range combinations, comma list, field-count mismatch,
  non-string input.
- `computeNextRun` with fixed `now`: every minute, 2am daily
  before / after, 15-minute step, Monday 9am from Sunday, month
  boundary to May 1, leap Feb 29 -> 2028-02-29, DOM/DOW OR semantics.
- `wallFields` + `cronMatches` UTC breakdown, matching true, month
  mismatch false.
- CRUD: create normalises + computes nextRun, invalid cron rejected
  before persist, duplicate id rejected, `listSchedules` filters by
  enabled / projectId, `updateSchedule` recomputes nextRun on cron
  change, `updateSchedule` rejects bad cron patch, `deleteSchedule`
  returns true/false, enable / disable toggles the flag.
- `runDueSchedules` gating: disabled schedules do not fire, enabled +
  past nextRun fires and advances.
- `forceRun` + history retention: bumps lastRun without advancing
  nextRun, history trimmed to `HISTORY_LIMIT`.
- Storage roundtrip: fresh ScheduleManager reads writes, missing file
  returns empty, malformed JSON falls back.
- `scheduleTick` dispatcher: invoked per due schedule, advances
  without a dispatcher, dispatcher exceptions do not abort the tick.
- Gantt render: per-enabled-schedule rows + correct run counts,
  disabled excluded, ASCII-only + label + marker, rowless header.

`tests/rbac.test.js` bumps `ALL_ACTIONS.length` to 18 and adds
`SCHEDULE_READ` / `SCHEDULE_MANAGE` enum checks.

Full suite 88 / 88 pass (up from 87).

## Limitations

- Google Calendar / MCP sync is deferred to a future iteration. The
  schedule store already carries `timezone`, so the sync layer can
  project scheduled runs onto an external calendar without a schema
  break.
- The Web UI Gantt / timeline view slots on top of the `/schedules`
  endpoints and ships under the 10.x UI track.
- Deadline-based priority auto-adjust (bumping project task priority
  when a milestone nears) remains a follow-up - today schedules fire
  on a fixed cadence only.
- Dispatcher creates local workers only. Fleet / remote dispatch via
  DGX alias is a natural extension once the dispatcher callback grows
  a `target` field; today the template string can name the machine but
  the worker is always created on `local`.
