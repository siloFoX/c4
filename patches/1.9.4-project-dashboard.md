# Patch 1.9.4 — Project-specific dashboard (10.3)

## Summary

Join the three 10.x backbone modules (project-mgmt, audit-log, cost-report)
into a single per-project dashboard snapshot. Project managers no longer
have to stitch three endpoints to see a project's health — one request
returns `tasks + workers + merges + tokens + velocity + contributors`.

## Motivation

The 10.x track is building out operator-facing observability:
- 10.1 delivered role-based access control (who can see what),
- 10.2 delivered the audit trail (what happened and when),
- 10.5 delivered cost/token aggregation (how much it cost),
- 10.8 delivered per-project task boards (what is planned and what is done).

Each of those is useful on its own but a project manager wants the
joined view: for project `demo`, show me `done-vs-open` tasks, active
workers on `c4/demo*` branches, the last N merges of those branches,
per-user token usage, per-user task count, and a rolling velocity. The
four sub-systems already store the data — the dashboard is the join.

## What changed

### New module: `src/project-dashboard.js`

Pure aggregation layer — no PTY, no filesystem of its own, no network.
Callers wire up collaborators through the constructor so tests run
against tmpdir-backed board/audit/cost instances and the daemon wires
shared singletons.

```js
const dashboard = new ProjectDashboard({
  board,          // ProjectBoard (10.8)
  auditLogger,    // AuditLogger  (10.2)
  costReporter,   // CostReporter (10.5)
  workers: () => manager.list().workers,
  cacheTtlMs: 30000,       // default 30s
  velocityWeeks: 4,        // default 4 weeks
});

const snap = dashboard.getSnapshot('demo');
// {
//   project:       { id, name, tasks, milestones, sprints, backlog, ... },
//   activeWorkers: [{ name, branch, status, startedAt }, ...],
//   recentMerges:  [{ branch, commit, mergedAt }, ...],
//   todoStats:     { open, done, total, done_pct },
//   tokenUsage:    { total, byUser: {...}, byModel: {...} },
//   contributors:  [{ user, tasks, tokens }, ...],
//   velocity:      { tasksPerWeek, mergesPerWeek, windowWeeks, tasks, merges,
//                    windowStart, windowEnd },
//   generatedAt:   '2026-04-17T12:00:00.000Z'
// }
```

### Caching

`getSnapshot(projectId)` memoizes per project for `cacheTtlMs` (30s
default). The cache key also carries a signature derived from the
project's `tasks.length + milestones.length + sprints.length + max(updatedAt)`
so any `POST /projects/<id>/tasks` or task status flip auto-invalidates
the snapshot on the next read — operators never see a stale dashboard
after editing a task. Callers can also force invalidation via
`invalidate(projectId)` or `invalidateAll()`.

### Matching rules

- **activeWorkers**: a worker belongs to the project when its `branch`
  equals the projectId, is `c4/<projectId>`, starts with `c4/<projectId>-`
  or `c4/<projectId>/`, OR its explicit `project` field matches.
- **recentMerges**: `merge.performed` audit events are filtered by the
  same branch rule, sorted newest-first, capped at 20.
- **tokenUsage / contributors**: cost-report records whose `project`
  field equals the projectId.

### Daemon endpoints

```
GET /projects/:id/dashboard        -> full snapshot (404 on missing)
GET /projects/:id/contributors     -> { projectId, contributors, count }
GET /projects/:id/velocity[?weeks=N]-> velocity window
GET /projects/:id/tokens           -> { projectId, total, byUser, byModel }
```

All four are gated by `requireRole(authCheck, rbac.ACTIONS.PROJECT_READ,
{type:'project', id})` so RBAC protects the dashboard the same way it
protects `GET /projects/:id`.

`POST /config/reload` drops the cached `ProjectDashboard` alongside
`ProjectBoard` so a new `projects.path` or cost-model change picks up
on the next request without a daemon restart.

### CLI

```
c4 project dashboard <id>           # compact human summary
c4 project dashboard <id> --json    # raw snapshot JSON
c4 project contributors <id>        # per-user tasks + tokens
c4 project velocity <id> [--weeks N]# sliding-window velocity
c4 project tokens <id>              # total + per-user + per-model tokens
```

The compact dashboard view prints:
```
Project: Demo
  tasks:    12 (4 done / 8 open, 33.33%)
  workers:  2 active
  merges:   5 recent
  tokens:   142000 total
  velocity: 1 tasks/wk, 0.5 merges/wk (over 4 wk)
  contributors: 3
```

### Tests

`tests/project-dashboard.test.js` adds 28 tests / 80+ assertions across
seven suites, all using `fs.mkdtempSync` paths so no real `~/.c4`
pollution:

- **exports + helpers**: module surface, `projectBranchMatches` contract,
  constructor defaults vs overrides.
- **snapshot shape**: all top-level fields present, missing-project null,
  empty-project zero-filled counters.
- **todoStats**: 50% and 100% cases.
- **activeWorkers**: branch-prefix matching, workers-as-function callback,
  field preservation (name/branch/status/startedAt).
- **recentMerges**: newest-first ordering, non-merge event types ignored.
- **tokenUsage**: input+output sum, per-user + per-model buckets,
  unknown-label fallback.
- **contributors**: task + token aggregation, accessor parity,
  tokens-desc sort.
- **velocity**: default 4-week window, `--weeks` override changes results.
- **cache**: hit within TTL, signature-based auto-invalidation on task
  add, `invalidate(projectId)`, `invalidateAll()`, TTL expiry forces
  recompute.
- **resilience**: getSnapshot works without auditLogger/costReporter,
  null board returns null, malformed worker entries skipped.

Full suite 85 / 85 pass.

## Limitations / follow-ups

- Project-branch matching assumes the c4 worktree branch convention
  (`c4/<projectId>` prefix). Workers driving a project through an
  ad-hoc branch name must set the `project` field on their worker
  record to show up in `activeWorkers`.
- Web UI project-dashboard cards slot on top of the four REST endpoints
  and ship under the 10.x UI track.
- Velocity is currently a simple arithmetic mean over the window.
  Weighted-recency or burn-down charts are a 10.x follow-up (additive,
  no schema break).
