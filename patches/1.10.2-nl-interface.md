# 1.10.2 — Natural Language Interface (11.4)

## Summary

Closes TODO 11.4. Adds a rule-based natural-language layer so operators
can drive the daemon with English instead of memorising flags:

- Web UI: new **Chat** top-level tab backed by `POST /nl/chat`.
- CLI: `c4 chat "query"`, `c4 chat --interactive`, `c4 chat sessions`,
  `c4 chat history <id>`.
- Daemon: `POST /nl/chat`, `GET /nl/sessions`, `GET /nl/sessions/:id`,
  `DELETE /nl/sessions/:id`.

## What landed

### New module `src/nl-interface.js`

- `parseIntent(text) -> {intent, params, confidence}` — regex-first
  classifier over eight intents:
  `list_workers`, `create_worker`, `send_task`, `get_status`,
  `get_history`, `read_output`, `close_worker`, `unknown`.
- `executeIntent(intent, params, {adapter}) -> result` — thin dispatch
  layer. `adapter` exposes `listWorkers` / `createWorker` / `sendTask` /
  `getStatus` / `getHistory` / `readOutput` / `closeWorker`. Adapter
  throws are captured as `{ok:false, error}` so callers never have to
  distinguish parser errors from adapter errors.
- `formatResponse(result, intent) -> string` — terse chat-style reply
  shared by the Web UI, the `c4 chat` one-shot, and the REPL.
- `SessionStore` — persists `ChatSession {id, history, lastWorker,
  createdAt, updatedAt}` to `~/.c4/nl-sessions.json`. Methods:
  `createSession`, `getSession`, `listSessions`, `appendMessage`,
  `setLastWorker`, `deleteSession`, `reload`. Tolerates missing and
  malformed JSON by falling back to an empty store.
- `NlInterface.handle(sessionId, text)` — parse -> execute -> format ->
  persist in one call. Auto-creates a session when the id is missing or
  unknown. Resolves pronouns (`it`/`that`/`this`/`him`/`her`/`them`)
  against `lastWorker` so "close it" reuses the most recently referenced
  worker without another trip through the parser.
- `buildActions(parsed, result)` — emits `{type, worker?, label}` chips
  the Web UI renders as quick-action buttons (Task / Read / Close /
  Daemon status).

### Daemon wiring

`src/daemon.js` gains a `getNlInterface()` singleton that builds an
adapter from the in-process `PtyManager` (no HTTP self-calls) and four
endpoints:

- `POST /nl/chat`  — body `{sessionId?, text}`, returns
  `{sessionId, response, intent, params, confidence, result, actions}`.
- `GET  /nl/sessions`        — list sessions with `messageCount` + `lastWorker`.
- `GET  /nl/sessions/:id`    — full session (history included).
- `DELETE /nl/sessions/:id`  — remove a session.

All four routes are gated by the new `nl.chat` RBAC action. The shared
`_nlInstance` singleton drops on `/config/reload` so a new
`config.nl.sessionsPath` kicks in without a daemon restart.

Audit events: `nl.chat` (carries `intent` + `confidence`) and
`nl.session.deleted`.

### RBAC (`src/rbac.js`)

- New canonical action `NL_CHAT = 'nl.chat'`.
- `ALL_ACTIONS.length` grows from 20 to 21.
- Default matrix: both `manager` and `viewer` get `nl.chat` since chat
  itself is read-ish; destructive operations the chat may dispatch
  still go through their own gates (`worker.create`, `worker.task`,
  `worker.close`). `admin` keeps `*`.

### CLI (`src/cli.js`)

New `c4 chat` subcommand:

```
c4 chat "query"            # one-shot NL query
c4 chat --interactive      # readline REPL; type 'exit' or 'quit' to leave
c4 chat sessions           # list stored chat sessions
c4 chat history <id>       # dump a session's messages
```

The session id is pinned at `~/.c4-nl-session` so subsequent commands in
the same shell keep conversation context ("close it" works across
invocations).

### Web UI (`web/src/components/Chat.tsx` + `web/src/App.tsx`)

- New `Chat` component: input box, message list, auto-scroll, reset
  button, quick-action chips rendered from `actions[]`, session id
  persisted in `localStorage` under `c4.nl.sessionId`.
- `App.tsx` adds a third **Chat** tab in the top-level tablist alongside
  **Workers** and **History**.

### Tests

`tests/nl-interface.test.js` adds 56 assertions across 11 suites:

- exports + constants (3)
- `parseIntent` list_workers (5), create_worker (4), send_task (4),

## Limitations and future work

- **Rule-based parser.** `parseIntent` covers roughly the top 80% of
  everyday phrasings but will miss long / unusual queries. An LLM-backed
  classifier (either local via 8.9 or the Claude API) is a natural next
  step; the spec mentions "LLM 기반 의도 파악" and that remains open. The
  module is structured so `NlInterface` can accept a different parser
  implementation without touching the adapter or the session layer.
- **Chained workflows.** "Train a model on DGX and Slack me when it
  finishes" is out of scope for this iteration. Those multi-step,
  multi-tool plans belong to the 11.3 workflow engine; the NL interface
  should eventually call into that engine once it lands.
- **Web UI.** The current Chat component is a minimal panel (list +
  input + chips). A richer view with command suggestions, per-session
  history pane, and intent confidence shading is left for a follow-up.
- **Confirmation prompts.** All dispatches execute immediately. A
  future iteration should require confirmation for destructive intents
  (`close_worker`, send_task with `--auto-mode`) when `confidence` is
  below a threshold.

## Migration notes

- No breaking changes. Operators who do not hit the new endpoints or the
  `c4 chat` subcommand see the same behaviour as before.
- Existing RBAC stores keep working: the default role matrix is only
  consulted for new users. Existing `viewer`/`manager` users pick up
  `nl.chat` automatically because it is derived from
  `DEFAULT_PERMISSIONS` at resolve time.
- Chat sessions are stored under `~/.c4/nl-sessions.json`. Safe to
  delete at any time; the next call will recreate the file.
