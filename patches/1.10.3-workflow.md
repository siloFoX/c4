# Patch 1.10.3 - Workflow engine (11.3)

## Summary

Add a graph-based multi-worker orchestration layer on top of the
existing PtyManager + worker dispatcher. Operators define a workflow
as a small DAG of `task`, `condition`, `parallel`, `wait`, and `end`
nodes connected by directed `edges`; the daemon walks the graph,
dispatches `task` nodes through the same path `c4 task` uses, and
records every node's status / output into a per-run history file. A
view-only DAG visualizer lands in the Web UI under a new `Workflows`
tab; full drag-and-drop authoring is deferred.

## Motivation

11.4 (Natural Language Interface) shipped single-turn chat over the
daemon, but anything requiring a chained "do A, then if X do B, else
do C" plan still required the operator to fan out individual `c4 task`
calls by hand. 11.1 (MCP Hub) handles per-worker tool wiring but does
not orchestrate across workers. 11.3 fills the gap: workflows are the
multi-step glue between intents and worker lifecycles. The TODO entry
calls out the canonical case of "every morning check email -> if it's
work-related forward via KakaoTalk -> if it's code-related spawn a
worker", and the workflow engine is the layer that lets that chain
become a single persisted definition instead of a script.

## What changed

### New module: `src/workflow.js`

Pure storage + decision layer plus the executor. Writes definitions to
`~/.c4/workflows.json` (`config.workflows.path` override) and run
history to `~/.c4/workflow-runs.json` (capped at 200 most recent runs;
`config.workflows.runsPath` override). No shell-outs from the executor
itself - the dispatcher is an injected async function so unit tests
drive runs without touching PTY state.

```js
const { WorkflowManager, WorkflowExecutor } = require('./workflow');

const wfMgr = new WorkflowManager({
  workflowsPath: '~/.c4/workflows.json',
  runsPath: '~/.c4/workflow-runs.json',
});

wfMgr.createWorkflow({
  id: 'morning-triage',
  name: 'Morning triage',
  description: 'Check inbox, branch on category, optionally spawn a worker.',
  nodes: [
    { id: 'inbox', type: 'task', name: 'Read inbox', config: { workerName: 'inbox-reader', taskTemplate: 'summarize unread mail' } },
    { id: 'cat',   type: 'condition', name: 'Code-related?', config: { expression: 'output.category === "code"' } },
    { id: 'spawn', type: 'task', name: 'Spawn worker', config: { workerName: 'morning-worker', taskTemplate: 'investigate today\'s tickets' } },
    { id: 'forward', type: 'task', name: 'Forward to KakaoTalk', config: { workerName: 'kakao-fwd', taskTemplate: 'forward important mail summary' } },
    { id: 'done',  type: 'end', name: 'Done', config: {} },
  ],
  edges: [
    { from: 'inbox', to: 'cat' },
    { from: 'cat', to: 'spawn',   condition: 'output.value === true' },
    { from: 'cat', to: 'forward', condition: 'output.value === false' },
    { from: 'spawn',   to: 'done' },
    { from: 'forward', to: 'done' },
  ],
});

const exec = new WorkflowExecutor({ manager: wfMgr });
const run = await exec.executeWorkflow('morning-triage', {}, {});
// run.status === 'completed' | 'failed'
// run.nodeResults[<nodeId>] === { status, output, error, startedAt, completedAt }
```

`validateGraph(nodes, edges)` is the single source of truth and runs
before every persist call. It returns `{valid, errors}`. Errors fired:

- empty graph
- duplicate node ids
- edges referencing unknown nodes
- duplicate `(from, to)` edges
- self-edges
- cycles (Kahn's algorithm)
- fully-orphan nodes (no incoming AND no outgoing edges) when more
  than one node exists
- missing terminal `end` node
- unknown `node.type`
- non-string `edge.condition`

`WorkflowExecutor.executeWorkflow(workflowId, inputs, context)` topo-
logically sorts the graph, then iterates: a node executes when at
least one upstream edge has fired into it. After it runs, every
outgoing edge's optional `condition` is evaluated against the
just-produced output; truthy means the target is activated and the
output is threaded as `prev` for that successor. Failed nodes halt
downstream propagation and flip the run to `failed`. Disabled
workflows throw with `code: 'WORKFLOW_DISABLED'` so the daemon (and
CLI) can surface a typed error instead of a string-match.

Conditions evaluate inside a hardened `new Function(...)` sandbox:

- the expression body is wrapped in `"use strict"; return (...)`;
- the parameter list shadows `globalThis`, `global`, `process`,
  `Buffer`, `console`, `setTimeout`, `setInterval`, `setImmediate`,
  `clearTimeout`, `clearInterval`, `clearImmediate`,
  `queueMicrotask`, `fetch`, `module`, `exports` with `undefined`;
- a small whitelist (`Math`, `JSON`, `Date`, `Number`, `String`,
  `Boolean`, `Array`, `Object`, `RegExp`) is passed as actual values;
- a regex pre-check rejects `eval`, `Function`, `require`, `import`,
  `export`, `throw`, `while`, `for`, `do`, `class`, `new <Anything>Process`
  before parsing;
- the source is capped at 1024 chars.

This is a *basic* sandbox suitable for trusted-but-typo'd workflow
YAML, not a defence against an actively hostile author with full
code-execution intent. Operators who care about that should keep
`workflow.manage` admin-only via the RBAC matrix.

### RBAC: `src/rbac.js`

Two new canonical actions on the `ACTIONS` enum (23 total, up from
21):

- `WORKFLOW_READ` -> `workflow.read`
- `WORKFLOW_MANAGE` -> `workflow.manage`

Default matrix:

- `manager`: read + manage
- `viewer`: read
- `admin`: wildcard (already covered)

`tests/rbac.test.js` bumps `ALL_ACTIONS.length` to 23 and adds the
new enum checks.

### Daemon endpoints: `src/daemon.js`

```
GET    /workflows                   # workflow.read; ?enabled=&nameContains=
POST   /workflows                   # workflow.manage; validateGraph before persist
GET    /workflows/:id               # workflow.read
PUT    /workflows/:id               # workflow.manage; re-validates on graph change
DELETE /workflows/:id               # workflow.manage
POST   /workflows/:id/run           # workflow.manage; body {inputs?}; returns WorkflowRun
GET    /workflows/:id/runs          # workflow.read; per-workflow history
GET    /workflow-runs/:runId        # workflow.read; single run by id
```

The shared `_workflowManager` + `_workflowExecutor` singletons drop on
`POST /config/reload` so a new `config.workflows.path` takes effect
without a daemon restart.

The dispatcher passed to `WorkflowExecutor` resolves `node.config.workerName`
+ `node.config.taskTemplate` into a `manager.create` + `manager.sendTask`
call so a workflow node spawns the same workers `c4 task` does. Worker
names are mangled to `<base>-<runIdSuffix>` so two concurrent runs of
the same workflow do not collide.

Audit events: `workflow.created`, `workflow.updated`, `workflow.deleted`,
`workflow.run` (includes `runId` + `status`).

### CLI: `src/cli.js`

```
c4 workflow list [--enabled] [--disabled] [--name N]
c4 workflow create --file <workflow.yaml|workflow.json>
c4 workflow show <id>
c4 workflow run <id> [--inputs '{...}']
c4 workflow runs <id>
c4 workflow delete <id>
c4 workflow export <id>
```

The `--file` parser tries `JSON.parse` first; on failure it falls
through to a minimal in-CLI YAML loader covering nested mappings,
sequences-of-mappings, scalars (string / number / boolean / null /
quoted), and `#` comments. That is enough for the workflow-definition
shape and avoids pulling a runtime YAML dependency. Anything more
complex (anchors, multi-line scalars, flow-style collections) should
ship the file as JSON.

### Web UI: `web/src/components/WorkflowEditor.tsx`

View-only iteration. Layout:

- left rail: workflow list with `Refresh` button, name + node/edge
  counts, on/off pill;
- main: selected workflow header (name + description + `Run` button),
  SVG DAG canvas, two cards below for the selected node's properties
  (id / type / pretty-printed config) and the most recent runs
  (status colour-coded, started-at timestamp).

The DAG layout assigns each node a column equal to the longest path
to it from any source, then stacks nodes vertically within a column
in stable id order. Edges are bezier curves with arrow markers; edges
with a condition string render the (truncated) expression as a label.
Click a node to load it in the property panel.

`web/src/App.tsx` adds a fourth `Workflows` top-level tab alongside
Workers / History / Chat. Selection is persisted in `localStorage`
under the existing `c4.topView` key.

Drag-to-edit, palette, zoom, inline edge condition editing, and
live-running animations are deferred to a follow-up patch.

### Tests: `tests/workflow.test.js`

42 tests / 80+ assertions across eight suites:

- helpers + constants (NODE_TYPES, RUN_STATUS, NODE_STATUS,
  RUN_RETENTION, defaults under `~/.c4`, `isId`, `genWorkflowId`)
- `validateGraph` (empty, duplicate ids, missing edge endpoints,
  duplicate edges, self-edges, cycles, orphan nodes, missing
  terminal, unknown types, linear example, `topoSort` cycle null)
- sandboxed condition evaluator (operates on `output` / `input` /
  `Math.max`, blocks `eval` / `Function` / `require` / `import` /
  `throw`, shadows `process` / `globalThis` / `setTimeout` / `console`
  to `undefined`, rejects empty / non-string / oversized expressions)
- `WorkflowManager` CRUD (validates + persists, duplicate-id
  rejection, invalid-graph rejection with `errors[]` on the thrown
  Error, `updateWorkflow` re-validates when nodes/edges change,
  `deleteWorkflow` returns false on miss, `listWorkflows` filters on
  `enabled` + `nameContains`, `enableWorkflow` / `disableWorkflow`
  toggle the flag)
- storage roundtrip (workflows + runs survive a fresh manager,
  `ensureWorkflowsShape` drops malformed entries, `normalizeWorkflow`
  defaults `enabled=true` + fills timestamps)
- run history retention (custom retention of 5 keeps only last 5 of 8
  runs)
- `WorkflowExecutor` (linear chain in topo order with prev-output
  threading, conditional branch true-path activates `t` and skips
  `f`, conditional branch false-path activates `f` and skips `t`,
  parallel fan-out + join executes the join exactly once, disabled
  workflow rejected with `code: 'WORKFLOW_DISABLED'`, wait node
  honours `waitImpl(ms)`, failed dispatcher marks the run failed and
  skips downstream nodes, executeWorkflow throws when the workflow
  does not exist)
- `WorkflowStore` retention edges (insertion-order trim, `getRun`
  returns null on miss)

Full suite 91 / 91 pass (up from 90).

## Non-goals (deferred)

- Drag-to-edit Web UI - the editor ships view-only this iteration;
  authoring still happens via `c4 workflow create --file`.
- True parallel concurrency - the in-process executor still walks the
  topo order single-threaded; `parallel` is a fan-out marker that
  activates multiple branches but the branches sequence through the
  same await loop. True concurrency lands when 11.5 introduces a
  worker-pool runtime.
- Wait-for-event nodes - the `wait` node currently only honours
  `delayMs`; named events are recorded as `{event, fired:true}` but
  the daemon does not yet have an event bus to wire them to.
- Versioning / rollback of workflow definitions - every PUT
  overwrites; a future patch can add an immutable history table
  alongside `workflow-runs.json`.
- YAML feature surface - the in-CLI loader handles nested mappings,
  sequences-of-mappings, and scalars only. Anchors, multi-line
  scalars, and flow-style `[a, b, c]` are not supported; ship those
  workflows as JSON.
