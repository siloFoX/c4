# Patch 1.11.6 — Planner Back-propagation loop (TODO 9.12)

## Summary

`c4 plan` previously generated a plan document and stopped. If a worker
discovered mid-execution that the plan was wrong (missing dep, infeasible
step, ambiguous spec) it had no way to feed that back into the planner —
it either improvised or surfaced an error and the manager had to write a
fresh plan by hand. This patch closes the loop:

1. Workers carry a `plan_doc_path` that points at the plan they were
   dispatched against.
2. `c4 plan-update` appends a structured "needs revision" block to that
   plan document.
3. With `--replan`, the planner factory runs again with the original task
   + failure reason + partial state, and a new revision file lands in
   `docs/plans/<name>-rev<N>.md`.
4. With `--redispatch`, the worker is re-tasked with the revised plan,
   carrying its own snapshots forward via `contextFrom=<name>`.
5. A loop limit (default 3, configurable) prevents runaway re-planning,
   and breach pages the operator via Slack.

## Behavioural contract

### Worker metadata

After `c4 task w1 "..." --plan-doc docs/plans/w1.md`, the worker record
holds:

```js
{
  plan_doc_path: 'docs/plans/w1.md',
  replan_count: 0,
  plan_revisions: [],
  // ... existing fields
}
```

Path may be absolute or relative to the worktree; `_resolvePlanPath`
joins it against `worker.worktree` when needed.

### `c4 plan-update`

```
c4 plan-update <name> --reason <text> [--evidence <text>] [--replan] [--redispatch]
```

| Flags | Effect |
|---|---|
| (none) | Append a `## Needs Revision` block to `plan_doc_path`. |
| `--replan` | Same + invoke planner factory + write `docs/plans/<name>-rev<N>.md` + bump `replan_count` + repoint `plan_doc_path`. |
| `--redispatch` | Implies `--replan` + call `manager.sendTask` with the revised plan prompt and `contextFrom=<name>`. |

The CLI emits the daemon JSON response as-is. On loop-limit breach the
response is `{ success: false, error: "Loop limit 3 exceeded — manual intervention required", loopLimitExceeded: true, replanCount, maxReplans }`.

### Planner factory

```js
factory({
  workerName,        // 'w1'
  originalTask,      // worker._taskText
  reason,            // from --reason
  evidence,          // from --evidence
  previousPlanPath,  // current plan_doc_path
  revisionNumber,    // N (1, 2, 3...)
  revisionPath,      // absolute path of the rev file we'll write
  partialState,      // _describePartialState(worker)
}) => ({ content }) | ({ error })
```

If no factory is wired, `_defaultPlannerFactory` returns a deterministic
templated plan that records the failure context. Tests use
`planner.setPlannerFactory(mockFn)` to inject a mock.

### Loop limit

`_resolveMaxReplans` precedence: per-call `options.maxReplans` →
`config.plannerLoop.maxReplans` → `3`. Set to 0 to disable. On breach:

- `replan` returns `{ error: "Loop limit N exceeded — manual intervention required", loopLimitExceeded: true, ... }` without calling the factory.
- `manager._notifications.pushAll(...)` fires a `[PLANNER LOOP LIMIT]` message and `_flushAll()` is invoked best-effort.
- `updateAndMaybeReplan` propagates the error and never reaches `redispatch`.

### Revision history

`listRevisions(name)` returns:

```js
{
  success: true,
  worker: 'w1',
  current: '/.../docs/plans/w1-rev2.md', // or null if never replanned
  replanCount: 2,
  maxReplans: 3,
  revisions: [
    { rev: 1, path: '...w1-rev1.md', reason: '...', evidence: '...', when: '2026-04-18T...' },
    { rev: 2, path: '...w1-rev2.md', reason: '...', evidence: '...', when: '...' }
  ]
}
```

Daemon mirror: `GET /plan-revisions?name=w1`.

## Tests

`tests/planner-loop.test.js` covers:

1. `setPlanDocPath` stores metadata + initialises `replan_count`/`plan_revisions`.
2. Rejects unknown worker / empty path; `getPlanDocPath` returns null when missing.
3. `appendNeedsRevision` writes the When/Reason/Evidence block to the plan file.
4. Fails when `plan_doc_path` is missing or the file does not exist.
5. `replan` calls the factory once with the right input and writes `<name>-rev1.md`.
6. Multiple replans bump `replan_count` to 2 and write rev1 + rev2 files.
7. Default loop limit (3) rejects further replans with the exact message and pings Slack.
8. `config.plannerLoop.maxReplans` overrides the default.
9. `redispatch` calls `manager.sendTask` once with the revised prompt + `contextFrom=<name>`.
10. `updateAndMaybeReplan` chains append + replan + redispatch.
11. Without `--replan`/`--redispatch` flags, only append runs (factory untouched).
12. Loop-limit breach short-circuits redispatch (no `sendTask` call).
13. `listRevisions` returns the recorded history + `maxReplans`.
14. `listRevisions` rejects unknown worker.
15. Default factory produces a valid revised plan when none is wired.
16. `replan` rejects factory errors and empty content without mutating state.

Existing `tests/planner.test.js` (8 assertions across `Planner` suite)
runs unchanged — the new constructor signature is backward-compatible
(`new Planner(manager)` still works because the second arg defaults to
`{}`).
