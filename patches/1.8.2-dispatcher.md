# 1.8.2 - Fleet task dispatcher (TODO 9.7)

## Summary

TODO 9.7 (Dispatcher - fleet task distribution) lands on top of the 9.6
fleet registry. C4 gains a ranking + placement pipeline that picks
which fleet machine a task runs on based on live load + role tags +
explicit location pins, plus a top-level `c4 dispatch` CLI so operators
can fan tasks across peers with one command.

- `src/dispatcher.js` (new) - pure-node ranking + placement pipeline
- `src/fleet.js` - tags support on `addMachine` / `getMachine` /
  `listMachines` + `normalizeTags` helper
- `src/daemon.js` - new `POST /dispatch` endpoint
- `src/cli.js` - new `c4 dispatch` subcommand + `--tags` on `c4 fleet add`
- `tests/dispatcher.test.js` (new) - 42 assertions across 13 suites

No central broker. The dispatch command runs on the machine the
operator is sitting on; that daemon samples every registered peer's
`/health` + `/list` in parallel, picks slots, and fans out `/task`
calls using the existing JWT auth (8.14) so each peer authenticates as
itself.

## Strategies

Three strategies are available through `--strategy`:

| strategy | primary sort | secondary | tertiary | use case |
|----------|--------------|-----------|----------|----------|
| `least-loaded` (default) | workers asc | tag count desc | alias asc | spread load evenly |
| `tag-match` | match count desc | workers asc | alias asc | pin gpu / build roles |
| `round-robin` | alphabetical | (stateless) | - | even distribution ignoring load |

Placement is **incremental**: `pickLeastLoadedIncremental` and
`pickTagMatchIncremental` increment a simulated worker count after each
chosen slot, so a 4-slot batch against 2 equally-loaded peers lands
`[a, b, a, b]` instead of piling on the first least-loaded machine.

## Fallback paths

`dispatcher.dispatch(...)` never throws on transport failure. It
returns one of:

- `fallback: 'no-machines'` - empty fleet AND no local sample
- `fallback: 'local-only'` - fleet empty, local daemon reachable
- `fallback: 'all-unreachable'` - every remote sample failed and no
  local daemon in the mix
- `fallback: 'tags-no-match'` - tag filter emptied the candidate pool
- `fallback: null` - everything worked

## Fleet tags

`~/.c4/fleet.json` machine entries grow an optional `tags` array:

```json
{
  "machines": {
    "dgx":   { "host": "192.168.10.222", "port": 3456, "tags": ["gpu", "high-mem"] },
    "build": { "host": "192.168.10.50",  "port": 3456, "tags": ["build", "web"] }
  }
}
```

Tags are lowercased + deduped + validated against
`/^[a-z0-9][\w.-]*$/`. `c4 fleet add <alias> <host> --tags gpu,high-mem`
writes them; `c4 fleet add <alias> <host>` (no `--tags`) preserves the
existing set; `c4 fleet add <alias> <host> --clear-tags` wipes them.

## Daemon endpoint

```
POST /dispatch
{
  "task": "implement feature X",
  "count": 3,
  "strategy": "least-loaded",
  "tags": ["gpu"],
  "location": null,        // optional explicit alias pin
  "namePrefix": "dispatch", // worker name prefix; default "dispatch"
  "branch": "feature",      // branch prefix; each slot becomes feature-N
  "profile": "ml",
  "autoMode": true,
  "dryRun": false           // returns plan without issuing /task calls
}
```

Response shape:

```
{
  "strategy": "least-loaded",
  "count": 3,
  "tags": ["gpu"],
  "fallback": null,
  "plan": [
    {"slot": 0, "name": "dispatch-1", "branch": "feature-1", "task": "...",
     "machine": {"alias": "dgx", "host": "...", "port": 3456, "tags": ["gpu"]},
     "score": {"strategy": "least-loaded", "workers": 0, "tagCount": 1}},
    ...
  ],
  "samples": [{"alias": "...", "ok": true, "workers": 0, "tags": [...], ...}],
  "created": [{"slot": 0, "name": "dispatch-1", "alias": "dgx", "ok": true,
               "result": {...}}],
  "dryRun": false
}
```

## CLI

```bash
# Same task to 3 workers, picking least-loaded machine each time
c4 dispatch "migrate db" --count 3

# Pin the gpu-tagged machine for an ML job
c4 dispatch "train model" --count 2 --strategy tag-match --tags gpu

# Even distribution across every machine
c4 dispatch "crawl URLs" --count 12 --strategy round-robin

# Dry-run: see the plan without spawning workers
c4 dispatch "risky op" --count 5 --dry-run

# Force every slot onto a specific machine
c4 dispatch "gpu-only task" --count 1 --location dgx

# Provide a custom name + branch prefix
c4 dispatch "refactor auth" --count 3 --name auth --branch feature/auth
```

## Tests

1. `normalizeStrategy` defaults to `least-loaded`, accepts
   case-insensitive known names, throws on unknown.
2. `rankLeastLoaded` orders by workers asc, tag count desc, alias asc;
   `Infinity`-sorts unknown workers so `null` lands last.
3. `rankTagMatch` orders by matches desc, workers asc, alias asc;
   handles zero-wanted-tags edge.
4. `rankRoundRobin` alpha-sorts for deterministic cyclic walks.
5. `filterByTags` drops missing-tag peers case-insensitively; empty
   wanted-tags returns the input unchanged.
6. `filterReachable` drops `ok:false`.
7. `buildPool` filters invalid entries, honors `locationPin` to a single
   alias, returns `[]` when the pin misses.
8. `sampleFleet` folds pool `tags` + `authToken` onto sample rows;
   empty pool returns `[]`.
9. `pickLeastLoadedIncremental` increments simulated load so 4 slots
   against 2 equal machines land `[a, b, a, b]`; respects preexisting
   worker counts.
10. `pickRoundRobin` cycles when `count > pool`; empty input returns `[]`.
11. `dispatch()` top-level fallback matrix: `no-machines`, `local-only`,
    `all-unreachable`, `tags-no-match`, all-remote-unreachable-with-local
    routes to local, round-robin spreads 5 slots across 3 peers
    deterministically, tag-match beats lower load, least-loaded avoids
    hot machine, `location` forces a single alias, plan enrichment.
12. Fleet tags persistence (store / preserve on re-add / `clearTags` /
    casing + dedup / reject invalid chars).
13. Daemon + CLI source-grep wiring.

## Limitations

- The dispatcher does not create the *worker* on the remote peer
  directly; it sends `/task` which triggers that peer's auto-create via
  the existing worker lifecycle. If an operator wants a bare `/create`
  followed by a separate `/task` flow, call `/dispatch` with
  `dryRun:true` then fan out their own create + task per slot.
- Tag matching is set-intersection, not substring. `--tags gpu` matches
  a machine tagged `gpu` but not one tagged `gpu-pool`; use the exact
  label.
- Round-robin is stateless per call. Two sequential 1-slot dispatches
  both land on the first alpha-sorted alias. Use `least-loaded` if you
  want sequential dispatches to spread.
- The fallback chain ends at "local ok". If the local daemon is also
  unreachable, the response carries `fallback: 'all-unreachable'` and
  an empty plan - in practice the CLI cannot even reach the endpoint in
  that case.
