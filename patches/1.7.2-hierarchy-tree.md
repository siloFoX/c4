# Patch 1.7.2 - Recursive hierarchy tree for workers (8.2)

## Date: 2026-04-17

## Summary

TODO 8.2 delivers parent/child visualization for C4 workers across both the
CLI and the Web UI, plus a daemon-side `GET /tree` endpoint for third-party
consumers. Worker metadata gains an optional `parent` field that flows
through `create()` -> list -> persisted state, a new pure-ASCII tree
renderer powers `c4 list --tree`, and a React `HierarchyTree` component
replaces the flat sidebar with an expand/collapse forest when the operator
toggles the sidebar mode. The daemon API stays backwards compatible --
parent is optional everywhere, so existing install bases that never set it
see `parent: null` and keep working.

## Motivation

Under unattended autonomy the manager typically spawns workers that in
turn spawn workers (for example `auto-mgr` -> `scribe` + N task workers,
or a batch run that fans out to `batch-1..N`). Phase 8.1 listed every
worker flat in a sidebar, which made it impossible to tell at a glance
which worker was a child of which, which subtree was blocked on an
approval, and whether failures were localized to one branch of the tree
or fanned out everywhere. 8.2 fixes that by making parentage first-class
metadata and rendering it at both the terminal and the browser.

## Implementation

### `src/hierarchy-tree.js` (new)

Pure, dependency-free module. Kept separate from `pty-manager.js` so the
CLI (which does not link against node-pty) can require it directly, and
tests can exercise it without spinning up a PTY.

- `buildTree(workers)` groups the flat worker array into a forest by
  looking up each worker's `parent` in a `Map`. Orphans whose parent name
  does not match any worker get promoted to roots instead of silently
  dropped. Cycles -- including self-cycles (`X.parent=X`) and mutual
  cycles (`A.parent=B, B.parent=A`) -- are detected via an upward walk
  with a `Set` guard, and the offending edge is broken by re-rooting
  the current node. Siblings are sorted by name for a stable render.
- `computeRollup(node)` recursively aggregates counts of each status
  (`idle` / `busy` / `exited`), intervention, and error into the parent
  so a manager can see `[3 idle, 2 busy, 1 intervention]` on a root
  without walking every descendant by hand. Intervention is counted
  independently of status because a worker parked at an approval prompt
  is still `busy` to the scheduler; the rollup makes the parked state
  visible at the parent level.
- `renderTree(roots)` emits pure-ASCII (`+--`, `|`, space) -- no
  box-drawing glyphs -- so the output survives logs, non-UTF8 terminals,
  and copy/paste. Each node line gets `<name>  [status]  <rollup>?
  (branch)?` and children are indented with the usual tree branches.
- `flatten(roots)` / `isInterventionActive(node)` / `statusBadge(node)` /
  `formatRollup(r)` are small helpers reused by both the CLI render path
  and the React component mirror.

### `src/pty-manager.js`

Worker metadata now carries parent end-to-end:

1. `create(name, command, args, options)` reads `options.parent`, stores
   it on the worker record as a string-or-null, and forwards it into the
   node-pty spawn env as `C4_PARENT` so any tool that shells into the
   worker can see who spawned it. The env also gains `C4_WORKER_NAME`
   so a `claude` process running `c4 new <child>` from inside a worker
   auto-detects its own name as the child's parent without having to
   pass `--parent` explicitly.
2. `list()` echoes `parent: w.parent || null` on every entry. No schema
   break: clients that ignore the field keep working.
3. `_saveState()` persists `parent` into the state.json worker records,
   and `_loadState()` propagates it onto `lostWorkers` entries so the
   tree survives a daemon bounce (otherwise a restarted daemon would
   forget every ancestry link on recovery).

### `src/daemon.js`

- `POST /create` now reads `parent` from the request body and hands it
  through to `manager.create(... { parent })`. Missing `parent` -> `null`,
  no schema break.
- New `GET /tree` wires `manager.list()` through `hierarchy-tree.buildTree`
  and returns `{roots, queuedTasks, lostWorkers}`. Exposed for third-party
  consumers; the Web UI continues to reuse `/list` to avoid a double fetch.

### `src/cli.js`

- `c4 new <name> [--parent <name>] ...`: parses `--parent`; when the flag
  is absent, falls back to `process.env.C4_WORKER_NAME` so nested calls
  record the spawning worker automatically. Explicit `--parent` always
  wins (operators can override).
- `c4 list --tree`: routes the list response through
  `hierarchy-tree.buildTree` + `renderTree` and prints the forest, then
  tacks on queued tasks and lost workers below. The existing flat list
  output (default) is untouched.

### Web UI

- `web/src/components/HierarchyTree.tsx` (new): mirrors the backend
  rollup logic in TypeScript, renders each node with an expand/collapse
  button (disabled for leaves), a status pill (green idle / yellow busy
  / red intervention / gray exited), and per-subtree rollup badges under
  parent rows. Shares `/api/list` + the same SSE subscription as
  `WorkerList` so switching tabs does not double-fetch. Expand-all /
  Collapse-all buttons at the top for quick manipulation on large trees.
- `web/src/App.tsx`: adds a `List` / `Tree` tab pair in the sidebar
  header, persists the selection to `localStorage.c4.sidebar.mode` so
  the operator's preference survives reload. Both tabs use the same
  selected-worker state so clicking a node in the tree drives the same
  detail view as clicking in the flat list.
- `web/src/types.ts`: adds `parent: string | null` to the `Worker`
  interface.

## Tests

`tests/hierarchy-tree.test.js` (21 assertions, 5 suites):

- **`buildTree`**: empty/invalid input returns `[]`; roots sort
  alphabetically; parent/child nesting resolves by name; unknown parent
  names demote to root instead of dropping; self-cycle + mutual-cycle
  edges get broken and every worker appears exactly once in the forest;
  entries without a `name` are skipped instead of throwing.
- **`computeRollup`**: counts idle/busy/exited/error across a subtree;
  counts intervention independently of status; null/missing fields do
  not crash the aggregator.
- **`renderTree`**: ASCII byte-level check (every char code <= 0x7e or
  is `\n`); rollup surfaces on multi-node subtrees; empty input renders
  empty string; active intervention emits `[intervention]`.
- **Source-wiring**: regex-grep `src/pty-manager.js` / `src/daemon.js` /
  `src/cli.js` to lock in that parent is stored in `create()`, echoed
  in `list()`, persisted in `_saveState` + `lostWorkers`, injected into
  spawn env as `C4_WORKER_NAME`, forwarded on `/create`, exposed via
  `/tree`, accepted on the CLI as `--parent`, falls back to
  `C4_WORKER_NAME`, and that `c4 list --tree` calls `renderTree`. These
  tests catch drift between the implementation and the spec without
  having to pull in `node-pty`.
- **End-to-end render**: nested grandchildren indent further than their
  parents; intervention on a grandchild surfaces in the rendered line.

Full suite 69/69 pass.

## Backwards compatibility

- `parent` is optional everywhere. Clients, configs, and persisted state
  that never set it continue to work unchanged (`null` is the default).
- `POST /create` ignores unknown body keys on the way in and treats a
  missing `parent` as null on the way out.
- `c4 list` default output is unchanged; `--tree` is an opt-in flag.
- Web UI defaults to the existing flat list; Tree view is a toggle.
- No dependency additions (bcryptjs / jsonwebtoken / vite / etc. all
  untouched). The utility is pure Node, no external packages.
