# Patch 1.11.9 - CLI /api prefix auth fix (8.19)

## Summary

After the 1.7.0 session-auth work (TODO 8.14), every daemon handler that
mutates state (`/create`, `/task`, `/merge`, `/close`, ...) was hidden
behind an `auth.checkRequest` middleware that only runs for requests
that come in under the `/api` prefix. The CLI (`src/cli.js`), however,
still addressed handlers by their legacy bare paths (`/create`, etc.).
The middleware therefore skipped the request, `authCheck.decoded` stayed
`null`, and the downstream RBAC gate in `requireRole` returned
`401 Authentication required` on every CLI write even though the same
token posted to `/api/create` by curl succeeded.

`c4 auth/status` (the separate patch in `src/auth.js` that adds
`/auth/status` to `OPEN_API_ROUTES`) was also needed for the Web UI to
render the login form; that fix is orthogonal but lives in the same
commit train because both were surfaced by the same operator report.

## Files changed

| File | Change |
|---|---|
| `src/cli.js` | New `withApiPrefix(p)` helper; `request()` runs its `path` argument through it so every HTTP call lands on `/api/<route>` while call sites keep writing `/create`, `/list`, etc. The `c4 watch` SSE stream previously hit `/watch` with no token; now it hits `/api/watch?name=<n>&token=<jwt>` because EventSource-style callers cannot set the `Authorization` header. `main()` is guarded by `require.main === module` and `withApiPrefix` is exported so tests can require the module without triggering argv parsing. |
| `src/auth.js` | `OPEN_API_ROUTES` gains `/auth/status`. The Web UI polls that endpoint pre-login to decide whether to render the login form; previously the 401 fall-through made `fetchAuthStatus()` report `{enabled: false}` and the UI skipped the login step entirely. |
| `tests/cli-api-prefix.test.js` | New file (3 suites / 11 assertions). (a) `withApiPrefix` unit tests — idempotent on `/api/...`, prepends on bare, safe on non-string input. (b) Integration spawn of `src/cli.js` against an in-process HTTP capture server; asserts that `c4 health` / `c4 list` land on `/api/*` with the `Authorization: Bearer <token>` header when `C4_TOKEN` is set (and no header when it is not). spawn is async-plus-promise because a sync child would block the parent's event loop and the capture server could never respond. (c) Path classification over `auth.checkRequest` — `/auth/login`, `/auth/status`, `/health` open; every other route requires a bearer and unlocks with a valid one. |

## Reproduction

Pre-fix, with `config.auth.enabled=true` and a valid bearer token in
`~/.c4-token`:

```
$ c4 new test-auth-1 --no-branch
{
  "error": "Authentication required"
}
```

Direct curl against `/api/create` with the same token succeeded, which
made the CLI look like it was sending a malformed bearer and wasted
operator time.

Post-fix the same command creates the worker:

```
$ c4 new test-auth-1 --no-branch
{
  "name": "test-auth-1",
  "pid": 3866196,
  "target": "local",
  "status": "running",
  "tier": "worker"
}
```

And blocks correctly when the token is missing:

```
$ C4_TOKEN= c4 new no-token --no-branch
{
  "error": "Authentication required"
}
```

## Why the fix is small

Every CLI call already flowed through a single `request(method, path,
body, timeout)` helper, so the prefix work is centralised there. The
only caller that bypassed the helper was `c4 watch` (uses `http.get`
directly for the SSE stream); that call site is patched in place and
attaches the token via the `?token=` query-param fallback that
`auth.extractBearerToken` already honours.

No daemon-side route classification was tightened. The spec explicitly
allowed that option if a catch-all gate was 401ing non-`/api` paths, but
no such gate exists - `auth.checkRequest` is only invoked when
`isApiPrefixed || route === '/dashboard'`, so the conservative fix is
the CLI-only change. A future operator who wants to formally retire the
bare-route shape can flip the route table to 404 on non-`/api` paths;
that change is not in scope for 8.19.
