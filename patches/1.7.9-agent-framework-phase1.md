# 1.7.9 - Agent Framework phase 1 (TODO 9.1)

## Summary

TODO 9.1 (Agent Framework transition) in a single batch: extract the
explicit Agent Adapter interface from `src/terminal-interface.js` and move
the Claude Code integration behind it as the first concrete adapter. No
PTY behavior change, no breaking CLI change, no second adapter yet - that
is deliberate scope for phase 1.

- `src/agents/adapter.js` - new abstract base class `Adapter` + runtime
  shape validator `validateAdapter(instance)`
- `src/agents/claude-code.js` - new `ClaudeCodeAdapter extends Adapter`
  carrying every pattern + keystroke that used to live on
  `TerminalInterface`
- `src/agents/index.js` - new `createAdapter({type, options}, legacyOpts)`
  factory with a single-entry registry (`claude-code` today) and a
  helpful `Unknown agent type: X. Registered: claude-code` error
- `src/terminal-interface.js` - refactored into a thin compatibility
  wrapper whose constructor returns the adapter, so every pre-9.1 call
  site keeps working unchanged
- `config.example.json` - new `agent: {type, options: {...}}` section
  documenting all planned adapter slots
- `tests/agent-framework.test.js` - 30 node:test assertions across 4
  suites covering conformance + forwarding + factory + backward compat

Full suite: 75 / 75 pass (was 74 / 74 before this patch; 29 pre-existing
`terminal-interface.test.js` assertions stay green unchanged).

## Why phase 1 (not "land everything")

TODO 9.1 eventually ships four adapters (Claude Code today, Local LLM
under 9.2, Codex, claude-agent-sdk) plus hybrid routing. Landing them in
one batch would mean touching the daemon's PTY lifecycle, idle detection,
permission gate, intervention state, and scope guard all at once. Phase 1
instead locks in the contract and proves it by migrating the current
agent behind it without a single behavior change. Subsequent phases add
registry entries.

## Interface signature

```js
// src/agents/adapter.js
class Adapter {
  get metadata() { return { name: string, version: string }; }
  get supportsPause() { return boolean; }            // pause/resume support
  init(workerCtx) {}                                 // attach PTY context
  sendInput(text: string) {}                         // raw text
  sendKey(key: string) {}                            // Enter/Escape/C-c/...
  onOutput(cb: (chunk: string) => void) {}           // returns unsubscribe fn
  detectIdle(chunk: string): boolean {}              // idle at prompt?
}
function validateAdapter(instance): true             // throws on mismatch
```

`metadata` (identity + version) and `supportsPause` (can the adapter
pause cleanly mid-task?) are inspectable properties so the daemon can
branch on capabilities without knowing the adapter type.

Required methods:

| method       | contract                                                             |
|--------------|-----------------------------------------------------------------------|
| init         | adapter captures `{proc, screen, name}` so later calls can reach the PTY |
| sendInput    | writes raw bytes to the agent; no-op when no proc attached            |
| sendKey      | maps `Enter` -> `\r`, `Escape` -> `\x1b`, `C-c` -> `\x03`, etc.       |
| onOutput     | registers a chunk listener; returns an unsubscribe fn                 |
| detectIdle   | returns true when the agent is idle at its prompt                     |

`validateAdapter` throws with a specific message on the first violation
(missing method / empty metadata name / non-boolean supportsPause) so a
bad adapter fails at wire-up, not in the middle of a user session.

## Design

### File layout

```
src/
  agents/
    adapter.js          (base Adapter + validateAdapter)
    claude-code.js      (ClaudeCodeAdapter)
    index.js            (createAdapter factory + REGISTRY + listAdapterTypes)
  terminal-interface.js (thin wrapper, factory-backed)
```

### Factory

```js
// src/agents/index.js
const REGISTRY = { 'claude-code': ClaudeCodeAdapter };

function createAdapter(agentConfig = {}, legacyOpts = {}) {
  const type = agentConfig.type || 'claude-code';
  const AdapterClass = REGISTRY[type];
  if (!AdapterClass) {
    throw new Error(`Unknown agent type: ${type}. Registered: ${Object.keys(REGISTRY).join(', ')}`);
  }
  const merged = { ...legacyOpts, ...(agentConfig.options || {}) };
  const instance = new AdapterClass(merged.patterns || {}, merged);
  validateAdapter(instance);
  return instance;
}
```

`legacyOpts` exists so callers that still pass the old
`(patterns, options)` shape get their values merged under
`agentConfig.options` without restructuring. `agentConfig.options` wins
on key collisions - the new API is authoritative.

### Claude Code adapter

`ClaudeCodeAdapter` is a strict superset of the pre-9.1
`TerminalInterface`: every method (`isTrustPrompt`, `isPermissionPrompt`,
`isReady`, `isModelMenu`, `getPromptType`, `extractBashCommand`,
`extractFileName`, `countOptions`, `getApproveKeys`, `getDenyKeys`,
`getTrustKeys`, `getModelMenuKeys`, `getEffortKeys`, `getEscapeKey`) is
carried over byte-for-byte behavior. On top of that it adds:

- `metadata: {name: 'claude-code', version: '1.0.0'}`
- `supportsPause: false` (Claude Code interrupts via Ctrl-C; no pause)
- `init(workerCtx)` storing `{proc, screen, name}`
- `sendInput(text)` delegating to `proc.write` (no-op when detached)
- `sendKey(key)` with KEY_MAP (Enter, Escape, Tab, Backspace, Up, Down,
  Left, Right, C-c, C-d) and passthrough for unknown keys
- `onOutput(cb)` with unsubscribe + error-swallow so a buggy listener
  cannot kill the PTY loop
- `detectIdle(chunk)` delegating to `isReady`

The class exports its default pattern dictionary + key map so other
adapters can import them when they want to mimic Claude Code's UI
semantics.

### TerminalInterface compatibility shim

```js
// src/terminal-interface.js (reduced)
class TerminalInterface {
  constructor(patterns = {}, options = {}) {
    const agentCfg = options.agent || { type: 'claude-code' };
    const legacyOpts = {
      patterns: patterns || {},
      alwaysApproveForSession: !!options.alwaysApproveForSession,
    };
    return createAdapter(agentCfg, legacyOpts);  // constructor-return trick
  }
}
```

Returning the adapter from the constructor means every existing call
site in `src/pty-manager.js` resolves unchanged:

```js
this._termInterface = new TerminalInterface(patterns, { alwaysApproveForSession });
// ... later:
this._termInterface.isReady(text);       // works
this._termInterface.getDenyKeys(text);   // works
this._termInterface.isTrustPrompt(text); // works
```

23 call sites across PTY lifecycle, permission gate, effort setup, and
scope guard were inspected; all resolve correctly because
`ClaudeCodeAdapter` exposes the same surface as the old class.

## Migration notes

### For daemon code (src/pty-manager.js today)

Nothing to change in this phase. `new TerminalInterface(patterns, options)`
still works identically. The daemon can migrate to the factory directly
in a future phase:

```js
// Future (non-blocking) migration
const { createAdapter } = require('./agents');
this._adapter = createAdapter(
  this.config.agent || { type: 'claude-code' },
  { patterns: this.config.compatibility?.patterns,
    alwaysApproveForSession: this.config.autoApprove?.alwaysApproveForSession }
);
```

### For adapter authors

1. Implement a class extending `Adapter` in `src/agents/<name>.js`.
2. Override `metadata`, `supportsPause`, and the five required methods.
3. Register it in `src/agents/index.js` REGISTRY under a stable string key.
4. Document its options block in `config.example.json` under
   `agent.options.<name>`.
5. Add a test suite following the 4-suite pattern in
   `tests/agent-framework.test.js` (conformance + forwarding + factory
   selection + backward compat if the adapter is meant to replace a
   legacy surface).

### For operators

No action required. `agent.type` defaults to `claude-code` and existing
`config.json` files without the `agent` section work unchanged because
the factory falls back to the same default.

## What phase 1 does NOT do

- No second adapter lands here. Local LLM (9.2), Codex, and
  claude-agent-sdk are deliberately deferred - phase 1 proves the
  contract by migrating the existing agent behind it.
- No hybrid routing. `config.agent.type` is a single string; there is no
  per-task adapter selection, no cost-based dispatch, no fallback chain.
- No daemon API change. `POST /create` + `POST /task` payloads are
  unchanged; a future phase may add an `agentType` override.
- No new CLI surface. `c4 new <name>` still spawns the configured agent;
  a future phase may add `--agent <type>`.

## Follow-up

- 9.2 Local LLM adapter: `src/agents/local-llm.js` that speaks to
  Ollama / llama.cpp / vLLM via HTTP rather than PTY. Expected shape:
  adapter spawns no PTY; `init` opens an HTTP session; `sendInput`
  POSTs prompt deltas; `onOutput` wires SSE streaming. The Adapter
  interface is deliberately designed so the caller never needs to know
  whether the adapter is PTY-backed or HTTP-backed.
- Codex adapter: similar HTTP shape but speaking the Codex protocol;
  `supportsPause: true` is plausible depending on the API surface.
- claude-agent-sdk adapter: in-process via the SDK; no PTY at all.
- Hybrid routing (9.2 overlap): per-task agent selection based on
  cost/latency signals; the daemon can consult `adapter.metadata` +
  `adapter.supportsPause` at dispatch time without hard-coding types.
