# 1.7.7 - MCP server upgrade to 2025-06-18 spec (TODO 9.4)

## Summary

`src/mcp-handler.js` was a five-tool JSON-RPC shim over the old 2024-11-05
MCP draft. This patch rewrites it as a full MCP server targeting the
2025-06-18 specification while keeping clients that still speak
2024-11-05 or 2025-03-26 working via protocol version negotiation.

Key deltas vs. the prior handler:

- Protocol version negotiation across 2025-06-18, 2025-03-26, 2024-11-05
- Capability set expanded from `{tools}` to `tools + resources + prompts
  + logging + experimental.sampling`
- Tool count 5 -> 14
- Resources + resource templates for daemon-wide and per-worker state
- Prompt catalogue with argument validation
- `logging/setLevel` support across all syslog levels
- `config.mcp.allowedTools` whitelist
- Strict JSON-RPC 2.0 semantics (notifications produce no response)
- New stdio transport bridge for Claude Desktop (`c4 mcp start`)

## Protocol

- Latest supported: **2025-06-18**
- Also accepted: 2025-03-26, 2024-11-05
- Unknown versions fall back to the latest so the handshake never aborts

### initialize response shape

```json
{
  "protocolVersion": "2025-06-18",
  "capabilities": {
    "tools":       { "listChanged": false },
    "resources":   { "subscribe": false, "listChanged": false },
    "prompts":     { "listChanged": false },
    "logging":     {},
    "experimental": { "sampling": {} }
  },
  "serverInfo": {
    "name":    "c4-mcp",
    "title":   "C4 Worker Orchestrator",
    "version": "1.7.7"
  },
  "instructions": "C4 orchestrates Claude Code workers..."
}
```

## Tool catalogue

| Name | Purpose |
|------|---------|
| `create_worker` | Spawn a new PTY worker (optional parent for the 8.2 hierarchy tree) |
| `send_task` | Dispatch a task with auto branch/worktree, honoring scope, contextFrom, plan, autoMode, budgetUsd, maxRetries |
| `list_workers` | Full roster from `PtyManager.list()` |
| `get_worker_state` | Single worker record (name, status, branch, intervention, etc.) |
| `read_output` | Read snapshots / current screen / wait until idle |
| `get_scrollback` | Last N scrollback lines |
| `approve_worker` | Approve permission prompts by option number |
| `cancel_task` | Cancel queued / pending / in-flight task without killing the worker |
| `restart_worker` | Respawn the PTY while keeping the worktree and branch |
| `rollback_worker` | `git reset --soft` to the pre-task commit |
| `merge_worker` | Merge the branch into main via `manager.mergeBranch` (falls back to a clean isError when the daemon does not expose it) |
| `close_worker` | Close + cleanup worktree |
| `get_token_usage` | Daily token usage with optional per-task rollup |
| `get_validation` | 9.9 structured completion object for a worker |

### Example tools/call

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "send_task",
    "arguments": {
      "name": "refactor-auth",
      "task": "Split AuthService into smaller modules",
      "branch": "refactor/auth",
      "scope": { "allowFiles": ["src/auth/**"] },
      "budgetUsd": 2.5,
      "maxRetries": 2
    }
  }
}
```

The handler returns `content: [{type:"text", text:"..."}]` with
`isError:false` for success, `isError:true` for any manager-level error
(duplicate worker, missing name, etc.).

## Resources

Static entries returned by `resources/list`:

- `c4://workers` - full roster, `application/json`
- `c4://token-usage` - daily token aggregate, `application/json`
- `c4://session-context` - scribe session context tail, `text/markdown`

URI templates returned by `resources/templates/list`:

- `c4://worker/{name}/state` - single worker record
- `c4://worker/{name}/scrollback` - last 200 lines
- `c4://worker/{name}/validation` - .c4-validation.json contents

### Example resources/read

```json
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "resources/read",
  "params": { "uri": "c4://worker/refactor-auth/scrollback" }
}
```

Response:

```json
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "contents": [
      {
        "uri": "c4://worker/refactor-auth/scrollback",
        "mimeType": "text/plain",
        "text": "..."
      }
    ]
  }
}
```

## Prompts

`prompts/list` returns three entries that build user messages for common
operator flows:

- `run-task` - dispatch `send_task` then `read_output` mode=wait
- `triage-worker` - inspect `get_worker_state` + `get_scrollback` + `get_validation`
- `review-merge` - summarize validation + diff before merging

Required arguments (`worker`, `task`) are enforced - missing values
return `-32602 Missing required argument`.

## Logging

`logging/setLevel` accepts every syslog level:

```
debug | info | notice | warning | error | critical | alert | emergency
```

Invalid levels return `-32602`. The default level is `info` unless
overridden via `options.logLevel` or `config.mcp.logLevel`.

## allowedTools filter

```jsonc
// config.json
{
  "mcp": {
    "enabled": true,
    "allowedTools": ["list_workers", "read_output", "get_scrollback"]
  }
}
```

Effect:
- `tools/list` returns only the whitelisted entries
- `tools/call` of an excluded tool returns `-32602` with a message
  pointing at `config.mcp.allowedTools`

An empty array (`[]`) is treated as "allow all" so the out-of-the-box
config keeps the full surface exposed.

## Transports

### HTTP (existing)

`POST /mcp` on the daemon accepts a single JSON-RPC request and returns
the response. Auth-gated by `config.auth.enabled` just like every other
`/api/*` route (8.14).

### stdio (new)

`c4 mcp start [--base URL]` reads newline-delimited JSON-RPC from
stdin, POSTs each message to the daemon's `/mcp` endpoint (using the
saved `~/.c4-token` JWT when auth is enabled), and writes responses to
stdout. Notifications correctly produce no output. This is the
transport Claude Desktop uses.

### Claude Desktop registration

`~/Library/Application Support/Claude/claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "c4": {
      "command": "c4",
      "args": ["mcp", "start"]
    }
  }
}
```

Windows: `%APPDATA%\Claude\claude_desktop_config.json`.

### Streamable-HTTP / SSE

The daemon's existing `/events` SSE stream stays as-is for live worker
events. Wiring SSE as a formal MCP streamable-http endpoint is out of
scope for this patch - the HTTP POST transport plus stdio cover Claude
Desktop, claude.ai connectors, and direct curl usage.

## CLI

New `mcp` subcommand:

```
c4 mcp start [--base URL]   Stdio proxy (for Claude Desktop)
c4 mcp status               Probe the daemon and print negotiated version + serverInfo
c4 mcp tools                List exposed MCP tools
```

## Migration notes

Clients on the previous 2024-11-05 shim keep working:

- Known tool names (`create_worker`, `send_task`, `list_workers`,
  `read_output`, `close_worker`) are unchanged
- `initialize` still returns `protocolVersion`, `capabilities`,
  `serverInfo` - the version string is negotiated to whatever the client
  requested when it is in the supported set
- `tools/call` response shape (`content`, `isError`) is unchanged
- `notifications/initialized` sent as a request with `id` still returns
  `{}` for backwards compatibility; sent correctly as a notification (no
  `id`) now produces no response per JSON-RPC 2.0

Breaking change: sending notifications with no `id` no longer receives a
`{}` response. Clients that depended on that behaviour (rare - it was a
spec violation) need to treat notifications as fire-and-forget.
