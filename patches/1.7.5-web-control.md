# 1.7.5 - Web UI Worker Control Panel (8.8)

Branch: `c4/web-control`
Date: 2026-04-17

## Problem

TODO 8.8 required per-worker control in the Web UI: stop/close, pause
(Ctrl+C), resume, rollback, restart, cancel current task, and batch
control across multiple workers. Before this patch the Web UI only
exposed the reduced set in `WorkerActions.tsx` (Merge / Approve / Ctrl+C
/ Close) and had no bulk-close or bulk-cancel. There was also no daemon
surface for "cancel the pending/queued/in-flight task" or "restart this
worker on the same branch" - operators had to drop to the CLI for both.

## Design

### Daemon endpoints (backwards-compatible additions)

Two new POST routes on `src/daemon.js`:

| Route | Body | Manager call | Response |
|-------|------|--------------|----------|
| `/cancel` | `{name}` | `manager.cancelTask(name)` | `{success, kind: 'queued' \| 'pending' \| 'interrupt', name, task}` or `{error}` |
| `/restart` | `{name}` | `manager.restart(name)` | `{success, restarted:true, name, branch, worktree, target}` or `{error}` |

Both routes sit inside the existing `auth.checkRequest` gate (no new
auth surface) and reject a missing `name` with a 400. Every pre-existing
endpoint (`/close`, `/send`, `/key`, `/rollback`, `/merge`, `/approve`)
is left untouched so the change is fully backwards compatible - the CLI
and any downstream tooling keep working.

### `PtyManager.cancelTask(name)`

Three-branch resolver in priority order so the user never has to
reason about "which kind of task is pending" before clicking:

1. **Queued entry** - if `_taskQueue` has an entry matching the name,
   splice it, `_saveState`, return `{kind:'queued', task}`.
2. **Pending task on a worker that has not yet been flushed to the PTY**
   - clear `_pendingTask`, reset `_pendingTaskSent=false`, clear all
     three pending-task timers (`_pendingTaskTimer`,
     `_pendingTaskTimeoutTimer`, `_pendingTaskVerifyTimer`), return
     `{kind:'pending', task}`.
3. **Live worker** - write `\x03` to the PTY, clear `_taskText`,
   return `{kind:'interrupt', task}` (a worker that is alive but
   idle still gets the Ctrl+C as a best-effort interrupt).

Exited workers and unknown names both return a clear `{error}` so the
UI can surface the reason instead of silently no-oping.

### `PtyManager.restart(name)`

Unlike `close()`, restart deliberately **keeps the worktree and the
`c4/` branch alive** - "restart on same branch" means the branch stays
checked out on disk so the next task picks up exactly where the old
session left off.

Flow:

1. Snapshot `{branch, worktree, worktreeRepoRoot, target, parent,
   _startCommit, _autoWorker}` from the current worker.
2. Parse `command` + `args` out of the stored `worker.command` string;
   fall back to `claude` with no args when empty.
3. Clear every pending-task timer so they do not fire against the
   dead proc.
4. `proc.kill()` the old PTY if it was alive, end the raw log stream,
   delete the worker from the Map.
5. `this.create(name, command, args, {target, parent?, cwd:worktree?})`.
   Any `create()` error propagates unchanged and the workers Map stays
   clean (no half-dead entry).
6. Re-stamp `branch`, `worktree`, `worktreeRepoRoot`, `_startCommit`,
   `_autoWorker` onto the fresh worker record.
7. `_saveState` so the daemon-restart recovery path sees the new
   process right away.

### `web/src/components/ControlPanel.tsx`

Two stacked sections:

1. **Single-worker controls** - a grid of six labelled buttons:
   - Pause (send `C-c`)           - no confirm, instant
   - Resume (send `Enter`)        - no confirm, instant
   - Cancel task                  - confirm "Cancel the current or pending task for X?"
   - Restart                      - confirm "Restart X? The current process will be killed..."
   - Rollback                     - confirm "Rollback X to its start commit? Commits will be unstaged."
   - Stop / Close                 - confirm "Close X? Terminates the session and removes its worktree and branch."

   Colour tone per action - neutral (Pause / Resume), warn
   (Cancel / Restart), danger (Rollback / Close).

2. **Batch section** - `/api/list` polled every 5s, checkbox per
   worker, Select all / Clear helpers, plus two bulk buttons:
   - **Cancel selected** (warn, confirm-gated) - loops `/api/cancel`.
   - **Close selected** (danger, confirm-gated) - loops `/api/close`,
     clears the selection set on success.

   The last batch run renders inline as a per-name green-ok / red-error
   list, and a toast summarises `{ok}/{failed}`. Looping the existing
   per-worker endpoints keeps the daemon API minimal and reuses the
   existing permission checks.

All fetches route through the shared `apiFetch` wrapper from
`web/src/lib/api.ts` so the JWT from 8.14 attaches automatically
and the 401 handler still flips the app back to login.

### `web/src/App.tsx`

- `DetailMode` gets a third literal `'control'` alongside
  `'terminal'` and `'chat'`.
- `readDetailMode()` accepts the new value from `c4.detail.mode`
  `localStorage` so the tab selection round-trips across reloads.
- A third tab button is added to the detail-area tablist with
  `aria-selected={detailMode === 'control'}` and
  `setDetailMode('control')`.
- `<ControlPanel key={`control-${selectedWorker}`}` mounts when
  `detailMode === 'control'`; the per-worker React key guarantees
  state does not leak when the operator jumps between workers.

Existing Terminal and Chat tabs are untouched; switching between
modes remounts through the `term-*` / `chat-*` / `control-*` keys
pattern that 8.6 and 8.13 already established.

## Testing

`tests/web-control.test.js` - 26 assertions across 5 node:test suites:

1. **`PtyManager.cancelTask` (7 cases)** - missing name rejected;
   queued splice returns `kind='queued'` and actually removes the
   entry; pending-task branch clears `_pendingTask` + `_pendingTaskSent`
   + `_pendingTaskTimer`; in-flight interrupt writes exactly `'\x03'`
   to the fake PTY and clears `_taskText`; alive-but-idle still
   interrupts; exited worker returns `/exited/` error; unknown name

   happy path captures all snapshot fields, kills the old proc

   worktree cwd + parent options, restores branch / worktree /
   `_startCommit` / `_autoWorker` onto the new worker; `create()`
   error passthrough leaves the workers Map clean; empty command
   falls back to `claude` + `[]`.
3. **`daemon.js` source wiring (3 cases)** - regex-grep locks
   `route === '/cancel'` -> `manager.cancelTask(name)` and
   `route === '/restart'` -> `manager.restart(name)`, plus the
   `Missing name` guard in both blocks.

## Backwards compatibility

- Existing `/api/close`, `/api/send`, `/api/key`, `/api/rollback`,
  `/api/merge`, `/api/approve` routes are unchanged.
- `WorkerActions.tsx` (merge / approve / Ctrl+C / close) is untouched,
  so the existing buttons on the WorkerDetail view keep working.
- `PtyManager.close()` and `PtyManager.rollback()` are unchanged.
- The new `/cancel` and `/restart` endpoints are additive; no existing
  client needs to be updated.
- `c4.detail.mode` localStorage still accepts the previous values
  (`terminal`, `chat`); the new `control` value is handled by an
  explicit branch in `readDetailMode` and any legacy value falls
  back to `terminal`.
