# 1.11.1 - dep smoke check (TODO 8.16)

Closes the regression class where a branch declares a new runtime
dependency in `package.json` but main never gets `npm install` run
against the new lockfile, so the first consumer on main hits
`Cannot find module <newDep>`.

## Why this exists

8.14 ("Web UI session management + auth injection hardening") added
`bcryptjs` and `jsonwebtoken` to `package.json` + `package-lock.json`.
Pre-merge `npm test` passed because the worker's worktree had
`node_modules/` populated from an earlier install -- the lockfile
update and the on-disk `node_modules` were in sync inside the
worktree. After the merge to main, `npm install` was never rerun on
main. A user then ran `c4 init --user silofox` and the daemon crashed
with `Error: Cannot find module 'bcryptjs'`.

The existing pre-merge gate catches:

- Dirty tree on main (7.28).
- TODO.md / CHANGELOG.md not updated (1.11).
- `.c4-validation.json` missing or `test_passed=false` (9.9).
- `validation.test_count` disagreeing with `npm test` output (9.9).
- `package-lock.json` environment drift (7.29).

None of those fire when package.json gains a new dep, the worker
test run passes inside a pre-populated worktree, and main's own
`node_modules` is stale. Hence 8.16.

## What it does

Four new/edited pieces:

1. **`src/dep-smoke.js` (new).** Pure helpers, no network, no npm
   shell-out of their own:
   - `detectNewDeps(baseSha, headSha, repoRoot)` runs
     `git -C <repoRoot> diff-tree --no-commit-id --name-only -r
     <base> <head>` and short-circuits with
     `{hasChanges:false, reason:'unchanged'}` when `package.json` is
     not in the changed-files list. When it is, it `git show`s
     `package.json` at both SHAs, parses each side, and returns the
     newly added entries in `dependencies` + `devDependencies`
     separately. Version-only changes are deliberately ignored --
     the gate is about new names, not version bumps.
   - `verifyDepsLoadable(depNames, cwd)` spawns
     `node -e "require(<name>)"` per dep so (a) the caller's
     require-cache cannot mask a missing module, and (b) a broken
     package cannot crash the caller. Returns
     `{ok, failed:[{name, error}]}` with stderr captured for
     diagnostics.
   - `runCheck({baseSha, headSha, repoRoot, cwd, afterInstall})` is
     the detect -> install -> verify glue, kept thin so the CLI
     merge flow owns the `npm ci` call (it needs to surface
     progress, honor `--skip-checks`, and clean up on failure).
   - `formatFailure(result)` builds the operator-facing multi-line
     message ("Missing / unloadable dependencies: ... Run `npm ci`
     in the merge target before retrying.").
   - `parsePkgJson` / `depMap` / `diffDepMaps` are exported so the
     unit tests can exercise pieces in isolation.

2. **`src/validation.js` -- `checkPackageDepsInstalled(opts)`.** Wraps
   the detect + install + verify flow as a merge-gate compatible
   function. Takes `{baseSha, headSha, repoRoot, skipInstall?,
   includeDev?}`. Returns
   `{ok, skipped, reason, detail, detect, prod, dev}` -- never
   throws. Special reasons: `missing-args` (inputs not provided,
   ok:true skipped), `package-json-unchanged`, `no-new-deps`,
   `invalid-package-json`, `npm-ci-failed`, `require-failed`, `ok`.
   `skipInstall:true` exists for unit tests; production callers omit
   it. `includeDev:false` is the default because a missing
   `devDependencies` entry does not break a runtime consumer.

3. **`src/cli.js` -- merge flow Check 4.** After the CHANGELOG.md
   check, computes:
   ```
   baseSha = git merge-base main <branch>
   headSha = git rev-parse <branch>
   ```
   and calls `validationLib.checkPackageDepsInstalled`. Output:
   ```
   [check] package-deps-installed ... PASS (verified: bcryptjs)
   [check] package-deps-installed ... SKIP (package-json-unchanged)
   [check] package-deps-installed ... FAIL
      require-failed
      Missing / unloadable dependencies:
        - bcryptjs: Error: Cannot find module 'bcryptjs'
      Run `npm ci` in the merge target before retrying.
   ```
   FAIL gates `allPassed` so the merge is rejected before `git
   merge` runs. Passes go through to the existing merge block
   unchanged.

4. **`tests/dep-smoke.test.js` (new, 35 assertions / 9 suites).**
   Uses tmpdir git fixtures (`mkTmpRepo` + `writePkg` + `commitAll`)
   so `detectNewDeps` runs against real commits rather than mocks.
   `verifyDepsLoadable` is exercised through the real
   `node -e "require(x)"` path with `path` (built-in, always passes)
   + a synthetic fake name (always fails), with a stub-spawn variant
   covering the call-count contract. The
   `checkPackageDepsInstalled` suite uses `skipInstall:true` to
   avoid a live `npm ci` in the CI run (the flag is the sole reason
   it exists) and the require-failed branch is driven by a fake dep
   name that is guaranteed not to resolve. Final suite is
   source-wiring greps so silent rename / drop cannot bypass the
   gate.

## Where it can still miss

- A branch that ships a new dep *and* the consumer never uses the
  code path that requires it. `require()` has to actually execute
  for the probe to fire, which is the same constraint Node itself
  imposes. This is acceptable: a dep that nothing imports is not
  actually a new dep.
- `optionalDependencies` are not probed. They are opt-in by
  definition.
- Installed-but-broken transitive deps (C++ addon build failure
  after `npm ci`) can still slip through when the root-level
  require works but a deeper `require()` in the addon chain throws
  later. Out of scope for 8.16; handled by the existing
  `install-verify` runbook (8.11) for full fresh-install
  reproducibility.

## Opting out

`c4 merge <name> --skip-checks` keeps bypassing every pre-merge
check, 8.16 included. This is unchanged from the existing escape
hatch (docs-only commits etc.); nothing about 8.16 asks for a
separate override because the gate already no-ops when
`package.json` did not change.
