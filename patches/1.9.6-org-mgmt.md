# Patch 1.9.6 ā€” Department / team management (10.6)

## Summary

Add an organizational layer on top of RBAC (10.1), cost report (10.5),
and project management (10.8). Departments own projects, machines, and
worker quotas; teams group users under a department. Operators can ask
"which budget does user X charge against?" and "has team Y blown its
monthly token limit?" without stitching three modules together by hand.

## Motivation

10.1 gives us user -> role -> permission. 10.8 gives us projects. 10.5
gives us per-project / per-user cost. The missing piece is the
organizational container: a multi-team workspace needs to see budgets
and worker counts rolled up to a department, not to a single user or a
single project. That roll-up is 10.6.

## What changed

### New module: `src/org-mgmt.js`

Pure storage + aggregation. Writes to `~/.c4/org.json` by default
(`config.org.path` override). No shell-outs, no direct daemon coupling,
and `getQuotaUsage(deptId, ctx)` accepts an injected `costReporter` +
`workers` so unit tests run in-process.

```js
const org = new OrgManager({ storePath: '~/.c4/org.json' });
org.createDepartment({ id: 'eng', name: 'Engineering' });
org.createDepartment({ id: 'platform', parentId: 'eng' });
org.addMember('eng', 'alice', 'manager');
org.createTeam({ id: 'sre', deptId: 'platform', name: 'SRE' });
org.assignMember('sre', 'bob');
org.setQuota('eng', { maxWorkers: 20, monthlyBudgetUSD: 500, tokenLimit: 5_000_000 });

const usage = org.getQuotaUsage('eng', {
  costReporter,  // CostReporter from 10.5
  workers,       // manager.list().workers snapshot
});
// -> { usage: { workers, costUSD, tokens }, percent: {...}, exceeded: {...} }
```

Schema: `{ departments: {<id>: Department}, teams: {<id>: Team} }`
where
```
Department = {
  id, name, parentId | null,
  managerUserIds: [username...],
  memberUserIds:  [username...],
  projectIds:     [projectId...],
  machineAliases: [alias...],
  quotas: { maxWorkers, monthlyBudgetUSD, tokenLimit }
}
Team = { id, deptId, name, memberUserIds: [username...] }
```

### Tree + resolution helpers

`treeView()` returns a nested `[{ dept, subdepts, teams, members }]`
rooted at every `parentId === null` department. Each node's `members`
is the deduped union of `dept.memberUserIds` plus every
`team.memberUserIds` under it, so the UI renders a single roster.

`resolveUserDept(userId)` walks team membership first (most specific),
falls back to direct department membership, and tie-breaks by depth so
`"nearest department"` matches the spec even when a user appears at
multiple levels. Cycle-safe.

### New RBAC actions: `org.read` + `org.manage`

Added to `ACTIONS` in `src/rbac.js` and to the `DEFAULT_PERMISSIONS`
matrix: `manager` gets both, `viewer` gets `org.read`, `admin` already
has `*`. Existing operators pick up the new permissions automatically
when `checkPermission(username, 'org.manage')` runs ā€” no migration of
the on-disk `~/.c4/rbac.json` is required because the role matrix lives
in code.

### Daemon endpoints (`src/daemon.js`)

All protected behind the new actions. Org paths decoded up-front like
the existing `/projects/<id>/...` handlers so the dispatch is a flat
string match.

| Method | Route | Action |
|--------|-------|--------|
| `GET`  | `/orgs/tree` | `org.read` |
| `POST` | `/orgs/dept` | `org.manage` |
| `POST` | `/orgs/dept/:id/member` | `org.manage` |
| `POST` | `/orgs/team` | `org.manage` |
| `POST` | `/orgs/team/:id/member` | `org.manage` |
| `POST` | `/orgs/dept/:id/quota` | `org.manage` |
| `GET`  | `/orgs/dept/:id/usage` | `org.read` |

`/orgs/dept/:id/usage` joins a freshly built `CostReporter` (so a
`c4 config reload` flips the cost table) and `manager.list().workers`
so the response is a live snapshot, not a pre-rendered value.

The shared `_orgManager` singleton is reset on `POST /config/reload`
alongside `_projectBoard` and `_projectDashboard`, so a new
`config.org.path` is picked up without a daemon restart.

### CLI (`src/cli.js`)

```
c4 org tree
c4 org dept create --id ID --name N [--parent PID]
c4 org dept member add <deptId> <userId> [--role manager]
c4 org team create --id ID --dept DEPTID --name N
c4 org team member add <teamId> <userId>
c4 org quota set <deptId> [--max-workers N] [--budget USD] [--tokens N]
c4 org usage <deptId>
```

Thin wrapper over the daemon endpoints ā€” no local state, so operators
can run these from any machine that can reach the daemon.

### Tests (`tests/org-mgmt.test.js`)

30+ assertions covering:

- Helpers: `defaultOrgPath`, `isId`, `isUsername`, `normalizeQuotas`,
  `normalizeDept`, `normalizeTeam`, `ensureShape` (drops bad ids /
  orphan teams).
- Department CRUD: id validation, duplicate rejection, parent check,
  sub-department roundtrip.
- `addMember` / `removeMember`: manager role also registers as member,
  idempotent adds, bad userId rejection.
- Team CRUD + `assignMember`: dept-existence check, membership
  propagation to the parent dept, filtered `listTeams`.
- `setQuota`: partial update preserves fields the caller did not set.
- `treeView`: root discovery, depth, team + member roll-up, multi-root
  ordering. `parentChain` walks from root to leaf.
- `resolveUserDept`: null for unknown user; team membership wins;
  deepest hit wins when user appears at multiple levels.
- `getQuotaUsage`: cost aggregated only over dept members, active
  workers counted by user / project / machine, `exceeded` flag flips
  at the limit, zero-limit case, unknown-dept throw.
- Storage roundtrip: second OrgManager on the same file sees every
  write; `reload()` re-reads after an external edit; missing file and
  malformed JSON fall back to fresh state without crashing.

Also extended `tests/rbac.test.js` to cover the two new action
constants and the updated `ALL_ACTIONS.length = 16`.
