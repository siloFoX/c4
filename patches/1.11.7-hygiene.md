# Patch 1.11.7 — v8.1 hygiene pass

## Summary

A small follow-up to the v7 autonomous run. Two unrelated correctness
issues were spotted in the v7 changeset and are addressed here.

1. `src/merge-core.js` `runPreMergeChecks` built the worktree lookup
   path with a regex that left `/` intact, so for any branch carrying a
   slash (which is every worker branch by convention) the resulting
   path never existed on disk and the downstream `validation.test_passed`
   check silently degraded to `SKIP`.
2. `TODO.md` carried a duplicate `8.16` row — one **done** entry
   recording the actual implementation, and one **todo** entry that
   was the pre-implementation incident note. The stale copy has been
   removed.

No behavioural API changes; this is a hygiene pass.

## The bug

### Before

```js
const worktreePath = path.resolve(
  repoRoot,
  '..',
  'c4-worktree-' + branch.replace(/[^A-Za-z0-9._/-]/g, '-')
);
```

The negated character class `[^A-Za-z0-9._/-]` allowed `/` through.
For `branch = 'c4/slack-events'` the result was

```
<repoRoot>/../c4-worktree-c4/slack-events
```

which `path.resolve` collapsed to

```
<parentOfRepo>/c4-worktree-c4/slack-events
```

— a path that never existed because the actual worktree on disk is
`<parentOfRepo>/c4-worktree-c4-slack-events` (created by the worker
spawn flow, which sanitizes branch names to hyphens). `fs.existsSync`
therefore returned `false`, `validationObj` stayed `null`, and the
check produced

```
validation.test_passed   SKIP   no worktree for branch
```

for every worker branch in the repo. The pre-merge gate was effectively
disabled for the very check it was added to enforce.

### After

```js
const worktreePath = path.resolve(
  repoRoot,
  '..',
  'c4-worktree-' + branch.replace(/[^A-Za-z0-9._-]/g, '-')
);
```

`/` is no longer in the allowed set, so it is rewritten to `-`. The
resulting path now matches the on-disk worktree convention and the
validation check runs as intended.

### Audit of related code

`resolveBranchForWorker` (further down `src/merge-core.js`) was
inspected for the same defect. It builds its lookup path from the
worker name rather than the branch name and applies no regex, so it
does not share the bug and was not modified.

## Tests

`tests/merge-core.test.js` adds six assertions in one `node:test`
suite. All cases run as pure path computation — no filesystem touch
required, which keeps the test fast and portable across platforms.

| # | Case | Expected output |
|---|---|---|
| 1 | plain branch name `hygiene-v8` | `<parent>/c4-worktree-hygiene-v8` |
| 2 | single-slash branch `c4/slack-events` (regression) | `<parent>/c4-worktree-c4-slack-events` |
| 3 | multi-segment branch `feature/sub/leaf` | `<parent>/c4-worktree-feature-sub-leaf` |
| 4 | dot/underscore/hyphen preserved | `<parent>/c4-worktree-c4-v1.2_alpha-rc` |
| 5 | every other special char hyphenated | `<parent>/c4-worktree-feat-foo-bar-baz-1` |
| 6 | source-grep guard | `src/merge-core.js` contains the fixed regex and not the buggy one |

The grep guard is belt-and-suspenders: a future refactor cannot
quietly reintroduce `/` into the character class without case 6
flagging it.

## Why we care

- The pre-merge gate exists to keep red branches from landing on
  `main`. A SKIP-by-default is indistinguishable from a PASS in the
  CLI summary, so the gate had been silently no-op for any worker
  branch — which is to say, every branch the gate was designed to
  guard.
- Duplicate TODO rows make it easy to lose track of which entry is
  authoritative; the canonical "done" row is what hygiene tooling
  (and humans) should be reading.
