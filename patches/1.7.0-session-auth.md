# 1.7.0 — Web UI session auth (8.14)

## Problem

Before 1.7.0 the C4 daemon and Web UI shipped with **no authentication**.
Any HTTP client that could reach `http://<host>:3456/api/*` could:

- spawn workers (`POST /create`)
- send arbitrary tasks (`POST /task`) — including full Claude prompts
- approve TUI prompts (`POST /approve`, `POST /key` with Enter)
- merge worker branches into `main` (`POST /merge`)
- trigger `git push` via tasks
- read worker scrollback / PTY screens (`GET /scrollback`, `GET /read-now`)

In practice this made port-forwarding (or any LAN exposure via 8.10's
`bindHost: 0.0.0.0`) a direct RCE / injection vector: a remote client
just had to craft a `/task` body to pivot into the host shell.

2026-04-17 after a user flagged the risk) made it the blocker for all
external binding.

## What shipped in 1.7.0

- `src/auth.js` — bcryptjs password hashing, jsonwebtoken HS256 signing
  (24h expiry), `checkRequest(cfg, req, route)` HTTP middleware with a
  single open-route set (`/auth/login`, `/health`). Also accepts a
  `?token=` query param so EventSource (which cannot set custom
  headers) can still authenticate SSE streams.
- `src/auth-setup.js` — `provisionAuth()` for `c4 init`: generates
  `auth.secret` on first run only, bcrypt-hashes the password, stores
  only the hash, and never rewrites the source password file.
- `src/daemon.js` — middleware wired in before every `/api/*` handler;
  new `POST /auth/login`, `POST /auth/logout`, `GET /auth/status`.
- `src/cli.js` — attaches `Authorization: Bearer <token>` from
  `$C4_TOKEN` or `~/.c4-token` so existing `c4` commands keep working.
- Web UI:
  - `web/src/lib/api.ts` — central `apiFetch` wrapper, localStorage
    token, `AUTH_EVENT` on 401, `eventSourceUrl` for SSE.
  - `web/src/components/Login.tsx` — sign-in form.
  - `App.tsx` — gated rendering (`loading` / `anon` / `authed` /
    `disabled`), sign-out button.
  - All existing components migrated from raw `fetch()` to `apiFetch`.
- `c4 init` modes:
  - non-interactive: `c4 init --user <name> --password-file <path>`
  - interactive: TTY prompts for user + password (silent password).
- `package.json` bumped to `1.7.0`; adds `bcryptjs` + `jsonwebtoken`.

## Config schema addition

```json
{
  "auth": {
    "enabled": true,
    "secret": "<96 hex chars>",
    "users": {
      "admin": { "passwordHash": "$2a$10$..." }
    }
  }
}
```

When `auth` is absent or `auth.enabled` is `false`, the daemon behaves
exactly as it did pre-1.7.0 (no auth, open API). Once an operator runs
`c4 init --user ... --password-file ...`, `enabled` flips to `true` and
every `/api/*` route requires a valid JWT.

## Verification

From the 8.14 brief:

- (a) login success + failure (wrong password / unknown user / missing
  fields / missing secret) → `tests/session-auth.test.js`
- (b) middleware allows `/auth/login` without a token → same file
- (c) middleware rejects `/api/*` without a token (and with malformed /
  tampered tokens) → same file
- (d) `c4 init --user + --password-file` writes a bcrypt hash without
  touching the source file → same file (+ reuse-secret-across-runs and
  skip-existing-user assertions)

Plus a source-grep guard over `src/daemon.js` to lock the wiring in:
`require('./auth')`, `route === '/auth/login'`, `auth.checkRequest(`.

Full suite: 66 / 66 pass.

## Operational notes

- Existing installs with no `auth` block keep working unchanged.
- Turning auth on is a two-step migration:
  1. `c4 init --user <you> --password-file ./pw.txt`
  2. `c4 daemon restart`
  Then either `export C4_TOKEN=...` or let the CLI read `~/.c4-token`.
- Web UI logins persist in `localStorage` under `c4.authToken` and
  expire after 24h (JWT TTL). The frontend clears the token and
  redirects to the login form on any 401.
- `bindHost: 0.0.0.0` from 8.10 is now safe to enable *only if* auth is
  provisioned. Leaving `auth.enabled` false while binding externally is
  still the same RCE exposure as before - 8.14 did not block that
  combination automatically; that guard belongs in a follow-up.
