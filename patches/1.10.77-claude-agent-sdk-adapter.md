## Why dependency injection instead of `require('@anthropic-ai/...')`

Three reasons:

1. **The SDK iterates rapidly.** Hard-pinning a version in C4's
   package.json would force C4 releases on every SDK release.
   `peerDependencies` doesn't help — operators who don't use the
   SDK shouldn't have to add it to their dependency tree just to
   satisfy a peer warning.

2. **The SDK has its own setup.** API key env vars, MCP server
   registries, tool definitions, retry policies. Operators
   already know how to wire it. C4 just needs the protocol — give
   me a function, I'll call it with the prompt.

3. **DI keeps the door open.** Some operators may want to plug a
   different SDK with the same shape (an OpenAI Assistants port,
   an Aider library, etc.). The adapter doesn't care which SDK
   the queryFn delegates to as long as it returns an
   AsyncIterable of `{type, text}` events.

## queryFn contract

```ts
type QueryFn = (
  prompt: string,
  opts: {
    model?: string;
    systemPrompt?: string;
    signal?: AbortSignal;
  }
) => Promise<AsyncIterable<{
  type: 'text' | 'tool_use' | 'error';
  text?: string;
}>>;
```

`signal` is the adapter's AbortController — passed in so the
operator's queryFn can wire it through to the SDK's internal HTTP
client. When `dispose()` is called, the signal aborts and the
queryFn should bail.

`tool_use` events are ignored by the scaffold. Operators who need
tool dispatch (run a tool, feed the result back into the next
turn) subclass `ClaudeAgentSdkAdapter` and override `runQuery` or
intercept the event in their queryFn wrapper.

## Wiring pattern

```js
const { query } = require('@anthropic-ai/claude-agent-sdk');
const a = createAdapter({
  type: 'claude-agent-sdk',
  options: {
    model: 'claude-opus-4-7',
    systemPrompt: 'You are a senior backend engineer.',
  },
});
a.queryFn = (prompt, opts) => query({
  prompt,
  model: opts.model,
  systemPrompt: opts.systemPrompt,
  signal: opts.signal,
});
```

Operator wires this once in their daemon bootstrap (e.g. a
`scripts/wire-agent-sdk.js`) before constructing PtyManager.

## Conservative behaviour

- **Errors surface in-band** through `onOutput` with `[claude-
  agent-sdk] error: <msg>\n` prefix. No throws leak. Same pattern
  as `LocalLLMAdapter` — a stuck inference never pins the
  adapter.
- **Concurrent `runQuery` rejected** with a busy-guard error
  (also in-band). Sequential queries via the same adapter are
  fine; parallel calls are operator error.
- **detectIdle** returns `true` only when `_idle === true && !_
  busy`. Initial state is `false`; first successful runQuery
  flips `_idle` to `true`. Conservative — same reasoning as
  CodexAdapter's detectIdle, false-positive idle would silently
  declare tasks done.
- **dispose()** aborts in-flight queries via AbortController,
  clears handler list, safe to call repeatedly.

## metadata.model

Production audit / snapshot consumers want to know which model an
adapter is pointed at without reaching into options. The metadata
object carries `{ name: 'claude-agent-sdk', version, model }` so
`adapter.metadata.model` is enough to identify the runtime config.
