# 1.7.8 - Multi-machine fleet management (TODO 9.6)

## Summary

C4 used to assume one daemon per user. This patch introduces a
first-class fleet model so a single CLI install can target 40 + DGX +
15 peers on the LAN, each running its own daemon.

- `~/.c4/fleet.json` stores machines `{ alias: { host, port, authToken? } }`
- `~/.c4/fleet.current` (or `C4_FLEET` env) pins subsequent CLI commands
  to a specific alias
- Daemon exposes `GET /fleet/overview` that samples every registered
  peer in parallel
- CLI gains `c4 fleet <add|list|remove|use|current|status>`

No central broker. Each machine keeps running its own daemon; the CLI
on each host maintains its own registry of remote aliases.

## Setup walkthrough

### 1. Register a peer from the admin machine

```bash
c4 fleet add dgx 192.168.10.222 --port 3456
c4 fleet add build 192.168.10.50 --port 3456 --token eyJhbGciOi...
c4 fleet list
```

Output:

```
ALIAS           HOST                    PORT    TOKEN   PINNED
build           192.168.10.50           3456    yes
dgx             192.168.10.222          3456    no
```

### 2. Pin a peer for the current shell

```bash
c4 fleet use dgx
# => [ok] fleet: pinned dgx -> http://192.168.10.222:3456

c4 list            # runs against the DGX daemon, not localhost
c4 new mle --target local
```

The `request()` helper in `src/cli.js` reads `~/.c4/fleet.current` at
CLI start via `resolveBase()`, so every subcommand picks up the pin
without per-call flags. `readToken()` pulls the per-machine token from
`fleet.json` before falling back to `~/.c4-token`.

### 3. Unpin

```bash
c4 fleet use --clear
# => [ok] fleet: pin cleared (back to local daemon)
```

### 4. Aggregated health view

```bash
c4 fleet status
```

Output:

```
SELF:
  127.0.0.1:3456  workers=2  version=1.7.0

REMOTES:
  ALIAS   HOST            PORT    OK      WORKERS VERSION ELAPSED
  build   192.168.10.50   3456    yes     1       1.7.0   42ms
  dgx     192.168.10.222  3456    yes     5       1.7.0   51ms

Total: 8 workers across 3/3 reachable machines
```

Pass `--timeout 5000` to raise the per-machine timeout.

## Daemon endpoint

`GET /fleet/overview` sits inside the existing `auth.checkRequest` gate
(no new auth surface). Envelope:

```json
{
  "self": {
    "ok": true,
    "alias": "_self",
    "host": "127.0.0.1",
    "port": 3456,
    "workers": 2,
    "version": "1.7.0"
  },
  "machines": [
    { "alias": "dgx", "host": "192.168.10.222", "port": 3456,
      "ok": true, "workers": 5, "version": "1.7.0",
      "error": null, "elapsedMs": 51 },
    { "alias": "build", "host": "192.168.10.50", "port": 3456,
      "ok": false, "workers": null, "version": null,
      "error": "ECONNREFUSED", "elapsedMs": 3002 }
  ],
  "total": { "machines": 3, "reachable": 2, "workers": 7 },
  "generatedAt": "2026-04-17T18:42:00.000Z"
}
```

Query params:

- `timeout=<ms>` - override the default 3000 ms per-machine timeout

Each machine is probed in parallel (`/health` + `/list`). The
aggregator never rejects on transport errors, so one unreachable peer
never hides the rest.

## JWT handling

The fleet reuses the JWT from 8.14. When `c4 fleet add` receives a
`--token` the token is stored per-machine in `fleet.json`. At call time
`getPinnedBase()` resolves the token with this precedence:

1. `machine.authToken` (per-peer JWT)
2. `C4_TOKEN` env
3. `~/.c4-token` file

`sampleMachine` forwards the token as `Authorization: Bearer <jwt>` to
`/health` + `/list` so a peer daemon with `auth.enabled=true` can
authenticate the caller just like a local `c4 list`.

## Security notes

- JWTs in `fleet.json` are a convenience for scripting. File lives
  under `~/.c4/` (home-only permissions inherited from the OS). If
  you'd rather not persist tokens, omit `--token` on `c4 fleet add`
  and rely on `C4_TOKEN` env + `~/.c4-token`.
- `/fleet/overview` is auth-gated. A public daemon still requires a
  valid JWT before enumerating peers.
- Timeout is honored but capped by the Node `http.request` timeout, so
  a malicious caller cannot starve the daemon with a huge `?timeout=`.

## Limitations (what 9.6 does NOT do)

- No daemon-to-daemon state sync. Overview is a pull poll, not a push.
- No cross-peer worker dispatch. Piping a `c4 task` through a pinned
  alias forwards the task body unchanged, so the remote daemon's
  `projectRoot` / worktree config is what actually runs. Explicit
  `--repo` or `--cwd` on a path that exists on the remote is the
  recommended pattern until 9.7 (Dispatcher) lands.
- Pin is a single alias. For rapid peer rotation in the same shell use
  `C4_FLEET=<alias> c4 <cmd>` inline; there is no multi-pin queue.
