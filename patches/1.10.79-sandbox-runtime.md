# 1.10.79 — SandboxRuntime + DockerRuntime command builder (11.5 Stage 2 first cut)

First cut of 11.5 Stage 2 (execution-isolation framework).

## Why builder-first

Shadow execution of risky commands is itself risky:

1. **Container escapes exist and get found.** Even with `--cap-
   drop=ALL --user=nobody --security-opt=no-new-privileges`, a
   sufficiently motivated payload + a freshly-disclosed CVE can
   still bridge to host. C4 should not be the new attack surface
   that drives operators into containers they didn't ask for.

2. **Resource exhaustion can affect the host even with cgroup
   limits.** `--memory=128m --cpus=0.5 --pids-limit=64
   --tmpfs=/tmp:rw,size=64m` is the right bound, but a
   misconfigured runtime (operator drops `--memory` because they
   want room for a build) can fork-bomb the host.

3. **The classifier sometimes flags benign commands.** Running
   them in a sandbox just to "verify" intent burns cycles for no
   information — Stage 1's static `extractIntent()` already
   tells you what files / network peers / privileges the command
   would touch.

4. **Some risky commands are dangerous BECAUSE of side
   effects.** `rm -rf /` in the sandbox doesn't damage the host;
   shadow-running it doesn't add information beyond "it would
   delete files" — which the classifier already flagged at
   Stage 1.

Shipping the builder cleanly separates "what command would the
sandbox run?" (testable, useful) from "should we actually run
it?" (policy decision). The first informs the second.

## Runtime interface

```js
class SandboxRuntime {
  available(): { ok: boolean, reason?: string }
  describeIsolation(): { name, network, filesystem, resources }
  prepareArgs(command, opts?): {
    binary,    // 'docker', null for NullRuntime
    args,      // argv to pass binary
    env,
    command,   // echoed verbatim for the audit
    isolation, // copy of describeIsolation()
  }
}
```

Three implementations ship today: `SandboxRuntime` (abstract),
`NullRuntime` (no isolation, host everything), `DockerRuntime`
(full hardening).

## DockerRuntime defaults

```
image:          alpine:latest        (~5MB, common)
network:        'none'               (no egress)
--read-only:    true
--tmpfs=/tmp:   rw, size=64m        (writable scratch)
--memory:       128m
--cpus:         0.5
--pids-limit:   64                   (cap fork bombs)
--user:         nobody               (no root in container)
--security-opt: no-new-privileges
--cap-drop:     ALL
timeoutMs:      5000                 (host-side kill)
```

Operators override per call: `image / network / memory / cpus /
timeoutMs / mounts / env`. Mounts are off by default — the
read-only root + tmpfs combo is enough for "what does this do"
probes. Operators who want to mount their own workspace pass
`opts.mounts: [{src, dst, ro}]`.

`docker run` argv is built verbatim under `sh -c` so chains like
`cmd1 && cmd2 || cmd3` survive without argv splitting issues.

## CLI surface

```sh
$ c4 risk "rm -rf /tmp/test" --sandbox-preview docker
Level:    HIGH
Action:   review
Reasons:
  - [rm-rf-dir] rm -rf <directory>
Intent:
  dest:   rm /tmp/test

Sandbox runtime: docker
  available: true
  isolation: network=none, fs=read-only root + tmpfs /tmp (64m)
             memory=128m cpus=0.5 pids=64 timeout=5000ms
  command:
    docker run --rm --network=none --memory=128m --cpus=0.5 \
      --pids-limit=64 --read-only --tmpfs=/tmp:rw,size=64m \
      --user=nobody --security-opt=no-new-privileges --cap-drop=ALL \
      alpine:latest sh -c 'rm -rf /tmp/test'
```

## Bug fixed during rollout

The first cut of the positional filter
`args.filter((a, i) => !a.startsWith('--') && i !== (spIdx+1))`
also dropped `args[0]` when `--sandbox-preview` was absent
(because `spIdx === -1` makes `spIdx+1 === 0`). Caught by the
existing cli-risk cases. Fixed with a `spIdx >= 0` guard before
applying the filter.

## Pending follow-ups under 11.5 Stage 2

1. **Shadow execution path** — run the prepared argv, capture
   stdout/stderr/exit code, surface as audit event. The
   builder is the input; this is the output. Deliberately
   separate cut so the runtime interface settles first.
2. **Runtime config in config.json** — `riskClassifier.sandbox.
   {name, opts}` so an operator can pin the daemon to
   "shadow-mode by default" once the exec path lands.
3. **Audit chain integration** — new `risk.shadow_exec` event
   type, mirror to scribe-v2 timeline.

## Backwards compatibility

None broken. Pure addition (new module + new CLI flag). The
classification path of `c4 risk` is unchanged when
`--sandbox-preview` is absent.
