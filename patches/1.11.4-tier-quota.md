# 1.11.4 — Tier-based daily token quota + complexity-driven model selection (8.3)

## Why

Token spend was tracked globally (`config.tokenMonitor.dailyLimit`) but had
no way to bound a single class of worker. An autonomous manager session
that fanned out 30 workers could drain the daily budget in minutes, and
there was no mechanism to say "managers can spend 5x what a worker spends,
but only on Opus." Operators also kept hand-picking models with `--model`
when the right size was usually obvious from the task text (`design X` is
a planning task, `rename foo` is a one-line edit). 8.3 closes both gaps
with a thin layer that sits next to the existing token monitor rather
than replacing it.

## What lands

### `src/tier-quota.js` (new, pure module)

- `DEFAULT_TIERS` — three tiers with defaults:
  - `manager` — 500_000 tokens / `[opus]`
  - `mid`     — 200_000 tokens / `[opus, sonnet]`
  - `worker`  — 100_000 tokens / `[sonnet, haiku]`
- `mergeTiers(override)` — overlays `config.tierConfig` without losing
  untouched fields. Setting `worker.dailyTokens=50000` keeps the
  `[sonnet, haiku]` allow-list intact.
- `selectModel(taskDescription, tier, opts?)` — keyword + length heuristic:
  - opus keywords: `design`, `plan`, `architect`, `refactor`,
    `investigate`, `analyze`, `audit`, `spec`, `rfc`
  - haiku keywords: `typo`, `rename`, `format`, `lint`, `comment`,
    `docstring`, `whitespace`, `simple edit`, `spelling`
  - sonnet keywords: `implement`, `fix`, `add`, `update`, `write`,
    `create`, `build`, `wire`, `integrate`, `patch`
  - length fallback: >500 chars -> opus, <80 chars -> haiku, else sonnet
  - the wanted score is then constrained to the tier allow-list with a
    sensible nearest-allowed fallback (worker-tier "design" -> sonnet,
    manager-tier "rename" -> opus)
- `class TierQuota({baseDir, tiers, now})`:
  - `chargeTier(tier, tokens)` — increments + persists, throws
    `Error{code:'QUOTA_EXCEEDED', tier, used, requested, limit}` when
    the next charge would cross the cap. Failed charge does NOT advance
    the counter.
  - `getUsage(tier)` / `getRemaining(tier)` — the latter returns
    `Infinity` when `dailyTokens=0` (= unlimited).
  - `resetDaily(tier?)` — zeroes one tier or all.
  - `_rolloverIfNeeded()` — auto-loads a new file when the injected
    clock crosses midnight UTC. Day 1 totals stay intact in the
    original file; day 2 charges land in a brand-new file.
  - `snapshot()` — full read-only view:
    `{date, tiers:{tier:{dailyTokens, models, used, remaining}}}`.
    `remaining = -1` for unlimited tiers (so JSON consumers do not have
    to special-case `Infinity`).
  - `selectModel(task, tier)` — delegates to the module function with
    the instance tier override.
- Persistence shape: `~/.c4/tier-quota-YYYY-MM-DD.json` =
  `{"date":"YYYY-MM-DD","tiers":{"manager":N,"mid":N,"worker":N}}`.

### Daemon wiring (`src/daemon.js`)

- New module-level instance:
  `tierQuota = tierQuotaMod.getShared({tiers: cfg.tierConfig, force:true})`
  + `tierWorkerMap` (worker name -> tier) so `/task` can resolve the
  worker's tier without depending on pty-manager-internal worker fields.
- `POST /create` parses `{tier}` (default `'worker'`). Unknown tier
  returns `400 {error, allowed[]}`. On success the tier lands in the
  audit record + Slack `worker_spawn` event + `result.tier`.
- `POST /task` parses `{tier, model}`:
  - tier resolution: explicit `tier` > `tierWorkerMap.get(name)` > default
    `'worker'`.
  - quota gate: `tierQuota.getRemaining(tier) === 0` returns
    `429 {error, tier, remaining:0}` before touching `manager.sendTask`.
  - model resolution: `model === 'auto'` (or omitted) runs through
    `tierQuota.selectModel(task, tier)`; an explicit model passes
    through unchanged so operators can still pin a specific size.
  - tier + model land in the audit record + Slack `task_start` event +
    `result.tier` / `result.model`.
- `GET /quota` -> full snapshot. `GET /quota/:tier` -> one tier's
  slice (`{date, tier, dailyTokens, models, used, remaining}` or
  `{error, allowed[]}` on unknown).
- `POST /config/reload` calls `tierQuota.setTiers(newCfg.tierConfig)`
  so live edits of daily caps or model allow-lists take effect on the
  next dispatch without a daemon restart.

### CLI wiring (`src/cli.js`)

- `c4 new <name> [--tier manager|mid|worker]` — defaults to worker
  server-side when omitted.
- `c4 task <name> "task" [--tier T] [--model auto|opus|sonnet|haiku]` —
  default `--model auto` resolves through `tierQuota.selectModel`.
- `c4 quota [tier]` — pretty-prints either the full table:

  ```
  Tier quota (2026-04-18):
    manager  used=       0 / 500,000 (remaining=500,000) models=[opus]
    mid      used=       0 / 200,000 (remaining=200,000) models=[opus, sonnet]
    worker   used=       0 / 100,000 (remaining=100,000) models=[sonnet, haiku]
  ```

  or a single-tier detail block (date, daily limit, models, used,
  remaining; "unlimited" rendered when `dailyTokens=0`).
- Help text gains a `quota [tier]` line right under `token-usage`.

### Tests (`tests/tier-quota.test.js`)

23 node:test assertions across 6 suites. Every suite uses a fresh
`tmpdir()` as `baseDir` so the persisted JSON file is real but isolated,
and `now` is injected so the daily roll-over test can fast-forward time
without monkey-patching `Date`.

1. **defaults + mergeTiers (3 tests)** — the three required tier names
   exist with the spec'd defaults; `mergeTiers` overlays without losing
   untouched fields; non-object overrides ignored safely.
2. **chargeTier + getRemaining (5 tests)** — zero usage at start;
   `chargeTier` accumulates and persists across instances; canonical
   `~/.c4/tier-quota-YYYY-MM-DD.json` shape; unknown-tier rejects on
   every method; negative / non-numeric token amounts reject.
3. **quota exceeded reject (3 tests)** — `chargeTier` throws
   `QUOTA_EXCEEDED` with `tier`/`limit` populated and the failed charge
   does NOT advance the counter; cap exactly hit -> remaining 0 +
   subsequent +1 rejects; `dailyTokens=0` = unlimited (Infinity
   remaining, no reject even after a 999_999_999-token charge).
4. **daily reset + roll-over (3 tests)** — `resetDaily(tier)` zeroes
   one tier and persists; `resetDaily()` zeroes every tier; clock
   advance to next UTC day -> new file written, day-1 file kept intact
   with the original total.
5. **selectModel keyword + length heuristic (6 tests)** — opus
   keywords, haiku keywords, sonnet keywords; length fallback
   (>500 chars -> opus, <80 chars -> haiku); tier allow-list
   constraint (manager-only-opus pins haiku-keyword to opus,
   worker-no-opus pins design-keyword to sonnet, mid pins haiku-keyword
   to sonnet); unknown tier / empty allow-list -> null;
   `TierQuota.selectModel` honours the instance tier override.
6. **snapshot output (2 tests)** — exposes used / remaining / models
   for every tier; `remaining = -1` for unlimited tiers.

Full suite: **96 / 96 pass** (95 baseline + 1 new test file).

## Operational notes

- Default tier on `c4 new` is `worker`, so existing call sites keep
  working without breakage. Operators flip to `--tier mid` or
  `--tier manager` only when they want the higher daily cap or the
  Opus allow-list.
- The quota gate runs **before** worker spawn, so a drained tier never
  spawns a worker just to be cancelled.
- The keyword + length heuristic is intentionally simple — operators
  who disagree with a pick can still override with `--model opus`.
- `~/.c4/tier-quota-YYYY-MM-DD.json` is small (one int per tier) so
  storage growth is bounded; rotating off old files is left to
  housekeeping (no auto-prune in this slice).
- `chargeTier` is exposed but not yet called automatically by the
  daemon — the existing token monitor still owns global accounting.
  A follow-up slice can wire `chargeTier` into the per-task token
  delta so the gate becomes self-driving instead of advisory.
