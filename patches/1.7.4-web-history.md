# Patch 1.7.4 - Web UI conversation / task history (8.7)

## Date: 2026-04-17

## Summary

TODO 8.7 adds a dedicated History browsing experience to the Web UI so an
operator can look up past conversations + task records per worker,
including workers that have already been closed, with search and filter
affordances. The `c4 history` CLI (added by 3.7) already knows how to
read `history.jsonl`; this patch layers a richer daemon response shape on
top of that, exposes a per-worker endpoint, and surfaces the scribe
`session-context.md` alongside the task history so the two "what did I
do last session?" signals live in the same pane.

## Motivation

Before this change the only way to review a worker's past work from the
Web UI was:

- `/api/list` only shows *currently alive* workers, so once a worker is
  closed + its worktree cleaned up it disappears from the UI entirely.
- `/api/history` returned `{records}` with no per-worker aggregation, no
  server-side filters beyond the CLI's `worker=` / `limit=` pair, and no
  way to browse completed workers without knowing the exact name in
  advance.
- `docs/session-context.md` - the scribe's cross-session memory - had no
  Web UI surface at all. It was readable only by opening the file on
  disk.

After this change the Web UI has a first-class "History" tab that shows
every distinct worker name that has ever appeared in `history.jsonl`,
merges the current live status in, and lets the operator drill into any
one of them to see past tasks + commits + (when still running) live
scrollback.

## Implementation

### `src/history-view.js` (new, no node-pty dep)

Four pure functions:

- `normalizeRecord(rec)` — coerces a `history.jsonl` line into a stable
  shape (`{name, task, branch, startedAt, completedAt, commits, status}`)
  and drops entries that are not objects.
- `filterRecords(records, options)` — applies `worker`, `status`,
  `since`, `until`, `q` (case-insensitive match against name / task /
  branch), and `limit` filters, in that order. `limit` slices the last N
  so "last 20 tasks" is a natural request.
- `summarizeWorkers(records, liveWorkers)` — aggregates records by name
  into `{name, taskCount, firstTaskAt, lastTaskAt, lastTask, lastStatus,
  branches, alive, liveStatus}`. When a name also appears in the live
  workers list, `alive` + `liveStatus` are filled from there and any
  additional branches are unioned into the historical set. Result is
  sorted newest-first with an alphabetical tie-break.
- `readScribeContext(projectRoot, options)` — resolves
  `<root>/docs/session-context.md` (or `options.outputPath` when the
  config overrides the scribe output), `statSync`s for size + mtime,
  and either reads the whole file or, if it exceeds `options.maxBytes`
  (default 256 KiB), tail-truncates via `fs.readSync` with a
  `Buffer.alloc(limit)` so extremely long session summaries do not DOS
  the response. Returns `{exists, path, size, updatedAt, truncated,
  content}`. Missing file returns `exists:false` without throwing.

### `src/daemon.js`

- `require('./history-view')` at the top.
- Path-param regex for `/history/<name>` added next to the existing
  `/worker/<name>/validation` matcher so the REST shape matches the
  TODO spec `GET /api/history/:name`.
- `GET /history` now returns `{records, workers, total}`. Query params:
  `worker`, `limit`, `status`, `since` (ISO), `until` (ISO), `q`. The
  `{records}` sub-shape from 3.7 is preserved so existing
  `c4 history` CLI calls keep working.
- `GET /history/<name>` returns `{name, records, alive, status, branch,
  worktree, scrollback}`. `scrollback` pulls from
  `manager.getScrollback(name, 2000)` when the worker is still in the
  live Map and is `null` otherwise. `lines=` query param tunes the
  scrollback length.
- `GET /scribe-context` returns the scribe helper output directly.
  `config.scribe.outputPath` is honored. `maxBytes=` query overrides
  the 256 KiB tail cap.
- All three endpoints route through the shared `auth.checkRequest`
  gate, so when `config.auth.enabled` is true they require a JWT the
  same way every other `/api/*` route does (8.14).

### Web UI

- New `web/src/components/HistoryView.tsx`:
  - Left-column aggregated worker list (`/api/history` summary).
  - Search input (`q`), status select (`closed` / `exited`), and two
    `type="date"` inputs for since / until; changes fire a refetch via
    a stable `useCallback` identity.
  - Selecting a worker fetches `/api/history/<name>` and renders past
    tasks (task text, branch, status badge, formatted start → complete
    timestamps, commit hashes if any) plus the live scrollback in a
    separate `<pre>` below.
  - A "Scribe" button in the sidebar header opens a full-pane viewer
    for `/api/scribe-context` with path + size + updatedAt metadata and
    a truncation banner when the tail cap kicks in; `exists:false`
    renders a neutral empty-state instead of an error.
  - All fetches go through the shared `apiGet` wrapper so the 8.14 JWT
    + 401 handling flows through unchanged.
- `web/src/App.tsx` adds a `topView` state (`workers` | `history`)
  persisted under the `c4.topView` localStorage key, and a Workers /
  History tab pair next to the Sign-out button in the global header.
  `topView === 'history'` replaces the main content area with
  `<HistoryView />`; the workers sidebar (List / Tree) and detail tabs
  (Terminal / Chat) stay untouched when `topView === 'workers'` so 8.6
  / 8.2 / 8.13 continue to work exactly as before.

### Tests

`tests/history-view.test.js` - 32 assertions across 6 node:test
suites:

1. `filterRecords` — no-filter passthrough (returns every record),
   filter by worker / status / since+until / q (name or task or
   branch, case-insensitive), `limit` slices last N, malformed entries
   (null, undefined, numbers, strings, empty objects) are skipped.
2. `summarizeWorkers` — per-worker aggregation matches expected
   counts + last-task metadata, ordering is newest-first with
   alphabetical tie-break, live-worker merge sets `alive` +
   `liveStatus` and appends new branches, `status:'exited'` is not
   alive, records without a name are skipped.
3. `readScribeContext` — missing file returns `exists:false` without
   throwing, present file returns content + size + updatedAt, custom
   `outputPath` option resolves correctly, `maxBytes` truncation keeps
   the tail.
4. `daemon.js` source-wiring — `require('./history-view')`,
   `route === '/history'`, path-regex `^\/history\/([^\/]+)$`,
   `route === '/scribe-context'`, query-param extraction for
   `worker` / `status` / `since` / `until` / `q`.
5. `HistoryView.tsx` source-wiring — imports `apiGet` from
   `../lib/api`, builds `URLSearchParams` against `/api/history`,
   fetches `/api/history/${encodeURIComponent(name)}` and
   `/api/scribe-context`, renders search placeholder + status / date
   aria-labels, exposes the Scribe button + `openScribe` handler,
   and exports `default HistoryView` + `HistoryWorkerSummary` +
   `HistoryWorkerDetail` types.
6. `App.tsx` integration — imports HistoryView, stores
   `c4.topView` in localStorage, renders both Workers + History tab
   buttons, conditions on `topView === 'history'` to mount
   `<HistoryView />`.

Full suite 71 / 71 pass. `tsc --noEmit && vite build` produces a clean
production bundle (~178 KB, 54.9 KB gzip).

## Backwards compatibility

- `GET /history` still returns `{records}` inside the richer response,
  so `c4 history` + existing third-party callers keep working. The new
  keys (`workers`, `total`) are purely additive.
- `GET /history/:name` is new; no existing route was on that path.
- `GET /scribe-context` is new.
- The Web UI's default `topView` is `workers`, which renders the
  unchanged 8.6 / 8.2 / 8.13 layout.
