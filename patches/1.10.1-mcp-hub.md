# 1.10.1 — MCP Hub: dynamic per-worker MCP server registry (TODO 11.1)

## Why

Workers need MCP servers bundled per profile, not copy-pasted into every
worktree. Until now each `.mcp.json` had to be maintained by hand. This
patch introduces a shared hub with CRUD, RBAC, profile integration, and
`.mcp.json` generation at worker spawn time.

## What shipped

### Storage

- `~/.c4/mcp-servers.json` (override: `config.mcp.path`).
- Schema: `{servers: {<name>: {name, command, args, env, description,
  enabled, transport}}}`.
- Transport: `stdio` (default) or `http`. Name validated against
  `^[A-Za-z0-9._-]+$` at register time.
- Malformed JSON falls back to empty state; missing file returns empty
  state without throwing.

### Module (`src/mcp-hub.js`)

Class `McpHub`:

- `registerServer({name, command, args?, env?, description?, enabled?,
  transport?})` — duplicate name + invalid transport rejected.
- `updateServer(name, patch)` — partial patch; invalid transport
  rejected; throws when server missing.
- `unregisterServer(name)` — returns `false` when missing, `true` when
  deleted.
- `listServers({enabled?, transport?})` — returns sorted by name.
- `getServerConfig(name)` — returns server object or `null`.
- `enableServer(name)` / `disableServer(name)` — flip the gate without
  mutating other fields.
- `buildMcpJson(names)` — produces the Claude Code `.mcp.json` shape,
  filtering out disabled servers + unknown names.
- `writeWorkerMcpJson(worktreePath, names)` — writes `.mcp.json` to the
  worktree; returns the path, or `null` when no enabled server
  matches.
- `testServer(name)` — best-effort spawn+kill for stdio; reports URL
  for http without firing a request.
- Shared singleton via `getShared(opts)` / `resetShared()` so the
  daemon + CLI + pty-manager see the same live state.

### pty-manager integration

`_writeWorkerSettings(worktreePath, workerName, options)` calls the new
`_writeWorkerMcpJson(worktreePath, options)` after the Claude settings
land. That helper resolves the profile via `_getProfile(options.profile
|| options.template)` and forwards `profile.mcpServers` to the shared
hub. The hub is reloaded per spawn so a recent `c4 mcp add` reaches
the next worker without a daemon restart. Silent no-op when the
profile is missing or has no `mcpServers` list, so existing profiles
keep working unchanged.

`listProfiles()` exposes the `mcpServers` array so operators + Web UI
see which servers a profile will auto-load.

### RBAC (`src/rbac.js`)

Adds `MCP_READ` (`mcp.read`) and `MCP_MANAGE` (`mcp.manage`) to the
`ACTIONS` enum. `DEFAULT_PERMISSIONS.manager` gains both,
`DEFAULT_PERMISSIONS.viewer` gains `mcp.read`. `ALL_ACTIONS.length` is
now 20 (up from 18).

### Daemon (`src/daemon.js`)

New endpoints, all gated through `requireRole`:

| Method | Route | Action |
| --- | --- | --- |
| GET | `/mcp/servers` (filters `?enabled=&transport=`) | `mcp.read` |
| POST | `/mcp/servers` | `mcp.manage` |
| GET | `/mcp/servers/:name` | `mcp.read` |
| PUT | `/mcp/servers/:name` | `mcp.manage` |
| DELETE | `/mcp/servers/:name` | `mcp.manage` |
| POST | `/mcp/servers/:name/enable` | `mcp.manage` |
| POST | `/mcp/servers/:name/disable` | `mcp.manage` |
| POST | `/mcp/servers/:name/test` | `mcp.manage` |

`_mcpHub` shared singleton drops on `/config/reload` so a new
`config.mcp.path` takes effect without a daemon restart. Audit events:
`mcp.registered`, `mcp.updated`, `mcp.unregistered`, `mcp.enabled`,
`mcp.disabled`.

### CLI (`src/cli.js`)

`case 'mcp'` now routes hub subcommands ahead of the stdio-proxy
subcommands it already handled:

```
c4 mcp list [--enabled] [--disabled] [--transport T]
c4 mcp add --name N --command CMD [--args 'a,b,c'] [--env 'K=V,K2=V2']
           [--transport stdio|http] [--description D] [--disabled]
c4 mcp show <name>
c4 mcp enable <name>
c4 mcp disable <name>
c4 mcp remove <name>
c4 mcp test <name>
```

Existing `c4 mcp start | status | tools` continue to behave exactly as
before (the stdio proxy is untouched).

### Tests

`tests/mcp-hub.test.js` — 40 assertions across six suites:

1. helpers — `defaultStorePath`, `VALID_TRANSPORTS`, `NAME_PATTERN`,
   `isValidName`, `isValidTransport`, `normalizeServer`, `freshState`,
   `ensureShape`.
2. `registerServer` — persist + duplicate rejection + missing name +
   missing command + invalid transport + invalid name + http URL.
3. `list` / `update` / `delete` — enabled filter, transport filter,
   sorted output, partial patch, invalid-transport patch rejected,
   unregister idempotence, enable/disable flip.
4. storage roundtrip — fresh instance sees prior writes, missing file,
   malformed JSON, `reload()` picks up external mutation.
5. `buildMcpJson` / `writeWorkerMcpJson` — stdio + http shapes,
   disabled skipped, unknown skipped, writes `.mcp.json`, null on
   empty names, null when all blocked, re-enable lands in next spawn.
6. profile integration — `profile.mcpServers -> .mcp.json` content,
   missing profile yields no file, ghost server yields no file,
   `pty-manager.listProfiles` surfaces `mcpServers`.

`tests/rbac.test.js` — bumped `ALL_ACTIONS.length` to 20 and added
`MCP_READ` / `MCP_MANAGE` enum checks.

Full suite: 89 / 89 pass (up from 88). No skips.

## Limitations

- `c4 mcp test` only verifies the launch path — there is no MCP
  handshake yet. A real protocol probe lands under the follow-up
  observability track.
- Runtime hot-reload for an already-spawned worker is not supported:
  Claude Code reads `.mcp.json` at boot, so changes take effect on
  the next `c4 new` / `c4 task`. A restart-on-change flow is out of
  scope.
- Web UI registry panel is a 10.x follow-up; the six REST endpoints
  are the integration contract.

## Migration notes

- Existing profiles without `mcpServers` are untouched — no `.mcp.json`
  is written, no behaviour change.
- RBAC roles refresh on the next request because the daemon rebuilds
  its role table through `reload()`; no manual migration for `admin`
  users (wildcard covers the new actions).
