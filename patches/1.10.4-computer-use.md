# 1.10.4 — Computer Use agent (TODO 11.2)

Stub-first GUI automation pipeline. Ships a `ComputerUseAgent` +
`Backend` abstraction so future integrations (macOS CGEvent, Windows
SendInput, Wayland portal capture) drop in by subclassing `Backend` and
plugging into `selectBackend()` — without touching the daemon
endpoints, CLI commands, or session storage layout.

## Why stub-first

Real pixel capture and input injection need native dependencies that
are hard to set up inside a CI worker:

- Linux/X11 needs `xdotool` + `scrot` (or ImageMagick `import`) on
  `PATH`. Wayland needs `ydotool` or a portal-based capture pipeline.
- macOS needs `CGEventCreateMouseEvent` / `CGEventKeyboardSetUnicodeString`
  through a FFI binding; the daemon also needs Accessibility +
  Screen Recording entitlements.
- Windows needs `SendInput` / `BitBlt` through `user32.dll` +
  `gdi32.dll`.

Instead of pinning a native module today and leaving Windows / macOS
operators unable to install the CLI at all, this release ships the
**abstraction + interface layer** with a stub executor that records
every action to memory. The daemon endpoints, CLI commands, RBAC gate,
config flag, session storage, and audit trail are all wired up now so
the real backends drop in later as additive work.

The Linux `XdotoolBackend` is included today as a proof-of-concept: it
shells out to `xdotool` + `scrot` when both are on `PATH`, throws
`NotAvailable` when they aren't, and is selected automatically by
`selectBackend('auto')` on Linux hosts that have the binaries.

## Safety flag

`config.computerUse.enabled` defaults to **false**. `startSession`
throws `computerUse.enabled=false — refusing to start session` until an
operator explicitly opts in. Granting Computer Use to a worker is
effectively "remote desktop as the daemon user" — the process can
screenshot, type, click, and drag **anywhere on the logged-in
display** — so we do not auto-enable it the way we auto-enable
`nl.chat` for viewers.

The daemon's RBAC layer adds a **single** canonical action
`COMPUTER_USE = 'computer.use'` (24 total, up from 23). Every new
endpoint routes through `requireRole(authCheck, rbac.ACTIONS.COMPUTER_USE)`.
Default matrix:

- `admin`: allowed via the existing `*` wildcard.
- `manager`: **not** allowed by default. Broaden explicitly if a
  trusted operator needs it.
- `viewer`: **not** allowed by default.

We deliberately picked **one powerful action** over per-endpoint
scopes (click-only vs. type-only) because the threat model doesn't
split cleanly: a client that can move the mouse can also drag-select
text, and a client that can type can also activate menus via Alt.
Splitting the scope later is additive; collapsing a fine-grained
scheme would be a breaking change.

## Session storage

- Sessions → `~/.c4/computer-use-sessions.json` (FIFO-capped at 50;
  `config.computerUse.sessionsPath` overrides).
- Screenshots → `~/.c4/screenshots/<sessionId>/<shotId>-<timestamp>.png`
  (`config.computerUse.screenshotsDir` overrides).

Session entries keep `{id, backend, actions: [], screenshots: [],
startedAt, endedAt}`; `DELETE /computer-use/sessions/:id` **soft-ends**
the session (stamps `endedAt`) so the recorded action trail survives
for audit and post-hoc replay. `deleteSession` on the agent strips the
row entirely and is only used in tests.

Backends live in the agent's in-process `_backends` Map for the life
of the daemon. After a restart the agent re-hydrates a backend of the
same preference on next use — recorded actions stay intact, but any
live display-side state (focus, cursor position) is lost because the
backend is stateless by design.

## Coordinate + key validation

- Coordinates must be non-negative finite numbers. Negative, `NaN`,
  `Infinity`, or non-numeric inputs are rejected before the backend is
  called so a bad click never reaches the display.
- Key names pass through `normalizeKeyName()`: a small alias table
  collapses casual spellings (`enter`→`Return`, `esc`→`Escape`,
  `ctrl`→`ctrl`, `space`→`space`) and `+`-joined combos
  (`Ctrl+Shift+A`) are title-cased token-by-token. Unknown keys (`F1`,
  `KP_Enter`) pass through unchanged so backend-specific names still
  work.

## Future work

1. **macOS CGEvent backend.** FFI binding + Accessibility +
   Screen Recording entitlements. Ships under `1.10.x-cgevent.md`.
2. **Windows SendInput backend.** `user32.dll` + `gdi32.dll` FFI
   binding. Ships under `1.10.x-sendinput.md`.
3. **Wayland portal backend.** `ydotool` fallback today; portal-based
   capture via `xdg-desktop-portal-screencast` for distributions that
   disable XWayland.
4. **Tesseract OCR** (`screenshot → text`). Lets an agent locate a
   button by label instead of pixel coordinates. Adds a new
   `readText(sessionId, region?)` method on the agent without breaking
   the existing coordinate-first API.
5. **OpenCV element detection.** Template matching on reference images
   so an agent can find a button after a layout shift. Adds
   `findElement(sessionId, template)` → `{x, y, confidence}`.
6. **Per-endpoint RBAC scopes** (click-only vs. type-only). We'll add
   them when a concrete use case demands the split — today the single
   `COMPUTER_USE` action matches the threat model cleanly.
7. **Worker type "computer-use".** The worker pool can learn to spawn
   a dedicated worker with the `COMPUTER_USE` grant pre-wired; today
   any worker with the permission can drive the display via the CLI
   or HTTP API.
