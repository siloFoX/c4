# Patch 1.7.1 - Web UI terminal view resolution + resize (8.13)

## Date: 2026-04-17

## Summary

The Web UI `WorkerDetail` screen/scrollback view rendered the worker's PTY
output at a fixed 160x48 grid because `src/screen-buffer.js` and the node-pty
spawn defaults were locked to that size. On browser viewports narrower than
~160 columns the TUI wrapped inside the server's virtual terminal (not the
browser), producing the reported "lines are broken" symptom. Claude's TUI
and interactive prompts paint cursor-positioned output that relies on
knowing the terminal width, so wrapping one column short made every other
frame look garbled.

Fix:

1. `ScreenBuffer` gained a `resize(cols, rows)` method that pushes overflow
   rows into scrollback (respecting `maxScrollback`), truncates each line
   on cols shrink, pads with empty lines on cols/rows grow, and clamps the
   cursor, saved cursor, and scroll region into the new bounds.
2. `PtyManager.resize(name, cols, rows)` calls node-pty `proc.resize` then
   `screen.resize`, clamped through a static `_clampResizeDims` helper
   (defaults 20..400 cols / 5..200 rows, overridable via
   `config.pty.min*/max*`).
3. `POST /resize {name, cols, rows}` on the daemon wires the manager.
4. Web UI `WorkerDetail` adds a terminal toolbar: Auto-fit toggle, font
   size +/- (9-24px), manual cols input, live dims readout. Auto-fit
   measures a hidden 1-char ruler span with the same font stack, computes
   `cols = floor(pre-inner-width / char-width)`, and POSTs `/api/resize`
   on mount, font-size change, and debounced (120ms) window resize. A ref
   dedupe prevents identical dims from re-hitting the server. Prefs
   persist in `localStorage` (`c4.term.fontSize` / `c4.term.autoFit` /
   `c4.term.cols`).
5. Layout gains `min-w-0` + `min-h-0` on `<main>` and the detail column so
   the `<pre>` horizontal/vertical scroll works inside the flex row. On
   small viewports a hamburger toggle collapses the worker list sidebar,
   and padding drops from `p-6` to `p-3`.

## Evaluating xterm.js

Adopting xterm.js would require streaming raw PTY bytes to the browser and
removing the server-side virtual-terminal parse. The existing `ScreenBuffer`
already strips ANSI and maintains a clean text model that the existing
`/read-now` + `/scrollback` endpoints, `c4 scrollback` CLI, stall detection,
and hook event log all depend on. Swapping in xterm.js would require
maintaining both models in parallel or rewriting those consumers. The
8.13 symptom was a dims mismatch, not a rendering fidelity gap - resize +
auto-fit addresses the reported breakage without touching the shared text
model. xterm.js remains a reasonable follow-up if we need true color,
mouse support, or alt-screen handling, but it is out of scope here.

## Files changed

- `src/screen-buffer.js` - new `resize` method.
- `src/pty-manager.js` - static `_clampResizeDims`, instance `resize`.
- `src/daemon.js` - `POST /resize` route.
- `web/src/components/WorkerDetail.tsx` - toolbar, ruler, auto-fit,
  manual cols, localStorage, responsive `<pre>` sizing.
- `web/src/App.tsx` - sidebar collapse, `min-w-0`/`min-h-0` wiring,
  responsive padding.

## Tests

- `tests/screen-buffer-resize.test.js` - 10 assertions covering no-op,
  shrink-rows-to-scrollback, grow-rows-padding, cols truncate, cursor +
  saved cursor + scroll region clamp, `maxScrollback` respected on
  overflow, non-numeric coercion, continued writes after a resize.
- `tests/pty-resize.test.js` - 16 assertions: 6 for `_clampResizeDims`
  (min / max / custom limits / non-numeric / fractional), 7 for the
  instance `resize` method (unknown / dead worker errors, proc+screen
  forwarding, clamp, node-pty throw propagation, missing proc.resize
  fallback, config-overridden limits), 3 source-grep assertions pinning
  the `/resize` daemon wiring (`route === '/resize'`,
  `manager.resize(name, cols, rows)`, `Missing name, cols or rows`).

Full suite: 68 / 68 pass.

## Config knobs

Optional `config.pty` additions (no config change required for the
defaults):

```json
{
  "pty": {
    "minCols": 20,
    "maxCols": 400,
    "minRows": 5,
    "maxRows": 200
  }
}
```

## Manual smoke check

- Open the dashboard, select a worker, watch `dims:` update from the
  default (e.g. `120 x 48`) to the actual viewport-fit.
- Shrink the browser window; dims re-fit after the 120ms debounce.
- Uncheck Auto-fit, type `80` into the cols input; server confirms
  `dims: 80 x 48`, wrapped lines disappear.
- Toggle the mobile hamburger (under 768px); sidebar collapses and the
  `<pre>` takes the full width.
