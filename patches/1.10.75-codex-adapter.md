## Why a scaffold instead of a full integration

Two reasons:

1. **codex's UI is a moving target.** Hard-coding `readyPrompt` /
   `readyIndicator` substrings would break on the first codex
   release that touches them. Letting the operator configure those
   strings in `agent.options.codex.patterns` decouples C4's release
   cadence from codex's.

2. **C4 doesn't ship the codex binary.** The adapter is purely
   protocol — it knows how to drive a PTY-attached agent, not how
   to install one. Operators with codex already on their system
   point C4 at it via `agent.options.codex.binary`.

This matches how ClaudeCodeAdapter was originally introduced (TODO
9.1 phase 1, v1.7.9) — protocol + opt-in patterns, with the binary
already owned by the operator's environment.

## Conservative `detectIdle`

```js
detectIdle(chunk) {
  const text = String(chunk == null ? '' : chunk);
  const prompt    = this.patterns.readyPrompt;
  const indicator = this.patterns.readyIndicator;
  if (typeof prompt    !== 'string' || prompt.length    === 0) return false;
  if (typeof indicator !== 'string' || indicator.length === 0) return false;
  return text.includes(prompt) && text.includes(indicator);
}
```

The daemon's state machine treats `true` as "task done". If we
defaulted to a permissive prompt match, an unconfigured CodexAdapter
would silently declare tasks complete mid-flight every time codex's
output happened to contain a `>` character. Returning `false`
until BOTH patterns are explicitly configured AND present is
worst-case a hung task (the operator can interrupt via `c4 send
<name> "C-c"`); a false positive would silently lose work.

## Configuration shape

```json
{
  "agent": {
    "type": "codex",
    "options": {
      "binary": "/usr/local/bin/codex",
      "args": ["--no-color"],
      "patterns": {
        "readyPrompt": "...",
        "readyIndicator": "..."
      }
    }
  }
}
```

`binary` + `args` are informational — `PtyManager` owns the spawn
and reads them from the same config block. `patterns` accepts both
the legacy positional form and the per-type sub-bag form; positional
wins on conflict.

## Cross-adapter contract test integration

`tests/agent-adapter-contract.test.js` (shipped 1.10.74) iterates
over `listAdapterTypes()` and runs the same 8 shape checks against
each entry in `REGISTRY`. CodexAdapter required no special
construction options, so the test picked it up automatically:
41 cases → 49 cases. This is the framework working as intended —
new adapters that extend the contract get covered for free, and
adapters that drift out of contract break the canary first.

## Backwards compatibility

None broken. The `'codex'` key was unused before this patch;
existing `agent.type` values continue to behave as before. The
local-llm REGISTRY canary needed updating from 6 keys to 7.
