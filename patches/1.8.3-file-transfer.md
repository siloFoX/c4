# 1.8.3 - Machine-to-machine file transfer (TODO 9.8)

## Summary

TODO 9.8 lands on top of the 9.6 fleet registry + 9.7 dispatcher. C4
gains a native file transfer layer so an operator who has dispatched a
task to a fleet peer can stage source code, build artifacts, or model
weights to that peer without leaving the c4 CLI. Two transports:

- `rsync` over ssh for general files, with `--info=progress2` streamed
  as SSE events so dashboards can render a progress bar.
- `git push` over ssh for repositories, so both sides share commit
  history instead of rsync-ing a tree with a `.git` directory.

Files added / changed:

- `src/file-transfer.js` (new) - pure-node helper module
- `src/daemon.js` - new `POST /transfer` endpoint + SSE progress events
- `src/cli.js` - new `c4 send-file` + `c4 push-repo` subcommands
- `tests/file-transfer.test.js` (new) - 69 assertions across 18 suites

The HTTP response from `POST /transfer` returns immediately with
`{ started, pid, transferId }`; progress / completion / error events
arrive on the existing `/events` SSE bus with types
`transfer-progress`, `transfer-complete`, `transfer-error`. The caller
correlates events by `transferId`.

## Endpoints

```
POST /transfer
{
  "alias": "dgx",
  "type": "rsync",
  "src": "/root/c4/build/",
  "dest": "arps/build/",
  "opts": {
    "excludes": ["node_modules", ".git", "*.log"],
    "delete": true,
    "dryRun": false,
    "allowSystem": false
  }
}
```

Response:
```
{
  "started": true,
  "pid": 12345,
  "alias": "dgx",
  "type": "rsync",
  "transferId": "tx-1713384921345-482301",
  "cmd": "rsync",
  "args": ["-a", "-v", "-z", "-P", "--info=progress2", "--delete",
           "--exclude", "node_modules", ..., "shinc@192.168.10.222:arps/build/"]
}
```

For git transfers:
```
POST /transfer
{
  "alias": "dgx",
  "type": "git",
  "src": "/root/c4-worktree-file-transfer",
  "remoteRepoPath": "repos/c4.git",
  "branch": "c4/file-transfer",
  "opts": { "force": false }
}
```

SSE events (on `GET /events`):
```
{"type":"transfer-progress","alias":"dgx","transferId":"tx-...",
 "file":"src/big.bin","percent":25,"bytes":32768,"bytesPerSec":"32MB/s","eta":"0:00:02"}
{"type":"transfer-complete","alias":"dgx","transferId":"tx-...",
 "code":0,"ok":true}
{"type":"transfer-error","alias":"dgx","transferId":"tx-...",
 "error":"ssh: Could not resolve hostname dgx"}
```

## CLI

```bash
# rsync a build directory to the dgx machine
c4 send-file dgx /root/c4/build/ arps/build/ --delete \
  --exclude node_modules --exclude '*.log'

# dry-run (no changes, just rsync itemize)
c4 send-file dgx /root/project dest/ --dry-run

# push the current repo's worktree to a bare repo on dgx
c4 push-repo dgx c4/file-transfer \
  --repo /root/c4-worktree-file-transfer \
  --remote-repo repos/c4.git
```

Both commands POST to `/transfer` on the local daemon. They return as
soon as the process is spawned — progress shows up on the SSE stream
which the Web UI (or any EventSource consumer) can render live.

## Security notes

The module actively refuses unsafe operator input. The rules below are
not just code — they are the contract c4 relies on when it fans
transfers across a fleet.

1. **Source path guard.** `validateSrcPath` resolves the path, then
   checks it is under one of: `$HOME`, the c4 project root, or any
   explicit root passed in `opts.allowedRoots`. Paths resolving under
   `/etc`, `/bin`, `/sbin`, `/boot`, `/dev`, `/proc`, `/sys` are
   **always denied** — even with `--allow-system`. This catches the
   "traversal that resolves into /etc" case (e.g. `/root/../etc/passwd`)
   because the resolve happens before the root check, not after a
   naïve substring match.
2. **Destination path guard.** Remote dest paths starting with `/` are
   rejected unless `--allow-system` is passed. `..` segments are
   rejected unconditionally. Shell metacharacters in dest are rejected
   because the remote-side rsync receiver runs them through a shell.
3. **No `rm -rf` on remote.** We never pass `--rsync-path` with a
   constructed command. `--delete` is the only remote-deleting flag
   we emit, and it only removes files that were removed on the source
   side, under the rsync contract. Plain `git push --force` is never
   built — `--force` maps to `--force-with-lease` so a concurrent
   remote update cannot be silently overwritten.
4. **No password prompts.** Every ssh invocation passes
   `-o BatchMode=yes` so a missing key fails loud instead of prompting
   for a password that will never come. `StrictHostKeyChecking=accept-new`
   lets first-contact with a new host succeed but still pins the key
   against future MITM.
5. **Fleet alias is the ssh identity.** `resolveMachine` reads the
   fleet entry — there is no free-form host argument the caller can
   supply. Operators register machines via `c4 fleet add` once, and
   transfers from then on reference the alias only. ssh-config and
   ssh-keys provide the user / authentication.

### Threat model — what this does and does not block

- **Command injection via path strings** — blocked. Shell metachars
  are rejected at input.
- **Path traversal out of $HOME / project root** — blocked. Resolution
  happens before the root check.
- **Accidental destruction of system files on the source** — blocked.
  Reading `/etc`, `/dev`, etc. is never allowed.
- **Accidental destruction of system files on the destination** —
  blocked by default. Operators must pass `--allow-system` to target
  absolute paths, which makes the choice visible in the CLI history.
- **Active attacker with ssh key access** — out of scope. If the
  attacker has the ssh private key that c4 uses, they can push
  whatever they want by hand; c4 is not a secondary authentication
  layer for ssh.
- **Man-in-the-middle between source and destination on first
  contact** — partially out of scope. `accept-new` pins on first use
  but does not authenticate that first use. Operators who need
  stricter behavior should pre-populate `~/.ssh/known_hosts`.

## Progress parsing

`parseRsyncProgress` matches the cumulative progress line rsync emits
with `--info=progress2`:

```
     32,768  25%   32.68MB/s    0:00:00
    102,400 100%  100.23MB/s    0:00:02
```

And `parseRsyncFileLine` distinguishes filename lines from the
indented progress lines and the summary lines
(`sending incremental file list`, `sent N bytes received N bytes`,
`total size is N speedup is N`, `(xfr#1, to-chk=0/1)` etc.). The
driver threads the most recent filename into each progress event so
consumers always know *which* file the bytes belong to.

## Tests

`tests/file-transfer.test.js` - 69 assertions across 18 node:test
suites. Highlights:

1. `validateAlias` / `hasShellMetachars` guard the input surface.
2. `validateSrcPath` rejects paths outside allowed roots, system
   prefixes, and traversal, and honors `allowSystem` for non-deny
   paths.
3. `validateDestPath` / `validateRemoteRepoPath` reject absolute paths
   without `--allow-system`, `..` traversal, and shell metachars.
4. `buildRsyncArgs` produces `-avzP --info=progress2`, emits
   `--exclude <pattern>` per entry, `--delete` only when requested,
   `-n` for dry-run, and `-e "ssh -o BatchMode=yes ..."`.
5. `buildSshCommand` + `buildSshTarget` handle `sshUser`, `sshHost`,
   `sshPort`, and reject unsafe host characters.
6. `normalizeExcludes` trims + validates patterns, allowing glob
   chars (`*`, `?`) that rsync handles natively while rejecting `;`,
   `$()`, backticks, etc.
7. `parseRsyncProgress` / `parseRsyncFileLine` handle real-world
   sample lines, including the 100% completion case.
8. `resolveMachine` reads the fleet entry, surfaces `sshHost` /
   `sshUser` / `sshPort` overrides, falls back to the daemon host.
9. `buildGitPushArgs` produces `-C <local> push <url> <branch>`,
   maps `force` to `--force-with-lease` (never plain `--force`),
   omits the branch when absent, and rejects shell metachars in
   branch / localRepoPath.
10. `buildGitEnv` assembles `GIT_SSH_COMMAND` with BatchMode and
    threads `-p <port>` when provided.
11. `transferFiles` spawns rsync with the built args, drains progress
    lines from stdout, fires `onComplete` on exit, `onError` on error,
    and refuses unsafe src before spawning.
12. `pushRepo` spawns git with `-C <repo> push ...`, threads
    `GIT_SSH_COMMAND` via env, streams progress from stderr
    (git push convention), and refuses unsafe localRepoPath.
13. Daemon + CLI source-grep wiring verifies the module import, the
    `/transfer` route, the three SSE event types, and the CLI
    subcommands / flags / help text.

Full suite: 79/79 pass.

## Limitations

- Remote progress is best-effort. Rsync with `--info=progress2` emits
  cumulative bytes, not per-file. The `file` field on progress events
  is the most-recently-named file from the filename stream; if rsync
  is mid-batch on a large file, that file keeps appearing until the
  next file starts.
- The transfer process is owned by the daemon. Killing the daemon
  kills the rsync / git push child. There is no resume across daemon
  restarts yet — rsync's `-P` handles mid-transfer resume on the
  receiver side, but the HTTP response + SSE correlation is tied to
  the current daemon instance.
- `--force` on `c4 push-repo` maps to `--force-with-lease`. There is
  no escape hatch for a plain `git push --force` in this module;
  operators who need that can call git directly.
- The fleet alias must have working ssh-config on the source host.
  C4 does not manage ssh keys or host entries — fleet.json stores the
  host + port for the daemon HTTP layer only.
- Web UI dashboard integration (a transfer progress panel) is not in
  this slice. The SSE events are the contract the UI will build on.
