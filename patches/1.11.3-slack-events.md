# 1.11.3 - Slack autonomous event notification integration (8.15)

## Summary

Before 8.15 the only Slack traffic emitted by c4 was the buffered
stall/health digest that `src/notifications.js` flushed every five
minutes. Autonomous merges, approvals, and worker lifecycle changes
were invisible unless an operator tailed the Web UI, which defeats the
stated goal of "running c4 from Slack alone until the Web UI is
needed".

This patch adds a daemon-level event fabric so every interesting state
change surfaces in Slack as a single structured line
(`[c4:event] merge_success branch=c4/x sha=abc123 worker=w1`), with
level filtering, allowlist filtering, and sliding-window dedupe so a
flapping worker cannot spam the channel.

The work:

1. Introduces `src/slack-events.js` - a standalone emitter with
   `emit` / `configure` / `listen` / `recentEvents` / `clearRecent` /
   `getConfig` and a shared singleton via `getShared` / `resetShared`.
   Ten canonical event types with pinned default severity; payload
   `level` overrides the default per emit.
2. Wires the emitter into `src/daemon.js` at five lifecycle points
   (worker create / task send / worker close / approval / merge
   success+fail) plus a Notifications.notifyStall monkey-wrap that
   converts every halt / intervention / escalation into a
   `halt_detected` event without disturbing the legacy buffered send.
3. Adds two new endpoints: `GET /slack/events` (open to any
   authenticated caller) and `POST /slack/emit` (gated by the new
   `SLACK_WRITE` RBAC action).
4. Extends `src/merge-core.js` `performMerge` with an optional `emit`
   callback so the CLI path fires the same events as the daemon path.
5. Ships `c4 slack test` / `c4 slack status` subcommands and a new
   `c4 merge --push` flag that pushes to origin main and emits
   `push_success`.
6. Bumps RBAC `ALL_ACTIONS` to 27 (adds `slack.write` to manager + admin).

## Event types + default levels

| Event type         | Default level | Fired at                                     |
| ------------------ | ------------- | -------------------------------------------- |
| `task_start`       | info          | `POST /task` on success                      |
| `task_complete`    | info          | (reserved; future pty-manager hook)          |
| `worker_spawn`     | info          | `POST /create` on success                    |
| `worker_close`     | info          | `POST /close` on success                     |
| `merge_success`    | info          | `POST /merge` on success, CLI `c4 merge`     |
| `push_success`     | info          | CLI `c4 merge --push` on successful push     |
| `halt_detected`    | warn          | Any `notifications.notifyStall(...)` call    |
| `approval_request` | warn          | `POST /approve` on success                   |
| `merge_fail`       | error         | `POST /merge` on git failure, CLI `c4 merge` |
| `error`            | error         | Manual `POST /slack/emit` payload            |

## Config schema (config.slack)

```json
{
  "slack": {
    "enabled": false,
    "webhookUrl": "",
    "minLevel": "info",
    "dedupeWindowMs": 60000,
    "events": ["task_start", "merge_success", "merge_fail", "halt_detected"]
  }
}
```

- `enabled=false` (default) suppresses every emit, including manual
  `c4 slack test`. Nothing flows to Slack until the operator flips it.
- `webhookUrl` missing or empty keeps emit() succeeding but returns
  `webhook.reason = 'no-webhook'` instead of posting - useful for
  local dev so you can inspect `c4 slack status` without a real hook.
- `minLevel` filters by numeric priority (`info < warn < error`).
  Setting `minLevel = 'warn'` is the most common production choice -
  merge failures + stalls + approvals get through, routine
  spawn/close/task traffic does not.
- `dedupeWindowMs` defaults to 60000. A worker stuck emitting the same
  stall reason every 10s will fire exactly once per window. Window
  changes purge stale entries on the next emit.
- `events` is an allowlist. An empty or missing array means "all
  canonical types". A non-empty array that filters down to zero valid
  types falls back to "all canonical types" on the assumption the
  operator typoed rather than meant to mute everything.

## RBAC (new action)

- `ACTIONS.SLACK_WRITE = 'slack.write'`
- Added to `DEFAULT_PERMISSIONS.manager` so the manager role keeps its
  "can drive every daemon endpoint" guarantee.
- Viewer role does **not** get it - a viewer JWT cannot inject events
  into the operator's Slack channel.
- `ALL_ACTIONS.length` bumps from 26 to 27. `tests/rbac.test.js`
  asserts the new constant + count.

## Verification

```
npm --prefix /root/c4-worktree-slack-events test
# ... 95/95 pass
```

No external network traffic: the test suite injects a mock
`httpClient` so every Slack POST is captured in-memory, and
`recentCap=3` + controllable `now()` let the tests verify the buffer
bound + the sliding-window dedupe without real time.

## Operator notes

- The default config lands with `slack.enabled=false`, so upgrading
  does not change any behaviour until the operator opts in.
- When they do opt in, the daemon emits on every worker lifecycle
  event; setting `minLevel='warn'` and restricting the `events` list
  to `["halt_detected", "approval_request", "merge_fail"]` keeps the
  channel quiet for day-to-day work while still surfacing incidents.
- `c4 merge --push` is opt-in - the previous `c4 merge` behaviour
  (merge only, no push) stays unchanged when `--push` is absent.
- The in-memory recent buffer is capped at 100 and lives in the
  daemon process only, so restarting the daemon clears it.
