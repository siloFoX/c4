# 1.10.78 — PtyAdapterBase shared scaffolding

## Summary

Pure refactor; no production behavior change. Extracts the ~30
lines of input/key/init plumbing that claude-code.js and codex.js
each shipped on their own onto a shared base class. New PTY
adapter authors (e.g. an Aider integration) inherit the
boilerplate for free.

- `src/agents/pty-adapter-base.js` (new) — `PtyAdapterBase
  extends Adapter`
- `src/agents/claude-code.js` — `extends Adapter` → `extends
  PtyAdapterBase`; KEY_MAP re-exported via
  `PtyAdapterBase.DEFAULT_KEY_MAP`
- `src/agents/codex.js` — same treatment
- `tests/agent-pty-adapter-base.test.js` (new) — 23 cases / 7
  suites locking in the base contract

Suite 165 → 166.

## What lives on the base

```js
class PtyAdapterBase extends Adapter {
  constructor() {
    super();
    this._keyMap = DEFAULT_KEY_MAP;   // subclasses can override
  }

  init(workerCtx) {
    this._workerCtx = workerCtx || null;
  }

  sendInput(text) {
    if (typeof text !== 'string') {
      throw new TypeError('sendInput requires a string');
    }
    const proc = this._workerCtx && this._workerCtx.proc;
    if (proc && typeof proc.write === 'function') {
      proc.write(text);
    }
  }

  sendKey(key) {
    const mapped = Object.prototype.hasOwnProperty.call(this._keyMap, key)
      ? this._keyMap[key]
      : key;
    this.sendInput(mapped);
  }
}
```

`DEFAULT_KEY_MAP` is `Object.freeze`d so subclasses that want to
extend it must do so via spread (`{...DEFAULT_KEY_MAP, F1:
'\x1bOP'}`) — preventing one adapter's mutation from leaking to
another's runtime.

## What subclasses still own

- `metadata` getter — adapter-specific name / version (and any
  extra fields like `model` for ClaudeAgentSdkAdapter, though
  that one isn't PTY-driven so it doesn't extend PtyAdapterBase)
- `supportsPause` getter — pause/resume capability
- `detectIdle(chunk)` — adapter-specific idle test
- Any adapter-specific helpers (claude-code's
  `extractBashCommand`, codex's `binary`/`args` config, etc.)

`validateAdapter` passes once the subclass fills in the three
required surfaces. The cross-adapter contract test
(`tests/agent-adapter-contract.test.js` from 1.10.74) covers
those for every entry in REGISTRY automatically.

## Why inheritance, not composition

Both alternatives were on the table. Inheritance won because:

1. The PTY adapters share `_workerCtx` lifecycle. Putting `init`
   on a utility module would still need each adapter to declare
   the field; the base class encapsulates both the field and the
   methods that touch it.

2. The cross-adapter contract test iterates `listAdapterTypes()`
   — a shared base means new PTY adapters automatically pass the
   same shape checks for free, with no special-cased opts in
   `ADAPTER_OPTS`.

3. The class hierarchy stays shallow: `Adapter →
   PtyAdapterBase → ClaudeCodeAdapter / CodexAdapter`. Three
   levels is fine. Composition via mixin would have introduced
   either runtime mutation (Object.assign on the prototype, ick)
   or a delegation property (`adapter.pty.sendInput`, awkward
   from the daemon's call sites).

## What we did NOT extract

- `_emitOutput` per-handler error isolation — already lives on
  the Adapter base via `_outputHandlers` + `_emitOutput`. Both
  PTY adapters and the non-PTY adapters (mock,
  claude-agent-sdk, local-llm) share that path.
- ScreenBuffer ownership — local-llm has its own ScreenBuffer
  (it's not PTY-driven so it has to simulate one); claude-code
  doesn't (PtyManager owns it). Different ownership models, no
  shared base.
- Spawn lifecycle (`spawn` / `kill` / `dispose`) — local-llm
  has those because it needs to manage HTTP requests + an
  AbortController. PTY adapters defer all of that to
  PtyManager, so they don't have those methods. Different
  responsibility split.

## Test coverage

```
PtyAdapterBase — abstract / inheritance (3)
  - extends Adapter (transitive instanceof)
  - PtyAdapterBase itself is constructable (Adapter abstract guard
    only fires for new.target === Adapter)
  - subclass fillin makes validateAdapter() pass

PtyAdapterBase — DEFAULT_KEY_MAP (4)
  - frozen
  - covers Enter / Escape / Tab / Backspace / arrows / C-c / C-d
  - arrow keys produce CSI sequences (\x1b[A/B/C/D)
  - control sequences match POSIX (C-c=0x03, C-d=0x04)

PtyAdapterBase — init() lifecycle (3)
  - init(ctx) stores on _workerCtx
  - init(null) clears
  - init() with no arg falls through to null

PtyAdapterBase — sendInput (5)
  - writes to ctx.proc when attached
  - no-op when no proc attached
  - no-op when ctx has no proc
  - throws TypeError on non-string (number / null / undefined)
  - empty string is forwarded (not coerced to no-op)

PtyAdapterBase — sendKey (3)
  - maps named keys via DEFAULT_KEY_MAP
  - passes unknown names through as raw bytes
  - subclass-supplied _keyMap takes precedence (spread + override
    + new key)

PtyAdapterBase — onOutput inherited from Adapter base (3)
  - returns unsubscribe fn
  - rejects non-function callback
  - _emitOutput fans to all listeners with per-handler isolation

PtyAdapterBase — production adapters use it (2)
  - ClaudeCodeAdapter instanceof PtyAdapterBase
  - CodexAdapter instanceof PtyAdapterBase
```

The last suite is the regression guard — if a future "cleanup"
PR accidentally reverts one of the production adapters off
PtyAdapterBase, that test breaks first.
