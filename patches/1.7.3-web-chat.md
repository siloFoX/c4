# Patch 1.7.3 - Web UI chat interface per worker (8.6)

## Date: 2026-04-17

## Summary

TODO 8.6 replaces the `c4 send` + `c4 read` CLI ping-pong with a browser
chat UI per worker. The existing `WorkerDetail` terminal view stays
untouched and now shares the detail area with a new `ChatView` component
through a Terminal / Chat tab pair in `App.tsx`. The chat view renders
conversation turns as speech bubbles, posts user text through the daemon's
existing `/send` + `/key` endpoints, and streams worker output live off
the SSE `/watch` endpoint. Auth + 401 handling flow through the shared
`apiFetch` + `eventSourceUrl` wrappers so the JWT attaches automatically
to both the REST calls and the EventSource URL.

## Motivation

Until now the only way to talk to a worker from the Web UI was to read
the raw ANSI-stripped PTY screen in `WorkerDetail` and type into a plain
input that fires a single `/send` call. That is fine for one-off
intervention keystrokes (approving a prompt, firing an Enter) but it is
not a conversation view:

- The screen model is the full TUI buffer, so user input and worker
  output share a single rectangular region that scrolls by overwrite
  rather than by message order.
- There is no visual distinction between "I said this" and "the worker
  said that."
- The view polls `/read-now` every 3 seconds, so snappy back-and-forth
  feels laggy and rapid state changes get compressed.
- `c4 send` does not append Enter, so the operator has to click a second
  button every turn to actually submit the message.

A dedicated chat view separates those concerns: bubbles preserve turn
ordering, alignment + colour separate roles at a glance, SSE delivers
output the instant the daemon emits it, and submission is a single
Enter-press that fires both `/send` and `/key {Enter}` in order.

## Implementation

### `web/src/components/ChatView.tsx` (new)

- `type Role = 'user' | 'worker'` + `ChatMessage { id, role, text, ts }`.
- `stripAnsi(input)` removes:
  - OSC sequences terminated by BEL (`\x1b]...\x07`) or ST
    (`\x1b]...\x1b\\`) so title / hyperlink payloads do not bleed into
    the chat text.
  - CSI sequences (`\x1b[...<final>`) for colour, cursor moves, clears.
  - Other common ESC dispatches (`\x1b = / > / ( / )`).
  - Lone `\r` (not followed by `\n`) rewritten to `\n` so a naked
    carriage return does not collapse the current line.
  - C0/C1 control characters in `[\x00-\x08\x0b-\x1f\x7f]`, leaving
    `\t` and `\n` intact.
- `b64decode(b64)` goes through `atob` + `Uint8Array` + `TextDecoder`
  so UTF-8 payloads round-trip without the `atob`-only
  Latin-1 pitfall.
- SSE subscription via `new EventSource(eventSourceUrl('/api/watch?...'))`:
  the URL helper appends the JWT as `?token=<jwt>` because EventSource
  cannot set `Authorization`. Each frame decodes, strips, appends to
  `pendingBufRef.current`, and resets a 1200ms flush timer. When the
  timer fires the buffered text is pushed as a single worker bubble --
  this debounce is what turns a chatty render pass (dozens of small
  cursor-move frames) into one legible message.
- Submission handler:
  1. Flush any pending worker buffer first so the user bubble lands
     after the prior worker bubble instead of racing it.
  2. Append the user bubble (right-aligned, blue).
  3. `apiFetch('/api/send', ..., { name, input: text })`.
  4. `apiFetch('/api/key', ..., { name, key: 'Enter' })`.
  5. Focus the textarea again for rapid follow-ups.
  The composer is a `<textarea rows=2>` with
  `onKeyDown`: Enter submits, Shift+Enter inserts a newline.
- Auto-scroll: a `useLayoutEffect` pins the scroll container to the
  bottom whenever `messages` changes and `autoScroll` is true. The
  `onScroll` handler recomputes
  `distanceFromBottom = scrollHeight - scrollTop - clientHeight` and
  toggles `autoScroll` based on a 24px threshold. When paused, a
  "Jump to latest" button in the header re-pins and resumes.
- A live / disconnected pill mirrors the EventSource `onopen` /
  `onerror` callbacks so the operator can tell whether streaming is
  actually flowing.
- Resets its own state on `workerName` change (empties messages,
  clears the pending buffer, cancels the flush timer) so switching
  workers does not cross-pollinate transcripts.
- Caps messages at `MAX_MESSAGES=300` via trailing-window slice so
  long-running sessions do not pin unbounded memory.

### `web/src/App.tsx` (modified)

- New `type DetailMode = 'terminal' | 'chat'` with
  `DETAIL_MODE_KEY = 'c4.detail.mode'` localStorage persistence
  (identical shape to the existing sidebar mode persistence).
- Adds a Terminal / Chat tab pair in the detail header (next to the
  existing worker title area). The tabs follow the same visual pattern
  as the List / Tree sidebar tabs (rounded border, highlighted selected
  button, `role="tablist"` + `aria-selected`).
- Renders `<WorkerDetail key={`term-${selectedWorker}`} />` in the
  Terminal branch and `<ChatView key={`chat-${selectedWorker}`} />` in
  the Chat branch. The distinct key prefixes force remount on worker
  swap and ensure the two views do not share mounted state.

### `tests/chat-view.test.js` (new)

21 assertions across 4 suites (`node:test`):

1. **ChatView stripAnsi** -- 7 cases: CSI colour, CSI cursor moves, OSC
   BEL, OSC ST, lone CR -> LF, C0/C1 control chars vs tab + newline,
   ASCII passthrough.
2. **ChatView b64decode** -- 2 cases: UTF-8 round-trip, composition
   with stripAnsi on an ANSI-laden payload.
3. **ChatView source wiring** -- 8 greps over `ChatView.tsx`: apiFetch
   + eventSourceUrl imports, `/api/watch` subscription shape, `/api/send`
   + `/api/key {Enter}` posts, conditional `justify-end` vs
   `justify-start`, autoScroll state + `distanceFromBottom`,
   `WORKER_FLUSH_MS` debounce constant, `b64decode(data.data)` call
   site, and the `export function stripAnsi` + `export function
   b64decode` visibility hooks.
4. **App.tsx integration** -- 4 greps: ChatView import,
   `c4.detail.mode` key + `localStorage.setItem` write, both Terminal +
   Chat tab labels + `setDetailMode` click handlers, and the `term-` /
   `chat-` React key prefixes.

The helper functions are re-implemented in JS inside the test rather
than loaded from the TSX because there is no TS test harness in the
repo; if the regexes drift between TSX and test the behavioural cases
will fail, keeping the two in lockstep.

`WorkerDetail`), so no server-side rollback is needed.
