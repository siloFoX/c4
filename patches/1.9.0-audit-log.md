# 1.9.0 - Audit Log (TODO 10.2)

## Summary

Append-only, tamper-evident audit trail for every security-relevant daemon
action. Writes one JSON event per line to `~/.c4/audit.jsonl` and binds
each line into a SHA-256 hash chain so the log cannot be silently edited
after the fact.

## Why

Before this release the daemon had no persistent record of who did what
when. A manager approving a `rm -rf` via `c4 approve`, a peer user
triggering `c4 merge`, a first-run login against a freshly provisioned
auth config - none of it was captured anywhere durable, so incident
response (or any kind of after-action review) had nothing to work from
beyond `c4 history` and console output, both of which are lossy.

The spec (TODO 10.2) asked for "who, when, what worker, what task,
what approval" coverage with an append-only, tamper-evident guarantee.
Web UI filter/search slots on top of this layer and lives in 10.x.

## Event types

Twelve canonical types. Daemon and CLI use these spellings; `EVENT_TYPES`
exports the full frozen list so consumers can enumerate without typos.

| Type | Fires on | Actor | Target |
|------|---------|-------|--------|
| `worker.created` | `POST /create` success | JWT sub or `system` | worker name |
| `worker.closed` | `POST /close` success | JWT sub or `system` | worker name |
| `task.sent` | `POST /task` success | JWT sub or `system` | worker name |
| `task.completed` | reserved for future wiring (task lifecycle hook) | - | - |
| `approval.requested` | reserved for future wiring (intervention-detected hook) | - | - |
| `approval.granted` | `POST /approve` with optionNumber=1 or null | JWT sub or `system` | worker name |
| `approval.denied` | `POST /approve` with any other optionNumber | JWT sub or `system` | worker name |
| `merge.performed` | `POST /merge` success | JWT sub or `system` | branch name |
| `config.reloaded` | `POST /config/reload` success | JWT sub or `system` | `config` |
| `auth.login` | `POST /auth/login` (both success and failure) | user name (success) or `anonymous` (failure) | user name |
| `auth.logout` | `POST /auth/logout` | JWT sub | same user |
| `fleet.changed` | reserved for future wiring (CLI-side fleet mutation) | - | - |

Events flagged "reserved" are accepted by `record()` today but are not
yet fired from any code path - they are the canonical names to use when
the follow-up hooks land so the event vocabulary stays stable.

## Event shape

```json
{
  "timestamp": "2026-04-17T10:15:42.881Z",
  "type": "worker.created",
  "actor": "alice",
  "target": "feature-worker-1",
  "details": {
    "command": "claude",
    "args": [],
    "target": "local",
    "cwd": "",
    "parent": "",
    "pid": 12345
  },
  "hash": "3f7c1a...<64 hex>"
}
```

`hash` is `sha256(prevHash + canonicalize(event))` where
`canonicalize()` serializes the event in a fixed key order
(`timestamp -> type -> actor -> target -> details`) so V8's
JSON.parse ordering cannot drift the hash. The first event's hash
uses `prevHash = ''`.

## Query examples

```bash
# Dump the full log
c4 audit query

# Every task.sent event in the last hour
c4 audit query --type task.sent --from 2026-04-17T09:00:00.000Z

# Every event touching worker-7
c4 audit query --target worker-7

# Last 20 approvals
c4 audit query --type approval.granted --limit 20

# Everything in a one-hour window
c4 audit query --from 2026-04-17T10:00:00.000Z --to 2026-04-17T11:00:00.000Z
```

Output is one JSON object per line, machine readable. Pipe through `jq`
for pretty-printing or filtering on arbitrary fields.

## Verify examples

```bash
# Clean log
$ c4 audit verify
[ok] audit log valid (142 events, path=/home/ops/.c4/audit.jsonl)

# Tampered log
$ c4 audit verify
[tamper] hash chain broken at line 87 (total=142, path=/home/ops/.c4/audit.jsonl)
# exit code 2
```

`corruptedAt` is the 0-based index of the first broken line. Any edit to
an earlier line invalidates every hash after it, so the number is always
the earliest tamper point, not necessarily the one an attacker touched.
To find the tampered line itself, compare the stored `hash` of
`corruptedAt - 1` against the recomputed value; if they match, the break
is at `corruptedAt` - otherwise the chain was broken earlier and
propagated forward.

## Compliance notes

- **Append-only on disk.** `fs.appendFileSync` is the only writer. The
  module never calls `writeFileSync`, `truncate`, or any mutating op.
  Anyone wanting to delete or rewrite lines has to do it out of band -
  and `verify()` will flag it on the next run.
- **Tamper evidence, not tamper resistance.** The hash chain detects
  modifications but does not prevent them. An attacker with write
  access to `~/.c4/audit.jsonl` can truncate, delete, or replay lines
  - the chain will fail to verify but the file itself is gone. For
  stronger guarantees, mount `~/.c4` read-only for the daemon user and
  rotate the file via a privileged writer, or ship each line to an
  external WORM store.
- **Actor is best-effort.** When auth is disabled (`config.auth.enabled
  !== true`), every event records `actor: "system"`. When auth is
  enabled, the actor is the JWT subject (`decoded.sub`), which is the
  login name - so the audit trail is only as trustworthy as the JWT
  signing secret.
- **No PII scrubbing.** `details` is recorded verbatim from the request
  body. Task text is truncated to 500 chars on `/task` to keep lines
  bounded, but otherwise nothing is redacted. Callers that pass secrets
  in task text will see those secrets land in the log. The `target`
  field is likewise verbatim.
- **Concurrency.** `record()` is synchronous and uses appendFileSync,
  which runs to completion inside a single JS event-loop turn.
  Concurrent callers in the same process cannot interleave. Multi-
  process writers to the same file are not supported - the daemon is
  the only intended writer.

## Test coverage

30 tests across six describe blocks:
- (a)-(e) Helper shape: `defaultLogPath`, `EVENT_TYPES` membership,
  canonical key order, `hashEvent` determinism and chain sensitivity.
- (f)-(l) `record`: JSONL append + ISO-8601 regex + full field set +
  default actor `system` + first-event hash = sha256 of canonical event
  + subsequent-event chain binding + tail-hash recovery across new
  logger instances.
- (m)-(s) `query`: non-existent -> `[]`, no-filter order, type /
  target / from / to / limit filters, combined filters.
- (t)-(y) `verify`: non-existent + fresh = valid; edited timestamp /
  edited details / corrupted JSON / deleted middle line all report
  `corruptedAt` at the correct index.
- (z)-(dd) Concurrency: 30-call burst valid chain, 10-call burst FIFO
  order, Promise.all-wrapped 20-call atomic serialize.
- (bb)-(cc) Shared singleton: `getShared` stable, `resetShared` clears.

Full suite: **81 / 81 pass** (80 existing + audit-log).

## Limitations / follow-ups

- Web UI filter/search view (10.x) not included - CLI-only today.
- `task.completed`, `approval.requested`, `fleet.changed` event types
  are named and reserved but not yet fired from any code path.
- No log rotation. On a busy fleet the file grows unbounded; operators
  should rotate externally with `logrotate` or similar.
- No multi-process writer coordination - the daemon is the only
  intended writer to `~/.c4/audit.jsonl`.
