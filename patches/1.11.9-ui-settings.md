# 1.11.9 - UI Settings page (TODO 8.17)

Adds a top-level Settings destination and centralizes the dashboard's
UI preferences. Before this change, sidebar mode / detail mode / top
view were persisted directly from `App.tsx` via inline
`window.localStorage` calls, theme was hard-coded to dark via the
`class="dark"` attribute in `web/index.html`, and there was no
in-product surface for toggling any of them.

## Why this exists

Three gaps motivated the change:

1. **No theme control.** `web/src/index.css` already ships a complete
   light-mode token set alongside the dark palette, but the app had no
   way to activate it short of editing `web/index.html` by hand.
2. **Scattered preference state.** `App.tsx` duplicated the same
   `try { window.localStorage... } catch { /* private mode */ }`
   pattern for three separate keys. Any future preference would have
   to re-derive the same guard logic in another place.
3. **Discoverability.** Sidebar mode (List / Tree) and detail view
   (Terminal / Chat / Control) were only reachable as inline toggles
   inside the respective panes. New users had no single surface to see
   or reset their choices.

## What it does

Three additions:

1. **`web/src/lib/preferences.ts` (new).** Typed helpers over
   `window.localStorage`:
   - Namespaced keys (`c4.sidebar.mode`, `c4.detail.mode`, `c4.topView`,
     `c4.theme`) and `DEFAULT_*` constants.
   - `readSidebarMode` / `readDetailMode` / `readTopView` / `readTheme`
     validate the raw string against the allowed set and fall back to
     the default — any corrupt or stale value is transparently dropped.
   - `writeSidebarMode` / `writeDetailMode` / `writeTopView` /
     `writeTheme` swallow `QuotaExceededError` in private-mode
     browsers (preferences stay in-memory for the session).
   - `writeTopView` short-circuits when called with `'settings'` so a
     visit to the Settings tab never becomes the persisted restore
     target — the saved value stays on the last content tab.
   - `resetPreferences` removes all four keys.
   - `resolveTheme(mode)` maps `'system'` to the OS preference via
     `window.matchMedia('(prefers-color-scheme: dark)')`.
   - `applyTheme(mode)` calls `resolveTheme` and toggles the `dark`
     class on `document.documentElement`.

2. **`web/src/components/SettingsView.tsx` (new).** Card + Panel page
   with two groups:
   - **Appearance** -- a `radiogroup` for theme (Light / Dark /
     System, lucide `Sun` / `Moon` / `Monitor`).
   - **Layout** -- radiogroups for sidebar mode (List / Tree) and
     detail view (Terminal / Chat / Control).
   The footer shows `Using defaults` / `Custom preferences active`
   and a `Reset to defaults` button (disabled when nothing is
   customized) that calls `resetPreferences()` and notifies the
   parent to snap component state back to `DEFAULT_*`.

3. **Wiring.**
   - `web/src/components/layout/TopTabs.tsx` adds `settings` to the
     `TopView` union (with the lucide `Settings` icon). The saved
     preference list intentionally excludes `'settings'` so
     `readTopView` can never return it.
   - `web/src/App.tsx` drops its inline `readSidebarMode` /
     `readDetailMode` / `readTopView` helpers, switches to the shared
     module, threads `theme` state + `applyTheme` into an effect,
     re-applies on OS color-scheme changes while `theme === 'system'`,
     and listens for `storage` events so two open tabs converge on
     the same preferences. A new branch in the view switch mounts
     `SettingsView` when `topView === 'settings'`.

## Tests

`tests/web-ui-settings.test.js` (new, 7 assertions):

- preferences lib exports the four namespaced keys, the `DEFAULT_*`
  constants, and the `read*` / `write*` / `resetPreferences` /
  `applyTheme` / `resolveTheme` surface.
- `writeTopView` guards the `'settings'` value.
- `applyTheme` operates on `document.documentElement` and toggles the
  `dark` class.
- `TopTabs` includes `'settings'` in the `TopView` union and imports
  the lucide `Settings` icon.
- `SettingsView` renders Appearance / Theme / Sidebar mode / Detail
  view controls, uses `role="radiogroup"`, and calls
  `resetPreferences()` on reset.
- `App.tsx` imports `SettingsView`, mounts it when
  `topView === 'settings'`, and calls `applyTheme(theme)`.

`web/` typecheck + build (`tsc --noEmit` + `vite build`) pass clean.
