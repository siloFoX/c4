# 1.7.6 — Intelligent exception recovery (TODO 8.4)

## Problem

Before this change the daemon's only response to a worker failure was the
existing intervention + escalation state in `src/pty-manager.js`: once the
same error line repeated `intervention.escalation.maxRetries` (default 3)
times, the worker's `_interventionState` flipped to `'escalation'`, a Slack
`[STALL]` went out, and the automation stalled until a human intervened.
The loop surfaced the problem but never made progress on it, which meant
an unattended overnight swarm would reliably freeze on the first flake.

TODO 8.4 asks for something smarter: analyze the failure, pick a strategy
that redefines or narrows the task, retry, learn from the outcome — and
above all stay non-destructive so an unattended pass can't rm code, push
force, or bypass checks.

## What changed

### New module `src/recovery.js`

Pure Node, no `node-pty` dependency at require time, so tests can drive
it directly with fakes.

- **`classifyError(scrollback, options)`** — tail-pattern classifier. Scans
  the last ~8 KB (overridable via `options.tailBytes`) against ordered
  buckets and returns `{ category, signal, confidence }` where `category`
  is one of:
  - `tool-deny` — `permission denied`, `tool ... denied`, `EACCES`,
    `EPERM`, `denied by policy`. Evaluated **before** the generic error
    fallback so a permission cascade never gets filed as `unknown`.
  - `timeout` — `ETIMEDOUT`, `ECONNABORTED`, `ESOCKETTIMEDOUT`,
    `request timed out`, `operation timed out`.
  - `test-fail` — jest `Tests: N failed`, bare `FAIL `, `npm ERR!.*test`,
    `AssertionError`, `Expected…Received`, pytest failures.
  - `build-fail` — TypeScript `TS\d{4}:` codes, `SyntaxError`,
    `ParseError`, `ReferenceError`, `Cannot find module`,
    `Module not found`, webpack/vite/eslint errors.
  - `dependency` — `npm ERR!.*ENOENT`, `Could not resolve dependency`,
    `peer dep missing`.
  - `unknown` — generic `Error|Exception|FAIL|failed` fallback with
    confidence 0.3, or nothing matched (confidence 0).

- **Strategy registry** — four pluggable strategies, each exposing
  `transform(originalTask, context)`:
  - `retry-same` — pass the task through unchanged (good for flakes).
  - `retry-simpler` — prepend `[C4 RECOVERY]` banner + run
    `stripTaskOptions` (drop bullet / numbered / `options:` / `[opts:]`
    lines) so the re-ask focuses on the core verb.
  - `retry-with-smaller-scope` — prepend a banner asking the worker to
    narrow to ONE file or ONE failing test; keeps the original task as
    reference.
  - `ask-manager` — returns `null` from `transform`, signaling the
    orchestrator to notify instead of re-sending.

- **`pickStrategy(category, attempt, config)`** — picks a strategy for
  the given category + attempt, respecting
  `config.recovery.strategies.<category>` overrides and filtering
  unknown strategy names. Defaults:

  | category | ordering |
  | --- | --- |
  | `test-fail` | `retry-same -> retry-simpler -> retry-with-smaller-scope -> ask-manager` |
  | `build-fail` | `retry-simpler -> retry-with-smaller-scope -> ask-manager` |
  | `tool-deny` | `retry-simpler -> ask-manager` |
  | `timeout` | `retry-same -> retry-same -> ask-manager` |
  | `dependency` | `ask-manager` (npm install cascades are rarely fixable by re-asking the worker) |
  | `unknown` | `retry-simpler -> ask-manager` |

- **History** — `appendHistory(projectRoot, entry)` writes one JSON line
  to `<projectRoot>/.c4/recovery-history.jsonl` (creates `.c4/` on
  demand, tolerates write errors). `readHistory(projectRoot, filter)`
  reads the file with optional `worker` / `category` filters and a
  trailing `limit`. Malformed lines are skipped, not fatal.

- **`recoverWorker(manager, name, options)`** — orchestrator. Contract:
  - Gated on `config.recovery.enabled === true` unless `options.manual`
    is `true` (`c4 recover <name>`). When disabled, returns
    `{ skipped: true, reason: 'recovery-disabled' }` without touching
    anything.
  - Checks `worker._interventionState`. When it's `question` or
    `critical_deny` we return `{ skipped: true, reason: 'intervention:<kind>' }`
    — those states mean "a human has to make this call", and blindly
    clearing them would be exactly the kind of blind auto-approval the
    known-issues doc calls out.
  - Reads the scrollback via `manager.getScrollback(name, 400)`, runs
    `classifyError`, derives the attempt counter from existing history
    (so repeat invocations escalate through the strategy list), picks a
    strategy.
  - Over `config.recovery.maxAttempts` (default 3): emits `ask-manager`
    + `[RECOVERY] ... max attempts reached` Slack push, records
    `phase='give-up'` in history, returns without acting.
  - For acting strategies: calls `manager.sendTask(name, transformed,
    { reuse: true, autoMode: config.recovery.autoMode })`. `reuse: true`
    keeps the worker on its existing branch/worktree. The `autoMode`
    flag defaults to `false` so recovery never silently flips a worker
    into unattended mode unless the operator opted in.
  - **Never calls** `close` / `rollback` / `cleanup`, never forwards
    `skipChecks`, never rewrites git state. The test suite rigs those
    methods to throw if recovery ever touches them, so any accidental
    invocation would fail loudly.
  - Every pass (applied / send-failed / ask-manager / give-up) appends
    one audit line to `.c4/recovery-history.jsonl`.

### Daemon wiring (`src/daemon.js`)

1. `require('./recovery')`.
2. `POST /recover { name, category? }` -> `recovery.recoverWorker(manager,
   name, { manual: true, categoryHint: category })`. `manual: true`
   forces the orchestrator to run even when `config.recovery.enabled`
   is false so an operator can trigger a pass on demand.
3. `GET /recovery-history?name=&limit=` -> `recovery.readHistory(...)`
   with the worker's project root resolved via
   `config.worktree.projectRoot` (fallback to `path.resolve(__dirname, '..')`).
4. `manager.on('sse', event => ...)` listens for
   `{ type: 'error', escalation: true }` events (the same ones
   `PtyManager` already emits when `_interventionState` flips to
   `'escalation'`), filters on `config.recovery.enabled === true`, and
   fires `recoverWorker(manager, event.worker, { manual: false })`.
   A `_recoveryLastRun` `Map<workerName, timestamp>` +
   `RECOVERY_DEBOUNCE_MS = 30000` per-worker gate keeps a stuck loop
   from retry-storming every frame. Console logs surface the outcome
   (`strategy=...` / `skipped (...)` / `failed for ...`).

All three new routes sit inside the existing `auth.checkRequest` gate so
JWT auth (8.14) is honored transparently.

### CLI (`src/cli.js`)

`c4 recover <name> [--category X] [--history] [--limit N]`:

- Default mode: hits `POST /recover`, prints `strategy / category /
  attempt / action / recovered / history`. Skipped passes surface the
  reason (`recovery-disabled` / `intervention:question` / etc).
- `--category` forces a classifier category, useful when the operator
  already knows why the worker died and wants to pick a strategy from
  the matching override list.
- `--history [--limit N]` dumps the last N JSONL entries for the named
  worker. Useful for post-mortem after a night of unattended runs.

Help text updated with a two-line `recover <name>` section under
`rollback`.

### Config (`config.example.json`)

New `recovery` section sitting right under `intervention`:

```json
"recovery": {
  "enabled": false,
  "maxAttempts": 3,
  "autoMode": false,
  "strategies": {
    "test-fail": ["retry-same", "retry-simpler", "retry-with-smaller-scope", "ask-manager"],
    "build-fail": ["retry-simpler", "retry-with-smaller-scope", "ask-manager"],
    "tool-deny": ["retry-simpler", "ask-manager"],
    "timeout": ["retry-same", "retry-same", "ask-manager"],
    "dependency": ["ask-manager"],
    "unknown": ["retry-simpler", "ask-manager"]
  }
}
```

`enabled: false` keeps the change opt-in so existing installs don't
suddenly start auto-retrying after the bump. Turn it on for supervised
overnight runs.

## Tests

`tests/recovery.test.js` adds 45 assertions across 9 node:test suites.
The test module requires `../src/recovery` directly (no `node-pty` dep,
no regex-extract trickery) and drives `recoverWorker` with fake managers
that record sendTask / notify calls.

- Strategy transforms: retry-same unchanged; retry-simpler banner +
  stripped options; retry-with-smaller-scope banner keeps reference;

## Non-destructive guarantees

The whole point of the TODO is unattended self-healing. If the module
had a path to `rm -rf`, `git push --force`, or bypassing `c4 merge`
checks, it would be the biggest footgun in the repo. Explicit contract:

1. The only mutation the orchestrator performs on the worker is
   `manager.sendTask(name, transformed, { reuse: true, autoMode })`.
   `reuse: true` forbids creating a fresh branch / worktree.
2. `skipChecks` is never forwarded. `c4 merge` gating is untouched.
3. Git state is never modified (no reset, no checkout, no push).
4. `close` / `rollback` / `cleanup` on the manager are never invoked.
   Tests rig them to throw; the test suite would fail if that ever
   slipped.
5. `_interventionState === 'question'` or `'critical_deny'` short-circuits
   the pass. Human-needed states stay human-needed.
6. History is append-only. The module never deletes or rewrites its
   own JSONL.
7. `maxAttempts` (default 3) is a hard ceiling. Past it, the worker is
   handed off to the manager via notify; recovery never loops forever.
8. Auto-recovery is gated on `config.recovery.enabled === true` — default
   is `false`, so the feature is opt-in.
