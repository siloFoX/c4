# 1.8.4 - Local LLM adapter (Ollama / llama.cpp / vLLM) + hybrid routing

Closes TODO 9.2. Plugs a self-hosted inference backend into the 9.1 Agent
Framework so the daemon can drive Ollama / llama.cpp / vLLM with the same
state machine (scrollback, idle detection, hook pipeline, stall timer) it
uses for Claude Code. No PtyManager rewrite.

## What's new

### `src/agents/local-llm.js`

`LocalLLMAdapter` extends `Adapter` and acts as a pseudo-PTY. Three
backend-pinned subclasses are exported:

- `LocalOllamaAdapter`   -> `POST <url>/api/generate`            (JSONL stream)
- `LocalLlamaCppAdapter` -> `POST <url>/v1/chat/completions`     (OpenAI SSE)
- `LocalVllmAdapter`     -> `POST <url>/v1/chat/completions`     (OpenAI SSE)

Defaults:

| Backend    | URL                        | Model                          |
| ---------- | -------------------------- | ------------------------------ |
| ollama     | `http://localhost:11434`   | `llama3.1`                     |
| llama-cpp  | `http://localhost:8080`    | `local-model`                  |
| vllm       | `http://localhost:8000`    | `meta-llama/Llama-3.1-8B`      |

#### Adapter interface

Same contract as every other 9.1 adapter:

- `metadata` -> `{ name: 'local-llm', version: '1.0.0', backend }`
- `supportsPause` -> `true` (inference can be aborted via `AbortController`)
- `init(ctx)` / `sendInput(text)` / `sendKey(key)` / `onOutput(cb)` /
  `detectIdle(chunk)`

#### Pseudo-PTY lifecycle

In addition to the Adapter surface, the class exposes PTY-style methods so a
daemon can treat the adapter as both the runtime and the terminal:

- `spawn(opts)`       -> set up the internal `ScreenBuffer`, emit the initial
                         `> ` prompt, assign a synthetic pid
- `write(data)`       -> echo input to the screen, buffer until CR/LF, then
                         fire `runInference(prompt)` and return the Promise
- `resize(cols, rows)`-> clamp + forward to `ScreenBuffer.resize`
- `kill(signal?)`     -> abort any in-flight request
- `dispose()`         -> abort + clear listeners + reset state

#### Streaming

- Ollama: newline-delimited JSON. Tokens come from each `response` field;
  `done:true` terminates. History is not retained (single-shot API).
- OpenAI-compat: SSE frames separated by `\n\n`. Tokens come from
  `choices[0].delta.content`. `data: [DONE]` terminates. The adapter keeps a
  `_history` list of `{role:'user'|'assistant', content}` turns so multi-turn
  conversations stay coherent.

Both parsers handle fragmented streams that arrive across multiple TCP chunks.

#### Error handling

Every failure path is in-band: connection refused, HTTP 500, missing fetch,
stream decode errors. The adapter writes a single line to the screen and the
output fan-out:

```
[local-llm:ollama] error: <message>
```

No exception leaks to the caller; `_busy` is released in `finally`, so a
stuck inference never pins the adapter.

### `src/agents/index.js`

Three new REGISTRY keys: `local-ollama`, `local-llama-cpp`, `local-vllm`.
Per-type option sub-bags are resolved via `agentConfig.options[type]`, with
a flat-options fallback so legacy callers keep working.

#### Hybrid routing

When `agentConfig.type === 'hybrid'` (or any call passes
`legacyOpts.hybrid === true`), the factory picks a concrete type based on the
task string:

```
isComplexTask(task, { threshold, keywords }):
  task.length > threshold
    || task contains any keyword (case-insensitive)
```

Default threshold is 2000 chars. Default keywords:
`['refactor', 'architect', 'architecture', 'design']`. Complex tasks route to
`agentConfig.complex` (default `claude-code`), everything else routes to
`agentConfig.local` (default `local-ollama`).

### `config.example.json`

```json
"agent": {
  "type": "claude-code",
  "local": "local-ollama",
  "complex": "claude-code",
  "hybridThreshold": 2000,
  "complexKeywords": ["refactor", "architect", "architecture", "design"],
  "options": {
    "claude-code": {},
    "local-ollama":    { "url": "http://localhost:11434", "model": "llama3.1" },
    "local-llama-cpp": { "url": "http://localhost:8080",  "model": "local-model" },
    "local-vllm":      { "url": "http://localhost:8000",  "model": "meta-llama/Llama-3.1-8B" },
    "local-llm":       { "endpoint": "http://127.0.0.1:11434", "model": "qwen2.5-coder:7b" },
    "codex": {},
    "claude-agent-sdk": {}
  }
}
```

Flip `"type": "hybrid"` to use the heuristic, or pin to a specific backend
such as `"local-vllm"` for always-local operation.

## Running a local backend

The adapter never ships a server. Bring your own inference runtime:

### Ollama

```bash
# Install from https://ollama.com
ollama pull llama3.1
ollama serve                                 # binds localhost:11434
# sanity check:
curl -s http://localhost:11434/api/generate \
  -d '{"model":"llama3.1","prompt":"hello","stream":true}'
```

### llama.cpp

```bash
# Build llama.cpp and download a GGUF model
./llama-server -m /path/to/model.gguf --port 8080 --host 0.0.0.0
# sanity check:
curl -s http://localhost:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"local-model","messages":[{"role":"user","content":"hi"}],"stream":true}'
```

### vLLM

```bash
# Python env with vLLM installed
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-8B \
  --host 0.0.0.0 --port 8000
# sanity check:
curl -s http://localhost:8000/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"meta-llama/Llama-3.1-8B","messages":[{"role":"user","content":"hi"}],"stream":true}'
```

## Hybrid routing walkthrough

```js
const { createAdapter } = require('./src/agents');

// Short task -> local-ollama
createAdapter({ type: 'hybrid' }, { task: 'list tests in tests/' });
// Long task -> claude-code
createAdapter({ type: 'hybrid' }, { task: 'y'.repeat(3000) });
// Keyword match -> claude-code
createAdapter({ type: 'hybrid' }, { task: 'refactor the dispatcher' });
// Override knobs at config time
createAdapter(
  { type: 'hybrid', hybridThreshold: 500, complex: 'local-vllm' },
  { task: '...' }
);
```

## Tests

`tests/local-llm.test.js` adds 40 assertions across 8 node:test suites. All
suites stub `global.fetch` (via `opts.fetch`) and feed responses through an
in-process `ReadableStream`, so no real LLM server is contacted.

Coverage highlights:

- construction + defaults (3 subclasses, trailing slash stripping, unknown
  backend rejection, `BACKENDS` constant)
- `buildRequest` payload shape (ollama / llama-cpp / vllm, systemPrompt
  injection)
- ollama JSONL streaming, fragmented re-assembly, no-history invariant
- OpenAI SSE streaming, `[DONE]` halt, user+assistant history, fragmented
  re-assembly
- error handling (ECONNREFUSED, HTTP 500, `fetch:null`)
- adapter + PTY lifecycle (spawn prompt, resize, sendKey, detectIdle, write
  returns inference Promise, dispose aborts in-flight + clears listeners)
- hybrid heuristic (short / long / keyword, custom threshold + keywords +
  local / complex targets)
- factory integration (REGISTRY keys, explicit selection, nested options,
  hybrid type, `legacyOpts.hybrid:true` override, default `claude-code`)

Full suite: `80 / 80` pass.

## Limitations

1. **No token-level cost accounting.** Local backends do not charge per token,
   so `config.workerDefaults.maxBudgetUsd` has no effect when the adapter is
   the active runtime. `config.workerDefaults.maxRetries` still applies.
2. **No tool-calling surface.** Ollama + llama.cpp + vLLM can run base models
   that do not emit tool-call schemas by default. The adapter treats each
   response as free-form text; writing files / running shell commands from a
   local LLM is a follow-up.
3. **One in-flight request per adapter.** `write()` / `runInference()` short-
   circuit when `_busy` is true. Queueing is the daemon's job (same as every
   other adapter in the framework).
4. **No daemon wiring yet.** The adapter is self-contained and tested, but
   `src/pty-manager.js` still spawns real PTYs for Claude Code. Swapping the
   spawn path to consult `createAdapter()` and treat the returned adapter as
   both runtime + terminal is a follow-up phase of 9.1.

## Follow-ups (not in this patch)

- Tool-calling for models that expose an OpenAI-style function-call API.
- Daemon integration so `c4 new --agent local-ollama` spawns a worker whose
