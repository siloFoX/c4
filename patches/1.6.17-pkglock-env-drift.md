# Patch 1.6.17 - package-lock.json env-drift guard (7.29)

## Date: 2026-04-17

## Summary

`web/package-lock.json` had been showing up as `modified` at the start of
every session without any intentional edits, becoming a stash target during
`c4 merge` and producing diffs with no real content. Investigation identified
the root cause as npm version/platform drift between committers. The fix is
a static detector for the drift fingerprint + a `.githooks/pre-commit`
warning + documentation of the recommended workflow. `package-lock.json`
is explicitly NOT gitignored — that would break `npm ci` reproducibility.

## Root cause

The committed lockfile contains 8 `"peer": true` metadata entries emitted by
one npm/environment. The other environment's npm strips them on the next
install (e.g. `npm --prefix web install` / `npm --prefix web run dev`).
Real dependency updates always touch other fields (version, integrity,
resolved); a diff of `"peer": true` removals/additions only is a reliable
fingerprint of drift rather than a real change.

Reproduction (deterministic):

```
# Copy the committed lockfile + package.json to a temp dir
git -C /root/c4 show HEAD:web/package-lock.json > /tmp/pl/package-lock.json
cp /root/c4/web/package.json /tmp/pl/package.json

# Ask npm to refresh the lockfile only (no node_modules churn)
npm --prefix /tmp/pl install --package-lock-only

# 8 "peer": true lines removed
grep -c '"peer": true' /tmp/pl/package-lock.json   # 0 (committed: 8)
```

Affected packages (all reachable via peer edges as well as direct): the
`@babel/core` tree, `@types/react`, `browserslist`, `jiti`, `postcss`,
`react`, `yaml`, `@vitejs/plugin-react`.

No c4 code path runs `npm install` — `grep` for `npm install` / `npm ci` /
`package-lock` across `src/` returned 0 hits. The trigger is always a
manual user invocation of `npm` in `web/` or the repo root.

## Changes

### src/pkglock-guard.js (new)

Pure-JS module, spawn injectable for tests:

- `PEER_LINE` — regex for a `+`/`-` diff line whose only content is
  `"peer": true` with optional trailing comma and whitespace.
- `analyzeDiff(diff)` — scans a unified diff string. Skips `diff --git`,
  `index`, `---`, `+++`, `@@`, `\ No newline`, and context lines. For every
  `+`/`-` line, matches against `PEER_LINE` and counts `peerLines` vs
  `otherLines`. Returns `{ isPeerDriftOnly, peerLines, otherLines }` where
  `isPeerDriftOnly = peerLines > 0 && otherLines === 0`.
- `buildAdvice(path, peerLines)` — operator-facing message naming the path,
  the count, the `npm --version` check, the `git checkout --` revert, and
  the "warning only — the commit proceeds" line.
- `runCli(argv, { spawn, stderr })` — CLI entry. For each path in argv,
  runs `git diff --cached -- <path>`, passes stdout to `analyzeDiff`, and
  writes `buildAdvice` to stderr when the drift signature hits. Silent when
  the diff is empty, has real churn, or git exits non-zero.
- `require.main === module` branch so `.githooks/pre-commit` can
  `node src/pkglock-guard.js <path>`.

### .githooks/pre-commit (additive)

After the existing compound-command warning block, the hook now runs:

```sh
repo_root=$(git rev-parse --show-toplevel 2>/dev/null)
guard_script="$repo_root/src/pkglock-guard.js"
if command -v node >/dev/null 2>&1 && [ -f "$guard_script" ]; then
  staged=$(git diff --cached --name-only 2>/dev/null)
  for lock in web/package-lock.json package-lock.json; do
    if echo "$staged" | grep -qx "$lock"; then
      node "$guard_script" "$lock" || true
    fi
  done
fi
```

Staged path detection uses `grep -qx` to match only exact paths (avoids
false hits on `something/web/package-lock.json.bak`). `|| true` ensures
the hook never blocks the commit even if the guard throws. `node` absence
is treated as "skip" — not every developer machine has node in `$PATH`
when running git hooks.

### tests/pkglock-guard.test.js (new)

27 assertions across 5 suites:

- `PEER_LINE` regex: matches `-` / `+` / no-comma variants; rejects
  `integrity` and `dev:true`.
- `analyzeDiff`: empty/null/non-string guard; single/multiple peer
  removals; peer additions (reverse drift); mixed peer + non-peer (not
  drift); non-peer only (not drift); headers-only hunk (not drift);
  context + `\ No newline` ignored; CRLF handling; real-world fixture.
- `buildAdvice`: path + count + npm hint + git-checkout hint + 7.29
  reference + warning/proceeds wording.
- `runCli`: no-op on empty argv; advice on drift diff; silent on real
  churn / non-zero git / empty stdout; iterates every path argument. All
  tests run with a table-style spawn stand-in and a capture-stream stderr
  so no real git/node process is spawned.
- `.githooks/pre-commit` integration: source-level check that the hook
  references `pkglock-guard.js` and `web/package-lock.json`.

### tests/fixtures/pkglock-peer-drift.diff (new)

Captures the actual 8-line drift diff observed on 2026-04-17 (`git -C
/root/c4 diff web/package-lock.json`). Pinning the real payload makes the
regression test immune to future regex tweaks that might unintentionally
loosen the fingerprint.

## Operator notes

- The guard is a warning, not a block. Commits containing env-drift
  lockfile changes still succeed — the goal is to surface the cause so
  the committer knows to either revert or justify.
- When pulling from upstream and seeing `web/package-lock.json` as
  `modified` with the drift signature, the recommended action is
  `git checkout -- web/package-lock.json` + investigate `npm --version`
  mismatch. Do NOT `git add` blindly — that starts the ping-pong.
- If a long-term fix is pursued, the right lever is pinning npm across
  machines (via `engines.npm` / `.nvmrc` / volta) so the lockfile output
  is idempotent regardless of committer. That is out of scope for this
  patch because it requires coordination across developer machines.
