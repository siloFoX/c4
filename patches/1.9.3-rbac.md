# 1.9.3 - Role-Based Access Control (TODO 10.1)

## Summary

Three-tier role-based access control on top of session auth (8.14) and
the audit trail (10.2). Roles (`admin` / `manager` / `viewer`) plus
optional per-resource grants (project, machine) decide whether a JWT
holder can hit a given route. The decision point is a single
`requireRole(authCheck, action, resource?)` middleware in
`src/daemon.js`, fed by a `RoleManager` instance backed by
`~/.c4/rbac.json`. The CLI (`c4 rbac …`) and a `/rbac/*` REST surface
let operators introspect and mutate the store without restarting the
daemon.

## Why

8.14 (session auth) and 10.2 (audit log) closed the "anyone with a
token can do anything" gap and the "we cannot reconstruct who did
what" gap respectively. 10.1 closes the third one - "we cannot scope
what each authenticated user can do." The spec asks for admins
(everything), managers (worker lifecycle + project + fleet add +
config reload), and viewers (read-only), plus per-project and
per-machine grants so a manager only sees workers on their assigned
hardware.

Keeping RBAC in its own module - and keeping the JWT integration
purely a `roleResolver` callback in `src/auth.js` - means auth.js does
not gain a hard dependency on rbac.js, so existing operators who do
not opt into rbac get the existing behaviour and the JWT just gains a
`role: 'viewer'` claim by default.

## Storage

Single JSON file at `~/.c4/rbac.json` (overridable via
`config.rbac.path`). Schema:

```json
{
  "roles": {
    "admin":   ["*"],
    "manager": ["worker.create", "worker.close", "worker.task", "worker.merge",
                "project.create", "project.read", "project.update",
                "fleet.add", "config.reload", "audit.read"],
    "viewer":  ["project.read", "audit.read"]
  },
  "users": {
    "alice": { "role": "admin",   "projectIds": [],         "machineAliases": [] },
    "bob":   { "role": "manager", "projectIds": ["arps"],   "machineAliases": ["dgx"] },
    "carol": { "role": "viewer",  "projectIds": [],         "machineAliases": [] }
  },
  "resources": {
    "projects": {
      "arps":   { "allowedRoles": [],          "allowedUsers": ["bob"] },
      "secret": { "allowedRoles": ["admin"],   "allowedUsers": [] }
    },
    "machines": {
      "dgx": { "allowedRoles": [],             "allowedUsers": ["bob"] }
    }
  }
}
```

The shared `RoleManager` is cached in the daemon process and reloaded
on `POST /config/reload` so a `c4 rbac` mutation made via direct file
edit is picked up without a daemon restart. Tests construct their own
`RoleManager` pointed at `fs.mkdtempSync` and never touch the shared
instance.

## Role-action matrix

| Action              | admin | manager | viewer |
|---------------------|:-----:|:-------:|:------:|
| `worker.create`     |  Y    |   Y     |        |
| `worker.close`      |  Y    |   Y     |        |
| `worker.task`       |  Y    |   Y     |        |
| `worker.merge`      |  Y    |   Y     |        |
| `project.create`    |  Y    |   Y     |        |
| `project.read`      |  Y    |   Y     |   Y    |
| `project.update`    |  Y    |   Y     |        |
| `fleet.add`         |  Y    |   Y     |        |
| `fleet.remove`      |  Y    |         |        |
| `config.reload`     |  Y    |   Y     |        |
| `auth.user.create`  |  Y    |         |        |
| `audit.read`        |  Y    |   Y     |   Y    |

`admin` is `['*']` so any future action is auto-granted; `manager` and
`viewer` are explicit so a new action lands in a denied-by-default
state until reviewed.

## Resource scoping

`checkPermission(user, action, resource?)` returns true only when
(1) the user exists, (2) the role allows the action (or holds `*`),
and (3) the resource is reachable for that user. Resource shape is
`{type:'project'|'machine', id:'<projectId|alias>'}` and the rules:

- `admin` always passes resource scoping (matches the "admin = full
  access" guarantee in the spec).
- A resource without any registered ACL is unscoped - the call falls
  through to the role-only check. This means flipping rbac on does
  not retroactively lock everyone out of every existing project /
  machine.
- A resource with an ACL passes when the user's role appears in
  `allowedRoles` OR the user appears in `allowedUsers`. Either is
  sufficient.
- `grantProjectAccess(user, projId)` and `grantMachineAccess(user,
  alias)` append to both `users[user].projectIds|machineAliases` and
  the resource's `allowedUsers` so the inverse lookup is cheap.

## JWT integration

`src/auth.js` now exposes a `login(cfg, body, opts)` overload where
`opts.roleResolver(name) -> string|null` injects the RBAC role into
the signed payload. The daemon wires it to `roleFor(name)` which
checks the rbac store first then `config.auth.users[name].role`. The
fallback chain is: resolver -> `user.role` field -> `'viewer'`. The
token payload is now `{sub, role}` so downstream services that only
verify the JWT (web UI, fleet peers) can render permission-scoped UI
without an extra rbac round-trip.

## Daemon middleware

`requireRole(authCheck, action, resource?)` is the single decision
point. When `auth.enabled` is false it returns `{allow: true}` so the
existing behaviour stays intact for operators who have not enabled
auth yet. When auth is on it pulls the username from the JWT, calls
`rbacManager.checkPermission`, and returns `{allow: false, status:
403, body: {error, action, user}}` on a deny.

Routes gated:

| Route                                      | Action            | Resource scope          |
|--------------------------------------------|-------------------|-------------------------|
| `POST /create`                             | `worker.create`   | machine (target)        |
| `POST /task`                               | `worker.task`     | machine (target)        |
| `POST /merge`                              | `worker.merge`    | -                       |
| `POST /close`                              | `worker.close`    | -                       |
| `POST /config/reload`                      | `config.reload`   | -                       |
| `GET /audit/query` + `GET /audit/verify`   | `audit.read`      | -                       |
| `GET /projects`                            | `project.read`    | -                       |
| `POST /projects`                           | `project.create`  | -                       |
| `GET /projects/:id`                        | `project.read`    | per-project             |
| `POST /projects/:id/tasks` + PATCH         | `project.update`  | per-project             |
| `POST /rbac/role/assign`                   | `auth.user.create`| -                       |
| `POST /rbac/grant/project|machine`         | `auth.user.create`| -                       |
| `POST /rbac/revoke/project|machine`        | `auth.user.create`| -                       |

Read-only RBAC introspection (`GET /rbac/roles`, `GET /rbac/users`,
`POST /rbac/check`) is intentionally open so operators can see the
matrix without elevated privileges.

## CLI

```bash
# Roles + users
c4 rbac role list
c4 rbac role assign alice admin
c4 rbac role assign bob   manager
c4 rbac role assign carol viewer
c4 rbac users

# Per-resource grants
c4 rbac grant project bob arps
c4 rbac grant machine bob dgx
c4 rbac revoke project bob arps
c4 rbac revoke machine bob dgx

# Permission probe
c4 rbac check bob worker.merge --resource project:arps
```

## Tests

`tests/rbac.test.js` ships 37 tests across six suites, all writing to
`fs.mkdtempSync` paths so the operator's real `~/.c4/rbac.json` is
never touched:

1. Helpers - `defaultRbacPath`, `ROLES`, `ACTIONS` enum, validators
   (`isRole` / `isAction` / `isUsername`), `freshState`, `ensureShape`
   repair, `normalizeAcl` filtering.
2. `DEFAULT_PERMISSIONS` matrix - admin wildcard, manager lifecycle
   actions, manager denied auth.user.create + fleet.remove, viewer
   read-only.
3. `assignRole` + storage - file creation, reload from disk, role
   swap preserves grants, invalid role / username rejected,
   `listUsersByRole` filters, `removeUser` idempotent.
4. `checkPermission` - admin bypasses all 12 actions, viewer blocked
   from worker.create / task / merge + project.create / update +
   config.reload, viewer can project.read + audit.read, manager merge
   own project but not others, grant project / machine + revoke
   roundtrip, unknown user denied, unknown action denied for
   non-admin, admin bypasses resource scoping, unscoped resource
   falls through, allowedRoles in ACL grants whole role, revoke
   idempotent on missing grant.
5. `auth.login` JWT payload - resolver injects role, fallback to
   `user.role`, default to viewer, bad credentials still rejected.
6. Shared singleton - `getShared` returns same instance,
   `resetShared` clears, no real `~/.c4` pollution.

Full suite 84 / 84 pass (up from 83).

## Limitations

- Web UI permission-scoped views are a 10.x follow-up. The UI consumes
  `POST /rbac/check` per route to hide buttons the caller cannot
  reach; this patch ships the storage + REST + JWT + CLI layer so
  the UI can be added without schema churn.
- The `/rbac/role/assign` endpoint cannot self-bootstrap an admin.
  The first admin must be inserted via `c4 init` or by editing
  `~/.c4/rbac.json` directly because `auth.user.create` is admin-only.
- `fleet.remove` is `admin`-only by default. We considered granting it
  to managers but the spec explicitly calls out "manager: assigned
  machines" and removing a fleet entry deletes the alias for every
  user; that felt like an admin-tier action.
- `audit.read` is granted to viewers. Reading the trail is the only
  way for an SOC-style viewer to spot anomalies, and the audit log is
  already restricted to security-relevant events; per-event sensitivity
  filtering can land later if needed.
