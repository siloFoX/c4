# 1.9.1 - Cost Report + Billing (TODO 10.5)

## Summary

Aggregates token usage and USD cost across workers, projects, teams,
machines, and users, with month-boundary helpers and a budget check that
warns at 80% and blocks at 100% of the configured limit. Implemented as a
pure in-memory reporter (`src/cost-report.js`) so unit tests never touch
the daemon or filesystem, plus three daemon endpoints and a `c4 cost`
CLI subcommand that operators can point at any historical window.

## Why

The daemon already tracked daily token counts via `tokenMonitor` (2.5),
but the readout was a single "today" number with no way to slice by
worker, project, or machine, no USD conversion, and no per-model
breakdown. The spec (TODO 10.5) asked for per-project / per-team /
per-machine aggregation, automatic monthly reports, and a budget-limit
alarm - all of which require structured cost accounting on top of the
raw token counters.

Keeping the reporter pure (no node-pty, no network, no shared
singleton) makes the math auditable, testable, and reusable:
the same `CostReporter` drives the CLI, the daemon endpoints, and
future Web UI dashboards, and each layer chooses its own record source.

## Config

New `costs` section in `config.example.json`:

```json
"costs": {
  "models": {
    "claude-opus":   { "input": 15,  "output": 75 },
    "claude-sonnet": { "input": 3,   "output": 15 },
    "claude-haiku":  { "input": 0.8, "output": 4 },
    "local":         { "input": 0,   "output": 0 },
    "default":       { "input": 3,   "output": 15 }
  },
  "budget": {
    "defaultPeriod": "month",
    "warnAt": 0.8,
    "monthlyLimitUSD": 0
  }
}
```

Each model rate is USD per 1K tokens. `default` is the fallback for
records whose `model` field is unknown (e.g. a freshly rolled-out model
name) so reporting never blows up on unexpected input.

## Cost formula

For every record with `{ model, inputTokens, outputTokens }`:

```
cost = (inputTokens / 1000) * rate.input
     + (outputTokens / 1000) * rate.output
```

Totals are summed across records and rounded to four decimal places so
operators can eyeball sub-penny differences without format noise.
`byGroup` entries are sorted by `costUSD` descending, ties broken by
group name.

## Record shape

`CostReporter` accepts a flat array of usage records either at
construction (`{ records: [...] }`) or through a lazy loader
(`{ loadRecords: ({from, to}) => [...] }`). Each record is
`{ timestamp, project, team, machine, user, worker, model,
inputTokens, outputTokens }`. The daemon wiring feeds it rows from
`history.jsonl` via `loadHistoryRecords`; tests pass in-memory arrays
directly.

## CLI

```bash
# Full month-to-date report grouped by project
c4 cost report --from 2026-04-01T00:00:00Z --to 2026-04-30T23:59:59Z

# Monthly report with per-model breakdown, JSON output
c4 cost monthly 2026-04 --json

# Regroup by team and include per-model perModel field
c4 cost report --group team --models

# Budget check: daily limit 20 USD, alert on exceed
c4 cost budget --limit 20 --period day

# Per-project budget check
c4 cost budget --limit 100 --period month --group arps
```

`cost budget` prints one of three statuses:

- `[OK]` - used is under the warn threshold (default 80%).
- `[WARN]` - used is at or above `warnAt` but below the limit.
- `[EXCEEDED]` - used has reached or passed the limit.

## Daemon endpoints

```
GET  /cost/report?from=&to=&group=&models=
GET  /cost/monthly/:year/:month?group=
POST /cost/budget   { limit, period, group, groupBy, warnAt }
```

All three rebuild the `CostReporter` per request so `c4 config reload`
takes effect without a daemon restart. Each endpoint returns plain
JSON with `{ total, byGroup, period }` plus endpoint-specific fields
(`month`, `percent`, `warning`, `exceeded`).

## Tests

`tests/cost-report.test.js` ships 46 assertions across seven suites:

1. Helpers - `DEFAULT_COSTS` entries, `VALID_GROUP_BY` / `VALID_PERIODS`
   membership, `monthRange` for regular + leap-year + January + invalid,
   `periodRange` for day/week/month.
2. `getRate` - known model, unknown-model fallback to `default`,
   missing `default` falls back to zero, `costForRecord` per-1K math.
3. `groupBy` variations - project, team, machine, user, worker,
   unknown-label fallback, custom `unknownLabel`, invalid groupBy
   coerces to default.
4. Totals + time filter - `from`/`to` exclusion, hand-computed cost
   sum, `includeModels=true/false`, empty record set, zero-token
   record, malformed record dropping, sort-by-cost-descending.
5. `monthlyReport` - calendar bounds, month with a single record,
   `includeModels` default-true.
6. `budgetCheck` - limit validation, under-warn ok, warn at 0.8+,
   exceeded at 1.0+, group filter, custom `warnAt`, unknown period
   throws, missing group yields used=0.
7. `loadHistoryRecords` - JSONL roundtrip, missing file, malformed
   lines, `defaultHistoryPath` shape, and `CostReporter` integration
   via `loadRecords`.

Full suite: 82 / 82 pass.

## Limitations

- The reporter consumes rows from `history.jsonl`; enriching those rows
  with per-record input/output token counts is the next increment.
  Today records default to `inputTokens = 0 / outputTokens = 0` unless
  the history row already carries them, so operators using the CLI
  get accurate groupings but may see under-counted cost until the
  history writer is updated. This is an additive change - no schema
  break - and it will land as a follow-up PR.
- Web UI dashboard (10.5 spec item "manager cost dashboard") consumes
  the same endpoints but lives under the 10.x UI track.
- No hard blocking on budget exceed yet; `c4 cost budget` reports
  `exceeded: true` but workers are not prevented from running. The
  enforcement hook is a one-liner against `budgetCheck` and will land
  alongside the Web UI work.
